## 1. Code & Architecture Review

### 1.1 CRITICAL: Hardcoded Personal User Path in Production Script

**File:** `scripts/apply_verifier_fixes.py:9`

```python
BENCHMARKS = Path("/home/stephanie_jarmak/CodeScaleBench/benchmarks")
```

This is an absolute path to a specific developer's home directory on a specific
machine — hardcoded into a script that applies verifier fixes across all tasks.
The script will crash with `FileNotFoundError` for every other developer and in
every CI or Daytona environment.

Additionally, the path references `CodeScaleBench` (old name) rather than
`CodeContextBench`. This means the script has never been tested in any environment
other than the original developer's machine.

**Fix:** Replace with `Path(subprocess.check_output(["git", "rev-parse",
"--show-toplevel"]).decode().strip()) / "benchmarks"` — same pattern used elsewhere.

### 1.2 HIGH: Shell Injection in context_retrieval_agent.py

**File:** `scripts/context_retrieval_agent.py:432, 544, 552, 584`

`rerun_failed.py`'s `shell=True` injection was documented in CLAUDE.md, but
`context_retrieval_agent.py` has the same pattern across four call sites and is
notably more dangerous: the agent's code comment at line 429 explicitly states
"No allowlist — agent runs unrestricted." Combined with `shell=True` and dynamically
constructed commands, this creates a direct code execution path from untrusted task
config or LLM-generated content.

**Fix:** Convert to `subprocess.run([...], shell=False)` with list arguments.

### 1.3 HIGH: Non-Atomic File Writes (Data Corruption Risk)

**Files:**
- `scripts/aggregate_status.py:669` — writes JSON status directly
- `scripts/apply_verifier_fixes.py:103, 117, 134` — overwrites task files directly

Both use `.write_text()` without the write-to-temp-then-rename pattern. If the
process crashes mid-write (OOM, SIGKILL, disk full), the output file is left
partially written and silently corrupt. Downstream readers get garbage data with no
error.

Compare to `scripts/account_health.py:110-111` which does this correctly via a
temp file + `shutil.move()`.

**Fix:** Use `pathlib.Path` atomic write idiom:
```python
tmp = path.with_suffix(".tmp")
tmp.write_text(content)
tmp.replace(path)  # atomic on POSIX
```

### 1.4 MEDIUM: Bare `except:` Clauses Swallow KeyboardInterrupt

**Files:**
- `scripts/audit_v2_report_data.py:104-105`
- `scripts/ds_audit.py:244-245, 288`
- `scripts/extract_v2_report_data.py:144-145, 286-287`

Bare `except:` catches `KeyboardInterrupt` and `SystemExit`, making scripts
impossible to interrupt with Ctrl-C. They also mask `AttributeError`,
`TypeError`, and other programming mistakes that should surface during development.

The `ds_audit.py` case is particularly risky: it's inside a loop over results
where silently swallowing an exception causes the loop to continue with stale state.

**Fix:** Replace with `except (json.JSONDecodeError, KeyError, ValueError):` or the
appropriate specific exceptions.


## 2. Feature & UX Improvements

### 2.1 Dashboard: No Remediation Tracking View

`dashboard/app.py` shows benchmark results (pass rates, costs, IR metrics) but has
no view for repository health or technical debt. Operators have no dashboard-level
visibility into the 90+ open issues, their severity, or trends over time.

A "Repo Health" tab would surface:
- Issue count by severity (critical/high/medium/low)
- Trend line across nightly reports
- Which report introduced each issue
- Link to the relevant file:line

### 2.2 No Machine-Readable Output from Nightly Review

All 11 nightly reports are Markdown prose. Nothing in the pipeline parses them into
structured data. This means:
- Can't auto-create beads tickets from findings
- Can't deduplicate findings across reports
- Can't track when a finding is resolved
- Can't compute MTTR (mean time to remediation)

A `reports/nightly/issues.json` companion file (one entry per finding, with
`status: open|resolved`, `severity`, `file`, `line`, `first_seen`, `resolved_in`)
would enable automation and dashboard integration.

### 2.3 scripts/apply_verifier_fixes.py Is Unrunnable Anywhere Except One Machine

Beyond the hardcoded path bug (§1.1), `apply_verifier_fixes.py` has no `--dry-run`
flag, no confirmation prompt before overwriting task files, and no backup mechanism.
A script that bulk-modifies 400+ verifier copies across 13 suites should have all
three.

## 3. Research Recommendations

### 3.1 Ruff for Python Linting (Drop-in, Zero Config)

**URL:** https://docs.astral.sh/ruff/

Ruff can detect the majority of issues found in this report as built-in rules:

| Rule | Detects |
|------|---------|
| `S603` / `S604` | `shell=True` subprocess calls |
| `SIM115` | `open()` without context manager (FD leaks) |
| `BLE001` | Bare `except:` clauses |
| `S110` | `try/except/pass` swallowing exceptions |
| `PTH` | Non-pathlib path operations |

Single install: `pip install ruff`. Single config block in `pyproject.toml`.
Runs in milliseconds. Would have caught issues 1.2 through 1.5 automatically.

### 3.2 atomicwrites / pathlib.Path Atomic Pattern

The `atomicwrites` library (or the stdlib `os.replace()` idiom) provides atomic
cross-platform file writes in one line. Given that this codebase has at least 6
scripts performing non-atomic writes to important state files, a shared utility
`scripts/lib/atomic_write.py` would standardize the pattern.

### 3.3 Pre-commit Framework for Project-Specific Checks

**URL:** https://pre-commit.com/

Beyond Ruff (which handles generic Python quality), a `.pre-commit-config.yaml`
with custom local hooks could enforce project-specific invariants:

- No `~/CodeScaleBench` literals (54 stale paths still present)
- No `sourcegraph_full` config name references
- No hardcoded `/home/<username>/` paths
- `grep -P` usage in `.sh` files
- `DIR_PREFIX_TO_SUITE` defined outside of `suite_registry.py`

These patterns are specific enough that generic linters won't catch them but simple
enough that `grep`-based pre-commit hooks cover them.

### 3.4 Hypothesis for Property-Based Testing of Scoring Logic

The `csb_metrics/models.py` mutation bug and the falsy-value bugs in
`generate_eval_report.py` are the kind of bugs that property-based testing catches
trivially. Hypothesis generates thousands of edge-case inputs including `0`, `""`,
`None`, and `[]` automatically. Given that scoring correctness is the primary output
of this project, investing in Hypothesis tests for `models.py` and
`generate_eval_report.py` would give outsized confidence.

## 4. Recommended Next Feature

### Codebase-Specific Automated Code Quality Gate

**The problem in one sentence:** Issues are being discovered at 8–10 per report but
remediated at 0 per week, because nothing in the contribution workflow prevents the
same classes of bugs from being introduced again.

**What to build:** A pre-commit + CI quality gate tailored to the specific failure
patterns this project accumulates. Unlike generic linters, it enforces the ~15 rules
that keep appearing in nightly reports.

**Scope:**

1. **`pyproject.toml` with Ruff** — enable rules S603, S604, SIM115, BLE001, PTH.
   Catches shell injection, FD leaks, bare excepts in one step.

2. **`.pre-commit-config.yaml` with custom local hooks:**
   - `check-stale-paths`: grep for `CodeScaleBench`, `stephanie_jarmak`,
     `sourcegraph_full` — block commit if found
   - `check-grep-p`: grep for `grep -P` in `.sh` files — block commit
   - `check-shell-true`: grep for `shell=True` in Python — warn with context
   - `check-suite-registry`: grep for `DIR_PREFIX_TO_SUITE =` outside
     `csb_metrics/suite_registry.py` — block commit

3. **CI enforcement:** Add a `lint.yml` workflow that runs Ruff + pre-commit hooks
   on every PR. Gate merges on passing.

4. **Retroactive fix pass:** Once the gate is in place, run `ruff check --fix` and
   the custom hooks in audit mode to generate a list of all violations, then fix
   them in a single batch commit.

**Why this is the highest-leverage investment:**

- Prevents future accumulation of the top 5 issue classes entirely
- Costs ~1 day to implement
- Provides immediate feedback to contributors instead of waiting for nightly report
- Turns 11 reports' worth of discovered patterns into executable enforcement
- Unblocks downstream work: once FD leaks / shell injection / bare excepts are
  auto-enforced, those categories drop out of future nightly reports permanently

**PRD inputs:**
- Gate must not require pyproject.toml rewrite first (can coexist with sys.path hacks)
- Custom hooks must be runnable locally with `pre-commit run --all-files`
- CI workflow must fail fast with a human-readable summary of violations
- Ruff config must suppress false positives in the detection-logic scripts that
  contain fake keys (add `# noqa: S105` or per-file ignores)
- Scope excludes Ruff auto-fix for semantic bugs (models.py mutation, etc.) —
  those need manual review
