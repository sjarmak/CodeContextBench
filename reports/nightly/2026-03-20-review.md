### 1.3 Complex mktemp+mv Pipeline Race Condition (NEW — MEDIUM)

`configs/run_selected_tasks.sh` lines 648, 699, and 711 contain a complex
case-normalization pipeline:

```bash
_bl_temp_dir=$(
  _td=$(mktemp -d "/tmp/bl_${task_id}_XXXXXX") &&
  _td_lower=$(echo "$_td" | tr '[:upper:]' '[:lower:]') &&
  [ "$_td" != "$_td_lower" ] && mv "$_td" "$_td_lower";
  echo "${_td_lower:-$_td}"
)
```

**Problems:**
1. **Silent failure path:** If `mv "$_td" "$_td_lower"` fails (permissions, target
   exists, case-insensitive filesystem where the comparison `!=` evaluates false), the
   subshell exits non-zero but `${_td_lower:-$_td}` still echoes the original path —
   which may not exist after a partial `mv` attempt. The subsequent `cp -a` then
   silently targets a missing directory.
2. **Concurrent task collision:** Two tasks with the same `task_id` (e.g., retried
   runs) racing through this block can both call `mktemp -d
   "/tmp/bl_${task_id}_XXXXXX"` and one's `mv` may clobber the other's temp dir.
3. **No `trap` cleanup:** Previously noted (report #10) that there is no `trap` for
   temp dir removal. This issue compounds that: orphaned temp dirs accumulate if `cp`
   fails after the bad `mv`.

This pattern appears three times (lines 648, 699, 711) — each copy of the pipeline
has the same vulnerability.

### 1.4 FD Leak in `csb_metrics/extractors.py:669` (NEW — LOW)

```python
if tp.is_file():
    try:
        for line in tp.open():   # ← file handle never closed
            line = line.strip()
            if not line:
                continue
            entry = json.loads(line)
```

`tp.open()` returns a file object that is never explicitly closed. Normal iteration
exhaustion does not close it. On CPython it will be GC'd eventually, but on PyPy or
in tight loops this leaks FDs. The prior FD-leak audit (which reached 17+ sites)
targeted the `json.load(open(...))` pattern specifically; `.open()` called on a
`pathlib.Path` object uses a different call form and was missed by grep.

**Fix:** `with tp.open() as fh: for line in fh:`

## 2. Feature & UX Improvements

### 2.1 No Cross-Platform Validation for Runner Scripts

The `configs/` and `scripts/` directories contain 15+ shell scripts. There is no CI
step that validates these scripts run on macOS (the developer's primary platform) AND
Linux (the CI/CD and container platform). The split has caused multiple silent failures:
`grep -P` (POSIX portability), sparse array handling (Bash 3.2 vs 4+), and now the
`mktemp+mv` pipeline.

**Proposed improvement:** Add a `scripts/validate_shell_portability.sh` smoke-test
that runs `shellcheck` on all shell scripts and checks for known non-portable patterns:
`grep -P`, `declare -A`, `readarray`, and the `mktemp+mv` case-normalization idiom.

### 2.2 Visualization Pipeline Has No End-to-End Path

`extract_build_diary.py → plot_build_diary_supplementary.py` produces figures for
supplementary analysis. There is no documented pipeline for how these are run, no
`Makefile` target, no CI step, and no test. The scripts crash silently on non-developer
machines (§1.5 above). A `scripts/generate_build_diary_report.sh` wrapper with documented
prerequisites would make this pipeline usable.

### 2.3 `rerun_fixed_tasks.sh` and `rerun_zero_mcp_tasks.sh` Have No Dry-Run Mode

Both scripts immediately invoke Harbor runs. Neither has a `--dry-run` flag to preview
which tasks would be targeted before committing to a potentially expensive batch run.
Given the deprecated model ID (§1.2) causing silent API failures, adding dry-run mode
would have surfaced the problem before wasting quota.

## 3. Research Recommendations

### 3.1 Repo-Root Discovery Pattern

All five files with hardcoded paths use the same anti-pattern. The standard Python fix
is one line:

```python
REPO_ROOT = Path(__file__).resolve().parent.parent
```

For shell scripts:
```bash
REPO_ROOT=$(git rev-parse --show-toplevel)
```

A custom Ruff rule or pre-commit hook that detects bare `/home/` absolute paths in
`.py` files would catch new regressions. This is simpler than S603/S604 and would
have caught all five files in this report.

### 3.2 Model ID Registry

The recurring pattern of deprecated model IDs across Python files, skill files, and
now shell scripts suggests a **centralized model ID registry** is needed. A single
file (e.g., `scripts/csb_metrics/model_ids.py`) or a config YAML that defines the
canonical model IDs — with all scripts importing from it — would make updates a
one-line change. As a simpler alternative, a grep-based CI check
(`grep -rn "claude-opus-4-5" scripts/ .claude/`) would catch stale IDs at commit time.

### 3.3 Shell Script Static Analysis via shellcheck

`shellcheck` (https://www.shellcheck.net/) would flag:
- `grep -P` portability issues (SC2143)
- Unquoted variables causing word splitting
- Subshell exit code swallowing (SC2155)
- The `mktemp+mv` pipeline complexity

Adding `shellcheck scripts/*.sh configs/*.sh` to CI would catch the class of shell
bugs that Ruff misses. `shellcheck` is available via `brew install shellcheck` on macOS
and in most CI base images.

### 3.4 `pathlib` Migration for Path Operations

The codebase mixes `os.path`, string concatenation, and `pathlib.Path` for file
operations. Migrating consistently to `pathlib` would make paths explicit and
machine-independent. Ruff rule `PTH` flags `os.path.*` calls that should be `pathlib`
equivalents — this is one of the rules already planned for the code quality gate PRD.

## 4. Recommended Next Feature

**Automated Mass-Fix Script (`scripts/fix_common_patterns.py`)**

The backlog now exceeds 110 open issues, but 0 have been fixed in 8 days. The active
PRD (task metadata + DuckDB) adds new capability; the code quality gate PRD (Ruff +
pre-commit) prevents future regressions. But neither addresses the *existing* 110-issue
backlog.

The highest-leverage action is a one-shot **automated remediator** that applies all
known safe, mechanical fixes across the codebase in a single invocation. This is
distinct from both the quality gate (preventive) and DuckDB (additive) PRDs.

**Scope:** All patterns where the fix is unambiguous and safe to apply automatically:

| Pattern | Count | Fix |
|---------|-------|-----|
| `json.load(open(...))` → `with open(...) as f:` | 17+ sites | SIM115 auto-fix |
| Hardcoded `/home/stephanie_jarmak/CodeScaleBench` | 5+ files | Replace with `REPO_ROOT` |
| `except:` → `except Exception:` | 5 sites | BLE001 auto-fix |
| Deprecated `claude-opus-4-5-20251101` | 5+ files | String replace |
| `DEFAULT_REPO_BLOB_BASE` stale org | 1 site | String replace |

**Interface:**
```bash
python3 scripts/fix_common_patterns.py --dry-run   # preview all changes
python3 scripts/fix_common_patterns.py --fix        # apply all changes
python3 scripts/fix_common_patterns.py --category fd-leaks  # apply one category
```

**Design constraints:**
- Each fix category is independent and idempotent
- All writes are atomic (temp+rename)
- Script exits non-zero if any fix fails; changes so far are left in place
- Generates a report of all changes made (file:line → new content)
- Does NOT touch test files (separate pass after human review)

**Estimated scope:** 40–60 files, 100+ individual changes. Running this once would
eliminate ~60% of the open backlog in a single commit, unblocking the remediation
velocity crisis that has persisted since March 12.

**Why now:** The patterns are fully catalogued across 13 reports. The fix for each is
deterministic. The longer this waits, the more the backlog diverges from documented
fixes as the code evolves.
