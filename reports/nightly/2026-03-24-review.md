## 1. Code & Architecture Review

### 1.1 XSS in export_official_results.py HTML Viewer (NEW — SECURITY)

`scripts/export_official_results.py` generates `docs/official_results/index.html` with an
interactive JavaScript table. The table rows are built via `innerHTML` using unescaped
template literals (`export_official_results.py:2657–2668`):

```javascript
tr.innerHTML = `
  <td class="mono">${t.suite || 'unknown'}</td>
  <td class="mono">${t.run_dir}</td>
  <td class="mono">${t.config}</td>
  <td><a href="${t.task_page}">${t.task_name}</a>...</td>
  <td><span class="pill ${t.status}">${t.status}</span></td>
  ...
  <td>${(t.flags || []).join(', ') || '-'}</td>
`;
```

The Python-side escaping function `esc()` (line 1960) is only applied when generating
static per-task HTML pages. The JavaScript in `index.html` loads `official_results.json`
via `fetch()` and interpolates every field — `task_name`, `suite`, `config`, `flags`,
`status` — directly into `innerHTML` without sanitization.

**Attack vector**: If any field in `official_results.json` contains HTML (e.g., a task
name like `<img src=x onerror=alert(1)>`), it executes when the viewer is opened. While
the JSON is generated internally, it derives values from task.toml files and run
directories — surfaces where a maliciously crafted task definition or a corrupted run
artifact could inject markup.

**Fix**: Replace `innerHTML` with DOM API (`createElement`, `textContent`) for all
user-derived fields, or apply a JavaScript `escapeHtml()` utility before interpolation.
Add a `<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src
'self' 'unsafe-inline'">` header as defense in depth.

### 1.2 generate_eval_report.py Retrieval Comparison Also Broken for New Config Names (NEW)

Report #17 identified `compare_configs.py` as non-functional for new-style config names.
The same class of bug exists in the **main report generator** at
`generate_eval_report.py:704–706`:

```python
baseline_configs = [c for c in configs if "sourcegraph" not in c.lower() and "mcp" not in c.lower()]
mcp_configs = [c for c in configs if "sourcegraph_full" in c.lower() or "mcp_full" in c.lower()]
```

For current-generation runs using `baseline-local-direct` and `mcp-remote-direct`:
- `baseline_configs` correctly excludes `mcp-remote-direct` (contains "mcp"), so
  `baseline-local-direct` is selected.
- `mcp_configs` requires "sourcegraph_full" or "mcp_full" in the name — neither matches
  `mcp-remote-direct`. **Result: `mcp_configs` is empty.**

When `mcp_configs` is empty, `_build_retrieval_comparison()` returns `None` at line 709,
and the retrieval comparison table is **silently omitted** from every evaluation report.
No warning is emitted.

This is independent of the compare_configs.py fix — both files have their own hardcoded
config-name heuristics that need updating.

**Fix**: Apply the same dynamic config detection pattern proposed in report #17 for
`compare_configs.py`. Use `any("mcp" in c for c in configs)` or import a shared helper
from `config_utils.py`.

### 1.3 TOML Parser Aborts on Multi-Line Strings (NEW — HIGH SEVERITY)

`scripts/validate_tasks_preflight.py:132` contains a `break` statement that terminates
all TOML parsing when a triple-quoted string is encountered:

```python
def parse_task_toml_simple(path: Path) -> dict:
    ...
    for line in path.read_text().splitlines():
        ...
        if "=" in line:
            if '"""' in line:
                break            # <-- stops parsing the ENTIRE file
            key, _, val = line.partition("=")
            ...
    return result
```

Any task.toml file that uses multi-line strings (common for `description`, `instruction`,
or `expected_output` fields) causes the parser to **stop reading after that line**. All
subsequent key-value pairs — including critical fields like `mcp_suite`, `category`,
`language`, `difficulty` — are silently dropped.

**Impact**: Preflight validation for any task with multi-line TOML strings operates on
**incomplete metadata**. Missing fields are treated as absent rather than unreadable,
which means checks like difficulty calibration, language mismatch detection, and suite
classification silently pass with incomplete data.

**Scope**: Without scanning every task.toml for `"""`, the exact count of affected tasks
is unknown. However, task description fields are natural candidates for multi-line
strings, so this likely affects a significant fraction of the 275 canonical tasks.

**Fix**: Replace the `break` with a skip loop that consumes lines until the closing
`"""`:

```python
if '"""' in line:
    # Skip multi-line value
    if line.count('"""') < 2:  # opening but no closing on same line
        for line in lines_iter:  # consume until closing """
            if '"""' in line:
                break
    continue  # continue parsing, don't break
```

### 1.4 Oracle Criteria Truncation Inconsistency Creates Judge Scoring Bias (NEW)

`scripts/csb_metrics/judge/oracle.py` uses different criteria limits depending on which
discovery strategy produces the oracle bundle:

| Strategy | Line | Criteria limit | Context files limit |
|----------|------|----------------|---------------------|
| Instruction extraction (5.0) | 373 | `[:20]` | `[:20]` |
| Verifier rubric (5.5) | 453 | `[:20]` | `[:20]` |
| Merged instruction + verifier (5.5) | 690 | **`[:25]`** | `[:20]` |

When both instruction.md and verifier rubric contribute criteria (strategy 5.5 merged
path at line 690), the judge receives **up to 25 criteria items**. All other paths cap at
20. This creates an asymmetric evidence bundle: tasks with both instruction and verifier
criteria get 25% more evaluation dimensions than tasks with only one source.

Additionally, the confidence logic at lines 684–686 uses OR semantics:
```python
if (instruction_bundle and instruction_bundle.confidence == "high") or \
   (verifier_bundle and verifier_bundle.confidence == "high"):
    confidence = "high"
```

A high-confidence instruction bundle paired with a low-confidence verifier bundle still
produces a "high" confidence merged bundle, overstating certainty.

**Fix**: Standardize all criteria limits to 20 (change line 690 from `[:25]` to `[:20]`).
For confidence, use MIN logic instead of OR: `confidence = min(instr.confidence,
verifier.confidence)` by establishing a confidence ordering.

### 1.5 Unquoted `find` Output in _common.sh Rate-Limit Loops (NEW)

`configs/_common.sh` assigns `find` output to variables and iterates with unquoted
expansion at lines 1057, 1076, 1084, and 1110:

```bash
result_files=$(find "$log_dir" -name "result.json" ... 2>/dev/null || true)
for rf in $result_files; do            # <-- unquoted: word splitting + glob expansion
    if grep -qEi "$RATE_LIMIT_PATTERNS" "$rf" 2>/dev/null; then
        return 0
    fi
done
```

Four instances follow this pattern across `_is_rate_limited()` and
`_has_rejected_rate_limit_marker()`. When `$result_files` is unquoted in the `for` loop:
- Filenames containing spaces are split into separate tokens
- Filenames containing glob characters (`*`, `?`, `[`) are expanded against the
  filesystem

While task IDs in this codebase are unlikely to contain spaces, this is a latent shell
correctness bug. If any run directory or task ID ever includes spaces (e.g., from a
manually created test run), rate-limit detection silently breaks.

**Fix**: Use `mapfile` or a `while read` loop:
```bash
while IFS= read -r rf; do
    grep -qEi "$RATE_LIMIT_PATTERNS" "$rf" 2>/dev/null && return 0
done < <(find "$log_dir" -name "result.json" -path "*${task_id}*" 2>/dev/null)
```

### 1.7 generate_eval_report.py Silently Succeeds with Zero Runs After Filtering (NEW)

After `filter_runs_to_selected()` at line 839, `generate_eval_report.py` does not check
whether `runs` is empty:

```python
runs = filter_runs_to_selected(runs, task_index)
post_count = sum(r.task_count for r in runs)

```

If all discovered runs are filtered out (e.g., stale runs that don't match
`selected_benchmark_tasks.json`), the script continues with an empty `runs` list,
generates empty JSON/CSV/Markdown reports, and **exits with code 0** (success). No
warning is emitted.

Downstream consumers of the report (dashboards, CI steps) see a successful exit code and
process the empty reports as authoritative data.

**Fix**: Add after line 841:
```python

    print("ERROR: No tasks remain after filtering. Check runs-dir and selection file.", file=sys.stderr)
    return 1
```

## 2. Feature & UX Improvements

### 2.1 Schema Validation Is Write-Only

The `schemas/` directory contains 13 JSON schema files. Of these, only **3 have active
validation consumers** (`judge_result`, `enterprise_report`, `result`). Four analysis
scripts (`economic_analysis.py`, `failure_analysis.py`, `reliability_analysis.py`,
`workflow_metrics.py`) produce JSON matching their respective schemas but **never validate
output against them**. The schemas are documentation-only — the code that writes the JSON
doesn't import or reference the schema.

Additionally, the three most critical data artifacts have **no schema at all**:
- `task.toml` — parsed by a hand-rolled TOML parser (see 1.3) with no structural validation
- `selected_benchmark_tasks.json` — 275 entries with ad-hoc field checks
- `MANIFEST.json` — 1.3 MB artifact generated by `generate_manifest.py` with no schema

Adding `jsonschema.validate()` calls to the 4 analysis scripts (each ~3 lines of code)
would catch structural regressions in report output. Defining schemas for task.toml and
selected_benchmark_tasks.json would replace the fragile ad-hoc parsing.

### 2.2 Export Deduplication Hides Run Variations

`export_official_results.py:2501–2509` deduplicates results by
`(suite, config, task_name)` — keeping only the latest timestamp. The `run_dir` is not
part of the dedup key, so when the same task runs in multiple runs with the same config,
only the latest result survives. The JavaScript viewer offers "All task runs" vs "Latest"
toggle, but the "All" view still reflects Python-side filtering.

For analysts comparing results across runs (e.g., "did score improve between run A and
run B?"), the deduplication silently hides earlier results. Adding `run_dir` as an
optional fourth component of the dedup key (controlled by a `--dedup-scope` flag) would
preserve cross-run comparison capability.

### 2.3 Status Fingerprints Cover Only 12 of ~25 Error Categories

`scripts/status_fingerprints.py` defines 12 regex patterns for error classification.
Missing patterns include: SSH/git authentication failures, DNS/connectivity errors, Python
`MemoryError`, HTTP 502/503/504 errors, disk-full conditions, and Python version
mismatches. These gaps cause unrecognized errors to fall into a generic "unknown" bucket,
reducing the value of the error classification for triage. Expanding to ~20 patterns
(adding 8 for the categories above) would cover the long tail of operational failures.

## 3. Research Recommendations

### 3.1 JavaScript DOM API for Export Viewer

The innerHTML XSS issue (1.1) is a symptom of building HTML strings from data. Modern
browsers support `textContent` for safe text insertion and `createElement`/`setAttribute`
for structure. Converting the table-building loop in `index.html` from template-literal
innerHTML to DOM API calls eliminates the entire class of XSS vulnerabilities at the
source. The conversion is mechanical (~30 lines to change) and has no performance impact
for the table sizes involved (< 1000 rows).

### 3.2 Shared Config Detection Utility

Both `compare_configs.py` (report #17) and `generate_eval_report.py` (this report) have
independent, broken config-name heuristics. A single function in `config_utils.py`:

```python
def detect_config_pair(config_names: set[str]) -> tuple[str, str]:
    """Return (baseline_config, mcp_config) from whatever configs are present."""
    bl = next((c for c in config_names if c.startswith("baseline")), None)
    mcp = next((c for c in config_names
                if "mcp" in c or c == "sourcegraph_full"), None)
    return bl or "baseline", mcp or "sourcegraph_full"
```

would fix both call sites and any future scripts that need to identify the two primary
configs. This is a 10-line utility with zero new dependencies.

### 3.3 Python 3.11+ tomllib for TOML Parsing

The hand-rolled TOML parser in `validate_tasks_preflight.py` (see 1.3) breaks on
multi-line strings, doesn't handle escape sequences, and can't parse arrays or inline
tables. Python 3.11+ includes `tomllib` in the standard library. Since the CI baseline is
Python 3.10, a compatibility shim is straightforward:

```python
try:
    import tomllib
except ModuleNotFoundError:
    import tomli as tomllib  # pip install tomli for 3.10
```

This eliminates an entire class of parsing bugs and enables validation of TOML fields
that the current parser silently ignores.

## 4. Recommended Next Feature

### Eval Pipeline Config Compatibility Fix

**Why this**: The evaluation pipeline's two primary output tools — `generate_eval_report.py`
(retrieval comparison table) and `compare_configs.py` (cross-config analysis) — are both
broken for all runs using current-generation config names. Together they constitute the
entire statistical analysis capability of the project. Neither tool produces errors or
warnings; both silently produce empty results.

Report #17 recommended fixing `compare_configs.py` alone (4 targeted changes). This
report extends that recommendation to include `generate_eval_report.py:704–706` and
proposes a shared utility to prevent the same bug from recurring in future scripts.

**Scope (3 files, backward compatible):**

**`scripts/config_utils.py`** — add shared config detection:
```python
def detect_config_pair(config_names: Iterable[str]) -> tuple[str, str]:
    """Detect baseline and MCP config from a set of config names."""
    configs = set(config_names)
    bl = next((c for c in configs if c.startswith("baseline")), "baseline")
    mcp = next((c for c in configs
                if "mcp" in c or c == "sourcegraph_full"), "sourcegraph_full")
    return bl, mcp
```

**`scripts/compare_configs.py`** — replace 4 hardcoded lookups (lines 147–148, 227–228,
308–309, 401–402) with calls to `detect_config_pair()`. Add `--baseline-config` /
`--mcp-config` CLI overrides.

**`scripts/generate_eval_report.py`** — replace the heuristic at lines 704–706 with
`detect_config_pair()`:
```python
from config_utils import detect_config_pair
bl_config, mcp_config = detect_config_pair(configs)
baseline_configs = [bl_config] if bl_config in configs else []
mcp_configs = [mcp_config] if mcp_config in configs else []
```

**Complement (same PR):**
1. Fix `validate_tasks_preflight.py:132` — replace `break` with multi-line string skip
   (prevents metadata loss for tasks with multi-line TOML values)
2. Fix `export_official_results.py:2657–2668` — replace innerHTML with DOM API

**Why these three together**: The config detection fix restores the project's statistical
analysis capability. The TOML parser fix restores metadata integrity for preflight
validation. The XSS fix addresses the only security vulnerability found in the codebase.
All three are surgical changes with no architectural impact, totaling ~40 lines of
production code.
