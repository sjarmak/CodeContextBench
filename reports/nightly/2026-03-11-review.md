## 1. Code & Architecture Review

### 1.1 HIGH: Secret Sanitizer Allowlist Uses Overly Broad Substring Matching

`scripts/sanitize_secrets.py` line 69:

```python
_FAKE_INDICATORS = ("XXXX", "aBcDeF", "a1b2c3d4", "example", "test_key", "dummy")
```

Line 76-78 uses these as **substring matches**:

```python
for indicator in _FAKE_INDICATORS:
    if indicator in match_text:
        return True
```

The strings `"example"`, `"test_key"`, and `"dummy"` are generic enough
to appear as substrings in real production secrets. For instance:

- `sk-ant-api03-example_prod_deployment_key_abc123...` contains "example"
- `sk-proj-test_key_rotation_v2_abc123...` contains "test_key"
- `sgp_abcdef1234_dummy_service_account_0000...` contains "dummy"

Any real secret containing one of these substrings will be silently
exempted from redaction. The `FAKE_KEY_ALLOWLIST` set (lines 55-65)
uses exact match and is safe; the `_FAKE_INDICATORS` tuple is the
problem.

**Impact**: Real secrets could leak into exported HTML reports and
official results if they happen to contain any of these common English
words as substrings. The sanitizer was added in commit `cd779e70` to
prevent exactly this scenario.

**Fix**: Replace substring matching with more specific patterns:
- `"_XXXX"` (with underscore prefix, matching placeholder style)
- Remove `"example"`, `"test_key"`, `"dummy"` entirely -- if needed,
  add specific fake keys to the exact-match `FAKE_KEY_ALLOWLIST`

### 1.2 HIGH: Pass Rate Definition Diverges Between Two Modules

`scripts/generate_eval_report.py` lines 80-83:

```python
def _task_is_passed(task) -> bool:
    if task.passed is not None:
        return task.passed
    return task.status == "passed"
```

`scripts/csb_metrics/models.py` lines 172-175:

```python
return sum(
    1 for t in scored
    if t.passed is True or (t.passed is None and t.status == "passed")
) / len(scored)
```

When `task.passed = False` but `task.status = "passed"` (a data
inconsistency that can occur when verifier and status tracker disagree),
`generate_eval_report.py` counts it as **not passed** (returns
`task.passed` which is `False`), while `models.py` also counts it as
**not passed** (only counts `t.passed is True`). So the definitions
actually agree for this edge case.

However, the reverse case is problematic: when `task.passed = True`
but `task.status = "failed"`, both count it as passed. But the
`_task_is_scored` filter (line 76-77) includes this task because
`task.passed is not None` is True, while a caller filtering on
`task.status in ("passed", "failed")` would also include it. The
divergence is structural: `generate_eval_report.py` defines pass/score
logic as standalone functions with subtly different branching from
`models.py`'s inline property.

**Impact**: Any future change to pass rate semantics must be applied in
two places independently, with no shared code to enforce consistency.
This is a maintenance hazard that will eventually produce different pass
rates in eval reports vs RunMetrics objects.

**Fix**: Extract `_task_is_passed` and `_task_is_scored` into
`csb_metrics/models.py` as module-level functions. Import them in
`generate_eval_report.py`.

### 1.5 HIGH: Duplicate CI Workflow Runs Same Check Twice

`.github/workflows/docs-consistency.yml` runs:
```yaml
python3 scripts/docs_consistency_check.py        # Python 3.11
python3 scripts/refresh_agent_navigation.py --check
```

`.github/workflows/repo_health.yml` runs:
```yaml
python3 scripts/repo_health.py                   # Python 3.10
python3 scripts/refresh_agent_navigation.py --check
```

`repo_health.py` executes `docs_consistency_check.py` as its first
check (via `run_script_check()`). Both workflows also run
`refresh_agent_navigation.py --check`. Every push to `main` and every
PR triggers **two identical checks**: `docs_consistency_check.py` and
`refresh_agent_navigation.py --check`, run under different Python
versions (3.10 vs 3.11).

**Impact**: Doubled CI minutes with no additional coverage. Worse,
if a Python 3.10/3.11 behavioral difference causes one workflow to
pass and the other to fail, the conflicting signals confuse operators.

**Fix**: Remove `docs-consistency.yml` entirely. Its checks are fully
subsumed by `repo_health.yml`. If Python 3.11 compatibility testing
is desired, add a matrix strategy to `repo_health.yml` instead.

### 1.6 MEDIUM: 12 Python Scripts Use `json.load(open())` Without Context Manager

12 scripts in `scripts/` use the unclosed file handle pattern:

```python
data = json.load(open(path))  # File descriptor leaked
```

**Affected scripts**:
`scaffold_contextbench_tasks.py`, `generate_promoted_verifiers.py`,
`generate_csb_org_tasks.py`, `daytona_curator_runner.py` (2 instances),
`sync_oracle_files.py` (2 instances), plus 5 shell-embedded Python
one-liners in `scripts/rerun_*.sh` files.

**Impact**: File descriptor leak. In short-lived scripts this is
harmless, but `daytona_curator_runner.py` runs as a long-lived process
managing multiple sandbox lifecycle operations. Accumulated leaked
descriptors can exhaust the OS limit and cause unrelated I/O failures.

**Fix**: Replace with `with open(path) as f: json.load(f)`.

### 1.10 LOW: Cost Report Division-by-Zero When Baseline Has Zero Tasks

`scripts/cost_report.py` line 262:


`config_tasks` is populated from a `defaultdict(int)`. When a "baseline"

## 2. Feature & UX Improvements

### 2.1 No Automated Secret Sanitization in the Result Pipeline

`scripts/sanitize_secrets.py` exists as both a library and CLI tool,
but it is not integrated into any automated workflow:

- Not called by `export_official_results.py` (which generates the
  main HTML reports and official results JSON)
- Not called by `aggregate_status.py` or post-run processing
- Not integrated into the pre-commit hook
- Not referenced in any CI workflow

The script was added in commit `cd779e70` but requires manual
invocation (`python3 scripts/sanitize_secrets.py <path>`). Given
the credential-in-trajectory risk documented in CLAUDE.md, this
should be automated.

**Improvement**: Import `sanitize_secrets.redact_secrets()` in
`export_official_results.py` and apply it to all HTML and JSON
output before writing to disk. Add a `sanitize_secrets` check to
`repo_health.py` that scans `docs/official_results/` for unredacted
patterns.

### 2.2 No Validation That Harness-Agnostic Fallbacks Are Present

The March 10-11 migration updated 256+ test.sh files with
`TASK_WORKDIR` and `TASK_REPO_ROOT` fallback chains. However, there
is no automated check that prevents regression -- new tasks or
verifier edits could reintroduce hardcoded `/workspace` paths.

**Improvement**: Add a check to `validate_tasks_preflight.py` that
greps each task's `tests/test.sh` for hardcoded `/workspace` or
`/app` paths that are not wrapped in `${TASK_WORKDIR:-...}` fallback
syntax. This would have caught the 6 missed tasks (Section 1.4) and
would prevent future regressions.

```python
def check_harness_agnostic(test_sh: Path) -> list[str]:
    """Flag hardcoded workspace paths without fallback."""
    issues = []
    text = test_sh.read_text()
    for line_no, line in enumerate(text.splitlines(), 1):
        if '/workspace' in line and 'TASK_WORKDIR' not in line:
            if not line.strip().startswith('#'):
                issues.append(f"{test_sh}:{line_no}: hardcoded /workspace without TASK_WORKDIR fallback")
    return issues
```

### 2.3 Export HTML Browser Silently Truncates at 1200 Rows

`scripts/export_official_results.py` line ~2633 (within embedded
JavaScript):

```javascript
const limited = filtered.slice(0, 1200);
```

When a filter selection returns more than 1200 results, the browser
UI silently drops rows with no indication. Users see a partial result
set without knowing it is incomplete.

**Improvement**: Add a "Showing X of Y results" indicator in the UI.
If truncation occurs, display a warning: "Results truncated to 1200.
Apply filters to narrow your view."

### 2.4 No Test Coverage for Secret Sanitization Patterns

`scripts/sanitize_secrets.py` has no test file. Given that it handles
security-critical redaction logic (including the allowlist bypass
discussed in Section 1.1), it should have unit tests covering:

- Each of the 9 secret patterns matches its intended format
- The allowlist correctly exempts known fake keys
- The allowlist does NOT exempt real secrets containing indicator
  substrings
- HTML-entity-encoded secrets are handled correctly
- Edge cases: empty strings, binary content, nested JSON

**Improvement**: Create `tests/test_sanitize_secrets.py` with fixtures
for each pattern and allowlist edge case. This would have caught the
overly broad substring matching issue (Section 1.1) during development.

## 3. Research Recommendations

### 3.1 Add a Harness Compatibility Matrix to Task Validation

The harness-agnostic migration exposed that different task categories
require different fallback patterns:

| Pattern | Used By | Fallback Required |
|---------|---------|-------------------|
| `$TASK_WORKDIR` | Answer/analysis file paths | `${TASK_WORKDIR:-/workspace}` |
| `$TASK_REPO_ROOT` | Git diff, test execution | `${TASK_REPO_ROOT:-${VERIFY_REPO:-$TASK_WORKDIR}}` |
| `git diff origin/main HEAD` | no_changes_guard | Already correct for OH |
| Document search paths | understand/orient tasks | Hardcoded, not yet migrated |
| `answer.json` paths | onboarding tasks | Hardcoded, not yet migrated |

Add a `harness_compat` check to `validate_tasks_preflight.py` that
classifies each task's test.sh by which patterns it uses and whether
the correct fallbacks are present. Output a compatibility matrix
showing which tasks are ready for each harness (Harbor, OpenHands,
future harnesses).

### 3.2 Consolidate CI Workflows into a Single Matrix

The project has 4 GitHub Actions workflows (`repo_health.yml`,
`docs-consistency.yml`, `task_smoke_matrix.yml`, `roam.yml`) with
overlapping concerns:

| Check | repo_health | docs-consistency | task_smoke | roam |
|-------|:-----------:|:----------------:|:----------:|:----:|
| docs_consistency_check.py | Y | Y | | |
| refresh_agent_navigation.py | Y | Y | | |
| validate_tasks_preflight.py | Y | | Y | |
| repo_health.py (full) | Y | | | |
| contract audit | | | Y | |
| smoke runtime | | | Y | |
| roam fitness | | | | Y |

The duplicated `docs_consistency` and `refresh_agent_navigation`
checks waste CI minutes. Consolidate into:

1. **`ci.yml`**: `repo_health.py` (full, subsumes docs_consistency)
   + `refresh_agent_navigation.py --check` + roam analysis
2. **`task_smoke_matrix.yml`**: Keep as-is (path-triggered, expensive)

This reduces from 4 workflows to 2 and eliminates duplicate checks.

### 3.3 Add Integration Tests for `sanitize_secrets.py`

Beyond unit tests (Section 2.4), add integration tests that run the
sanitizer against synthetic trajectory files containing known secrets
in various encodings:

```python
def test_redact_trajectory_html():
    """Secrets in HTML-encoded trajectory should be redacted."""
    html = '<div>&quot;ANTHROPIC_API_KEY&quot;: &quot;sk-ant-api03-realkey123...&quot;</div>'
    result = redact_secrets(html)
    assert "[REDACTED:ANTHROPIC_KEY]" in result
    assert "realkey123" not in result

def test_allowlist_does_not_bypass_real_secrets():
    """Real secrets containing 'example' should still be redacted."""
    text = 'key = "sk-ant-api03-example_prod_key_that_is_real_abc123456789"'
    result = redact_secrets(text)
    # This currently FAILS due to the bug in Section 1.1
    assert "[REDACTED:ANTHROPIC_KEY]" in result
```

These tests serve as regression guards for the security-critical
redaction logic and would have caught the allowlist bypass bug.

### 3.4 Automate Dockerfile Registry Migration Verification

The `jefzda/` to `ghcr.io/sg-evals/` migration (Section 1.3) was
partially completed across two commits. Add a `repo_health.py` check:

```python
def check_deprecated_registries():
    """Flag Dockerfiles using deprecated image registries."""
    deprecated = ["jefzda/", "docker.io/jefzda/"]
    for dockerfile in Path("benchmarks").rglob("Dockerfile*"):
        if "backups" in str(dockerfile):
            continue
        text = dockerfile.read_text()
        for registry in deprecated:
            if registry in text:
                yield f"{dockerfile}: uses deprecated registry {registry}"
```

This prevents regression and catches any new tasks authored with
the old registry prefix.

### 3.5 Standardize the Pre-Commit Secret Detection

The two competing pre-commit hooks (Section 1.11) should be
consolidated. The repo-level hook (`.githooks/pre-commit.old`) has
patterns aligned with `sanitize_secrets.py`. Unarchive it, update
it to import from `sanitize_secrets.py` directly (eliminating
pattern duplication), and configure it via `.pre-commit-config.yaml`
or `core.hooksPath`:

```bash
#!/usr/bin/env bash
# Pre-commit: check staged files for real secrets
python3 scripts/sanitize_secrets.py --dry-run $(git diff --cached --name-only --diff-filter=ACM) | grep -q "redaction" && {
    echo "ERROR: Staged files contain real secrets. Run: python3 scripts/sanitize_secrets.py <files>"
    exit 1
}
```

This gives a single source of truth for secret patterns:
`sanitize_secrets.py` is used by pre-commit, post-run sanitization,
and export pipeline.

## 4. Recommended Next Feature

### Security and Sanitization Pipeline Hardening

**The single most impactful feature** is closing the security gaps
in the secret sanitization pipeline before the next round of official
result exports. The `sanitize_secrets.py` module landed in commit
`cd779e70` as a critical security control, but it has three problems
that undermine its effectiveness: (a) an overly broad allowlist that
can bypass real secrets, (b) no automated integration into the export
pipeline, and (c) no test coverage to catch regressions.

The March 6 report recommended CI testing (foundational). March 7
recommended operational recovery (unblocking runs). March 8
recommended evaluation data integrity. March 9 recommended benchmark
data integrity. March 10 recommended unified CSB operational
readiness. This report recommends **security pipeline hardening**
because the project has now reached the stage of exporting and
publishing official results, and the newly added sanitization layer
has a bypass vulnerability that could leak real API keys into public
artifacts.

**Part A -- Fix the allowlist bypass** (30 minutes)

1. Remove generic substrings (`"example"`, `"test_key"`, `"dummy"`)
   from `_FAKE_INDICATORS` in `scripts/sanitize_secrets.py` line 69
2. If any benchmark fixtures genuinely use these words in fake keys,
   add the specific full key to the exact-match `FAKE_KEY_ALLOWLIST`
3. Verify with: `python3 scripts/sanitize_secrets.py --dry-run benchmarks/`

**Part B -- Integrate sanitization into export pipeline** (1 hour)

1. Import `redact_secrets` in `scripts/export_official_results.py`
2. Apply `redact_secrets()` to all HTML content before
   `write_text()` and all JSON content before `json.dump()`
3. Add a `--skip-sanitize` flag for debugging (default: sanitize on)
4. Update `docs/EVALUATION_PIPELINE.md` to document the sanitization
   step

**Part C -- Create test suite** (1 hour)

1. Create `tests/test_sanitize_secrets.py` with tests for:
   - Each of the 9 secret patterns (positive match)
   - False positive prevention (non-secrets that look similar)
   - Allowlist exact match (legitimate fake keys pass through)
   - Allowlist bypass prevention (real secrets with indicator
     substrings are still redacted)
   - HTML-entity-encoded secrets
2. Add to CI workflow

**Part D -- Complete Dockerfile registry migration** (30 minutes)

1. Apply `jefzda/ -> ghcr.io/sg-evals/` to the remaining 33 active
   Dockerfiles
2. Add a `deprecated_registries` check to `validate_tasks_preflight.py`
3. Verify images exist on `ghcr.io/sg-evals/` before migrating

**Part E -- Fix remaining harness-agnostic gaps** (30 minutes)

1. Add `TASK_WORKDIR` fallbacks to the 3 understand tasks with
   hardcoded document search paths
2. Add `TASK_WORKDIR` fallbacks to the 3 onboarding tasks with
   hardcoded `answer.json` paths
3. Add `harness_compat` check to `validate_tasks_preflight.py`

**Why this is highest impact**: The project is now generating and
exporting official results for publication. The sanitization pipeline
is the last line of defense against credential leaks in public
artifacts. A single leaked API key in an exported HTML trace could
compromise infrastructure access. The allowlist bypass (Section 1.1)
means this defense has a hole: any real secret containing common
English words like "example" or "dummy" will pass through unredacted.
Fixing this before the next export cycle prevents a security incident
that would require key rotation and artifact recall.
