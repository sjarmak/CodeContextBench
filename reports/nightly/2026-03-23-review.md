## 1. Code & Architecture Review

### 1.1 compare_configs.py Non-Functional for New-Style Config Names (NEW HIGH SEVERITY)

`compare_configs.py` is the primary tool for comparing baseline vs MCP performance. It
has **four separate hardcoded lookups** for "baseline" and "sourcegraph_full" — the
legacy config names from before the three-dimensional naming scheme (`baseline-local-direct`,
`mcp-remote-direct`) was introduced:

| Function | Line(s) | Hardcoded lookup | Impact |
|----------|---------|-----------------|--------|
| `_build_comparison()` | 147–148 | `configs.get("baseline", {})`, `configs.get("sourcegraph_full", {})` | Comparison table shows "?" for all tasks |
| `format_comparison_table()` | 227–228 | `_config_label("baseline")`, `_config_label("sourcegraph_full")` | Column headers reference nonexistent configs |
| `_compute_statistical_tests()` | 308–309 | Same hardcoded lookups | Returns "Too few paired tasks (0)" |
| `gather_paired_analysis()` | 401–402 | Same | Zero common tasks across all suites |

For any run using "baseline-local-direct" and "mcp-remote-direct" (the current
canonical config names), every one of these functions silently returns empty data. There
is no error or warning — the tool simply shows no comparisons.

`config_utils.py` already has the right infrastructure (`LEGACY_CONFIGS`, `NEW_CONFIGS`,
`_SHORT_NAMES`, `_MCP_INTERNAL`), and `aggregate_status.py:CONFIGS` correctly lists both
old and new names. Only `compare_configs.py` still hardcodes the old names in its display
and analysis paths.

**Fix**: Replace the four hardcoded config lookups with dynamic resolution — detect
the two configs present in the data (or accept `--baseline-config` / `--mcp-config`
CLI arguments). The `TIER_LABELS` dict at line 191 already knows both naming schemes;
the lookup just needs to flip from hardcoded keys to "whichever non-baseline config is
present."

### 1.2 cost_report.py Has a Separate Hardcoded Pricing Table (NEW — PRD Gap)

The active PRD adds `claude-sonnet-4-6` and `claude-haiku-4-6` to
`scripts/csb_metrics/extractors.py:MODEL_PRICING`. However, `cost_report.py` has its
**own independent pricing dict at lines 29–34** that is separate from `MODEL_PRICING`:

```python
# scripts/cost_report.py lines 29–34
# Anthropic pricing (per token, as of 2025)
# Claude Opus 4.5
PRICING = {
    "input_per_token":       15.0 / 1_000_000,   # $15/MTok
    "output_per_token":      75.0 / 1_000_000,   # $75/MTok
    "cache_write_per_token": 18.75 / 1_000_000,  # $18.75/MTok
    "cache_read_per_token":  1.875 / 1_000_000,  # $1.875/MTok
}
```

This `PRICING` dict is used in `extract_cost_data()` (line 94–100) as a fallback when
`cost_usd` is not present in `task_metrics.json`. It is **model-unaware** — always uses
Opus 4.5 rates regardless of the run model. After the PRD fix lands (extractors.py gets
correct per-model pricing), this separate table in cost_report.py will remain stuck at
Opus rates and continue producing 5× / 19× overstatements for fallback cost estimates.

The PRD's US-006 adds a disclaimer to `compare_configs.py` but does not update or
remove this separate table in `cost_report.py`.

**Fix**: Remove the `PRICING` dict from `cost_report.py` and replace the fallback cost
calculation at line 94–100 with a call to `calculate_cost_from_tokens()` (from
`csb_metrics/extractors.py`), passing the run model if available. This makes cost_report.py
model-aware with zero new code.

### 1.3 cost_report.py "errored" vs "error" Status Mismatch (NEW BUG)

`csb_metrics/extractors.py` writes status strings `"passed"`, `"failed"`, and `"error"`
(line 223–229) to `task_metrics.json`. `cost_report.py` checks for `"passed"` correctly
at line 151, but checks for `"errored"` (not `"error"`) at line 153:

```python
# scripts/cost_report.py lines 150–156
status = cost_data.get("status", "")
if status == "passed":
    a["passed"] += 1
elif status == "errored":       # ← bug: extractors.py writes "error", not "errored"
    a["errored"] += 1
else:
    a["failed"] += 1            # ← all "error" tasks fall here instead
```

When `task_metrics.json` is the data source (the preferred path since line 45 checks
for it first), tasks with `status = "error"` are classified as "failed" instead of
"errored". The `errored` count in every cost report is **always 0** for runs where
task_metrics.json was written by the extractors pipeline.

The `"errored"` string is only produced by cost_report.py's own internal path (line 76)
when loading from `result.json` as a fallback. The two code paths use different status
vocabularies and were never reconciled.

**Fix**: Change line 153 from `elif status == "errored":` to
`elif status in ("errored", "error"):` — one character change.

### 1.4 benchmarks/csb/ Contains 275 Undocumented Tasks in Old Organization (NEW)

The repository contains a complete second task collection at `benchmarks/csb/` with **275
task.toml files** organized by work type (flat dirs: `crossrepo/`, `fix/`, `refactor/`,
etc.) rather than the canonical suite structure (`csb_sdlc_*`, `csb_org_*`). This
collection is:

- **Tracked in git** (not gitignored, not in `backups/`)
- **Excluded from `selected_benchmark_tasks.json`** — not in any current benchmark run
- **Referenced in official results HTML** (`docs/official_results/tasks/*.html`)
  contains paths like `benchmarks/csb/refactor/ccx-migration-107/tests/` — evidence
  these tasks were run in historical evaluations
- **Not mentioned in the README, CLAUDE.md, or any docs** — no guidance on what this
  collection is, whether it's active, or when tasks from it were last executed

The tasks in `benchmarks/csb/fix/` (SWE-bench Pro format) use a different schema with
`[verifier]` and `[agent]` sections instead of the standard `[verification]` and
`[environment]` sections, plus `pre_fix_rev` for diff-based testing. Tasks in other
subdirectories (`crossrepo/`, `refactor/`) use the standard schema with `[task]`.

**Impact**: The README states "275 benchmark tasks across 9 developer work types" — but
there are actually 550 tracked task.toml files (275 canonical + 275 in `csb/`). Scripts
that discover benchmarks by scanning the `benchmarks/` directory (e.g., `abc_audit.py`)
may include `csb/` tasks in their results, double-counting with the canonical suites.

**Needed**: A short explanation in the README (or `benchmarks/csb/README.md`) stating
whether these are active, archived, or in-development tasks, plus a check in
`validate_tasks_preflight.py --all` that explicitly excludes or handles `csb/`.

### 1.5 statistics.py Uses Normal Approximation for Welch's t-Test (LOW — Methodology)

`csb_metrics/statistics.py:91` computes Welch's t-test p-values using a normal CDF
approximation:

```python
# statistics.py line 91
p_value = 2 * (1 - _normal_cdf(abs(t_stat)))
```

The comment at line 90 notes this is "adequate for df > ~30." However, when running
statistics on a single benchmark suite (e.g., csb_sdlc_fix with 19 tasks, or
csb_sdlc_document with 11), the effective degrees of freedom are 9–17. At these small
sizes, the normal approximation underestimates p-values by 15–40%, producing more
"significant" results than warranted.

This does not affect correctness for whole-benchmark statistics (all 275 tasks give
df ≈ 273), but per-suite statistical tests in `compare_configs.py --suite` will
overstate significance. Since per-suite analysis is the most actionable comparison
(comparing MCP benefit in, say, crossrepo vs fix tasks), this affects the conclusions
analysts draw from per-suite reports.

**Fix**: Replace `_normal_cdf` with a t-distribution CDF. Python's `math.lgamma` enables
a direct implementation of the incomplete beta function for the t-distribution without
any dependency; scipy is optional. Alternatively, add a guard in `welchs_t_test()`:
warn if `df < 30` and set a minimum sample-size requirement.

### 2.1 compare_configs.py Needs Config Auto-Detection

The immediate path to unblocking `compare_configs.py` for modern runs is to detect
which configs are present in the loaded data rather than hardcoding "baseline" and
"sourcegraph_full". A minimal fix:

```python
# In gather_comparison():
all_configs_found = set()
for (suite, task_name), configs in task_matrix.items():
    all_configs_found.update(configs.keys())

# Identify the two "primary" configs:
bl_config = next((c for c in all_configs_found if c.startswith("baseline")), "baseline")
mcp_config = next((c for c in all_configs_found if "mcp" in c or c == "sourcegraph_full"), "sourcegraph_full")
```

This keeps backward compatibility with legacy runs while making modern runs work without
any CLI changes. Adding `--baseline-config` / `--mcp-config` flags as explicit overrides
would be the clean long-term solution.

After the PRD fix (`extractors.py` gets per-model pricing), `cost_report.py` should be
updated to import `calculate_cost_from_tokens` and remove its local `PRICING` dict. The
fallback cost path (lines 88–100) should:

1. Attempt to read the model from the parent directory's `config.json` (same logic as
   PRD US-003 adds to `extract_task_metrics.py`)
2. Call `calculate_cost_from_tokens(inp, out, cache_w, cache_r, model=model)`

This ensures cost_report.py and the extractors pipeline are always in sync on pricing.

## 3. Research Recommendations

### 3.1 compare_configs.py Config Detection via Manifest

Rather than auto-detecting configs from run directory names, `compare_configs.py` could
read the config list from `MANIFEST.json` (which is generated by `generate_manifest.py`
and already tracks all run configs). This would make the config detection reliable and
version-stable:

```python
manifest = json.loads((RUNS_DIR / "MANIFEST.json").read_text())
known_configs = set(manifest.get("configs", []))
```

The manifest already contains the authoritative config list; compare_configs.py could
derive its pairing logic from it rather than hardcoding names.

### 3.2 Pricing Table Centralization

Having pricing data in two files (`extractors.py:MODEL_PRICING` and
`cost_report.py:PRICING`) creates a maintenance trap. A single source of truth in
`csb_metrics/pricing.py` (or extending `extractors.py` to export
`calculate_cost_from_tokens` as a public function) would eliminate this. Both
`cost_report.py` and `extract_task_metrics.py` would import from the same place.

Additionally, pricing tables are a config concern — they change more frequently than
code. Moving them to a `configs/model_pricing.json` TOML/JSON would allow updates
without code changes, and would enable the cost disclaimer to automatically identify
which models had stale pricing.

### 3.3 t-Distribution CDF for statistics.py

Python stdlib provides `math.lgamma` and `math.log`. A pure-stdlib Student's t-CDF
implementation for statistics.py (for the small-sample regime, df < 100) requires only
the regularized incomplete beta function, which can be computed via a continued-fraction
expansion using `math.lgamma`. This would remove the dependency on the normal
approximation for all sample sizes, improving per-suite statistical test accuracy.

### Fix compare_configs.py for New-Style Config Names

**Why this**: `compare_configs.py` is broken for all runs using the current canonical
config names (`baseline-local-direct`, `mcp-remote-direct`). The tool's four hardcoded
"baseline" / "sourcegraph_full" lookups produce silently empty results (no error, just
"?" in all cells and "Too few paired tasks (0)" for stats). This affects:

- Every analyst comparing baseline vs MCP performance on recent runs
- All statistical tests run via `compare_configs.py --stats`
- All paired analysis output from `compare_configs.py --paired`
- All suite-level comparisons

**Scope (1 file, ~4 targeted changes, backward compatible):**

The fix replaces the four hardcoded config lookups with dynamic detection while
preserving backward compatibility for legacy runs:

**`scripts/compare_configs.py`:**

1. Add a helper after `gather_comparison()`:
```python
def _detect_config_pair(task_matrix: dict) -> tuple[str, str]:
    """Return (baseline_config, mcp_config) from whatever configs are in data."""
    seen = set()
    for configs in task_matrix.values():
        seen.update(configs.keys())
    bl = next((c for c in seen if c.startswith("baseline")), "baseline")
    mcp = next((c for c in seen if "mcp" in c or c == "sourcegraph_full"), "sourcegraph_full")
    return bl, mcp
```

2. Pass `bl_config, mcp_config = _detect_config_pair(task_matrix)` into
   `_build_comparison()` and replace the four hardcoded strings at lines
   147–148, 227–228, 308–309, 401–402.

3. Add `--baseline-config` / `--mcp-config` CLI flags to `parse_args()` as optional
   overrides.

**Why not bigger**: The TIER_LABELS dict, `_config_label()`, and `config_utils.py`
already handle both naming schemes correctly. The fix is purely in where `_build_comparison()`,
`_compute_statistical_tests()`, `format_comparison_table()`, and `gather_paired_analysis()`
look up their two primary configs — switching from static strings to detected names.
