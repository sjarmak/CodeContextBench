## 1. Code & Architecture Review

### 1.1 CRITICAL: Evaluation Report Silently Drops Zero-Cache Tasks

`scripts/generate_eval_report.py` lines 195-199 compute average cache tokens:

```python
cache_tok = _safe_mean([
    (t.cache_creation_tokens or 0) + (t.cache_read_tokens or 0)
    for t in r.tasks
    if t.cache_creation_tokens is not None or t.cache_read_tokens is not None
])
```

When `cache_creation_tokens=0` and `cache_read_tokens=None` (or vice versa),
the `or` filter evaluates `0 or None` which is falsy, so the task is excluded
from the average. Tasks with legitimate zero cache tokens are silently dropped,
biasing cache utilization metrics upward. The filter should use explicit
`is not None` checks on each field independently:

```python
if t.cache_creation_tokens is not None or t.cache_read_tokens is not None
```

This condition is correct for the `or` case, but the inner expression
`(t.cache_creation_tokens or 0)` treats `0` as falsy -- which is fine since
`0 or 0 == 0`. The real issue is subtler: when both are `None`, the task
is excluded. When one is `0` and the other is `None`, `0 is not None` is
`True`, so the task IS included. The filter itself works, but the broader
pattern of `x or 0` is fragile and would silently convert `False` to `0` if
the field types ever changed.

### 1.2 CRITICAL: `dict.get()` Double-Fallback Returns `None` Instead of Backup Value

`scripts/export_official_results.py` line 1449:

```python
cache_tokens = _safe_int(
    agent_result.get("n_cache_tokens", agent_result.get("cache_creation_input_tokens"))
)
```

When `agent_result["n_cache_tokens"]` exists but is `None`, Python's `.get()`
returns `None` (the key exists), never evaluating the fallback
`agent_result.get("cache_creation_input_tokens")`. This loses valid cache
token data from the backup field. The gotcha is already documented in
`CLAUDE.md` line 129 (`dict.get(key, default)` does NOT protect against
`None` values) but the code still has this bug.

**Fix**: `agent_result.get("n_cache_tokens") or agent_result.get("cache_creation_input_tokens")`

### 1.3 HIGH: Task ID Type Inversion Loses Task Path Metadata

`scripts/export_official_results.py` line 1502:

```python
task_id = result_payload.get("task_id") if isinstance(result_payload.get("task_id"), dict) else {}
task_path_from_result = task_id.get("path") if isinstance(task_id, dict) else None
```

When `task_id` in `result_payload` is a string (the common case, e.g.
`"csb_sdlc_fix/ansible-abc-imports-fix-001"`), the ternary assigns `{}` to
`task_id`. The second line then calls `{}.get("path")` which always returns
`None`. Task path metadata from `result.json` is lost, and the function
falls back to heuristic path discovery. String task IDs should be handled
as a separate branch to extract path information.

### 1.4 HIGH: OpenHands Launcher Leaks File Descriptors

`agents/harnesses/openhands/agent.py` lines 141 and 147-153 generate a
launcher script with unclosed file handles:

```python
f"task = open('{self._TASK_FILE}').read()",          # Line 141 -- never closed
"log = open('/logs/agent/openhands.txt', 'wb')",     # Line 147 -- closed only on success path
...
"log.close()",                                        # Line 153
```

The task file read uses `open().read()` without a context manager, leaking
a file descriptor. The log file has no `try/finally` protection -- if the
subprocess write loop throws an exception, `log.close()` is never reached.
In long-running container sessions, this accumulates leaked descriptors.

### 1.5 HIGH: Token Refresh Has No File Locking (TOCTOU Race)

`agents/claude_baseline_agent.py` lines 1325-1450 implement OAuth token
refresh with a read-check-refresh-write pattern but no file locking:

1. Process A reads `.credentials.json`, sees token expired (line 1337)
2. Process B reads `.credentials.json`, sees same expired token
3. Process A refreshes using `refresh_token` (single-use), writes new creds
4. Process B tries to refresh using the now-consumed `refresh_token` -- gets 403

The code has a mitigation (lines 1429-1446): on 403, it re-reads the file
to check if another process already refreshed. But there is a second TOCTOU
window: between the 403 re-read and the caller using the token, a third
process could start yet another refresh cycle. File locking (e.g.,
`fcntl.flock`) would eliminate the race entirely.

### 1.6 HIGH: Bootstrap CI Index Boundary Bug

`scripts/csb_metrics/statistics.py` lines 276-277:

```python
lo_idx = int(alpha / 2 * n_bootstrap)
hi_idx = int((1 - alpha / 2) * n_bootstrap) - 1
```

When `n_bootstrap=1` (e.g., single-task suites or edge-case configurations):
- `lo_idx = int(0.025 * 1) = 0`
- `hi_idx = int(0.975 * 1) - 1 = -1`

Index `-1` in Python accesses the last element, which happens to be
`resamples[0]` in a length-1 list -- technically not a crash but semantically
wrong (CI lower bound = upper bound = mean, which is correct by accident).
The same pattern appears in `paired_bootstrap_delta` at lines 348-349.
For `n_bootstrap=0` (defensive case), `resamples` is empty and both
indices crash with `IndexError`.

**Fix**: Guard with `if n_bootstrap < 2: return (mean_val, mean_val, mean_val)`.

### 1.7 MEDIUM: Auth Proxy Startup Race Condition

`agents/harnesses/openhands/agent.py` lines 253-262 use a hardcoded
1-second sleep before checking if the Sourcegraph auth proxy is ready:

```python
await environment.exec("sleep 1")
result = await environment.exec(
    f"curl -s -o /dev/null -w '%{{http_code}}' http://127.0.0.1:{self._SG_PROXY_PORT}/ || echo 'proxy_down'"
)
```

On CPU-throttled containers, the proxy may not bind to the port in 1 second.
When the health check fails, lines 258-261 log an error and `return` silently
(no exception), so the calling code cannot distinguish "MCP not needed" from
"MCP setup failed." The agent proceeds to use MCP tools that will fail at
runtime with unrelated errors.

**Fix**: Retry the health check with exponential backoff (max 10s), and raise
an exception on failure instead of returning silently.

### 1.8 MEDIUM: Proxy Retry Logic Has No Jitter

`agents/harnesses/openhands/sg_auth_proxy.py` lines 95-96 and 100-101 use
a fixed 2-second retry delay:

```python
_RETRY_DELAY = 2  # seconds
time.sleep(_RETRY_DELAY)
```

When multiple concurrent agent containers hit upstream Sourcegraph rate
limits simultaneously, all retry at the same cadence, creating a thundering
herd. Exponential backoff with random jitter is standard practice for
upstream-facing retry loops.

### 1.9 MEDIUM: Proxy Error Handler Exposes Internal Details

`agents/harnesses/openhands/sg_auth_proxy.py` lines 153-157:

```python
except Exception as e:
    self.send_response(502)
    self.wfile.write(str(e).encode())
```

The raw exception message is sent to the client without sanitization. If the
exception contains file paths, token fragments, or internal state, this
information is disclosed to the container environment. The exception is also
not logged to the server's stderr, making debugging harder.

### 1.10 MEDIUM: Import Monkeypatching Has Order Dependency

`agents/harnesses/openhands/agent.py` lines 40-44 patch a Harbor utility
function at import time:

```python
harbor_utils.get_api_key_var_names_from_model_name = _get_api_key_var_names_from_model_name
import sys
_openhands_mod = sys.modules.get("harbor.agents.installed.openhands")
if _openhands_mod is not None:
    _openhands_mod.get_api_key_var_names_from_model_name = _get_api_key_var_names_from_model_name
```

If `harbor.agents.installed.openhands` was imported before this module,
it captured a reference to the original function. The `sys.modules` check
handles this case, but only if the module was already fully loaded. During
circular imports or lazy loading, the patch may not take effect, causing
OpenHands runs to look for wrong API key environment variables.

### 1.11 MEDIUM: `declare -A` Bash-Specific Syntax in Shell Scripts

`configs/run_selected_tasks.sh` lines 250-252 and `configs/_common.sh`
lines 1197, 1202 use `declare -A` for associative arrays:

```bash
declare -A BENCHMARK_TASK_IDS
declare -A BENCHMARK_TASK_DIRS
declare -A BENCHMARK_COUNTS
```

These scripts have `#!/usr/bin/env bash` shebangs, so they are bash-specific
by design. However, macOS ships Bash 3.2 (2007) which supports `declare -A`
but with different behavior than Bash 4+. In Bash 3.2, `declare -A` is
technically supported but has known issues with compound assignment
(`declare -A foo=([a]=1 [b]=2)`). The code uses incremental assignment
(`foo[$key]+=...`), which works in 3.2, but this is fragile -- any future
use of compound assignment would break on macOS without warning.

The project's own `CLAUDE.md` (line 133) already notes that macOS Bash 3.2
lacks associative arrays. These scripts use associative arrays anyway.

## 2. Feature & UX Improvements

### 2.1 Multi-Account Orchestration Lacks Observability

The recently added multi-account support (`account_health.py` +
`_common.sh` expansion) introduces significant new state management:
account readiness classification, task-to-account assignment, rate-limit
tracking, and retry queues. However, there is no centralized view of this
state during or after a run:

- **During runs**: Operators see per-task log lines but no aggregate view
  of account utilization, queue depth, or retry counts.
- **After runs**: No post-run report shows which account ran which tasks,
  whether any accounts were deactivated, or how many retries occurred.

**Improvement**: Add an `--account-report` flag to `aggregate_status.py`
that reads the account assignment log and produces a summary table:

```
Account       Tasks  Retries  Deactivated  Rate-Limited
account1        45       2       No           1x at 14:32
account2        38       0       Yes (15:01)  3x
account3        42       1       No           0
```

### 2.2 No Validation of `daytona_cost_guard.py` Pricing Constants

`scripts/daytona_cost_guard.py` lines 37-43 hardcode Daytona infrastructure
pricing:

```python
CPU_HOURLY_USD = 0.0504
MEMORY_GIB_HOURLY_USD = 0.0162
STORAGE_GIB_HOURLY_USD = 0.000108
```

These are correct as of March 2026, but Daytona pricing can change without
notice. There is no mechanism to:
1. Verify pricing against the Daytona API
2. Warn when pricing constants are older than N days
3. Override pricing via environment variables or config file

Combined with the conflicting pricing constants for Claude API tokens
(reported March 7, still unfixed), the project now has two independent
sets of hardcoded pricing with no validation or centralized management.

**Improvement**: Create a `scripts/pricing.py` constants file covering both
Daytona infrastructure and Claude API model pricing, with a `--check` flag
that validates against live APIs.

### 2.3 Run Resumability Requires External Tracking

The `--skip-completed` flag in `run_selected_tasks.sh` (with its known bugs)
is the only resumability mechanism. But there is no durable record of:
- Why a run stopped (crash, rate limit, manual Ctrl-C, cost guard)
- Which tasks were in-flight when the run stopped
- What account was assigned to each in-flight task

If a run crashes, the operator must manually inspect run directories to
determine state. The new multi-account system makes this worse because
tasks may be assigned to different accounts.

**Improvement**: Write a `run_state.json` file at the start of each batch
that records the full task manifest, account assignments, and task status
(pending/running/completed/failed). Update it atomically as tasks complete.
On resume, read this file instead of scanning directories.

### 2.4 Stale Sandbox Cleanup Is Aggressive Without Confirmation

`scripts/daytona_cost_guard.py` includes stale sandbox detection with
default thresholds as low as 15 minutes (`DEFAULT_FAILED_SANDBOX_STALE_HOURS
= 0.25`). While the cost guard currently only reports stale sandboxes (it
doesn't delete them), the config scripts in `run_selected_tasks.sh` call
the cost guard with `--cleanup` in some code paths. An operator running a
slow task could have their sandbox flagged as stale and potentially cleaned
up while still active.

**Improvement**: The stale detection should cross-reference active Harbor
process state (e.g., check if a local process is still communicating with
the sandbox) before classifying a sandbox as stale.

### 2.5 No Diff-Aware Evaluation for Iterative Development

When developing or calibrating tasks, operators typically make small changes
to `instruction.md`, `oracle_answer.json`, or `test.sh` and want to re-run
only the affected tasks. Currently, `run_selected_tasks.sh` has no way to
filter to "tasks whose definition changed since last run." The operator must
either re-run the entire suite or manually specify task IDs.

**Improvement**: Add `--changed-since <commit>` flag that uses
`git diff --name-only <commit> -- benchmarks/` to identify tasks with
modified definitions and runs only those.

## 3. Research Recommendations

### 3.1 Add File Locking for Shared Credential Files

The token refresh TOCTOU race (Section 1.5) affects all multi-account
parallel runs. Python's `fcntl.flock()` provides advisory file locking
on Unix:

```python
import fcntl

with open(creds_file, 'r+') as f:
    fcntl.flock(f, fcntl.LOCK_EX)
    creds = json.load(f)
    # ... refresh logic ...
    f.seek(0)
    json.dump(creds, f, indent=2)
    f.truncate()
    fcntl.flock(f, fcntl.LOCK_UN)
```

This eliminates the race window entirely. The `_common.sh` token refresh
logic should use the same locking mechanism (via `flock` command on Linux/
macOS).

### 3.2 Add Health Check Retry with Backoff to Auth Proxy

Replace the single `sleep 1` + `curl` pattern in the OpenHands agent with
a retry loop:

```python
max_wait = 10
interval = 0.5
elapsed = 0
while elapsed < max_wait:
    await environment.exec(f"sleep {interval}")
    result = await environment.exec(f"curl -s -o /dev/null -w '%{{http_code}}' ...")
    if result.stdout.strip() == "200":
        break
    interval = min(interval * 2, 2)
    elapsed += interval
else:
    raise RuntimeError("Sourcegraph auth proxy failed to start within 10s")
```

This handles slow container startup and provides a clear error path.

### 3.3 Centralize Run State in a Lightweight State Machine

The project has accumulated multiple mechanisms for tracking run state:
- `result.json` (per-task, written by Harbor)
- `task_metrics.json` (per-task, written by post-processing)
- `status.json` (per-directory, written by `aggregate_status.py`)
- Account assignment logs (per-account, written by `_common.sh`)
- Cost guard cache (per-repo, written by `daytona_cost_guard.py`)

These are independent, sometimes contradictory, and require different
scripts to query. A lightweight state machine in `scripts/run_state.py`
that provides a single `RunState` object with `task_status()`,
`account_status()`, and `cost_status()` methods would unify these views
and enable features like:
- `--resume` from any failure point
- Real-time progress dashboards
- Post-run audit trails

### 3.4 Add `ruff` Pre-Commit Check for Data Loss Patterns

Several bugs in this report (Sections 1.1, 1.2, 1.3) involve Python
patterns that a custom `ruff` rule could catch:

- `dict.get(key, fallback)` where `fallback` is another `.get()` call
  (the default is only used when key is absent, not when value is `None`)
- `x or 0` where `x` could be `0` or `False` (use `x if x is not None else 0`)

While custom `ruff` rules require plugin development, the project could
use a simpler approach: a `grep`-based check in `repo_health.py` that
flags `.get(.*,.*.get(` patterns in Python files.

### 3.5 Investigate Daytona SDK Sandbox Lifecycle Events

The current stale sandbox detection in `daytona_cost_guard.py` relies on
polling sandbox state and comparing timestamps. The Daytona SDK may support
webhook or event-stream notifications for sandbox lifecycle events (created,
started, stopped, failed). If available, this would enable:
- Real-time cost tracking instead of periodic polling
- Immediate cleanup of failed sandboxes
- Accurate per-task cost attribution

### 3.6 Add Structured Logging to Agent Harnesses

The agent harnesses (`claude_baseline_agent.py`, `openhands/agent.py`)
use Python's `logging` module with unstructured string messages. For
post-run analysis, structured JSON logging would enable:
- Automated extraction of timing data (sandbox creation, agent startup,
  task execution, teardown)
- Account-to-task mapping without parsing log text
- Error categorization without regex fingerprinting

Replace `logger.info(f"...")` with `logger.info("event", extra={"event_type": "sandbox_created", "sandbox_id": sid, "account": account})` and configure a JSON formatter.

### Evaluation Data Integrity Layer

**The single most impactful feature** is fixing the silent data loss bugs in
the evaluation pipeline that produce incorrect metrics in published reports.
Three independently broken code paths corrupt the same output artifacts
(`eval_report.json`, official results export, statistical comparisons):

**Part A -- Fix `dict.get()` None-value bugs** (30 minutes)

1. Fix `export_official_results.py` line 1449: change nested `.get()` to
   `agent_result.get("n_cache_tokens") or agent_result.get("cache_creation_input_tokens")`
2. Fix `export_official_results.py` line 1502: handle string `task_id` values
   as a separate branch instead of collapsing to `{}`
3. Audit all `.get(key, .get())` patterns in `scripts/` for the same bug
   (preliminary grep finds 4 additional instances)

**Part B -- Fix bootstrap CI edge case** (15 minutes)

1. Add `if n_bootstrap < 2` guard in `statistics.py` `bootstrap_ci()` and
   `paired_bootstrap_delta()` to return `(mean, mean, mean)` for degenerate
   cases
2. Add unit tests for `n_bootstrap=1` and `n_bootstrap=0` cases

**Part C -- Add evaluation pipeline regression tests** (2 hours)

1. Create `tests/test_eval_report.py` with fixtures for edge cases:
   - Tasks with `None` token fields
   - Tasks with zero cache tokens
   - Single-task suites (bootstrap edge case)
   - Mixed dict/string `task_id` values
2. Create `tests/test_export_official.py` with fixtures for:
   - `result.json` with `None` values in all optional fields
   - `task_id` as string vs dict
   - Missing `task_metrics.json`
3. Add these tests to the CI workflow (when it is created)

**Part D -- Add `repo_health.py` check for known anti-patterns** (30 minutes)

1. Add a check that greps Python files for `.get(.*,.*\.get(` and flags
   instances for review
2. Add a check that greps for `(x or 0)` patterns where `x` could be
   numeric
3. These checks prevent future regressions of the same bug class

**Why this is highest impact**: The March 6 report recommended CI test
infrastructure (foundational). The March 7 report recommended operational
recovery (unblocking runs). This report recommends evaluation data integrity
because **the metrics pipeline is the project's primary output**. Every
report, comparison, and leaderboard entry flows through
`generate_eval_report.py` and `export_official_results.py`. Silent data loss
in these scripts means published results may contain incorrect cache token
metrics, missing task path metadata, and potentially wrong confidence
intervals. Unlike operational bugs (which operators notice immediately),
data integrity bugs persist in published artifacts indefinitely.

**PRD-ready description**: "Implement an evaluation data integrity layer
consisting of: (1) fixes for the `dict.get()` None-value antipattern in
`export_official_results.py` (lines 1449 and 1502) that silently lose cache

(3) regression test suites for `generate_eval_report.py` and

token fields, task IDs, and task metrics, and (4) a `repo_health.py` lint
check that flags `.get(key, .get())` patterns to prevent future instances.
Success criteria: all existing evaluation reports produce identical output
after fixes (no data was being *added* incorrectly, only *lost*), new test
suites cover the 4 identified edge cases with 100% pass rate, and
`repo_health.py` flags zero instances of the antipattern in the codebase."
