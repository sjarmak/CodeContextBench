## 1. Code & Architecture Review

### 1.1 HIGH: Agent Uses Fixed `/tmp` Filenames Creating Race Condition

`agents/claude_baseline_agent.py` lines 1134-1141:

```python
file_cmds += f" && echo '{prompt_b64}' | base64 -d > /tmp/claude_system_prompt.txt"
file_cmds += f" && echo '{script_b64}' | base64 -d > /tmp/claude_run.sh && chmod +x /tmp/claude_run.sh"
run_cmd = "su claude -s /bin/bash /tmp/claude_run.sh"
```

Both the system prompt and the wrapper script are written to fixed
paths (`/tmp/claude_system_prompt.txt`, `/tmp/claude_run.sh`). When
Harbor launches concurrent tasks in the same container or host,
processes race on these files:

1. Task A writes its system prompt to `/tmp/claude_system_prompt.txt`
2. Task B overwrites with its own prompt
3. Task A reads `/tmp/claude_system_prompt.txt` and gets Task B's
   prompt

The system prompt controls whether MCP tools are available and which
repos are scoped -- cross-contamination silently corrupts the
baseline/MCP distinction that is the benchmark's core evaluation axis.

**Impact**: Concurrent Harbor runs within a single sandbox can silently
exchange system prompts, causing baseline tasks to receive MCP
instructions or vice versa. Results appear valid but measure the wrong
configuration.

**Fix**: Use `mktemp` for unique temporary files:
```bash
PROMPT_FILE=$(mktemp /tmp/claude_system_prompt.XXXXXX)
SCRIPT_FILE=$(mktemp /tmp/claude_run.XXXXXX.sh)
```
Or, since each Harbor task gets its own sandbox, embed the task ID
in the filename: `/tmp/claude_system_prompt_${TASK_ID}.txt`.

### 1.2 HIGH: Token Refresh Missing Exception Handlers for Network Errors

`agents/claude_baseline_agent.py` lines 1523-1559:

```python
try:
    with urllib.request.urlopen(req, timeout=15) as resp:
        token_data = json.loads(resp.read())
except urllib.error.HTTPError as e:
    body = e.read().decode() if e.fp else ""
    ...
```

Only `urllib.error.HTTPError` is caught. Three common failure modes
are unhandled:

| Exception | Trigger | Outcome |
|-----------|---------|---------|
| `urllib.error.URLError` | DNS failure, connection refused, proxy error | Unhandled crash |
| `socket.timeout` | OAuth endpoint slow (>15s) | Unhandled crash |
| `json.JSONDecodeError` | Malformed token response body | Unhandled crash |

Additionally, in the `HTTPError` handler (line 1527), `e.read()` reads
the error body but never closes the error response's file pointer
(`e.fp`), leaking a socket file descriptor per failed refresh attempt.

**Impact**: Any network disruption during multi-account token rotation
causes the agent launcher to crash with an unhandled exception rather
than a descriptive error. In long-running batch sessions with periodic
refreshes, socket leaks from repeated 403 retries (line 1529-1551) can
exhaust file descriptor limits.

**Fix**: Wrap in broader exception handling:
```python
except urllib.error.HTTPError as e:
    with e:  # ensures e.fp is closed
        body = e.read().decode() if e.fp else ""
    ...
except (urllib.error.URLError, socket.timeout) as e:
    raise RuntimeError(f"Token refresh network error: {e}")
```

### 1.3 HIGH: Runner Exit Code Lost Through Pipeline

`configs/run_selected_tasks.sh` lines 681-682 and 756-757:

```bash
    2>&1 | tee -a "${bl_jobs_dir}.log" \
    || echo "WARNING: $BASELINE_CONFIG failed: $task_id"
```

The `||` operator applies to the pipeline's exit code, which is the
exit code of the **last command** (`tee`), not `harbor_run_guarded`.
Since `tee` almost always succeeds (exit 0), the warning branch never
executes regardless of whether `harbor_run_guarded` failed.

This pattern appears twice: once for baseline runs (line 681-682) and
once for MCP runs (line 756-757).

**Impact**: Failed Harbor runs produce no warning output. Operators
monitoring batch runs see no indication of per-task failures until they
run `aggregate_status.py` after the batch completes, delaying triage
by the full batch duration.

**Fix**: Use `set -o pipefail` in the subshell, or capture the exit
code explicitly:
```bash
(
    set -o pipefail
    harbor_run_guarded ... 2>&1 | tee -a "${bl_jobs_dir}.log" \
        || echo "WARNING: $BASELINE_CONFIG failed: $task_id (exit $?)"
) &
```

### 1.5 MEDIUM: Runner Missing Trap Handler for Temporary Directory Cleanup

`configs/run_selected_tasks.sh` creates temporary directories for each
task pair (lines 648, 699, 711) and tracks them in `_MCP_TEMP_DIRS`.
These are cleaned up in `_drain_pool()` at the end of execution.
However, no `trap` handler is registered, so early exits (Ctrl+C, error
before `_drain_pool`, `set -e` triggering) leave orphaned temporary
directories in `/tmp`.

**Impact**: Over repeated batch runs, each interrupted run leaves
behind task-sized directory copies in `/tmp`. For benchmarks with large
repos (e.g., Kubernetes, Linux), a single orphaned temp directory can
be 1-5 GB. Ten interrupted runs can fill `/tmp` on systems with limited
temp storage.

**Fix**: Register a cleanup trap after `_MCP_TEMP_DIRS` is declared:
```bash
trap '_drain_pool 2>/dev/null; rm -rf "${_MCP_TEMP_DIRS[@]}" 2>/dev/null' EXIT INT TERM
```

### 1.6 MEDIUM: Runner `mktemp` Failure Silently Produces Empty Paths

`configs/run_selected_tasks.sh` line 648:

```bash
_bl_temp_dir=$(_td=$(mktemp -d "/tmp/bl_${task_id}_XXXXXX") && \
  _td_lower=$(echo "$_td" | tr '[:upper:]' '[:lower:]') && \
  [ "$_td" != "$_td_lower" ] && mv "$_td" "$_td_lower"; \
  echo "${_td_lower:-$_td}")
```

If `mktemp -d` fails (disk full, permissions error), `_td` is empty.
The subsequent `echo "${_td_lower:-$_td}"` outputs an empty string.
Line 649-650 then runs `cp -a "${abs_path}/." "${_bl_temp_dir}/"` which
copies the entire task directory into the current working directory
(since `$_bl_temp_dir` is empty, the target becomes `./`).

The same pattern appears at lines 699 and 711 for MCP temp directories.

**Impact**: On a full `/tmp`, task files are silently copied into the
runner's working directory instead of a temporary directory. Harbor
then operates on the polluted working directory. Results are invalid
but no error is reported.

**Fix**: Check the temp dir exists before proceeding:
```bash
[ -d "$_bl_temp_dir" ] || { echo "FATAL: mktemp failed for $task_id"; exit 1; }
```

### 1.7 MEDIUM: `grep -P` Not Portable to macOS

`configs/run_selected_tasks.sh` line 726:

```bash
_sg_repos=$(grep -oP '(?<=ENV SOURCEGRAPH_REPOS=")[^"]+' \
  "${_mcp_task_path}/environment/Dockerfile" 2>/dev/null || true)
```

The `-P` flag (Perl-compatible regex) is a GNU `grep` extension.
On macOS (which ships BSD `grep`), this flag is not supported. The
`2>/dev/null || true` suppresses the error, causing `_sg_repos` to
be silently empty even when `SOURCEGRAPH_REPOS` is defined in the
Dockerfile.

**Impact**: MCP tasks run on macOS (local development, CI with macOS
runners) have empty `SOURCEGRAPH_REPOS`, potentially causing the agent
to query all repos instead of the task-scoped set. This changes the
evaluation conditions without any visible error.

**Fix**: Use POSIX-compatible alternatives:
```bash
_sg_repos=$(sed -n 's/^ENV SOURCEGRAPH_REPOS="\([^"]*\)".*/\1/p' \
  "${_mcp_task_path}/environment/Dockerfile" 2>/dev/null || true)
```

### 2.1 No Schema Validation Anywhere in the Pipeline

Meanwhile, Section 1.4 shows that 3 of 8 schemas have stale enum

Without any consumer, schemas drift silently. They cannot be trusted
as documentation because no automated process verifies their accuracy.

**Improvement**: Add a `schema_integrity` check to `repo_health.py`
that:
1. For each `*.schema.json` file, validates that at least one
   corresponding data file in the repo passes validation against it
2. For enum fields, verifies that at least one real data file uses
   each enum value (dead enum detection)

Alternatively, integrate `jsonschema` validation into the scripts that
produce these data formats (`run_judge.py` validates output against
`judge_result.schema.json`, etc.).

### 2.3 Agent Token Refresh Lacks Exponential Backoff

`agents/claude_baseline_agent.py` lines 1485-1559 handle OAuth token
refresh as a one-shot attempt with no retry logic. When the Anthropic
OAuth endpoint returns a transient error (500, 502, 503, 429), the
agent raises `RuntimeError` and the task fails.

In multi-account batch runs with 4+ accounts refreshing tokens in
parallel, transient OAuth failures are expected at low rates.
Currently, each transient failure kills a Harbor task that then needs
manual rerun.

**Improvement**: Add retry with exponential backoff for transient
HTTP errors:
```python
TRANSIENT_CODES = {429, 500, 502, 503, 504}
for attempt in range(3):
    try:
        with urllib.request.urlopen(req, timeout=15) as resp:
            return json.loads(resp.read())
    except urllib.error.HTTPError as e:
        if e.code not in TRANSIENT_CODES or attempt == 2:
            raise
        time.sleep(2 ** attempt)
```

### 2.4 No `task.toml` Schema File

Every task directory contains a `task.toml` file that
`validate_tasks_preflight.py` parses (line 227:
`parse_task_toml_simple(toml_path)`). The parser is a custom 70-line
regex-based TOML subset parser (lines 114-186) that handles only
top-level key=value pairs and `[section]` headers.

There is no `schemas/task_toml.schema.json` to document the expected
structure. Contributors authoring new tasks must reverse-engineer the
schema from existing tasks or the parser code.

**Improvement**: Create `schemas/task_toml.schema.md` (since TOML
doesn't have a standard schema format like JSON Schema) documenting:
- Required sections (`[task]`, `[environment]`)
- Required fields per section (`task.id`, `task.category`,
  `environment.dockerfile`, etc.)
- Optional fields with defaults
- Valid enum values for fields like `task.difficulty`

### 3.1 Centralize Suite Name Mapping (Eliminate 16-Way Duplication)

The `DIR_PREFIX_TO_SUITE` mapping is independently maintained in 16
scripts (Section 1.8). Each copy must be updated when suite naming
changes. Three naming eras exist simultaneously in the codebase:

| Era | Prefix | Example | Active Data |
|-----|--------|---------|-------------|
| Original | `ccb_` | `ccb_pytorch` | Historical runs only |
| SDLC/Org reorg | `csb_sdlc_`, `csb_org_` | `csb_org_security` | Current 274 tasks |
| CSB unified | `csb_` | `csb_security` | Future (benchmarks/csb/) |

Create a single `scripts/csb_metrics/suite_registry.py`:

```python
# Canonical source of truth for all suite name mappings
CURRENT_SUITES = {
    "csb_sdlc_debug", "csb_sdlc_design", "csb_sdlc_document",
    "csb_sdlc_feature", "csb_sdlc_fix", "csb_sdlc_refactor",
    "csb_sdlc_secure", "csb_sdlc_test", "csb_sdlc_understand",
    "csb_org_compliance", "csb_org_crossorg", "csb_org_crossrepo",
    "csb_org_crossrepo_tracing", "csb_org_domain", "csb_org_incident",
    "csb_org_migration", "csb_org_onboarding", "csb_org_org",
    "csb_org_platform", "csb_org_security",
}

LEGACY_DIR_PREFIX_TO_SUITE = {
    "k8s_docs_": "ccb_k8sdocs",
    "locobench_": "ccb_locobench",
    "pytorch_": "ccb_pytorch",
    "tac_": "ccb_tac",
    ...
}

def detect_suite(dir_name: str) -> str:
    """Detect suite from directory name, supporting all eras."""
    ...
```

All 16 scripts import `detect_suite` from this module. Suite naming
changes become a one-file update.

### 3.2 Add Pipeline Error Propagation Tests

The runner's exit-code-through-pipeline bug (Section 1.3) is a common
shell scripting mistake. Add a test that validates error propagation
in critical pipeline paths:

```bash
#!/usr/bin/env bash
# tests/test_runner_pipelines.sh
# Verify that harbor_run failures are not swallowed by tee

# Mock harbor_run_guarded to fail
harbor_run_guarded() { return 1; }
export -f harbor_run_guarded

# Run the pipeline pattern from run_selected_tasks.sh
output=$(
    set -o pipefail
    harbor_run_guarded 2>&1 | tee /dev/null
    echo "EXIT:$?"
)

# Assert exit code propagated
[[ "$output" == *"EXIT:1"* ]] || {
    echo "FAIL: harbor_run exit code swallowed by tee pipeline"
    exit 1
}
```

This catches regressions in the pipeline pattern even if the specific
`set -o pipefail` fix is accidentally removed.

### 3.3 Schema Synchronization via Code Generation

The 3 stale schemas (Section 1.4) and 8 schema files with no consumers
(Section 2.1) suggest that manually maintained schemas will always
drift. Instead of maintaining schemas by hand:

1. Define canonical data models in Python (e.g., in
   `scripts/csb_metrics/models.py` using `dataclass` or Pydantic)
2. Generate JSON schemas from the Python models:
   ```python
   from pydantic import BaseModel
   class JudgeResult(BaseModel):
       task_id: str
       benchmark: str
       judge_score: float  # 0.0-1.0
       ...
   schema = JudgeResult.model_json_schema()
   ```
3. Run `python3 scripts/generate_schemas.py` to regenerate all schemas
4. Add a `schema_freshness` check to `repo_health.py` that compares
   generated schemas against committed schemas

This eliminates the drift category entirely -- schemas are always
derived from the code that produces the data.

### 3.4 Investigate Container Isolation for Agent Temp Files

The `/tmp` race condition (Section 1.1) is mitigated if each Harbor
task gets its own container (which is the normal case for Daytona
sandboxes). However, the code should not rely on container isolation
as its only safety net:

1. Audit whether any execution mode (local Docker, shared sandbox)
   runs multiple tasks in the same `/tmp` namespace
2. If container isolation is guaranteed, document this assumption
   prominently in the agent code
3. If not guaranteed, implement unique temp file naming as described
   in Section 1.1

The fix is inexpensive (one line change per temp file) and eliminates
an entire class of concurrency bugs regardless of the execution
environment.
