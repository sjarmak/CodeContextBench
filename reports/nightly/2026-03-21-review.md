## 1. Code & Architecture Review

### 1.1 Two Confirmed Broken Quality-Gate Checks in abc_audit.py (CRITICAL)

Running `python3 -m pytest tests/test_abc_audit.py` today reveals **2 failing tests, 40
passing** â€” a partial test suite failure that has been invisible because no CI workflow invokes
pytest:

```
FAILED tests/test_abc_audit.py::TestT5NoSolutionLeak::test_detect_leak
FAILED tests/test_abc_audit.py::TestR2NoContamination::test_fail_contaminated
```

**Failure 1 â€” T5 solution leak check:**
`check_t5_no_solution_leak` returns `Status.PASS` for a task whose `instruction.md` contains
`"Apply this exact change:\npatch -p1 < fix.patch\n"` with a matching `solve.sh`. The function
should return `Status.WARN`. Any task that embeds its solution in the instruction currently passes
this gate silently.

**Failure 2 â€” R2 contamination check:**
`check_r2_no_contamination` returns `Status.PASS` for a task whose `instruction.md` contains
`"Fix the bug. Use sourcegraph to search the code."` â€” a classic tool-contamination pattern.
The function should return `Status.FAIL` with "sourcegraph" in evidence. Tasks instructing agents
to use specific search tools currently pass contamination review without a flag.

**Root cause:** `abc_audit.py` has the known duplicate function definition issue (report #8:
"Python silently uses last definition"). For both `check_t5_no_solution_leak` and
`check_r2_no_contamination`, the second definition is the live one â€” and it is broken.

**Impact:** Any audit run over the existing 274 tasks with `python3 scripts/abc_audit.py` is
producing incorrect T5/R2 verdicts. Tasks that should be flagged are passing clean.

### 1.2 Cost Pipeline Never Uses Actual Run Model (CRITICAL)

| File | Line | Call |
|------|------|------|
| `scripts/extract_task_metrics.py` | 266â€“269 | `calculate_cost_from_tokens(tm.input_tokens, tm.output_tokens, ...)` â€” no `model` |
| `scripts/csb_metrics/discovery.py` | 310â€“313 | Same pattern |

Both calls fall back to `_DEFAULT_MODEL = "claude-opus-4-5-20250514"` regardless of the
model actually used in a run. The function docstring acknowledges this: *"Unknown models
fall back to default Opus 4.5 pricing"* â€” but the intent was for callers to pass the model

**Severity by model:**

| Run model | Fallback rates | Cost error |
|-----------|----------------|------------|
| Sonnet 4.5 (`claude-sonnet-4-5-20250929`) | Opus 4.5 rates | ~5Ă— overstatement |
| Haiku 4.5 (`claude-haiku-4-5-20251001`) | Opus 4.5 rates | ~19Ă— overstatement |
| Opus 4.6 (`claude-opus-4-6`) | Opus 4.5 rates | ~0% error (same tier) |

`anthropic/claude-sonnet-4-6`, making this a live issue for any OpenHands runs.

**No `tm.model` field exists:** `TaskMetrics` has no model attribute, and no extractor
reads the model ID from `result.json`. The model ID is therefore lost entirely after

### 1.5 Non-Atomic Credential Writes (MEDIUM)

`scripts/daytona_runner.py:234` writes OAuth credentials with a direct `write_text`:

```python
creds_path.write_text(json.dumps(creds, indent=2))  # not atomic
```

`scripts/account_health.py:109â€“111` already implements the correct pattern:

```python

temp_path.write_text(...)
temp_path.replace(path)
```

If `daytona_runner.py` crashes mid-write (e.g., during token refresh under load), the
credentials file is left partially written and subsequent runs fail authentication entirely
with no descriptive error.

## 2. Feature & UX Improvements

Every cost figure in `generate_eval_report.py`, `compare_configs.py`, and the HTML export
currently reflects Opus 4.5 pricing regardless of the actual model. A one-line annotation â€”
*"Estimated at claude-opus-4-5-20250514 rates (model not extracted)"* â€” would at least alert

`verify_retrieval_eval_smoke.py` should accept `--fix-run`, `--debug-run`, and `--pooled-runs`

stage.

The ABC audit is described as a quality gate, but it is only run manually. Adding
`python3 scripts/abc_audit.py --all --summary` to `repo_health.yml` (after pytest) would
make audit regressions visible on every commit.

## 3. Research Recommendations

### 3.1 Centralized Model Registry with Startup Validation

All three model ID issues (missing Sonnet 4.6/Haiku 4.6 in pricing, callers not passing
model, stale model IDs in scripts) stem from the same root: model IDs are scattered strings
with no central authority. A `csb_metrics/model_registry.py` module defining:

```python
CURRENT_OPUS    = "claude-opus-4-6"
CURRENT_SONNET  = "claude-sonnet-4-6"
CURRENT_HAIKU   = "claude-haiku-4-6"
```

with a startup assertion that all `MODEL_PRICING` keys are recognized would turn this into
a build-time failure instead of a silent runtime mis-pricing. The existing `_WARNED_UNKNOWN_PRICING_MODELS` one-time warning is drowned out in large batch logs.

### 3.2 pytest + Coverage Baseline in CI

Adding `pytest tests/ --cov=scripts --cov-report=term-missing --cov-fail-under=40` to CI
serves two purposes:
1. Gates the build on test passage (catching the two T5/R2 regressions that exist today)
2. Establishes a coverage floor that prevents future PRDs from claiming "tested" without
   meeting the threshold

The 9 existing test files already cover the most critical modules. Adding the CI step is a
force multiplier on all future fixes.

### 3.3 Pytest Fixtures for Oracle Check Integration Tests

`check_test_ratio` at `oracle_checks.py:478` has no test coverage. Adding a test that
verifies a known-safe `test_command` string rejects injection metacharacters (`; rm -rf`)
would both document the expected behavior and prevent future shell injection from going
unnoticed.
