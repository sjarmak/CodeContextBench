Architecture of the Current Allocation Framework\n\n### 2.1 Entry Point: `ShardsAllocator`\n\n`ShardsAllocator` (`org.elasticsearch.cluster.routing.allocation.allocator`) is the top-level interface for all allocation logic. It declares three key operations:\n\n| Method | Purpose |\n|---|---|\n| `allocate(RoutingAllocation)` | Assign unassigned shards, relocate shards that must move, and rebalance |\n| `allocate(RoutingAllocation, ActionListener<Void>)` | Async variant; defaults to calling synchronous `allocate` |\n| `decideShardAllocation(ShardRouting, RoutingAllocation)` | Explain the allocation decision for a single shard (used by the cluster explain API) |\n\n### 2.2 Default Implementation: `BalancedShardsAllocator`\n\n`BalancedShardsAllocator` is the production implementation. It uses a multi-factor **weight function** to score each (node, index) pair and greedily assigns shards to the lowest-weight node that passes all `AllocationDecider` checks.\n\nWeight factors (all dynamically configurable via cluster settings):\n\n| Setting | Default | Description |\n|---|---|---|\n| `cluster.routing.allocation.balance.shard` | 0.45 | Weight for global shard count balance |\n| `cluster.routing.allocation.balance.index` | 0.55 | Weight for per-index shard count balance |\n| `cluster.routing.allocation.balance.write_load` | 10.0 | Weight for write load balance |\n| `cluster.routing.allocation.balance.disk_usage` | 2e-11 | Weight for disk usage balance |\n| `cluster.routing.allocation.balance.threshold` | 1.0 | Minimum imbalance before rebalancing triggers |\n\nThe internal `Balancer` class performs three passes in order:\n1. `allocateUnassigned()` — assigns all unassigned shards\n2. `moveShards()` — relocates shards that must leave their current node (e.g., due to decider veto)\n3. `balance()` — rebalances to even out the weight function across nodes\n\nUnassigned shards are processed in priority order via `PriorityComparator` (which already respects `index.priority`), but node selection within that ordering is purely weight-based with no tier awareness.\n\n### 2.3 Guard Layer: `AllocationDecider` and `AllocationDeciders`\n\n`AllocationDecider` (abstract) is the per-node veto mechanism. Each implementation answers questions of the form \"can shard X go on node Y right now?\" by returning one of:\n- `Decision.YES` — allowed\n- `Decision.NO` — forbidden\n- `Decision.THROTTLE` — allowed but rate-limited\n- `Decision.ALWAYS` — unconditional yes (skipped in non-debug mode)\n\n`AllocationDeciders` composes a collection of `AllocationDecider` instances. In non-debug mode it short-circuits on the first `NO`. In debug mode (explain API) it collects all decisions. The composed result feeds back into `BalancedShardsAllocator`'s node selection.\n\n**Key existing deciders:**\n\n| Decider | What it enforces |\n|---|---|\n| `DiskThresholdDecider` | Rejects nodes below disk watermark thresholds |\n| `AwarenessAllocationDecider` | Distributes replicas across awareness zones/racks |\n| `ShardsLimitAllocationDecider` | Caps shards per node or index |\n| `FilterAllocationDecider` | Respects `index.routing.allocation.include/exclude/require` |\n| `ThrottlingAllocationDecider` | Limits concurrent recoveries |\n| `MaxRetryAllocationDecider` | Prevents infinite retry loops on failed shards |\n\n### 2.4 Existing Priority Handling\n\n`IndexMetadata.SETTING_PRIORITY` (`index.priority`, default 1) already exists and is used exclusively by `PriorityComparator` in the gateway to order the *queue* of unassigned shards before allocation. It does **not** influence which node a shard is placed on. The new design extends priority to affect node selection.\n\n---\n\n## 3. Proposed Design: `PriorityShardsAllocator`\n\n### 3.1 Goals\n\n1. Allow indices to declare a routing priority tier (`hot`, `warm`, `cold`, `frozen`).\n2. Ensure high-priority shards land on high-tier nodes whenever possible.\n3. Degrade gracefully: if no suitable high-tier node is available, fall back to any eligible node (soft constraint by default, configurable to hard constraint).\n4. Remain composable with existing deciders — do not bypass `DiskThresholdDecider`, `AwarenessAllocationDecider`, etc.\n\n### 3.2 New Index Setting: `index.routing.allocation.priority`\n\n```\nindex.routing.allocation.priority: hot   # hot | warm | cold | frozen (default: hot)\n```\n\nThis is a new per-index setting stored in `IndexMetadata`. It maps to a numeric tier level used during node scoring:\n\n| Value | Tier Level |\n|---|---|\n| `hot` | 3 (highest) |\n| `warm` | 2 |\n| `cold` | 1 |\n| `frozen` | 0 (lowest) |\n\n### 3.3 Node Tier Attribute\n\nNodes advertise their tier via an existing or new node attribute:\n\n```yaml\n# elasticsearch.yml on a hot node\nnode.attr.tier: hot\n```\n\nThis follows the existing pattern used by `AwarenessAllocationDecider` and the ILM data-tier feature. No changes to node-level configuration are needed beyond setting this attribute.\n\n### 3.4 `PriorityAllocationDecider` (new `AllocationDecider`)\n\nA new decider enforces tier compatibility:\n\n```\norg.elasticsearch.cluster.routing.allocation.decider.PriorityAllocationDecider\n```\n\n**Behavior for `canAllocate(ShardRouting, RoutingNode, RoutingAllocation)`:**\n\n1. Read `index.routing.allocation.priority` from the index metadata.\n2. Read `node.attr.tier` from the candidate node's attributes.\n3. If the node has no tier attribute, return `YES` (untiered nodes are always eligible — backwards compatible).\n4. If the node's tier level is **below** the index's required tier level:\n   - In **soft** mode (default): return `THROTTLE` so the balancer prefers higher-tier nodes but can fall back.\n   - In **hard** mode (`index.routing.allocation.priority.strict: true`): return `NO`.\n5. Otherwise return `YES`.\n\n**Behavior for `canRemain(IndexMetadata, ShardRouting, RoutingNode, RoutingAllocation)`:**\n\nSame logic as `canAllocate` — a shard cannot remain on a node whose tier has been lowered below the index's required tier (e.g., a node re-labeled from `hot` to `warm`).\n\n### 3.5 `PriorityShardsAllocator` Class\n\n`PriorityShardsAllocator` extends `BalancedShardsAllocator` and overrides node scoring to add a priority affinity bonus. Rather than replacing the weight function, it applies a **priority bias** to the node scores before sorting:\n\n```\nadjustedWeight(node, index) = balancedWeight(node, index) - priorityBonus(node, index)\n```\n\nWhere `priorityBonus` is large when the node's tier matches or exceeds the index's required tier, pulling it toward the front of the candidate list without completely overriding balance considerations.\n\nThis ensures:\n- Within the same tier, normal balance applies.\n- Across tiers, priority preference dominates (controlled by a tunable factor).\n\n#### Constructor Parameters\n\n| Parameter | Source |\n|---|---|\n| `ClusterSettings clusterSettings` | Injected; watches all balance and priority settings |\n| `WriteLoadForecaster writeLoadForecaster` | Injected |\n| `AllocationDeciders allocationDeciders` | Injected; includes the new `PriorityAllocationDecider` |\n\n#### New Cluster Setting: `cluster.routing.allocation.priority.weight`\n\nControls how strongly priority preference overrides balance. Default `5.0`. Set to `0.0` to disable the priority bias entirely (decider still active).\n\n---\n\n## 4. Interaction with Existing Deciders\n\n### 4.1 `DiskThresholdDecider`\n\nNo conflict. `PriorityAllocationDecider` runs alongside `DiskThresholdDecider` inside `AllocationDeciders`. A shard cannot be placed on a hot node that has exceeded the high watermark, even if it is the preferred tier. The allocator will fall back to the next eligible node (respecting tier order).\n\nPriority **does not** override disk watermarks. This is intentional: placing a high-priority shard on an overfull node risks cluster stability.\n\n### 4.2 `AwarenessAllocationDecider`\n\nNo conflict. Awareness zone/rack constraints apply independently. A shard requiring rack awareness AND hot-tier placement must satisfy both. In practice, hot nodes should be spread across racks, so both constraints are typically satisfiable simultaneously.\n\nIf awareness constraints prevent placement on any hot node, the soft-mode fallback allows placement on a lower-tier node in a different rack, preserving availability over strict tier preference.\n\n### 4.3 `FilterAllocationDecider`\n\n`index.routing.allocation.require/include/exclude` settings take precedence over the new priority setting (they return hard `NO`). This is consistent with the existing override hierarchy. Operators can use attribute filters to pin specific shards to specific nodes regardless of tier.\n\n### 4.4 `ThrottlingAllocationDecider`\n\nUnaffected. Throttling limits concurrent recoveries per node and remains in effect. High-priority shards cannot bypass throttle limits, but the `THROTTLE` decision causes them to be retried on the next allocation round.\n\n### 4.5 Rebalancing\n\n`PriorityShardsAllocator` also adjusts rebalance decisions. A shard already on a lower-tier node when a higher-tier node becomes available will be flagged for relocation (via `moveShards`). The rebalance threshold (`cluster.routing.allocation.balance.threshold`) still applies, preventing churn for marginal weight differences.\n\n---\n\n## 5. API Changes\n\n### 5.1 New Index Settings\n\n| Setting | Type | Default | Dynamic | Description |\n|---|---|---|---|---|\n| `index.routing.allocation.priority` | `String` | `hot` | Yes | Desired tier: `hot`, `warm`, `cold`, `frozen` |\n| `index.routing.allocation.priority.strict` | `Boolean` | `false` | Yes | If `true`, hard-reject under-tier nodes |\n\n### 5.2 New Cluster Settings\n\n| Setting | Type | Default | Dynamic | Description |\n|---|---|---|---|---|\n| `cluster.routing.allocation.priority.weight` | `Float` | `5.0` | Yes | Priority bias factor added to the weight function |\n| `cluster.routing.allocation.priority.attribute` | `String` | `tier` | No | Node attribute name carrying the tier label |\n\n### 5.3 Cluster Explain API\n\n`decideShardAllocation()` is updated to surface the tier decision. The response includes:\n\n```json\n{\n  \"node_allocation_decisions\": [\n    {\n      \"node_id\": \"...\",\n      \"deciders\": [\n        {\n          \"decider\": \"priority\",\n          \"decision\": \"THROTTLE\",\n          \"explanation\": \"node tier [warm] is below required tier [hot] for index [my-index]\"\n        }\n      ]\n    }\n  ]\n}\n```\n\n### 5.4 No REST API Changes Required\n\nThe existing `PUT /<index>/_settings` and `PUT _cluster/settings` APIs are sufficient to configure the new settings.
