Must be non-null and non-empty. Replaces any existing subscription atomically on the next `poll()`.\n- `pattern` — Java `Pattern` object matched against all topic names visible to the consumer. Kafka re-evaluates the pattern on each metadata refresh, so newly created topics that match will be automatically consumed.\n- `listener` — Optional `ConsumerRebalanceListener` invoked during rebalance events (see below). If not provided, partition ownership changes occur silently.\n\n**Behavior:**\n\n- Calling `subscribe()` with a different topic set replaces the subscription; no error is thrown.\n- Requires `group.id` to be set. Without it, an `InvalidGroupIdException` is thrown.\n- Subscription changes take effect at the next `poll()` call, not immediately.\n- Pattern subscriptions perform regex matching server-side (Kafka 2.4+) or client-side depending on broker version.\n\n**Exceptions:**\n\n- `IllegalArgumentException` — `topics` is null, empty, or contains null/empty strings.\n- `IllegalStateException` — Manual assignment is currently active.\n- `InvalidGroupIdException` — `group.id` is not configured.\n\n### Manual Assignment (Standalone Consumer)\n\n```java\nvoid assign(Collection<TopicPartition> partitions)\n```\n\n**Parameters:**\n\n- `partitions` — The exact set of `TopicPartition` objects to assign. Replaces any existing assignment. Pass an empty collection to unassign all partitions.\n\n**Behavior:**\n\n- No group coordinator is involved. No rebalances occur. No `ConsumerRebalanceListener` is called.\n- The caller is responsible for all partition tracking and offset management.\n- Useful for stream-processing frameworks, exactly-once semantics implementations, and replay scenarios.\n- Heartbeats are not sent; `session.timeout.ms` and `max.poll.interval.ms` enforcement does not apply.\n\n**Exceptions:**\n\n- `IllegalArgumentException` — `partitions` is null or contains null elements.\n- `IllegalStateException` — Dynamic subscription is currently active.\n\n### Unsubscribing\n\n```java\nvoid unsubscribe()\n```\n\nRemoves all dynamic subscriptions and manual assignments. The consumer leaves the group (if it was a member) and sends a `LeaveGroup` request to the coordinator. Subsequent `poll()` calls return empty records until a new subscription or assignment is made.\n\n### `ConsumerRebalanceListener` Callbacks\n\nCallbacks are invoked **synchronously inside `poll()`** on the thread that calls `poll()`. They block `poll()` from returning until they complete.\n\n#### `onPartitionsRevoked(Collection<TopicPartition> partitions)`\n\nCalled **before** partition ownership is transferred away from this consumer, while the consumer is still a group member. This is the correct place to:\n- Commit offsets for the partitions being revoked (so the next owner starts from the right place).\n- Flush any in-progress work for those partitions.\n- Release partition-specific resources.\n\nThe `partitions` collection contains only those being revoked from this consumer. After this method returns, Kafka transfers ownership.\n\n**Eager rebalance (default before Kafka 2.4):** `onPartitionsRevoked` is called with **all** currently assigned partitions even if they will be reassigned back to the same consumer. This causes a stop-the-world pause.\n\n**Cooperative/incremental rebalance (CooperativeStickyAssignor):** `onPartitionsRevoked` is called only with partitions actually being moved away. Partitions retained by this consumer continue to be consumed without interruption.\n\n#### `onPartitionsAssigned(Collection<TopicPartition> partitions)`\n\nCalled **after** partition ownership has been transferred to this consumer. The `partitions` collection contains only newly assigned partitions. At this point, calls to `seek()`, `position()`, or `committed()` are valid for these partitions. Use this callback to:\n- Initialize per-partition state.\n- Seek to a custom starting position (e.g., a checkpoint stored outside Kafka).\n- Warm up partition-specific caches.\n\n#### `onPartitionsLost(Collection<TopicPartition> partitions)`\n\nCalled when partitions are lost involuntarily (e.g., after a session timeout where the consumer was removed from the group without a clean rebalance). Unlike `onPartitionsRevoked`, this may be called without a prior `onPartitionsRevoked` for the same partitions. Cleanup should be defensive—in-flight offsets for these partitions cannot be committed.\n\nDefault implementation delegates to `onPartitionsRevoked`.\n\n#### Callback Ordering\n\nFor a complete rebalance cycle:\n1. `onPartitionsRevoked` on the losing consumer(s).\n2. Kafka reassigns partitions.\n3. `onPartitionsAssigned` on the gaining consumer(s).\n\n---\n\n## Polling and Data Fetching\n\n### `poll()`\n\n```java\nConsumerRecords<K, V> poll(Duration timeout)\n\n@Deprecated\nConsumerRecords<K, V> poll(long timeoutMs)\n```\n\nThe central method of the consumer. On each call it:\n1. Sends any pending fetch requests (initiated after the previous `poll()`).\n2. Processes any pending coordinator responses (heartbeats, group join/sync).\n3. Runs `ConsumerRebalanceListener` callbacks if a rebalance is in progress.\n4. Commits offsets if auto-commit is enabled and the interval has elapsed.\n5. Collects fetched records up to `max.poll.records` and returns them.\n\n**Parameters:**\n\n- `timeout` — Maximum time to block waiting for records. `Duration.ZERO` causes an immediate return (with whatever records were already buffered). `Duration.ofMillis(Long.MAX_VALUE)` blocks indefinitely until at least one record is available.\n\n**Return value:** `ConsumerRecords<K, V>` — Never null. May be empty if no records arrived within the timeout, or if the consumer has no subscriptions/assignments.\n\n**Blocking behavior nuances:**\n\n- If `timeout` elapses with no records and no rebalance callbacks, `poll()` returns an empty `ConsumerRecords`.\n- If a rebalance is triggered during `poll()`, the `ConsumerRebalanceListener` callbacks execute **within the same `poll()` call**. If the callbacks are slow, `poll()` may return significantly later than `timeout`. This can cause `max.poll.interval.ms` to be exceeded on the *next* poll.\n- If `wakeup()` was called before or during `poll()`, a `WakeupException` is thrown immediately. Unlike interruption, the consumer remains usable after catching `WakeupException`.\n- Fetch requests for the next batch are sent at the end of the current `poll()` call (pipelining), so there is typically no network round-trip latency penalty in a tight poll loop.\n\n**`max.poll.interval.ms` enforcement:**\n\nIf the time between two consecutive `poll()` calls exceeds `max.poll.interval.ms`, the broker-side coordinator removes the consumer from the group and triggers a rebalance. The consumer itself detects this on the next `poll()` call and throws `InvalidGroupIdException` or re-initiates a group join. This mechanism protects the group from a consumer that is alive at the network level (heartbeats are sent by a background thread) but has stopped processing.\n\n**Exceptions:**\n\n- `WakeupException` — `wakeup()` was called.\n- `InterruptException` — The calling thread was interrupted (wraps `InterruptedException`). The consumer's interrupt flag is cleared.\n- `AuthenticationException` — Authentication with the broker failed.\n- `AuthorizationException` — Caller lacks `READ` permission on the topic.\n- `IllegalStateException` — Consumer is closed.\n- `KafkaException` — Unrecoverable internal error.\n\n---\n\n## Offset Management Methods\n\n### Offset Concepts\n\n| Concept | Description |\n|---|---|\n| **Position** | The offset of the **next record** to be fetched for a partition. Updated after each `poll()`. |\n| **Committed offset** | The offset stored in Kafka (or an external store) representing the last successfully processed record. Should be `position - 1 + 1 = position` at the time of commit (i.e., the next offset to fetch). |\n| **Beginning offset** | The earliest available offset in the partition (changes as log segments are deleted). |\n| **End offset** | The offset one past the last record currently in the partition (the offset of the next produced record). |\n| **LSO (Last Stable Offset)** | With `isolation.level=read_committed`, the consumer will not read past the LSO, which is the offset of the first open (not yet committed or aborted) transaction. |\n\n**Off-by-one rule:** Always commit `offset + 1` (the offset of the record you just processed, plus one). This is what `OffsetAndMetadata(record.offset() + 1)` achieves.\n\n### Automatic Offset Commits\n\nWhen `enable.auto.commit=true` (default), the consumer commits the current position for all assigned partitions every `auto.commit.interval.ms` milliseconds. The commit happens at the start of a `poll()` call if the interval has elapsed.\n\n**Pitfalls of auto-commit:**\n- Records may be re-processed after a crash if they were fetched but not processed before the next auto-commit window.\n- The consumer commits the fetch position, not the last record the application confirmed it processed.\n- With auto-commit, there is no guarantee of at-least-once processing unless `poll()` is called frequently.\n\nSet `enable.auto.commit=false` for exactly-once or at-least-once processing with manual control.\n\n### Synchronous Commit\n\n```java\nvoid commitSync()\nvoid commitSync(Duration timeout)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)\n```\n\nCommits offsets and **blocks** until the broker acknowledges or an error occurs.\n\n**Parameters:**\n\n- No-arg / timeout-only variants — Commits the current position for **all** assigned partitions.\n- `offsets` — A specific map of `TopicPartition` to `OffsetAndMetadata` to commit. Only the specified partitions are committed.\n- `timeout` — Maximum time to wait for acknowledgment. If omitted, uses `default.api.timeout.ms`.\n\n**Behavior:**\n\n- Retries on retriable errors (e.g., coordinator changes) until timeout.\n- Throws `CommitFailedException` on fatal commit failures.\n- Safe to call after processing each record for maximum durability, but high-frequency calls add latency.\n\n**Exceptions:**\n\n- `CommitFailedException` — Commit was rejected because the consumer was fenced out of the group (e.g., due to `max.poll.interval.ms` exceeded). The consumer must rejoin. **Do not retry after this exception.**\n- `TimeoutException` — Broker did not respond within the timeout.\n- `WakeupException` — `wakeup()` interrupted the blocking wait.\n- `InterruptException` — Thread was interrupted.\n- `AuthorizationException` — Missing `WRITE` permission on the `__consumer_offsets` topic.\n- `KafkaException` — Unrecoverable error.\n\n### Asynchronous Commit\n\n```java\nvoid commitAsync()\nvoid commitAsync(OffsetCommitCallback callback)\nvoid commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets, OffsetCommitCallback callback)\n```\n\nSends a commit request and returns immediately. The optional callback is invoked (on the polling thread, during a subsequent `poll()`) when the commit completes or fails.\n\n**Parameters:**\n\n- No-arg variant — Commits current positions, no notification on completion.\n- `callback` — `OffsetCommitCallback.onComplete(Map<TopicPartition, OffsetAndMetadata> offsets, Exception exception)` called on completion. `exception` is `null` on success.\n- `offsets` — Explicit partition-to-offset map.\n\n**Behavior:**\n\n- Does **not** retry on failure. Unlike `commitSync`, if the commit request fails transiently, it is lost.\n- Multiple in-flight async commits can coexist. Callbacks are delivered in the order requests were sent.\n- **Do not retry** in the callback: a later async commit may have already succeeded with a higher offset, and retrying an older commit with a lower offset would regress the committed position.\n- For a clean shutdown: call `commitAsync()` in the loop and `commitSync()` once after the loop exits to ensure the final commit is durable.\n\n**`OffsetCommitCallback` interface:**\n\n```java\npublic interface OffsetCommitCallback {\n    void onComplete(Map<TopicPartition, OffsetAndMetadata> offsets, Exception exception);\n}\n```\n\n### Querying Committed Offsets\n\n```java\nOffsetAndMetadata committed(TopicPartition partition)\nOffsetAndMetadata committed(TopicPartition partition, Duration timeout)\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions)\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions, Duration timeout)\n```\n\nFetches the last committed offset for the given partition(s) **from the broker**. This is a synchronous, blocking network call.\n\n**Return value:** `null` (single-partition variants) or a map entry with `null` value for partitions that have no committed offset. This is distinct from offset 0; it means the partition has never been committed for this group.\n\n**Use cases:**\n- Inspecting lag from outside the polling loop.\n- Initializing positions in `onPartitionsAssigned` without using auto-reset.\n- Verifying that commits succeeded.\n\n**Exceptions:** `TimeoutException`, `AuthorizationException`, `KafkaException`.\n\n---\n\n## Flow Control and Position Queries\n\n### Current Position\n\n```java\nlong position(TopicPartition partition)\nlong position(TopicPartition partition, Duration timeout)\n```\n\nReturns the offset of the **next record** to be fetched for the partition. This reflects how far the consumer has fetched, not how far it has committed.\n\n**Note:** After a rebalance, if no seek is performed, the position is determined by the committed offset (if any) or `auto.offset.reset`. Calling `position()` before any `poll()` triggers a fetch of the committed offset or application of the reset policy.\n\n**Exceptions:** `IllegalStateException` (partition not assigned), `TimeoutException`, `InvalidOffsetException` (on `auto.offset.reset=none` with no committed offset).\n\n### Seek Operations\n\n```java\nvoid seek(TopicPartition partition, long offset)\nvoid seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)\nvoid seekToBeginning(Collection<TopicPartition> partitions)\nvoid seekToEnd(Collection<TopicPartition> partitions)\n```\n\nManually repositions the fetch position for one or more partitions. The next `poll()` will fetch starting from the new position.\n\n**Parameters:**\n\n- `seek(partition, offset)` — Set position to an absolute offset.\n- `seek(partition, offsetAndMetadata)` — Set position and also record the leader epoch for fencing stale leaders.\n- `seekToBeginning(partitions)` — Seek to the earliest available offset. Evaluated **lazily**: the actual offset lookup happens on the next `poll()`, not at call time.\n- `seekToEnd(partitions)` — Seek to the end offset (the offset of the next record to be produced). Also evaluated lazily.\n\n**Validation:**\n\n- The partition must be currently assigned.\n- `seek()` with a negative offset throws `IllegalArgumentException`.\n- Seeking to an offset below the log start offset or above the log end offset does not throw immediately; the consumer will reset to `auto.offset.reset` policy on the next fetch.\n\n**Exceptions:** `IllegalStateException` (partition not assigned), `IllegalArgumentException` (negative offset).\n\n### Offset Discovery\n\n```java\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions)\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions, Duration timeout)\n\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions)\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions, Duration timeout)\n\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch)\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch, Duration timeout)\n```\n\n**`beginningOffsets`:** Returns the earliest available offset for each partition. The result reflects the current log start (after log compaction or retention deletion). Does **not** require the partition to be assigned to this consumer.\n\n**`endOffsets`:** Returns the offset one past the last record (the next-produce offset). Use `endOffset - position` to compute consumer lag. Does **not** require assignment.\n\n**`offsetsForTimes`:** For each `TopicPartition → timestamp` entry, returns the first offset whose timestamp is >= the given timestamp. Returns `null` for a partition if no such record exists (timestamp is beyond the end of the log). Useful for time-based replay.\n\n**Timestamp semantics:** Kafka uses `CREATE_TIME` or `LOG_APPEND_TIME` depending on the topic configuration (`message.timestamp.type`). `offsetsForTimes` searches using the same timestamp type configured for the topic.\n\n**Exceptions:** `TimeoutException`, `AuthorizationException`, `KafkaException`.\n\n### Pause and Resume\n\n```java\nvoid pause(Collection<TopicPartition> partitions)\nvoid resume(Collection<TopicPartition> partitions)\nSet<TopicPartition> paused()\n```\n\n**`pause(partitions)`:** Suspends fetching for the specified partitions. The consumer continues to own the partitions and send heartbeats, but `poll()` will not return any records from them. Already-fetched records buffered internally for paused partitions are also suppressed.\n\n**`resume(partitions)`:** Re-enables fetching for previously paused partitions.\n\n**`paused()`:** Returns the set of currently paused partitions.\n\n**Use cases:**\n- **Back-pressure:** When a downstream system is saturated, pause the consumer to stop buffering more records. Resume when capacity is restored.\n- **Multi-partition coordination:** Pause partitions that are \"ahead\" while waiting for partitions that are \"behind\" to catch up (e.g., when joining streams).\n- **Worker pool handoff:** After handing records to a worker thread, pause the partition to prevent the polling thread from fetching more records for that partition until the worker finishes.\n\n**Behavior notes:**\n- Pausing does not affect offset commits or group membership.\n- Pausing a partition that is not assigned is silently ignored.\n- After a rebalance, paused state is reset: partitions that are reassigned to this consumer start out unpaused.\n\n---\n\n## Metadata and Monitoring\n\n### Topic and Partition Metadata\n\n```java\nList<PartitionInfo> partitionsFor(String topic)\nList<PartitionInfo> partitionsFor(String topic, Duration timeout)\nMap<String, List<PartitionInfo>> listTopics()\nMap<String, List<PartitionInfo>> listTopics(Duration timeout)\n```\n\n**`partitionsFor`:** Returns partition metadata (leader, replicas, ISR) for a topic. Triggers a metadata fetch if the topic is not in the local cache.\n\n**`listTopics`:** Returns partition metadata for all topics visible to this consumer. May return a large result set on clusters with many topics.\n\n**`PartitionInfo` fields:**\n\n| Method | Type | Description |\n|---|---|---|\n| `topic()` | `String` | Topic name |\n| `partition()` | `int` | Partition index |\n| `leader()` | `Node` | Current partition leader |\n| `replicas()` | `Node[]` | All replicas |\n| `inSyncReplicas()` | `Node[]` | In-sync replicas (ISR) |\n| `offlineReplicas()` | `Node[]` | Replicas currently offline |\n\n### Assignment Queries\n\n```java\nSet<TopicPartition> assignment()\nSet<String> subscription()\n```\n\n**`assignment()`:** Returns the set of partitions currently assigned to this consumer (either via `assign()` or after a group rebalance following `subscribe()`). Returns an empty set if the consumer has no assignment.\n\n**`subscription()`:** Returns the set of topics currently subscribed to via `subscribe()`. Empty if manual assignment is in use.\n\n### Group ID\n\n```java\nOptionalLong groupMetadata()   // Returns ConsumerGroupMetadata\nConsumerGroupMetadata groupMetadata()\n```\n\n```java\nConsumerGroupMetadata groupMetadata()\n```\n\nReturns metadata about the consumer's current group membership, including `group.id`, `member.id`, `generation.id`, and optional `group.instance.id`. Used when integrating with Kafka transactions for fencing zombie producers in EOS (exactly-once semantics).\n\n### Wakeup\n\n```java\nvoid wakeup()\n```\n\nThe **only thread-safe method**. Causes a pending or future `poll()` call to throw `WakeupException`. Use this from a shutdown hook or separate control thread to interrupt the polling loop. After catching `WakeupException`, the consumer is still usable (call `close()` for clean shutdown).\n\n---\n\n## Lifecycle and Resource Management\n\n### Closing the Consumer\n\n```java\nvoid close()\nvoid close(Duration timeout)\n```\n\nCloses the consumer, releasing all network connections and local state.\n\n**Behavior:**\n\n- Sends a `LeaveGroup` request to the coordinator (for subscribed consumers), triggering an immediate rebalance so other group members can take over the partitions without waiting for `session.timeout.ms`.\n- Commits auto-commit offsets if `enable.auto.commit=true`.\n- Closes all network connections and frees memory.\n- `close(timeout)` waits up to `timeout` for pending requests (commits, leave-group) to complete. `close()` with no argument uses `default.api.timeout.ms`.\n- After `close()` returns, any further method call throws `IllegalStateException`.\n\n**Best practice:** Always close the consumer in a `finally` block or via `try-with-resources`. Failing to close causes the group to experience a dead-member delay of `session.timeout.ms` before reassigning partitions.\n\n```java\ntry (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {\n    consumer.subscribe(List.of(\"my-topic\"));\n    while (running.get()) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n        // process records\n    }\n} // close() called automatically\n```\n\n---\n\n## Exception Handling Guide\n\n### `WakeupException`\n\n**Package:** `org.apache.kafka.common.errors`\n\nThrown by `poll()` when `wakeup()` has been called. Signals an intentional interruption, typically from a shutdown hook. The consumer remains functional after this exception.\n\n**Recovery:** Catch at the outer loop level, do not re-throw into the processing path. Perform a final `commitSync()` if needed, then `close()`.\n\n### `CommitFailedException`\n\n**Package:** `org.apache.kafka.clients.consumer`\n\nThrown by `commitSync()` when the commit is rejected by the broker. Common causes:\n\n1. **Poll timeout exceeded:** `max.poll.interval.ms` elapsed between two consecutive `poll()` calls. The broker removed the consumer from the group. Offsets from the old generation cannot be committed. **Solution:** Increase `max.poll.interval.ms`, reduce processing time per batch, or reduce `max.poll.records`.\n2. **Concurrent consumer on same partition:** Another consumer (e.g., a zombie) committed to the same partition with a higher generation id. **Solution:** Ensure only one consumer instance per `group.id`.\n3. **Authorization error:** Consumer lacks permission to commit. **Solution:** Grant `WRITE` on `__consumer_offsets`.\n\nAfter `CommitFailedException`, the consumer must re-subscribe and rejoin the group. Do not retry the commit with the same consumer instance.\n\n### `TimeoutException`\n\n**Package:** `org.apache.kafka.common.errors`\n\nThrown when a blocking call (metadata fetch, commit, offset query) does not receive a response within the configured timeout. Retriable—the operation can be attempted again.\n\n### `AuthenticationException`\n\nThrown when the broker rejects the client's credentials. Check `security.protocol`, SASL configuration, and credential validity.\n\n### `AuthorizationException`\n\nThrown when the client is authenticated but lacks permission for the operation. Common scenarios:\n- Missing `READ` on the topic (`poll()`).\n- Missing `READ` on the group resource (`subscribe()`).\n- Missing `WRITE` on `__consumer_offsets` (commit).\n- Missing `DESCRIBE` on the topic (metadata calls).\n\n### `InterruptException`\n\n**Package:** `org.apache.kafka.common.errors`\n\nWraps `java.lang.InterruptedException`. Thrown when the calling thread is interrupted while a blocking call is waiting. The consumer's interrupt flag is cleared. Re-interrupt the thread if appropriate.\n\n### `InvalidOffsetException` and `NoOffsetForPartitionException`\n\nThrown by `position()` or internally during `poll()` when `auto.offset.reset=none` and there is no committed offset for an assigned partition. **Resolution:** Commit an initial offset or change the reset policy.\n\n### `KafkaException`\n\nBase class for most Kafka client exceptions. Wraps internal errors. Check the message and `getCause()` for details.\n\n### `TopicAuthorizationException`\n\nSpecialization of `AuthorizationException` for topic-level authorization failures.\n\n### `GroupAuthorizationException`\n\nSpecialization for group-level authorization failures (e.g., missing permission to join group).\n\n---\n\n## Configuration-Driven Behaviors\n\n### Auto-Reset Policy (`auto.offset.reset`)\n\n| Value | Behavior when no committed offset exists |\n|---|---|\n| `earliest` | Start from the first available record in the partition |\n| `latest` | Start from the end of the partition (skip all existing records) |\n| `none` | Throw `NoOffsetForPartitionException` |\n\nApplies only when the consumer starts without a committed offset (first run for a group, or partition added to topic after group was created). Once a committed offset exists, this setting is ignored.\n\n### Isolation Level (`isolation.level`)\n\n| Value | Behavior |\n|---|---|\n| `read_uncommitted` (default) | Reads all records, including those from open and aborted transactions |\n| `read_committed` | Reads only records from committed transactions and non-transactional records; blocks at LSO |\n\nWith `read_committed`:\n- Records from open transactions are invisible until the transaction commits.\n- Records from aborted transactions are permanently filtered out (control records signal the abort to the consumer).\n- The consumer's position may lag the actual log end offset by the duration of the longest open transaction.\n- `endOffsets()` returns the LSO, not the actual log end offset.\n\n### Static Group Membership (`group.instance.id`)\n\nWhen `group.instance.id` is set, the consumer uses **static membership**:\n- The member ID is stable across restarts (no group rejoin on reconnect within `session.timeout.ms`).\n- Partition assignment is preserved across consumer restarts as long as the instance reconnects within `session.timeout.ms`.\n- Reduces rebalance frequency for deployments with predictable rolling restarts.\n\n### Partition Assignment Strategy\n\n| Strategy | Behavior |\n|---|---|\n| `RangeAssignor` | Assigns contiguous ranges of partitions per topic; may be uneven across consumers |\n| `RoundRobinAssignor` | Distributes partitions evenly in round-robin order |\n| `StickyAssignor` | Minimizes partition movement on rebalance while maintaining balance |\n| `CooperativeStickyAssignor` | Like `StickyAssignor` but uses incremental (cooperative) rebalancing—revokes only partitions that need to move |\n\n`CooperativeStickyAssignor` enables incremental rebalances where only partitions actually being transferred are revoked. All other partitions continue to be consumed without interruption. This is the recommended default for most applications.\n\n---\n\n## Common Patterns and Best Practices\n\n### Pattern 1: Basic Subscription and Polling Loop\n\n```java\nProperties props = new Properties();\nprops.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, \"broker1:9092,broker2:9092\");\nprops.put(ConsumerConfig.GROUP_ID_CONFIG, \"my-group\");\nprops.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());\nprops.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());\nprops.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, \"earliest\");\n// auto-commit is enabled by default\n\ntry (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {\n    consumer.subscribe(List.of(\"orders\", \"payments\"));\n\n    while (!Thread.currentThread().isInterrupted()) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(500));\n        for (ConsumerRecord<String, String> record : records) {\n            System.out.printf(\"topic=%s partition=%d offset=%d key=%s value=%s%n\",\n                record.topic(), record.partition(), record.offset(),\n                record.key(), record.value());\n        }\n    }\n}\n```\n\n### Pattern 2: Manual Offset Commit (At-Least-Once)\n\nDisable auto-commit and commit explicitly after processing. Design callbacks to complete quickly, or implement incremental work with periodic checks.\n\n### `seekToBeginning` / `seekToEnd` Are Lazy\n\nThese methods do not perform network lookups immediately. They record the intent, which is evaluated on the next `poll()`. If you call `seekToBeginning(partitions)` and then `position(partition)` before the next `poll()`, the position query will trigger the evaluation synchronously.\n\n### Committed Offset `null` vs Offset 0\n\n`committed()` returning `null` means the group has **never committed** for that partition. The consumer fetches the committed offset from the broker again (or applies the `auto.offset.reset` policy). Any records fetched since the last commit but not yet committed will be re-fetched.\n\n### Auto-Commit Does Not Guarantee At-Least-Once\n\nWith `enable.auto.commit=true`, if the consumer crashes after auto-committing but before the application has finished processing those records, the records are lost (at-most-once). It is designed to interrupt a `poll()` call that is blocking in another thread.\n- The standard pattern is one consumer per thread, or one thread dedicated solely to calling `poll()` and dispatching work to a thread pool.\n\n### Event Loop Model\n\nThe consumer operates as a cooperative, single-threaded event loop. Executes `ConsumerRebalanceListener` callbacks when partition assignments change.\n4. Commits offsets (if auto-commit is enabled and the interval has elapsed).\n5. Returns batched records to the caller.\n\nAll callbacks and commit processing happen **inside** `poll()`. Nothing happens between calls to `poll()`.\n\n### Consumer Group Mechanics — High-Level\n\nA consumer group is a set of consumer instances sharing a `group.id`. Kafka balances topic-partition ownership across all live members. When a member joins, leaves, or crashes, a **rebalance** redistributes partitions. Rebalances are coordinated by the **group coordinator** broker and are triggered only when at least one consumer calls `poll()`.\n\n---\n\n## Core Types\n\n### `KafkaConsumer<K, V>`\n\nThe main consumer class. The consumer leaves the group (if it was a member) and sends a `LeaveGroup` request to the coordinator. They block `poll()` from returning until they complete.\n\n#### `onPartitionsRevoked(Collection<TopicPartition> partitions)`\n\nCalled **before** partition ownership is transferred away from this consumer, while the consumer is still a group member. This is the correct place to:\n- Commit offsets for the partitions being revoked (so the next owner starts from the right place).\n- Flush any in-progress work for those partitions.\n- Release partition-specific resources.\n\nThe `partitions` collection contains only those being revoked from this consumer. After this method returns, Kafka transfers ownership.\n\n**Eager rebalance (default before Kafka 2.4):** `onPartitionsRevoked` is called with **all** currently assigned partitions even if they will be reassigned back to the same consumer. This causes a stop-the-world pause.\n\n**Cooperative/incremental rebalance (CooperativeStickyAssignor):** `onPartitionsRevoked` is called only with partitions actually being moved away. Partitions retained by this consumer continue to be consumed without interruption.\n\n#### `onPartitionsAssigned(Collection<TopicPartition> partitions)`\n\nCalled **after** partition ownership has been transferred to this consumer. The `partitions` collection contains only newly assigned partitions. At this point, calls to `seek()`, `position()`, or `committed()` are valid for these partitions. Use this callback to:\n- Initialize per-partition state.\n- Seek to a custom starting position (e.g., a checkpoint stored outside Kafka).\n- Warm up partition-specific caches.\n\n#### `onPartitionsLost(Collection<TopicPartition> partitions)`\n\nCalled when partitions are lost involuntarily (e.g., after a session timeout where the consumer was removed from the group without a clean rebalance). Unlike `onPartitionsRevoked`, this may be called without a prior `onPartitionsRevoked` for the same partitions. Cleanup should be defensive—in-flight offsets for these partitions cannot be committed.\n\nDefault implementation delegates to `onPartitionsRevoked`.\n\n#### Callback Ordering\n\nFor a complete rebalance cycle:\n1. `onPartitionsRevoked` on the losing consumer(s).\n2. Kafka reassigns partitions.\n3. `onPartitionsAssigned` on the gaining consumer(s).\n\n---\n\n## Polling and Data Fetching\n\n### `poll()`\n\n```java\nConsumerRecords<K, V> poll(Duration timeout)\n\n@Deprecated\nConsumerRecords<K, V> poll(long timeoutMs)\n```\n\nThe central method of the consumer. On each call it:\n1. Sends any pending fetch requests (initiated after the previous `poll()`).\n2. Processes any pending coordinator responses (heartbeats, group join/sync).\n3. Runs `ConsumerRebalanceListener` callbacks if a rebalance is in progress.\n4. Commits offsets if auto-commit is enabled and the interval has elapsed.\n5. Collects fetched records up to `max.poll.records` and returns them.\n\n**Parameters:**\n\n- `timeout` — Maximum time to block waiting for records. `Duration.ZERO` causes an immediate return (with whatever records were already buffered). `Duration.ofMillis(Long.MAX_VALUE)` blocks indefinitely until at least one record is available.\n\n**Return value:** `ConsumerRecords<K, V>` — Never null. May be empty if no records arrived within the timeout, or if the consumer has no subscriptions/assignments.\n\n**Blocking behavior nuances:**\n\n- If `timeout` elapses with no records and no rebalance callbacks, `poll()` returns an empty `ConsumerRecords`.\n- If a rebalance is triggered during `poll()`, the `ConsumerRebalanceListener` callbacks execute **within the same `poll()` call**. If the callbacks are slow, `poll()` may return significantly later than `timeout`. This can cause `max.poll.interval.ms` to be exceeded on the *next* poll.\n- If `wakeup()` was called before or during `poll()`, a `WakeupException` is thrown immediately. Unlike interruption, the consumer remains usable after catching `WakeupException`.\n- Fetch requests for the next batch are sent at the end of the current `poll()` call (pipelining), so there is typically no network round-trip latency penalty in a tight poll loop.\n\n**`max.poll.interval.ms` enforcement:**\n\nIf the time between two consecutive `poll()` calls exceeds `max.poll.interval.ms`, the broker-side coordinator removes the consumer from the group and triggers a rebalance. The consumer itself detects this on the next `poll()` call and throws `InvalidGroupIdException` or re-initiates a group join. This mechanism protects the group from a consumer that is alive at the network level (heartbeats are sent by a background thread) but has stopped processing.\n\n**Exceptions:**\n\n- `WakeupException` — `wakeup()` was called.\n- `InterruptException` — The calling thread was interrupted (wraps `InterruptedException`). The consumer's interrupt flag is cleared.\n- `AuthenticationException` — Authentication with the broker failed.\n- `AuthorizationException` — Caller lacks `READ` permission on the topic.\n- `IllegalStateException` — Consumer is closed.\n- `KafkaException` — Unrecoverable internal error.\n\n---\n\n## Offset Management Methods\n\n### Offset Concepts\n\n| Concept | Description |\n|---|---|\n| **Position** | The offset of the **next record** to be fetched for a partition. Updated after each `poll()`. |\n| **Committed offset** | The offset stored in Kafka (or an external store) representing the last successfully processed record. Should be `position - 1 + 1 = position` at the time of commit (i.e., the next offset to fetch). |\n| **Beginning offset** | The earliest available offset in the partition (changes as log segments are deleted). |\n| **End offset** | The offset one past the last record currently in the partition (the offset of the next produced record). |\n| **LSO (Last Stable Offset)** | With `isolation.level=read_committed`, the consumer will not read past the LSO, which is the offset of the first open (not yet committed or aborted) transaction. |\n\n**Off-by-one rule:** Always commit `offset + 1` (the offset of the record you just processed, plus one). This is what `OffsetAndMetadata(record.offset() + 1)` achieves.\n\n### Automatic Offset Commits\n\nWhen `enable.auto.commit=true` (default), the consumer commits the current position for all assigned partitions every `auto.commit.interval.ms` milliseconds. The commit happens at the start of a `poll()` call if the interval has elapsed.\n\n**Pitfalls of auto-commit:**\n- Records may be re-processed after a crash if they were fetched but not processed before the next auto-commit window.\n- The consumer commits the fetch position, not the last record the application confirmed it processed.\n- With auto-commit, there is no guarantee of at-least-once processing unless `poll()` is called frequently.\n\nSet `enable.auto.commit=false` for exactly-once or at-least-once processing with manual control.\n\n### Synchronous Commit\n\n```java\nvoid commitSync()\nvoid commitSync(Duration timeout)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)\n```\n\nCommits offsets and **blocks** until the broker acknowledges or an error occurs.\n\n**Parameters:**\n\n- No-arg / timeout-only variants — Commits the current position for **all** assigned partitions.\n- `offsets` — A specific map of `TopicPartition` to `OffsetAndMetadata` to commit. Only the specified partitions are committed.\n- `timeout` — Maximum time to wait for acknowledgment. If omitted, uses `default.api.timeout.ms`.\n\n**Behavior:**\n\n- Retries on retriable errors (e.g., coordinator changes) until timeout.\n- Throws `CommitFailedException` on fatal commit failures.\n- Safe to call after processing each record for maximum durability, but high-frequency calls add latency.\n\n**Exceptions:**\n\n- `CommitFailedException` — Commit was rejected because the consumer was fenced out of the group (e.g., due to `max.poll.interval.ms` exceeded). The consumer must rejoin. **Do not retry after this exception.**\n- `TimeoutException` — Broker did not respond within the timeout.\n- `WakeupException` — `wakeup()` interrupted the blocking wait.\n- `InterruptException` — Thread was interrupted.\n- `AuthorizationException` — Missing `WRITE` permission on the `__consumer_offsets` topic.\n- `KafkaException` — Unrecoverable error.\n\n### Asynchronous Commit\n\n```java\nvoid commitAsync()\nvoid commitAsync(OffsetCommitCallback callback)\nvoid commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets, OffsetCommitCallback callback)\n```\n\nSends a commit request and returns immediately. The optional callback is invoked (on the polling thread, during a subsequent `poll()`) when the commit completes or fails.\n\n**Parameters:**\n\n- No-arg variant — Commits current positions, no notification on completion.\n- `callback` — `OffsetCommitCallback.onComplete(Map<TopicPartition, OffsetAndMetadata> offsets, Exception exception)` called on completion. `exception` is `null` on success.\n- `offsets` — Explicit partition-to-offset map.\n\n**Behavior:**\n\n- Does **not** retry on failure. Unlike `commitSync`, if the commit request fails transiently, it is lost.\n- Multiple in-flight async commits can coexist. Callbacks are delivered in the order requests were sent.\n- **Do not retry** in the callback: a later async commit may have already succeeded with a higher offset, and retrying an older commit with a lower offset would regress the committed position.\n- For a clean shutdown: call `commitAsync()` in the loop and `commitSync()` once after the loop exits to ensure the final commit is durable.\n\n**`OffsetCommitCallback` interface:**\n\n```java\npublic interface OffsetCommitCallback {\n    void onComplete(Map<TopicPartition, OffsetAndMetadata> offsets, Exception exception);\n}\n```\n\n### Querying Committed Offsets\n\n```java\nOffsetAndMetadata committed(TopicPartition partition)\nOffsetAndMetadata committed(TopicPartition partition, Duration timeout)\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions)\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions, Duration timeout)\n```\n\nFetches the last committed offset for the given partition(s) **from the broker**. This is a synchronous, blocking network call.\n\n**Return value:** `null` (single-partition variants) or a map entry with `null` value for partitions that have no committed offset. This is distinct from offset 0; it means the partition has never been committed for this group.\n\n**Use cases:**\n- Inspecting lag from outside the polling loop.\n- Initializing positions in `onPartitionsAssigned` without using auto-reset.\n- Verifying that commits succeeded.\n\n**Exceptions:** `TimeoutException`, `AuthorizationException`, `KafkaException`.\n\n---\n\n## Flow Control and Position Queries\n\n### Current Position\n\n```java\nlong position(TopicPartition partition)\nlong position(TopicPartition partition, Duration timeout)\n```\n\nReturns the offset of the **next record** to be fetched for the partition. This reflects how far the consumer has fetched, not how far it has committed.\n\n**Note:** After a rebalance, if no seek is performed, the position is determined by the committed offset (if any) or `auto.offset.reset`. Calling `position()` before any `poll()` triggers a fetch of the committed offset or application of the reset policy.\n\n**Exceptions:** `IllegalStateException` (partition not assigned), `TimeoutException`, `InvalidOffsetException` (on `auto.offset.reset=none` with no committed offset).\n\n### Seek Operations\n\n```java\nvoid seek(TopicPartition partition, long offset)\nvoid seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)\nvoid seekToBeginning(Collection<TopicPartition> partitions)\nvoid seekToEnd(Collection<TopicPartition> partitions)\n```\n\nManually repositions the fetch position for one or more partitions. The next `poll()` will fetch starting from the new position.\n\n**Parameters:**\n\n- `seek(partition, offset)` — Set position to an absolute offset.\n- `seek(partition, offsetAndMetadata)` — Set position and also record the leader epoch for fencing stale leaders.\n- `seekToBeginning(partitions)` — Seek to the earliest available offset. Evaluated **lazily**: the actual offset lookup happens on the next `poll()`, not at call time.\n- `seekToEnd(partitions)` — Seek to the end offset (the offset of the next record to be produced). Also evaluated lazily.\n\n**Validation:**\n\n- The partition must be currently assigned.\n- `seek()` with a negative offset throws `IllegalArgumentException`.\n- Seeking to an offset below the log start offset or above the log end offset does not throw immediately; the consumer will reset to `auto.offset.reset` policy on the next fetch.\n\n**Exceptions:** `IllegalStateException` (partition not assigned), `IllegalArgumentException` (negative offset).\n\n### Offset Discovery\n\n```java\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions)\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions, Duration timeout)\n\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions)\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions, Duration timeout)\n\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch)\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch, Duration timeout)\n```\n\n**`beginningOffsets`:** Returns the earliest available offset for each partition. The result reflects the current log start (after log compaction or retention deletion). Does **not** require the partition to be assigned to this consumer.\n\n**`endOffsets`:** Returns the offset one past the last record (the next-produce offset). Use `endOffset - position` to compute consumer lag. Does **not** require assignment.\n\n**`offsetsForTimes`:** For each `TopicPartition → timestamp` entry, returns the first offset whose timestamp is >= the given timestamp. Returns `null` for a partition if no such record exists (timestamp is beyond the end of the log). Useful for time-based replay.\n\n**Timestamp semantics:** Kafka uses `CREATE_TIME` or `LOG_APPEND_TIME` depending on the topic configuration (`message.timestamp.type`). `offsetsForTimes` searches using the same timestamp type configured for the topic.\n\n**Exceptions:** `TimeoutException`, `AuthorizationException`, `KafkaException`.\n\n### Pause and Resume\n\n```java\nvoid pause(Collection<TopicPartition> partitions)\nvoid resume(Collection<TopicPartition> partitions)\nSet<TopicPartition> paused()\n```\n\n**`pause(partitions)`:** Suspends fetching for the specified partitions. The consumer continues to own the partitions and send heartbeats, but `poll()` will not return any records from them. Already-fetched records buffered internally for paused partitions are also suppressed.\n\n**`resume(partitions)`:** Re-enables fetching for previously paused partitions.\n\n**`paused()`:** Returns the set of currently paused partitions.\n\n**Use cases:**\n- **Back-pressure:** When a downstream system is saturated, pause the consumer to stop buffering more records. Resume when capacity is restored.\n- **Multi-partition coordination:** Pause partitions that are \"ahead\" while waiting for partitions that are \"behind\" to catch up (e.g., when joining streams).\n- **Worker pool handoff:** After handing records to a worker thread, pause the partition to prevent the polling thread from fetching more records for that partition until the worker finishes.\n\n**Behavior notes:**\n- Pausing does not affect offset commits or group membership.\n- Pausing a partition that is not assigned is silently ignored.\n- After a rebalance, paused state is reset: partitions that are reassigned to this consumer start out unpaused.\n\n---\n\n## Metadata and Monitoring\n\n### Topic and Partition Metadata\n\n```java\nList<PartitionInfo> partitionsFor(String topic)\nList<PartitionInfo> partitionsFor(String topic, Duration timeout)\nMap<String, List<PartitionInfo>> listTopics()\nMap<String, List<PartitionInfo>> listTopics(Duration timeout)\n```\n\n**`partitionsFor`:** Returns partition metadata (leader, replicas, ISR) for a topic. Triggers a metadata fetch if the topic is not in the local cache.\n\n**`listTopics`:** Returns partition metadata for all topics visible to this consumer. May return a large result set on clusters with many topics.\n\n**`PartitionInfo` fields:**\n\n| Method | Type | Description |\n|---|---|---|\n| `topic()` | `String` | Topic name |\n| `partition()` | `int` | Partition index |\n| `leader()` | `Node` | Current partition leader |\n| `replicas()` | `Node[]` | All replicas |\n| `inSyncReplicas()` | `Node[]` | In-sync replicas (ISR) |\n| `offlineReplicas()` | `Node[]` | Replicas currently offline |\n\n### Assignment Queries\n\n```java\nSet<TopicPartition> assignment()\nSet<String> subscription()\n```\n\n**`assignment()`:** Returns the set of partitions currently assigned to this consumer (either via `assign()` or after a group rebalance following `subscribe()`). Returns an empty set if the consumer has no assignment.\n\n**`subscription()`:** Returns the set of topics currently subscribed to via `subscribe()`. Empty if manual assignment is in use.\n\n### Group ID\n\n```java\nOptionalLong groupMetadata()   // Returns ConsumerGroupMetadata\nConsumerGroupMetadata groupMetadata()\n```\n\n```java\nConsumerGroupMetadata groupMetadata()\n```\n\nReturns metadata about the consumer's current group membership, including `group.id`, `member.id`, `generation.id`, and optional `group.instance.id`. Used when integrating with Kafka transactions for fencing zombie producers in EOS (exactly-once semantics).\n\n### Wakeup\n\n```java\nvoid wakeup()\n```\n\nThe **only thread-safe method**. Causes a pending or future `poll()` call to throw `WakeupException`. Use this from a shutdown hook or separate control thread to interrupt the polling loop. After catching `WakeupException`, the consumer is still usable (call `close()` for clean shutdown).\n\n---\n\n## Lifecycle and Resource Management\n\n### Closing the Consumer\n\n```java\nvoid close()\nvoid close(Duration timeout)\n```\n\nCloses the consumer, releasing all network connections and local state.\n\n**Behavior:**\n\n- Sends a `LeaveGroup` request to the coordinator (for subscribed consumers), triggering an immediate rebalance so other group members can take over the partitions without waiting for `session.timeout.ms`.\n- Commits auto-commit offsets if `enable.auto.commit=true`.\n- Closes all network connections and frees memory.\n- `close(timeout)` waits up to `timeout` for pending requests (commits, leave-group) to complete. `close()` with no argument uses `default.api.timeout.ms`.\n- After `close()` returns, any further method call throws `IllegalStateException`.\n\n**Best practice:** Always close the consumer in a `finally` block or via `try-with-resources`. Failing to close causes the group to experience a dead-member delay of `session.timeout.ms` before reassigning partitions.\n\n```java\ntry (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {\n    consumer.subscribe(List.of(\"my-topic\"));\n    while (running.get()) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n        // process records\n    }\n} // close() called automatically\n```\n\n---\n\n## Exception Handling Guide\n\n### `WakeupException`\n\n**Package:** `org.apache.kafka.common.errors`\n\nThrown by `poll()` when `wakeup()` has been called. Signals an intentional interruption, typically from a shutdown hook. The consumer remains functional after this exception.\n\n**Recovery:** Catch at the outer loop level, do not re-throw into the processing path. Perform a final `commitSync()` if needed, then `close()`.\n\n### `CommitFailedException`\n\n**Package:** `org.apache.kafka.clients.consumer`\n\nThrown by `commitSync()` when the commit is rejected by the broker. Common causes:\n\n1. **Poll timeout exceeded:** `max.poll.interval.ms` elapsed between two consecutive `poll()` calls. The broker removed the consumer from the group. Offsets from the old generation cannot be committed. **Solution:** Increase `max.poll.interval.ms`, reduce processing time per batch, or reduce `max.poll.records`.\n2. **Concurrent consumer on same partition:** Another consumer (e.g., a zombie) committed to the same partition with a higher generation id. **Solution:** Ensure only one consumer instance per `group.id`.\n3. **Authorization error:** Consumer lacks permission to commit. **Solution:** Grant `WRITE` on `__consumer_offsets`.\n\nAfter `CommitFailedException`, the consumer must re-subscribe and rejoin the group. Do not retry the commit with the same consumer instance.\n\n### `TimeoutException`\n\n**Package:** `org.apache.kafka.common.errors`\n\nThrown when a blocking call (metadata fetch, commit, offset query) does not receive a response within the configured timeout. Retriable—the operation can be attempted again.\n\n### `AuthenticationException`\n\nThrown when the broker rejects the client's credentials. Check `security.protocol`, SASL configuration, and credential validity.\n\n### `AuthorizationException`\n\nThrown when the client is authenticated but lacks permission for the operation. Common scenarios:\n- Missing `READ` on the topic (`poll()`).\n- Missing `READ` on the group resource (`subscribe()`).\n- Missing `WRITE` on `__consumer_offsets` (commit).\n- Missing `DESCRIBE` on the topic (metadata calls).\n\n### `InterruptException`\n\n**Package:** `org.apache.kafka.common.errors`\n\nWraps `java.lang.InterruptedException`. Thrown when the calling thread is interrupted while a blocking call is waiting. The consumer's interrupt flag is cleared. Re-interrupt the thread if appropriate.\n\n### `InvalidOffsetException` and `NoOffsetForPartitionException`\n\nThrown by `position()` or internally during `poll()` when `auto.offset.reset=none` and there is no committed offset for an assigned partition. **Resolution:** Commit an initial offset or change the reset policy.\n\n### `KafkaException`\n\nBase class for most Kafka client exceptions. Wraps internal errors. Check the message and `getCause()` for details.\n\n### `TopicAuthorizationException`\n\nSpecialization of `AuthorizationException` for topic-level authorization failures.\n\n### `GroupAuthorizationException`\n\nSpecialization for group-level authorization failures (e.g., missing permission to join group).\n\n---\n\n## Configuration-Driven Behaviors\n\n### Auto-Reset Policy (`auto.offset.reset`)\n\n| Value | Behavior when no committed offset exists |\n|---|---|\n| `earliest` | Start from the first available record in the partition |\n| `latest` | Start from the end of the partition (skip all existing records) |\n| `none` | Throw `NoOffsetForPartitionException` |\n\nApplies only when the consumer starts without a committed offset (first run for a group, or partition added to topic after group was created). Once a committed offset exists, this setting is ignored.\n\n### Isolation Level (`isolation.level`)\n\n| Value | Behavior |\n|---|---|\n| `read_uncommitted` (default) | Reads all records, including those from open and aborted transactions |\n| `read_committed` | Reads only records from committed transactions and non-transactional records; blocks at LSO |\n\nWith `read_committed`:\n- Records from open transactions are invisible until the transaction commits.\n- Records from aborted transactions are permanently filtered out (control records signal the abort to the consumer).\n- The consumer's position may lag the actual log end offset by the duration of the longest open transaction.\n- `endOffsets()` returns the LSO, not the actual log end offset.\n\n### Static Group Membership (`group.instance.id`)\n\nWhen `group.instance.id` is set, the consumer uses **static membership**:\n- The member ID is stable across restarts (no group rejoin on reconnect within `session.timeout.ms`).\n- Partition assignment is preserved across consumer restarts as long as the instance reconnects within `session.timeout.ms`.\n- Reduces rebalance frequency for deployments with predictable rolling restarts.\n\n### Partition Assignment Strategy\n\n| Strategy | Behavior |\n|---|---|\n| `RangeAssignor` | Assigns contiguous ranges of partitions per topic; may be uneven across consumers |\n| `RoundRobinAssignor` | Distributes partitions evenly in round-robin order |\n| `StickyAssignor` | Minimizes partition movement on rebalance while maintaining balance |\n| `CooperativeStickyAssignor` | Like `StickyAssignor` but uses incremental (cooperative) rebalancing—revokes only partitions that need to move |\n\n`CooperativeStickyAssignor` enables incremental rebalances where only partitions actually being transferred are revoked. All other partitions continue to be consumed without interruption. This is the recommended default for most applications.\n\n---\n\n## Common Patterns and Best Practices\n\n### Pattern 1: Basic Subscription and Polling Loop\n\n```java\nProperties props = new Properties();\nprops.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, \"broker1:9092,broker2:9092\");\nprops.put(ConsumerConfig.GROUP_ID_CONFIG, \"my-group\");\nprops.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());\nprops.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());\nprops.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, \"earliest\");\n// auto-commit is enabled by default\n\ntry (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {\n    consumer.subscribe(List.of(\"orders\", \"payments\"));\n\n    while (!Thread.currentThread().isInterrupted()) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(500));\n        for (ConsumerRecord<String, String> record : records) {\n            System.out.printf(\"topic=%s partition=%d offset=%d key=%s value=%s%n\",\n                record.topic(), record.partition(), record.offset(),\n                record.key(), record.value());\n        }\n    }\n}\n```\n\n### Pattern 2: Manual Offset Commit (At-Least-Once)\n\nDisable auto-commit and commit explicitly after processing. Design callbacks to complete quickly, or implement incremental work with periodic checks.\n\n### `seekToBeginning` / `seekToEnd` Are Lazy\n\nThese methods do not perform network lookups immediately. The consumer fetches the committed offset from the broker again (or applies the `auto.offset.reset` policy). Any records fetched since the last commit but not yet committed will be re-fetched.\n\n### Auto-Commit Does Not Guarantee At-Least-Once\n\nWith `enable.auto.commit=true`, if the consumer crashes after auto-committing but before the application has finished processing those records, the records are lost (at-most-once).
