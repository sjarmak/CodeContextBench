REFACTORING RATIONALE
================================================================================

MOTIVATION:
The RecordAccumulator class manages batches of records (ProducerBatch objects),
not individual records. Its internal data structures and methods all operate
at the batch level:

  - Data Structure: ConcurrentMap<TopicPartition, Deque<ProducerBatch>>
  - Key Methods: ready(), drain(), append() all work with batches
  - Responsibility: Batch accumulation and flushing

BENEFIT:
Renaming to BatchAccumulator makes the class's true responsibility clear:
  - Improves code readability
  - Reduces developer confusion
  - Better aligns naming with functionality
  - Maintains backward compatibility at implementation level
