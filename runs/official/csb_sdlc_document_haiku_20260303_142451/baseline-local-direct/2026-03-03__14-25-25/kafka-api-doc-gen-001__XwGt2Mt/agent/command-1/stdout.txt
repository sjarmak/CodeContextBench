(Group-Managed)\n\nUse when you want Kafka to coordinate partition assignment across consumer group members:\n\n```java\n// Subscribe to topics - Kafka assigns partitions automatically\nconsumer.subscribe(Arrays.asList(\"topic1\", \"topic2\"));\n\n// Subscribe with rebalance listener\nconsumer.subscribe(\n    Arrays.asList(\"topic1\", \"topic2\"),\n    new ConsumerRebalanceListener() {\n        @Override\n        public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n            consumer.commitSync();  // Commit before losing ownership\n        }\n\n        @Override\n        public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n            System.out.println(\"Assigned: \" + partitions);\n            // Optionally seek to custom positions\n        }\n    }\n);\n\n// Subscribe to pattern (dynamic topic discovery)\nconsumer.subscribe(Pattern.compile(\"logs-.*\"));  // Subscribes to logs-foo, logs-bar, etc.\n```\n\n**Subscription Semantics:**\n- First `subscribe()` call in constructor triggers rebalance on next poll()\n- Changes to subscriptions trigger rebalances\n- Multiple `subscribe()` calls replace previous subscription\n- Rebalances only occur during `poll()` calls\n- Pattern subscriptions automatically discover new matching topics\n\n### Manual Assignment (Standalone Consumer)\n\nUse when you need explicit partition control or running a single-consumer application:\n\n```java\n// Manually assign specific partitions\nTopicPartition partition0 = new assigned partitions\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n```\n\n**Manual Assignment Semantics:**\n- No group coordination happens\n- Partitions // Returns Set<TopicPartition> if assigned, else throws\n```\n\n---\n\n## Polling and Data Fetching\n\n### Poll Semantics and Blocking Behavior\n\n```java\n// Poll with timeout\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(1000));\n```\n\n**What happens during `poll(timeout)`:**\n\n1. **Rebalancing (if needed):**\n   - If subscription changed or group needs rebalancing, triggers rebalance\n   - Calls `onPartitionsRevoked()` callback\n   - Calls `onPartitionsAssigned()` callback\n   - These callbacks can take significant time\n   - **poll() may block longer than timeout during rebalance**\n\n2. **Heartbeat (if group-managed):**\n   - Sends heartbeat if needed (managed internally)\n   - Receives group assignment updates\n\n3. **Fetching:**\n   - Sends fetch request to broker for assigned partitions\n   - Waits up to `timeout` for broker response\n   - Returns immediately if records available (up to `max.poll.records`)\n   - Returns empty if timeout expires with no records\n\n4. **Auto-commit (if enabled):**\n   - On some poll() calls (every `auto.commit.interval.ms`), commits offsets\n   - Happens asynchronously in background\n\n**Blocking Behavior:**\n\n```java\n// Scenario 1: Rebalance happening\nconsumer.poll(Duration.ofMillis(100));\n// May block for seconds if rebalance callbacks are slow!\n// Actual block time = rebalance callback duration + fetch timeout\n\n// Scenario 2: No records available\nconsumer.poll(Duration.ofMillis(100));\n// Blocks for full 100ms if no data\n\n// Scenario 3: Records immediately available\nconsumer.poll(Duration.ofMillis(100));\n// Returns immediately (may take <1ms)\n\n// Scenario 4: Rebalance + callback crashes\nconsumer.poll(Duration.ofMillis(100));\n// Throws CommitFailedException or other exception from callback\n```\n\n### Record Ordering Guarantees\n\n```java\n// Records are ordered per partition\nList<ConsumerRecord<String, String>> records = consumer.poll(...).records(partition);\n// records.get(0).offset() <= records.get(1).offset() <= ...\n\n// Records from different partitions have no ordering guarantee\nConsumerRecords<String, String> all = consumer.poll(...);\n// Partitions may be in any order: partition 2, then 0, then 1, etc.\n```\n\n### Poll Timeout Best Practices\n\n```java\n// Timeout should be set based on:\n// 1. Processing latency: poll should happen before max.poll.interval.ms expires\n// 2. Latency requirement: how quickly you want to process records\n// 3. Resource usage: longer timeout = less CPU spinning\n\n// Typical values:\nconsumer.poll(Duration.ofMillis(100));    // Low latency, frequent polling\nconsumer.poll(Duration.ofMillis(500));    // Balanced\nconsumer.poll(Duration.ofSeconds(1));     // High throughput, fewer CPU cycles\n\n// WRONG: Polling with very long timeout\nconsumer.poll(Duration.ofMinutes(1));  // May cause max.poll.interval.ms violation\n```\n\n### Empty Poll Handling\n\n```java\nwhile (running) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n    if (records.isEmpty()) {\n        System.out.println(\"No records, probably waiting for data\");\n        continue;\n    }\n\n    for (ConsumerRecord<String, String> record : records) {\n        process(record);\n    }\n}\n```\n\n---\n\n## Offset Management\n\n### Offset Concepts\n\n**Critical distinction between offset concepts:**\n\n| Concept | Definition | Usage |\n|---------|-----------|-------|\n| **Position** | Current offset the consumer will fetch next | Internal state, incremented on poll() |\n| **Committed Offset** | Last offset consumer agreed to process | Recovered on rebalance/restart |\n| **Beginning Offset** | Earliest offset available in partition | Seek target, retention boundary |\n| **End Offset** | Latest offset + 1 (high water mark) | Seek target, end-of-partition marker |\n| **Log Start Offset (LSO)** | Earliest offset after log compaction/retention | May differ from beginning if deleted |\n| **LSO (read_committed)** | Last stable offset for transactional reads | Messages after LSO are in-flight |\n\n**The Off-By-One Relationship:**\n\n```\nAssuming we read offset 41 and want to resume from there later:\n\nrecord.offset() = 41\n// To resume, we set:\nconsumer.seek(partition, 42)  // NOT 41! Next offset to read is 42\n// OR in commitSync:\ncommitSync(Map.of(\n    partition,\n    new OffsetAndMetadata(42)  // Committed offset is 42, next to read is 42\n))\n\n// The position is \"next offset to fetch\"\n// The committed offset is \"next offset to fetch after restart\"\n```\n\n### Automatic Offset Commit\n\n```java\n// Configuration\nprops.setProperty(\"enable.auto.commit\", \"true\");\nprops.setProperty(\"auto.commit.interval.ms\", \"5000\");\n\n// Behavior:\n// - Every ~5000ms during poll(), commits current position\n// - Commits happen asynchronously\n// - Last offset per partition is committed\n// - On exception/crash, may lose up to 5000ms of progress\n// - On rebalance, consumer group shares state (committed offsets)\n```\n\n**Risks with auto-commit:**\n\n```java\nwhile (running) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n    for (ConsumerRecord<String, String> record : records) {\n        // If we crash here, auto-commit may have already committed offset\n        // But we haven't actually processed the message!\n        process(record);  // <- CRASH HERE\n        // Message lost - already committed but not processed\n    }\n}\n```\n\n### Manual Offset Commit (Sync)\n\n```java\n// Disable auto-commit\nprops.setProperty(\"enable.auto.commit\", \"false\");\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\nconsumer.subscribe(Arrays.asList(\"topic\"));\n\nint processed = 0;\nwhile (running) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n    for (ConsumerRecord<String, String> record : records) {\n        process(record);\n        processed++;\n    }\n\n    // Commit after processing\n    if (records.count() > 0) {\n        try {\n            consumer.commitSync();  // Blocks until offset committed\n            System.out.println(\"Committed offsets\");\n        } catch (CommitFailedException e) {\n            // Lost group membership (rebalance, session timeout, fencing)\n            System.err.println(\"Commit failed, consumer fenced: \" + e);\n            break;  // Exit processing loop\n        }\n    }\n}\n```\n\n**Sync Commit Semantics:**\n\n```java\n// Commits current position (next offset to fetch)\nconsumer.commitSync();\n// Equivalent to:\nMap<TopicPartition, OffsetAndMetadata> offsets = consumer.committed(\n    consumer.assignment()\n);\n// (Actually commits the current position of each partition)\n\n// Commits specific offsets\nMap<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();\noffsets.put(tp, new OffsetAndMetadata(42));\nconsumer.commitSync(offsets);\n\n// Exceptions:\n// - CommitFailedException: Lost group membership (fenced by another member)\n// - IllegalStateException: Consumer not subscribed/assigned\n// - AuthenticationException: Credentials rejected\n```\n\n### Manual Offset Commit (Async)\n\n```java\n// Async commit with callback\nconsumer.commitAsync((offsets, exception) -> {\n    if (exception != null) {\n        System.err.println(\"Async commit failed: \" + exception);\n        // Exception types:\n        // - CommitFailedException: Lost membership\n        // - AuthenticationException: Auth failed\n        // - Retriable: Timeout, broker unavailable (will retry automatically)\n    } else {\n        System.out.println(\"Offsets committed: \" + offsets);\n    }\n});\n\n// Continue processing immediately without waiting\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n// Async commit without callback\nconsumer.commitAsync();  // Fire and forget\n```\n\n**Async Commit Semantics:**\n\n```java\n// Multiple pending commits are allowed\nfor (int i = 0; i < 3; i++) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n    for (ConsumerRecord<String, String> record : records) {\n        process(record);\n    }\n    consumer.commitAsync();  // Queue commit asynchronously\n}\n\n// On close(), pending commits are retried synchronously\n// This ensures graceful shutdown commits all processed offsets\nconsumer.close();\n\n// During normal operation, commits are retried in background\n// But NOT ordered - later commit may succeed before earlier one\n// Result: Earlier commit may be lost if later one succeeds\n// Risk: Offset may go backward temporarily during retries\n```\n\n### Hybrid Commit Strategy (Recommended for Exactly-Once)\n\n```java\nprops.setProperty(\"enable.auto.commit\", \"false\");\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\nconsumer.subscribe(Arrays.asList(\"topic\"));\n\ntry {\n    while (running) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n        for (ConsumerRecord<String, String> record : records) {\n            processRecord(record);\n        }\n\n        // Async commit during normal processing (fast path)\n        consumer.commitAsync();\n    }\n} finally {\n    try {\n        // Sync commit on shutdown (ensures all processed offsets committed)\n        consumer.commitSync();\n    } finally {\n        consumer.close();\n    }\n}\n```\n\n### Offset Seek and Reset Operations\n\n```java\n// Seek to specific offset\nTopicPartition tp = new TopicPartition(\"topic\", 0);\nconsumer.seek(tp, 100);  // Next poll returns offset 100 and beyond\n\n// Seek to beginning (earliest available)\nconsumer.seekToBeginning(Arrays.asList(tp));\n\n// Seek to end (latest, no records returned until new data arrives)\nconsumer.seekToEnd(Arrays.asList(tp));\n\n// Seek by timestamp\nMap<TopicPartition, Long> timestampMap = new HashMap<>();\ntimestampMap.put(tp, System.currentTimeMillis() - 3600000);  // 1 hour ago\nMap<TopicPartition, OffsetAndTimestamp> offsets = consumer.offsetsForTimes(timestampMap);\nif (offsets.get(tp) != null) {\n    consumer.seek(tp, offsets.get(tp).offset());\n}\n```\n\n### Committed Offset Queries\n\n```java\n// Get last committed offset for partition\nTopicPartition tp = new TopicPartition(\"topic\", 0);\nOffsetAndMetadata committed = consumer.committed(tp);\nif (committed != null) {\n    System.out.println(\"Committed offset: \" + committed.offset());\n    System.out.println(\"Metadata: \" + committed.metadata());\n} else {\n    System.out.println(\"No committed offset (first consumption)\");\n}\n\n// Get committed offsets for all assigned partitions\nMap<TopicPartition, OffsetAndMetadata> allCommitted = consumer.committed(\n    consumer.assignment()\n);\n```\n\n### Position Tracking\n\n```java\n// Get current position (next offset that will be fetched)\nTopicPartition tp = new TopicPartition(\"topic\", 0);\nlong position = consumer.position(tp);  // e.g., 42\n// Next poll will fetch from offset 42\n\n// After reading a record with offset 41:\n// position becomes 42\n\n// Seek changes position\nconsumer.seek(tp, 100);\nSystem.out.println(consumer.position(tp));  // 100\n\n// Position increments on poll\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n// For each record fetched, position incremented by 1\n```\n\n---\n\n## Flow Control and Position Management\n\n### Pause and Resume\n\n```java\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\nconsumer.subscribe(Arrays.asList(\"topic\"));\n\nwhile (running) {\n    // Check if worker pool has capacity\n    if (workerPool.getQueue().size() > QUEUE_THRESHOLD) {\n        // Pause this partition - poll() will not return records from it\n        consumer.pause(Arrays.asList(new TopicPartition(\"topic\", 0)));\n        System.out.println(\"Paused partition 0\");\n    }\n\n    // If resumed, start fetching again\n    if (workerPool.getQueue().size() < RESUME_THRESHOLD) {\n        consumer.resume(Arrays.asList(new TopicPartition(\"topic\", 0)));\n        System.out.println(\"Resumed partition 0\");\n    }\n\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n    // Paused partitions return no records\n\n    for (ConsumerRecord<String, String> record : records) {\n        workerPool.submit(() -> process(record));\n    }\n}\n```\n\n**Pause/Resume Semantics:**\n\n- Paused partitions still participate in fetch requests (heartbeat, metadata)\n- Position is maintained but records not returned\n- Useful for backpressure without losing offset position\n- Does not trigger rebalance\n- Can pause/resume different partitions independently\n\n### Position Reset on Rebalance\n\n```java\nconsumer.subscribe(\n    Arrays.asList(\"topic\"),\n    new ConsumerRebalanceListener() {\n        @Override\n        public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n            System.out.println(\"Assigned: \" + partitions);\n            // Consumer position is reset (cleared) on assignment\n\n            // Check if we have committed offset\n            for (TopicPartition tp : partitions) {\n                OffsetAndMetadata committed = consumer.committed(tp);\n                if (committed != null) {\n                    // Seek to committed offset (resume from last processed)\n                    consumer.seek(tp, committed.offset());\n                } else {\n                    // No committed offset - use auto.offset.reset\n                    // - \"earliest\": seekToBeginning()\n                    // - \"latest\": seekToEnd()\n                    // - \"none\": throw NoOffsetForPartitionException\n                }\n            }\n        }\n\n        @Override\n        public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n            // Commit offset before losing partition\n            consumer.commitSync();\n        }\n    }\n);\n```\n\n### Beginning and End Offset Queries\n\n```java\n// Query offsets without seeking\nSet<TopicPartition> partitions = consumer.assignment();\n\n// Beginning offsets\nMap<TopicPartition, Long> beginning = consumer.beginningOffsets(partitions);\nfor (TopicPartition tp : partitions) {\n    System.out.println(tp + \": earliest=\" + beginning.get(tp));\n}\n\n// End offsets (high water mark)\nMap<TopicPartition, Long> end = consumer.endOffsets(partitions);\nfor (TopicPartition tp : partitions) {\n    System.out.println(tp + \": latest=\" + (end.get(tp) - 1));\n}\n\n// Safe bounds checking\nTopicPartition tp = new TopicPartition(\"topic\", 0);\nlong beginning = consumer.beginningOffsets(Arrays.asList(tp)).get(tp);\nlong end = consumer.endOffsets(Arrays.asList(tp)).get(tp);\n\nconsumer.seek(tp, Math.max(beginning, Math.min(end - 1, 1000)));\n// Seek to offset 1000, but bounded by [beginning, end-1]\n```\n\n---\n\n## Consumer Group Mechanics\n\n### Group Coordination Protocol\n\n```\nConsumer Instance                Broker Coordinator\n       |                                 |\n       |--- JoinGroup Request ---------->|\n       |                                 |\n       |     [Rebalance in Progress]     |\n       |  (max.poll.interval.ms seconds) |\n       |                                 |\n       |<---- JoinGroup Response --------|\n       |  (Contains assigned partitions) |\n       |\n       |--- CommitOffset/LeaveGroup --->|\n       |<---- Response ------------------|\n```\n\n**Key Timing:**\n\n| Event | Timeout | Default | Trigger |\n|-------|---------|---------|---------|\n| Heartbeat | `session.timeout.ms` | 45s | No heartbeat for 45s → fenced |\n| Max Poll | `max.poll.interval.ms` | 5m | No poll for 5m → fenced |\n| Rebalance | `rebalance.timeout.ms` | 60s | Member doesn't join → excluded |\n\n### Session Timeout vs Max Poll Interval\n\n```java\n// session.timeout.ms: Heartbeat timeout\n// - Detected when heartbeat not sent\n// - Internal background heartbeat mechanism\n// - If missed, broker assumes consumer crashed\nprops.setProperty(\"session.timeout.ms\", \"45000\");\nprops.setProperty(\"heartbeat.interval.ms\", \"3000\");  // Should be < session.timeout.ms\n// Heartbeat sent every 3s, if missed for 45s -> fenced\n\n// max.poll.interval.ms: Poll timeout\n// - Detected when poll() not called\n// - Application might be stuck processing records\n// - If exceeded, broker removes consumer from group\nprops.setProperty(\"max.poll.interval.ms\", \"300000\");  // 5 minutes\n// If poll() not called for 5m -> fenced\n\n// Both can trigger fencing (removal from group)\n// session.timeout.ms: Usually faster (network/broker issue)\n// max.poll.interval.ms: Usually slower (application processing time)\n```\n\n### Rebalance Triggering and Timing\n\nRebalances occur ONLY during `poll()` calls:\n\n```java\n// Scenario 1: New member joins group\nThread newMember = new Thread(() -> {\n    KafkaConsumer<String, String> consumer2 = new KafkaConsumer<>(props);\n    consumer2.subscribe(Arrays.asList(\"topic\"));\n    consumer2.poll(Duration.ofMillis(100));  // <-- Triggers rebalance\n});\nnewMember.start();\n\n// Main consumer\nwhile (true) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n    // On next poll() call, discovers rebalance and participates\n    // poll() may block for seconds during rebalance\n}\n\n// Scenario 2: Topic pattern match changes\nconsumer.subscribe(Pattern.compile(\"logs-.*\"));\n// When new topic \"logs-2024\" created, discovered on next poll()\n// Triggers rebalance\nconsumer.poll(Duration.ofMillis(100));\n\n// Scenario 3: Consumer explicitly unsubscribes\nconsumer.unsubscribe();\n// Triggers rebalance on next poll()\n```\n\n### Rebalance Callback Ordering\n\n```java\nnew ConsumerRebalanceListener() {\n    @Override\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        System.out.println(\"1. Revoked: \" + partitions);\n        // Called FIRST\n        // Opportunity to commit last offsets before losing partition\n        consumer.commitSync();\n        // After this returns, you can no longer fetch from these partitions\n    }\n\n    @Override\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        System.out.println(\"2. Assigned: \" + partitions);\n        // Called SECOND (after revoke returns)\n        // Position is cleared on assignment\n        // Can seek to custom position\n        for (TopicPartition tp : partitions) {\n            OffsetAndMetadata committed = consumer.committed(tp);\n            if (committed != null) {\n                consumer.seek(tp, committed.offset());\n            } else {\n                consumer.seekToBeginning(Arrays.asList(tp));\n            }\n        }\n    }\n};\n```\n\n### Group Membership and Fencing\n\n```java\n// Scenario: Member A and Member B in same group\n// Member A is fenced (another member takes its partitions)\n\n// Member A's state:\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n// Returns empty or partially filled\n\n// Member A tries to commit:\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n    // Member A is fenced!\n    // Another member now owns the partitions\n    // Member A should stop processing\n    System.err.println(\"Fenced from group, exiting\");\n    break;\n}\n\n// Member A tries to seek:\nconsumer.seek(new TopicPartition(\"topic\", 0), 100);\n// IllegalStateException or similar - partition not owned\n\n// Member A continues polling:\nconsumer.poll(Duration.ofMillis(100));\n// May continue to work briefly, but subsequent operations fail\n// Eventually all operations fail when membership confirmed lost\n```\n\n### Proactive Leave and Graceful Shutdown\n\n```java\n// Option 1: Call close() for graceful shutdown\nconsumer.close();  // Triggers:\n// 1. Revoke all partitions (callback)\n// 2. Sends LeaveGroup request\n// 3. Waits for broker acknowledgement\n// 4. Consumer can be rebalanced off group immediately\n\n// Option 2: Explicit unsubscribe before close\nconsumer.unsubscribe();  // Triggers rebalance on next poll\nconsumer.poll(Duration.ofMillis(100));  // Rebalance happens here\nconsumer.close();  // Now just cleanup\n\n// Without explicit leave, broker waits for session.timeout.ms\n// Then removes consumer from group\n// Graceful leave is much faster\n```\n\n---\n\n## Metadata Queries\n\n### Topic and Partition Metadata\n\n```java\n// List all topics\nMap<String, List<PartitionInfo>> topics = consumer.listTopics();\nfor (String topic : Handling\n\n### Exception Hierarchy\n\n```\nException\n├── WakeupException\n│   └── Thrown by poll() when wakeup() called from another thread\n│   └── Consumer state still valid, can resume\n│\n├── CommitFailedException\n│   └── Thrown when commit fails due to lost group membership\n│   └── Indicates consumer fenced by another group member\n│   └── Should exit processing loop and create new consumer\n│\n├── AuthenticationException / AuthorizationException\n│   └── Credentials invalid or lacking permission\n│   └── Not retried, should fail fast\n│   └── Fix credentials and restart\n│\n├── IllegalStateException\n│   └── Invalid API call sequence\n│   └── Examples: subscribe after assign, poll from different thread\n│\n├── TimeoutException\n│   └── Operation exceeded timeout\n│   └── May Practices Summary\n\n### 1. **Choose Commit Strategy Carefully**\n\n- **Auto-commit**: Simple but loses progress on crash\n- **Manual sync commit**: Safe but slower (blocks)\n- **Manual async commit + sync on close**: Balanced (recommended)\n\n### 2. **Always Use Rebalance Listener**\n\n```java\nconsumer.subscribe(topics, new ConsumerRebalanceListener() {\n    @Override\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        consumer.commitSync();  // Don't lose work\n    }\n\n    @Override\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Seek to committed position or custom position\n    }\n});\n```\n\n### 3. **Handle CommitFailedException Properly**\n\n- Indicates consumer fenced by another member\n- Must stop processing and create new consumer\n- Don't retry or recover from same instance\n\n### 4. **Set max.poll.interval.ms Appropriately**\n\n- Must be larger than time to process one batch\n- Larger values tolerate slower processing but slower rebalance\n- Default 5 minutes usually sufficient\n\n### 5. **Use Explicit Pause/Resume for Backpressure**\n\n- Instead of discarding records, pause partitions\n- Allows worker threads to catch up\n- Maintains offset position\n\n### 6. **Graceful Shutdown Pattern**\n\n```java\ntry {\n    while (running) {\n        ConsumerRecords<String, String> records = consumer.poll(...);\n        // Process...\n    }\n} catch (WakeupException e) {\n    // Shutdown signal\n} finally {\n    consumer.commitSync();\n    consumer.close();\n}\n```\n\n### 7. **Test with Rebalancing**\n\n- Kill/start consumers in group\n- Verify offset commits work\n- Ensure no message loss or duplication\n- Test timeout scenarios\n\n### 8. **Monitor Consumer Lag**\n\n```java\nMap<TopicPartition, Long> endOffsets = consumer.endOffsets(consumer.assignment());\nlong totalLag = 0;\nfor (TopicPartition tp : consumer.assignment()) {\n    long position = consumer.position(tp);\n    long lag = endOffsets.get(tp) - position;\n    totalLag += lag;\n}\nSystem.out.println(\"Total lag: \" + totalLag);\n```\n\n### 9. **Document Offset Reset Behavior**\n\n- Make explicit decision about auto.offset.reset\n- Document what happens on first run (earliest vs latest)\n- Test offset reset scenarios\n\n### 10. **Thread Safety Discipline**\n\n- Never call consumer methods from different thread\n- Use `wakeup()` only for shutdown signaling\n- Synchronize shutdown flag with volatile or AtomicBoolean\n\n---\n\n## Gotchas and Common Mistakes\n\n### Gotcha 1: Poll Blocks Longer Than Timeout During Rebalance\n\n```java\n// Timeout is 100ms, but poll blocks for 5 seconds!\nconsumer.poll(Duration.ofMillis(100));  // Rebalance happens here\n// May block for seconds if rebalance callbacks are slow\n```\n\n**Fix:** Be aware rebalances don't respect timeout. Keep callback code fast.\n\n### Gotcha 2: Committed Offset is \"Next to Read\", Not \"Last Read\"\n\n```java\nConsumerRecord<String, String> record = records.get(0);\n// record.offset() = 42\n\n// To resume from this message:\nconsumer.seek(partition, 42);  // Wrong! Reads 42 again\nconsumer.seek(partition, 43);  // Correct! Next offset after 42\n\n// Or in commit:\ncommitSync(Map.of(partition, new OffsetAndMetadata(43)));  // Not 42!\n```\n\n### Gotcha 3: Position Cleared on Rebalance Assignment\n\n```java\nconsumer.seek(partition, 1000);\n\n// Rebalance happens\n// Position is cleared!\n\n// In onPartitionsAssigned:\nconsumer.seek(partition, 1000); Detected when heartbeat not sent\n// - Internal background heartbeat mechanism\n// - If missed, broker assumes consumer crashed\nprops.setProperty(\"session.timeout.ms\", \"45000\");\nprops.setProperty(\"heartbeat.interval.ms\", \"3000\");  // Should be < session.timeout.ms\n// Heartbeat sent every 3s, if missed for 45s -> fenced\n\n// max.poll.interval.ms: Poll timeout\n// -
