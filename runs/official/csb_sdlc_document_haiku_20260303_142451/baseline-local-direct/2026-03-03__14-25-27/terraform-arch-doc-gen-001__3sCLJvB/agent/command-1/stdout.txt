This is crucial for the plan safety mechanism - users can review planned changes before apply.\n\n#### Apply Phase\n\nFor each resource approved in the plan:\n\n1. **Call provider**: `ApplyResourceChange()` with:\n   - Current state\n   - Planned state\n   - Configuration\n2. **Execute change**: Provider performs actual infrastructure operation (API call, etc.)\n3. **Receive new state**: Provider returns:\n   - New state after successful operation\n   - Diagnostics\n4. **Update state**: Terraform updates local/remote state file with new state\n\n### 2.4 Resource State Representation\n\nEach resource in Terraform state is represented as:\n\n```\nResourceInstance {\n  type: string                    // \"aws_instance\"\n  name: string                    // Resource name from config\n  module_path: string             // Module path (empty for root)\n  address: string                 // Full address: \"aws_instance.web[0]\"\n\n  current: ObjectValue            // Current state from last apply\n  deposed: []ObjectValue          // Old versions during updates\n\n  attributes: map[string]interface{} // Actual state data\n  sensitive: map[string]bool      // Sensitive attribute markers\n  dependencies: []string          // Tracked dependencies\n}\n```\n\nDuring plan:\n- Current state is compared with configuration\n- Provider determines differences\n- New planned state is prepared\n\nDuring apply:\n- Planned changes are executed\n- Current state is replaced with new state\n- State modifications are atomic per resource\n\n## 3. State Management\n\n### 3.1 State File Structure and Persistence\n\nTerraform state is a JSON file containing the representation of managed infrastructure. The state file structure:\n\n```json\n{\n  \"version\": 4,\n  \"terraform_version\": \"1.0.0\",\n  \"serial\": 42,\n  \"lineage\": \"abc-123\",\n  \"outputs\": {},\n  \"resources\": [\n    {\n      \"type\": \"aws_instance\",\n      \"name\": \"web\",\n      \"instances\": [\n        {\n          \"schema_version\": 0,\n          \"attributes\": {\n            \"id\": \"i-1234567890abcdef0\",\n            \"ami\": \"ami-123456\",\n            \"instance_type\": \"t2.micro\",\n            ...\n          },\n          \"sensitive_attributes\": [],\n          \"private\": {...}\n        }\n      ]\n    }\n  ]\n}\n```\n\nKey fields:\n- **version**: State schema version (current: 4)\n- **serial**: Incremented on every write (detects concurrent modifications)\n- **lineage**: UUID identifying this state file's history (prevents mixing unrelated states)\n- **resources**: Array of resource objects tracking instances\n\n### 3.2 State Managers and Read/Write Operations\n\n#### State Manager Interface\n\n```\nStateManager {\n  State() *State                                    // Get current state\n  RefreshState() error                              // Re-read from backend\n  PersistState(state *State) error                  // Write to backend\n  Lock(info *LockInfo) string (lockID) error        // Acquire state lock\n  Unlock(lockID string) error                       // Release state lock\n  DeleteState() error                               // Delete state\n}\n```\n\n#### Filesystem State Manager\n\nFor local state files:\n1. **Read**: Load JSON file, parse into State object\n2. **Write**: Serialize State to JSON, atomically write to disk (write temp file, rename)\n3. **Lock**: Create separate lock file to prevent concurrent access\n\n#### Remote State Manager\n\nFor remote backends (S3, Azure Storage, Terraform Cloud, etc.):\n1. **Read**: Fetch state object via backend API\n2. **Write**: Send updated state via backend API\n3. **Lock**: Use backend's locking mechanism (DynamoDB for S3, etc.)\n\n### 3.3 State Snapshots and Concurrent Graph Evaluation\n\nState snapshots enable safe concurrent graph evaluation:\n\n#### Snapshot Semantics\n\nBefore graph execution begins:\n1. **Acquire state lock** (if remote backend or explicit locking enabled)\n2. **Create snapshot**: Copy current state to in-memory snapshot\n3. **Share snapshot**: All graph nodes operate on the snapshot as baseline\n4. **Collect changes**: Graph execution produces new state objects\n5. **Merge changes**: After execution completes, merge all changes atomically\n6. **Persist**: Write merged state to backend\n7. **Release lock**: Unlock state for other operations\n\nThis design ensures:\n- Graph nodes see a consistent view of state (no in-flight changes)\n- Multiple independent resources can be processed in parallel\n- Changes are merged safely (e.g., resource A and B can be created in parallel)\n- Conflicts are detected and reported\n\n#### State Merging Algorithm\n\nWhen resources are evaluated concurrently:\n\n```\n1. Start with baseline snapshot state\n2. For each completed resource operation:\n   - Extract state changes for that resource\n   - Merge into accumulated changes\n   - Detect conflicts (e.g., same resource modified twice)\n3. If conflicts detected:\n   - Apply conflict resolution policy (usually fail)\n4. Write final merged state\n```\n\n### 3.4 State Locking\n\nState locks prevent concurrent modifications that could corrupt state.\n\n#### Lock Information\n\n```\nLockInfo {\n  ID: string                      // Unique lock ID (UUID)\n  Path: string                    // State path/address\n  Timestamp: time.Time            // When lock acquired\n  Info: string                    // Description (user, operation)\n}\n```\n\n#### Locking Mechanisms by Backend\n\n**S3 Backend**:\n- Uses DynamoDB table for lock management\n- Creates item with lock ID as primary key\n- Lock expires via TTL if process crashes\n\n**Azure Storage**:\n- Uses blob storage with lease mechanism\n- Acquires exclusive lease on state blob\n- Lease auto-renews during operation\n\n**Local State**:\n- Creates `.terraform.lock.hcl` lockfile\n- Contains lock metadata\n- Prevents concurrent runs via filesystem\n\n**Terraform Cloud**:\n- Enforced by cloud backend\n- All state operations are serialized\n- Supports run queue for safety\n\n#### Deadlock Prevention\n\nTerraform prevents circular lock waits:\n- Timeout on lock acquisition (default: 0 = no timeout, must be explicit)\n- Lock acquisition is non-blocking or time-limited\n- Operator can force unlock if process crashed\n- Clean shutdown releases locks immediately\n\n### 3.5 State Synchronization During Plan/Apply\n\nThe state synchronization lifecycle:\n\n#### Before Plan\n```\n1. RefreshState()\n   └─ Call provider.ReadResource() for each managed resource\n      └─ Provider returns current real-world state\n2. Update in-memory state with real-world values\n3. Create state snapshot for planning\n```\n\n#### During Plan\n```\n1. Compare snapshot state with configuration\n2. Determine required changes per resource\n3. Provider suggests changes (plan)\n4. No state modifications during plan\n```\n\n#### During Apply\n```\n1. For each approved change:\n   - Call provider.ApplyResourceChange()\n   - Provider executes change (API call, etc.)\n   - Update state with result\n   - Check for errors\n2. Accumulate all state changes\n3. Merge into final state\n4. Persist to backend\n```\n\n#### Refresh Operations\n\nThe `terraform refresh` command (or `plan -refresh=true`):\n1. Calls provider's `ReadResource()` for all resources\n2. Updates state with real-world values\n3. Detects drift (state file ≠ real infrastructure)\n4. Reports drift to user\n5. Can be used to fix state when infrastructure changes outside Terraform\n\n## 4. Graph Builder System\n\n### 1.1 Dependency Graph Construction\n\nThe graph builder system is responsible for transforming Terraform configuration into an executable dependency graph. This process occurs in multiple stages:\n\n#### Configuration to Graph Translation\nWhen Terraform processes a configuration, each resource, data source, module, and variable becomes a node in the graph. The initial graph construction:\n\n1. **Resource Node Creation**: Each `resource`, `data`, `module`, `variable`, and `output` block creates corresponding graph nodes\n2. **Dependency Discovery**: Terraform's static analyzer scans HCL expressions to identify references between resources (e.g., `aws_instance.web.id`)\n3. **Implicit Dependencies**: Dependencies are recorded as edges in the graph, with explicit dependencies from `depends_on` also added as edges\n\nThe graph at this stage contains:\n- **Resource nodes**: Represent managed infrastructure instances\n- **Data source nodes**: Represent external data lookups\n- **Module nodes**: Represent modules (which expand into subgraphs)\n- **Output nodes**: Represent root module outputs\n- **Variable nodes**: Represent input and local variables\n- **Root node**: A synthetic node that depends on all root-level outputs\n\n#### Graph Structure\nThe dependency graph is a directed acyclic graph (DAG) where:\n- Nodes represent infrastructure objects and operations\n- Edges represent dependencies, pointing from dependent to dependency (if A depends on B, edge goes A → B)\n- The graph is guaranteed acyclic by validation before execution\n- Multiple edges can exist between nodes for various dependency types (data flow, ordering, etc.)\n\n### 1.2 Graph Transformers\n\nGraph transformers are a series of transformation passes applied to the graph before execution. Each transformer modifies the graph structure to prepare it for execution. Key transformers include:\n\n#### `TransitiveReductionTransformer`\nRemoves redundant edges from the graph. If A depends on B and B depends on C, and A also directly depends on C, the direct A→C edge is redundant and removed. This simplification reduces unnecessary sequential execution while maintaining correctness.\n\n#### `PruneTransformer`\nRemoves nodes that are not needed for the current operation:\n- In `plan` phase: removes resources that already exist and have no detected changes\n- In `apply` phase: removes resources not affected by the plan\n- Preserves nodes required by outputs or explicitly referenced resources\n\n#### `AttachResourceConfigTransformer`\nAssociates each resource node with its configuration from the parsed HCL. This attachment enables the node to:\n- Compare configuration with current state during planning\n- Generate resource creation, update, or destruction tasks\n\n#### `LocalTransformer`\nCreates nodes for local value assignments and ensures they're properly sequenced in the graph. Local values may have dependencies on resources, so this transformer ensures correct evaluation order.\n\n#### `OutputTransformer`\nTransforms output declarations into nodes with dependencies on the resources they reference. Outputs are expressed as graph nodes to ensure they're only computed after their dependencies are satisfied.\n\n#### `ModuleExpansionTransformer`\nExpands module nodes into subgraphs. For each module instantiation:\n- Creates a subgraph containing the module's resources and data sources\n- Connects the subgraph to the parent graph via module input/output nodes\n- Handles multiple instances created by `count` and `for_each` in modules\n\n#### `DynamicExpansionTransformer`\nHandles `count` and `for_each` resource expansion. This transformer:\n- Creates separate nodes for each resource instance (e.g., `aws_instance.web[0]`, `aws_instance.web[1]`)\n- Creates implicit dependencies between instances to enable proper ordering\n- Generates splat expressions that dynamically reference all instances\n\n#### `ResourceDestroyNodeTransformer`\nFor resources being destroyed, creates destroy nodes in the graph. Destroy operations have reverse dependency logic - a resource's destroy node must execute after resources that depend on it are destroyed.\n\n#### `ProviderTransformer`\nCreates nodes representing provider initialization and configuration. Ensures:\n- Providers are initialized before resources that require them\n- Provider configuration changes are properly sequenced\n- Provider dependencies are respected\n\n#### `ReferenceTransformer`\nAdds edges to the graph based on indirect references through computed values. When resource B references an attribute from resource A that isn't known until A is created, an edge A → B is added to preserve execution order.\n\n### 1.3 Node Representation\n\nEach graph node encapsulates an executable unit with:\n\n```\nNode Interface:\n├── Name() string                           // Unique identifier (e.g., \"aws_instance.web\")\n├── Dependencies() []string                 // List of node dependencies\n├── DependsOn() []string                    // Explicit depends_on relationships\n├── Execute(context, state) (NewState, error)  // Main execution logic\n├── ResourceType() string                   // For resource nodes: \"aws_instance\"\n├── ResourceAddress() string                // For resource nodes: \"aws_instance.web\"\n└── GraphNodeResourceProvider() string      // For resources: required provider\n```\n\nNode types include:\n- **Resource nodes**: `NodeResourceApply`, `NodeResourceDestroy`, `NodeResourcePlan`\n- **Module nodes**: `NodeModule` (expands into subgraph)\n- **Output nodes**: `NodeOutput`\n- **Data source nodes**: `NodeDataSource`\n- **Provider nodes**: `NodeProviderConfig`\n- **Local value nodes**: `NodeLocal`\n- **Root nodes**: Synthetic nodes for graph termination\n\n## 2. Provider Interface and Lifecycle\n\n### 2.1 Provider Plugin Architecture\n\nTerraform uses a plugin architecture where providers are separate processes communicating with the Terraform core via gRPC. This architecture provides:\n\n- **Isolation**: Provider crashes don't crash Terraform\n- **Language flexibility**: Providers can be written in any language\n- **Version compatibility**: Core and providers can be versioned independently\n- **Security**: Providers run with limited permissions\n\n#### Plugin Discovery and Loading\n\nTerraform discovers providers through:\n\n1. **Local search**: Current working directory, Terraform configuration directory\n2. **Plugin cache**: User's `~/.terraform.d/plugins` directory\n3. **Registry**: Official Terraform Registry (configured via provider requirements)\n\nThe loader validates:\n- Plugin protocol version compatibility (currently Protocol v6)\n- Plugin signatures (when signature verification is enabled)\n- Plugin checksums match expected values\n\n#### RPC Protocol\n\nCommunication between core and provider uses Protocol Buffer definitions over gRPC:\n\n```\nCore → Provider RPCs:\n├── GetProviderSchema()           // Returns provider capabilities\n├── PrepareProviderConfig()       // Validates and prepares config\n├── ValidateResourceConfig()      // Validates resource config\n├── ValidateDataSourceConfig()    // Validates data source config\n├── ReadResource()                // Read resource state\n├── PlanResourceChange()          // Plan resource changes\n├── ApplyResourceChange()         // Apply resource changes\n├── ImportResourceState()         // Import existing resource\n└── Diagnostics                   // Error/warning streaming\n\nProvider → Core callbacks:\n├── Logging                       // Provider debug/info logs\n└── Stop signal                   // Graceful provider shutdown\n```\n\n### 2.2 Provider Configuration and Initialization\n\n#### Provider Configuration Nodes\n\nFor each provider in the configuration, a `NodeProviderConfig` node exists:\n- Validates provider configuration against schema\n- Stores provider configuration in graph state\n- Tracks provider version and source\n\nDuring execution, provider config nodes:\n1. Evaluate configuration expressions (e.g., `aws_region = var.region`)\n2. Call provider's `PrepareProviderConfig()` to normalize configuration\n3. Store prepared config for use by resource operations\n\n#### Provider Instantiation\n\nEach provider instance represents a configured provider:\n\n```\nProviderInstance {\n  schema: *ProviderSchema          // From GetProviderSchema()\n  config: map[string]interface{}   // Prepared configuration\n  client: *grpc.ClientConn         // RPC connection to plugin\n  stop: chan struct{}              // Shutdown signal channel\n}\n```\n\nProvider instances are cached and reused:\n- Multiple resources using the same provider configuration share one instance\n- Different configurations (e.g., different AWS regions) create separate instances\n- Instances are initialized before any resource operations\n\n#### Provider Initialization Sequence\n\n```\n1. Validate provider configuration\n   └─ Call provider.PrepareProviderConfig() → normalized config\n\n2. Get provider schema\n   └─ Call provider.GetProviderSchema() → all resource/data source schemas\n\n3. Configure provider with validated config\n   └─ Store configuration for resource operations\n\n4. Ready for resource operations\n   └─ Call PlanResourceChange() / ApplyResourceChange() as needed\n```\n\n### 2.3 Resource Operation Delegation\n\nResource operations are delegated to providers through the plan and apply phases.\n\n#### Planning Phase\n\nFor each resource, the plan phase:\n\n1. **Read current state**: Retrieve resource state from Terraform state file\n2. **Validate configuration**: Call provider's `ValidateResourceConfig()`\n3. **Plan changes**: Call provider's `PlanResourceChange()` with:\n   - Current state\n   - Configuration\n   - Plan options (force replace, etc.)\n4. **Receive planned changes**: Provider returns:\n   - Planned state (what state will be after apply)\n   - Requires replacement (if true, resource must be destroyed and recreated)\n   - Sensitive attribute markers\n   - Diagnostics (warnings/errors)\n\nThe provider's `PlanResourceChange()` responsibility is to **determine what will change** without making changes. This is crucial for the plan safety mechanism - users can review planned changes before apply.\n\n#### Apply Phase\n\nFor each resource approved in the plan:\n\n1. **Call provider**: `ApplyResourceChange()` with:\n   - Current state\n   - Planned state\n   - Configuration\n2. **Execute change**: Provider performs actual infrastructure operation (API call, etc.)\n3. **Receive new state**: Provider returns:\n   - New state after successful operation\n   - Diagnostics\n4. **Update state**: Terraform updates local/remote state file with new state\n\n### 2.4 Resource State Representation\n\nEach resource in Terraform state is represented as:\n\n```\nResourceInstance {\n  type: string                    // \"aws_instance\"\n  name: string                    // Resource name from config\n  module_path: string             // Module path (empty for root)\n  address: string                 // Full address: \"aws_instance.web[0]\"\n\n  current: ObjectValue            // Current state from last apply\n  deposed: []ObjectValue          // Old versions during updates\n\n  attributes: map[string]interface{} // Actual state data\n  sensitive: map[string]bool      // Sensitive attribute markers\n  dependencies: []string          // Tracked dependencies\n}\n```\n\nDuring plan:\n- Current state is compared with configuration\n- Provider determines differences\n- New planned state is prepared\n\nDuring apply:\n- Planned changes are executed\n- Current state is replaced with new state\n- State modifications are atomic per resource\n\n## 3. State Management\n\n### 3.1 State File Structure and Persistence\n\nTerraform state is a JSON file containing the representation of managed infrastructure. The state file structure:\n\n```json\n{\n  \"version\": 4,\n  \"terraform_version\": \"1.0.0\",\n  \"serial\": 42,\n  \"lineage\": \"abc-123\",\n  \"outputs\": {},\n  \"resources\": [\n    {\n      \"type\": \"aws_instance\",\n      \"name\": \"web\",\n      \"instances\": [\n        {\n          \"schema_version\": 0,\n          \"attributes\": {\n            \"id\": \"i-1234567890abcdef0\",\n            \"ami\": \"ami-123456\",\n            \"instance_type\": \"t2.micro\",\n            ...\n          },\n          \"sensitive_attributes\": [],\n          \"private\": {...}\n        }\n      ]\n    }\n  ]\n}\n```\n\nKey fields:\n- **version**: State schema version (current: 4)\n- **serial**: Incremented on every write (detects concurrent modifications)\n- **lineage**: UUID identifying this state file's history (prevents mixing unrelated states)\n- **resources**: Array of resource objects tracking instances\n\n### 3.2 State Managers and Read/Write Operations\n\n#### State Manager Interface\n\n```\nStateManager {\n  State() *State                                    // Get current state\n  RefreshState() error                              // Re-read from backend\n  PersistState(state *State) error                  // Write to backend\n  Lock(info *LockInfo) string (lockID) error        // Acquire state lock\n  Unlock(lockID string) error                       // Release state lock\n  DeleteState() error                               // Delete state\n}\n```\n\n#### Filesystem State Manager\n\nFor local state files:\n1. **Read**: Load JSON file, parse into State object\n2. **Write**: Serialize State to JSON, atomically write to disk (write temp file, rename)\n3. **Lock**: Create separate lock file to prevent concurrent access\n\n#### Remote State Manager\n\nFor remote backends (S3, Azure Storage, Terraform Cloud, etc.):\n1. **Read**: Fetch state object via backend API\n2. **Write**: Send updated state via backend API\n3. **Lock**: Use backend's locking mechanism (DynamoDB for S3, etc.)\n\n### 3.3 State Snapshots and Concurrent Graph Evaluation\n\nState snapshots enable safe concurrent graph evaluation:\n\n#### Snapshot Semantics\n\nBefore graph execution begins:\n1. **Acquire state lock** (if remote backend or explicit locking enabled)\n2. **Create snapshot**: Copy current state to in-memory snapshot\n3. **Share snapshot**: All graph nodes operate on the snapshot as baseline\n4. **Collect changes**: Graph execution produces new state objects\n5. **Merge changes**: After execution completes, merge all changes atomically\n6. **Persist**: Write merged state to backend\n7. **Release lock**: Unlock state for other operations\n\nThis design ensures:\n- Graph nodes see a consistent view of state (no in-flight changes)\n- Multiple independent resources can be processed in parallel\n- Changes are merged safely (e.g., resource A and B can be created in parallel)\n- Conflicts are detected and reported\n\n#### State Merging Algorithm\n\nWhen resources are evaluated concurrently:\n\n```\n1. Start with baseline snapshot state\n2. For each completed resource operation:\n   - Extract state changes for that resource\n   - Merge into accumulated changes\n   - Detect conflicts (e.g., same resource modified twice)\n3. If conflicts detected:\n   - Apply conflict resolution policy (usually fail)\n4. Write final merged state\n```\n\n### 3.4 State Locking\n\nState locks prevent concurrent modifications that could corrupt state.\n\n#### Lock Information\n\n```\nLockInfo {\n  ID: string                      // Unique lock ID (UUID)\n  Path: string                    // State path/address\n  Timestamp: time.Time            // When lock acquired\n  Info: string                    // Description (user, operation)\n}\n```\n\n#### Locking Mechanisms by Backend\n\n**S3 Backend**:\n- Uses DynamoDB table for lock management\n- Creates item with lock ID as primary key\n- Lock expires via TTL if process crashes\n\n**Azure Storage**:\n- Uses blob storage with lease mechanism\n- Acquires exclusive lease on state blob\n- Lease auto-renews during operation\n\n**Local State**:\n- Creates `.terraform.lock.hcl` lockfile\n- Contains lock metadata\n- Prevents concurrent runs via filesystem\n\n**Terraform Cloud**:\n- Enforced by cloud backend\n- All state operations are serialized\n- Supports run queue for safety\n\n#### Deadlock Prevention\n\nTerraform prevents circular lock waits:\n- Timeout on lock acquisition (default: 0 = no timeout, must be explicit)\n- Lock acquisition is non-blocking or time-limited\n- Operator can force unlock if process crashed\n- Clean shutdown releases locks immediately\n\n### 3.5 State Synchronization During Plan/Apply\n\nThe state synchronization lifecycle:\n\n#### Before Plan\n```\n1. RefreshState()\n   └─ Call provider.ReadResource() for each managed resource\n      └─ Provider returns current real-world state\n2. Update in-memory state with real-world values\n3. Create state snapshot for planning\n```\n\n#### During Plan\n```\n1. Compare snapshot state with configuration\n2. Determine required changes per resource\n3. Provider suggests changes (plan)\n4. No state modifications during plan\n```\n\n#### During Apply\n```\n1. For each approved change:\n   - Call provider.ApplyResourceChange()\n   - Provider executes change (API call, etc.)\n   - Update state with result\n   - Check for errors\n2. Accumulate all state changes\n3. Merge into final state\n4. Persist to backend\n```\n\n#### Refresh Operations\n\nThe `terraform refresh` command (or `plan -refresh=true`):\n1. Calls provider's `ReadResource()` for all resources\n2. Updates state with real-world values\n3. Detects drift (state file ≠ real infrastructure)\n4. Reports drift to user\n5. Can be used to fix state when infrastructure changes outside Terraform\n\n## 4. Execution Flow and Hook System\n\n### 4.1 Overall Execution Pipeline\n\nThe complete execution flow from command invocation to completion:\n\n#### Phase 1: Initialization\n```\nterraform init/plan/apply\n    ↓\nLoad configuration\n    ↓\nLoad state (refresh optional)\n    ↓\nLoad provider schemas\n    ↓\nParse and validate HCL\n    ↓\nEvaluate variables and locals\n```\n\n#### Phase 2: Graph Building\n```\nBuild initial resource graph\n    ├─ Resource nodes\n    ├─ Module nodes\n    ├─ Data source nodes\n    ├─ Output nodes\n    └─ Variable nodes\n    ↓\nApply transformation passes\n    ├─ Reference transformer\n    ├─ Transitive reduction\n    ├─ Pruning\n    ├─ Module expansion\n    ├─ Dynamic expansion (count/for_each)\n    ├─ Destroy nodes (apply only)\n    └─ Provider config nodes\n    ↓\nTopologically sort graph\n```\n\n#### Phase 3: Plan Phase (for plan/apply)\n```\nLock state (if remote backend)\n    ↓\nCreate state snapshot\n    ↓\nFor each graph node in dependency order:\n    ├─ Read current state (if resource)\n    ├─ Evaluate configuration expressions\n    ├─ Call provider.PlanResourceChange()\n    ├─ Compare with current state\n    └─ Record planned changes\n    ↓\nGenerate plan file with changes\n    ↓\nRelease state lock\n```\n\n#### Phase 4: Apply Phase (apply only)\n```\nLock state (if remote backend)\n    ↓\nLoad plan (optional, can replan)\n    ↓\nCreate state snapshot\n    ↓\nWalk graph in parallel (respecting dependencies):\n    ├─ For each resource:\n    │   ├─ Execute planned operation\n    │   ├─ Call provider.ApplyResourceChange()\n    │   ├─ Update state with new values\n    │   └─ Record operation result\n    │\n    └─ Track completion and errors\n    ↓\nMerge state changes atomically\n    ↓\nPersist state to backend\n    ↓\nRelease state lock\n    ↓\nDisplay summary (resources created/updated/destroyed)\n```\n\n### 4.2 Graph Node Evaluation\n\n#### Plan Execution\n\nFor each node during plan:\n\n```\nPlanNode(node, state, options):\n  1. Evaluate node configuration\n     └─ Resolve variable references\n     └─ Compute expressions\n\n  2. For resource nodes:\n     a. Determine node action (create/update/delete/noop)\n        └─ Compare current state with config\n\n     b. Call provider.PlanResourceChange()\n        Inputs:\n          - priorState: current state from snapshot\n          - proposedNewState: state based on config\n          - config: resource configuration\n        Output:\n          - plannedState: what state will be after apply\n          - requiresReplace: if true, resource must be destroyed+recreated\n          - diagnostics: warnings/errors\n\n     c. Store plan for this resource\n\n  3. For data source nodes:\n     a. Call provider.ReadDataSource()\n     b. Store queried data\n\n  4. For output nodes:\n     a. Evaluate output expression\n     b. Store output value\n\n  5. Return plan result\n```\n\n#### Apply Execution\n\nFor each node during apply, in dependency-respecting parallel order:\n\n```\nApplyNode(node, plan, state, options):\n  1. Wait for all dependencies to complete\n\n  2. For resource nodes (create/update):\n     a. Get current state from latest snapshot\n     b. Call provider.ApplyResourceChange()\n        Inputs:\n          - priorState: current state\n          - plannedState: from plan phase\n          - config: resource configuration\n        Output:\n          - newState: actual state after operation\n          - diagnostics: success/failure info\n\n     c. If success:\n        - Update state snapshot with newState\n        - Record as created/updated\n\n     d. If failure:\n        - Record error\n        - Skip dependent resources\n\n  3. For resource nodes (destroy):\n     a. Call provider.DestroyResource()\n     b. If success: remove from state\n     c. If failure: leave in state, record error\n\n  4. Return result (success/failure)\n```\n\n### 4.3 The Walker Pattern\n\nThe graph walker is responsible for traversing the dependency graph and executing nodes. The walker implements:\n\n```\nWalker interface {\n  Walk(ctx Context) error {\n    return parallel_walk(graph, max_parallelism)\n  }\n}\n\nparallel_walk(graph, max_parallelism):\n  1. Perform topological sort of graph\n     └─ Assign level to each node based on dependencies\n\n  2. For each level (in order):\n     a. Collect all ready nodes at this level\n     b. Start execution for each:\n        ├─ Enqueue in worker pool\n        ├─ Up to max_parallelism workers active\n        └─ Track completion\n\n     c. Wait for all level nodes to complete\n\n     d. Check for errors:\n        ├─ If fatal error: stop, fail\n        ├─ If non-fatal: continue\n        └─ If stop requested: drain and exit\n\n  3. Return error if any fatal errors\n```\n\nThe walker supports:\n- **Concurrent execution**: Resources at same dependency level execute in parallel\n- **Dependency awareness**: Respects all edges in graph\n- **Early termination**: Stops on unrecoverable errors\n- **Cancellation**: Responds to context cancellation for graceful shutdown\n- **Progress reporting**: Reports completed/in-flight nodes via hooks\n\n### 4.4 Hook System\n\nHooks are extension points that allow monitoring and customizing Terraform's execution. The hook interface:\n\n```\nHook interface {\n  PreApply(address string, gen Plan, state State)\n  PostApply(address string, gen Plan, state State, err error)\n  PrePlan(address string, state State)\n  PostPlan(address string, plan Plan)\n  PreProviderConfigure(provider string, config Config)\n  PostProviderConfigure(provider string, err error)\n  ProviderOutput(provider string, message string)\n  DestoryDeposed(address string, id string)\n}\n```\n\n#### Built-in Hooks\n\n**UI Hook**:\n- Displays operations in progress: \"aws_instance.web: Creating...\"\n- Shows completion status: \"aws_instance.web: Creation complete!\"\n- Displays progress bars for long operations\n\n**State Hook**:\n- Tracks state mutations during execution\n- Records which resources changed\n- Enables detailed change summary\n\n**Logger Hook**:\n- Logs all operations to debug log\n- Captures provider output\n- Records timing information\n\n#### Custom Hooks\n\nUsers can implement custom hooks via:\n- Terraform Cloud/Enterprise: custom run tasks, policy as code\n- Terraform modules: in-module dynamic blocks for conditional logic\n- Provider plugins: resources that perform orchestration\n\nHook execution:\n1. Hooks are called synchronously before operation begins\n2. Hooks cannot block operation (timeout after 30s)\n3. Hook panics are caught and logged but don't stop execution\n4. Multiple hooks execute sequentially in registration order\n\n### 4.5 Dynamic Expansion for count/for_each\n\nResources using `count` or `for_each` are expanded into multiple instances.\n\n#### Count Expansion\n\nFor `resource \"aws_instance\" \"web\" { count = 3 }`:\n\n```\nSingle config node → Three graph nodes:\n├─ aws_instance.web[0]\n├─ aws_instance.web[1]\n└─ aws_instance.web[2]\n\nDependencies propagated to all instances\n```\n\nThe expander:\n1. Evaluates count expression (e.g., `count = var.instance_count`)\n2. Creates count.value and count.index variables\n3. Generates node for each index 0 to count-1\n4. Creates implicit ordering: [0] → [1] → [2] for state coherency\n\n#### For_Each Expansion\n\nFor `resource \"aws_instance\" \"web\" { for_each = var.instances }`:\n\n```\nSingle config node → N graph nodes (one per map/set entry):\n├─ aws_instance.web[\"web1\"]\n├─ aws_instance.web[\"web2\"]\n└─ aws_instance.web[\"web3\"]\n```\n\nThe expander:\n1. Evaluates for_each expression (must be map or set of strings)\n2. Creates each.key and each.value variables\n3. Generates node for each map key/set value\n4. No implicit ordering between instances (can execute in parallel)\n\n#### Splat Expression Handling\n\nWhen configuration references `aws_instance.web[*].id`:\n\n1. Expander creates splat reference node\n2. This node depends on all instances: [0], [1], [2]\n3. Splat expression evaluates to list of IDs from all instances\n4. Dependent resources wait for all instances to complete\n\n#### Dynamic Block Expansion\n\nFor blocks using `dynamic`:\n\n```hcl\nresource \"aws_security_group\" \"example\" {\n  dynamic \"ingress\" {\n    for_each = var.ingress_rules\n    content {\n      from_port   = ingress.value.from_port\n      to_port     = ingress.value.to_port\n    }\n  }\n}\n```\n\nDynamic blocks are expanded similarly:\n1. Evaluate `for_each` expression at plan time\n2. Generate multiple ingress blocks in resulting configuration\n3. Single resource with multiple blocks (not multiple instances)\n4. All blocks sent to provider in single resource\n\n### 4.6 Error Handling and Propagation\n\nError handling during execution:\n\n#### Error Classification\n\n**Fatal Errors** (stop execution):\n- Provider plugin crashes\n- State file corrupted\n- Configuration syntax errors\n- Resource dependency deadlock\n\n**Non-Fatal Errors** (continue execution):\n- Single resource creation fails\n- Provider API returns 409 Conflict\n- Data source query returns no results (depending on configuration)\n\n#### Error Propagation\n\n```\nprovider.Operation() error\n    ↓\nif error:\n  ├─ Log error\n  ├─ Store in node results\n  ├─ Update state (partial updates as needed)\n  ├─ Continue graph execution\n  │   └─ Skip dependent resources (can't proceed without dependency)\n  └─ Merge partial state changes\n    ↓\nFinal apply report shows:\n├─ Successfully applied: N resources\n├─ Failed: M resources (with error messages)\n└─ Skipped: K resources (dependencies failed)\n```\n\n#### Recovery Mechanisms\n\n**Partial State Persistence**:\n- State is saved even if some resources fail\n- Allows fixing failures and retrying\n- Failed resource remains in previous state\n\n**Idempotent Operations**:\n- Rerunning apply after failure is safe\n- Provider `PlanResourceChange()` detects already-applied changes\n- Failed resource creation can be retried (already exists → update)\n\n**Rollback (not automatic)**:\n- Terraform doesn't automatically undo successful changes\n- Failed apply leaves infrastructure in partially-modified state\n- User can modify configuration to destroy unwanted resources\n- Rollback requires explicit `terraform destroy` or config changes\n\n## 5. Extension Points and Customization\n\n### 5.1 Provider Plugin Extensibility\n\nProviders are the primary extension mechanism. Custom providers can:\n\n**Implement Resource Types**:\n- Define schema (attributes, validation rules)\n- Implement CRUD operations\n- Return state and computed values\n\n**Implement Data Sources**:\n- Define read-only data lookups\n- Query external systems\n- Return data for use in configuration\n\n**Implement Validators**:\n- Custom validation logic for attributes\n- Cross-attribute validation\n- Complex constraint checking\n\n**Example Custom Provider Operations**:\n```\nCustom provider \"acme\" {\n  - Manages DNS records (acme_dns_record)\n  - Manages certificates (acme_certificate)\n  - Provides data sources for domain info\n}\n```\n\n### 5.2 Terraform Graph Node Extension\n\nAdvanced users can extend graph behavior via:\n\n**Custom Resource Provisioners**:\n- `local-exec`: run commands locally\n- `remote-exec`: run commands on remote instance\n- Custom: implement provisioner plugin\n\n**Triggers for Redeployment**:\n```hcl\nresource \"aws_instance\" \"web\" {\n  ami = \"ami-123456\"\n\n  triggers = {\n    redeployment = var.redeployment_trigger\n  }\n}\n# Changing trigger value forces destroy+recreate\n```\n\n**Multiple Providers for Same Resource Type**:\n```hcl\nterraform {\n  required_providers {\n    aws_primary = {\n      source = \"hashicorp/aws\"\n      alias = \"primary\"\n    }\n    aws_dr = {\n      source = \"hashicorp/aws\"\n      alias = \"dr\"\n    }\n  }\n}\n# Resources can use different provider configurations\n```\n\n### 5.3 Hooks and Logging\n\n**Custom Operations via Hooks**:\n- Implement custom Hook interface\n- Monitor operations in real-time\n- Trigger custom logic on events\n\n**Logging**:\n- Set TF_LOG=DEBUG for detailed logs\n- Set TF_LOG_PATH=file for log file output\n- Includes: graph
