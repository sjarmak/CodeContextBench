## Summary

### Legacy ConsumerConnector
The `kafka.consumer.ConsumerConnector` interface is **not present** in any of the three repositories. It was part of the old Scala-based high-level consumer API that stored offsets in ZooKeeper and was removed in Kafka 2.0.

### New KafkaConsumer API Implementations (apache/kafka)

| File | Symbol | Role |
|------|--------|------|
| `clients/.../consumer/KafkaConsumer.java` | `KafkaConsumer` | Public API, implements `Consumer<K,V>`, delegates to internals |
| `clients/.../consumer/internals/ClassicKafkaConsumer.java` | `ClassicKafkaConsumer` | Classic group protocol, implements `ConsumerDelegate<K,V>` |
| `clients/.../consumer/internals/AsyncKafkaConsumer.java` | `AsyncKafkaConsumer` | New group protocol (KIP-848), implements `ConsumerDelegate<K,V>` |
| `clients/.../consumer/MockConsumer.java` | `MockConsumer` | Test implementation of `Consumer<K,V>` |

### Offset Management Behavioral Differences

- **Old ConsumerConnector**: ZooKeeper-stored offsets, `commitOffsets()` method, multi-threaded stream model
- **ClassicKafkaConsumer**: Offsets in `__consumer_offsets` topic, `commitSync()`/`commitAsync()` via `ConsumerCoordinator` on the calling thread, single-threaded poll loop
- **AsyncKafkaConsumer**: Same offset storage, but commits are processed via `CommitRequestManager` in a dedicated background `ConsumerNetworkThread` — fully event-driven architecture

### Camel (apache/camel)
`KafkaConsumer.java` and `KafkaFetchRecords.java` wrap the new KafkaConsumer API with `AsyncCommitManager`/`SyncCommitManager` for offset management.
