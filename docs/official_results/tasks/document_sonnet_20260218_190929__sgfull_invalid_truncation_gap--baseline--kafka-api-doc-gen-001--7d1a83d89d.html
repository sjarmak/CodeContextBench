## Objective

Generate comprehensive API reference documentation for the Apache Kafka `KafkaConsumer` Java API. The documentation should cover the complete API surface with emphasis on behavioral semantics, offset management strategies, rebalancing mechanics, and error handling patterns.

## Scope

Your documentation should cover the **KafkaConsumer API** from the `org.apache.kafka.clients.consumer` package, including:

1. **Core Consumer Lifecycle**
   - Constructor variants and configuration
   - Subscription methods (dynamic and manual assignment)
   - Polling and data fetching semantics
   - Resource cleanup and closing

2. **Offset Management**
   - Synchronous and asynchronous commit strategies
   - Offset queries and position control
   - Seek operations and offset discovery
   - Committed offset semantics

3. **Consumer Group Mechanics**
   - ConsumerRebalanceListener interface and callbacks
   - Rebalance triggers and timing
   - Group membership and heartbeat behavior
   - Partition assignment vs subscription models

4. **Flow Control and Position Management**
   - Pause and resume functionality
   - Position queries and manipulation
   - Offset-to-timestamp lookups
   - Beginning and end offset discovery

5. **Error Handling**
   - Exception types and recovery strategies
   - CommitFailedException and group fencing
   - WakeupException and thread interruption
   - Timeout and authentication errors

## Requirements

### API Methods Documentation (40%)

Document all public methods of the `KafkaConsumer` class with:
- Method signatures including all overloads
- Parameter semantics and validation rules
- Return types and their meanings
- Exception types thrown and conditions

### Behavioral Notes (30%)

Explain critical behavioral semantics:
- **poll() blocking behavior**: when it returns immediately vs when it blocks, timeout handling, rebalance callback execution during poll
- **Offset commit semantics**: difference between sync/async commits, retry behavior, commit failure handling
- **Rebalance coordination**: when rebalances occur (only during poll), callback ordering (revoked then assigned), partition ownership guarantees
- **Thread safety**: which methods are thread-safe, wakeup() special case, event loop model
- **Group membership**: max.poll.interval.ms enforcement, proactive leave behavior, session timeout vs poll timeout
- **Manual vs dynamic assignment**: mutually exclusive nature, use cases for each model
- **Position vs committed offset**: the off-by-one relationship ("committed should be next offset to read")
- **Transactional semantics**: read_committed isolation level, LSO boundary, filtered messages

### Usage Examples (20%)

Provide concrete code examples demonstrating:
- **Basic subscription and polling loop**: subscribe to topics, poll for records, process messages
- **Manual offset commit**: disable auto-commit, explicit commitSync/commitAsync after processing
- **Rebalance listener**: implement ConsumerRebalanceListener, commit offsets on revoke, initialize positions on assign
- **Seek operations**: seekToBeginning, seekToEnd, seek to specific offset, timestamp-based seeking
- **Multi-threaded processing pattern**: single consumer thread with pause/resume coordination and worker pool

### Documentation Structure (10%)

Organize documentation with clear sections:
- Overview and threading model
- Core types (KafkaConsumer, ConsumerRebalanceListener, ConsumerRecords, etc.)
- Subscription and assignment methods
- Polling and data fetching
- Offset management methods
- Flow control and position queries
- Metadata and monitoring
- Lifecycle and resource management
- Exception handling guide
- Configuration-driven behaviors
- Common patterns and best practices

It resides in the `org.apache.kafka.clients.consumer` package and implements the `Consumer<K, V>` interface.\n\n### Threading Model\n\n**KafkaConsumer is NOT thread-safe.** All methods except `wakeup()` must be called from a single thread. Concurrent access from multiple threads will throw `ConcurrentModificationException`.\n\nThe intended usage model is an **event loop** running on a single dedicated thread:\n\n```\nsingle thread:\n  subscribe / assign\n  loop:\n    poll()          \u2190 fetches records AND drives heartbeats, rebalances, offset commits\n    process records\n    commit offsets\n```\n\nThe only cross-thread interaction supported is `wakeup()`, which is safe to call from any thread (including signal handlers) and causes the consumer's next or current `poll()` call to throw `WakeupException`.\n\n### Group-Managed vs. Standalone Consumer\n\n| Aspect | Group-Managed (subscribe) | Standalone (assign) |\n|---|---|---|\n| API entry point | `subscribe(topics)` | `assign(partitions)` |\n| Partition assignment | Automatic, coordinated by broker | Manual, static |\n| Rebalancing | Yes \u2014 partition ownership can change | No |\n| Offset commits | To `__consumer_offsets` topic (group) | Optional; group.id not required |\n| Use case | Scalable, fault-tolerant consumer groups | Exact partition control, migration, replay |\n\nThese two modes are **mutually exclusive**: calling `subscribe()` after `assign()` (or vice versa) throws `IllegalStateException`.\n\n---\n\n## Core Types\n\n### KafkaConsumer\n\n**Package:** `org.apache.kafka.clients.consumer`\n**Implements:** `Consumer<K, V>`, `Closeable`\n\nThe central class for consuming records from Kafka. Manages connections, group membership, offset tracking, and record fetching.\n\n#### Constructors\n\n```java\n// From a configuration map with class-based deserializers\nKafkaConsumer(Map<String, Object> configs)\n\n// From a configuration map with explicit deserializer instances\nKafkaConsumer(Map<String, Object> configs,\n              Deserializer<K> keyDeserializer,\n              Deserializer<V> valueDeserializer)\n\n// From Properties (common in legacy code)\nKafkaConsumer(Properties props)\n\n// From Properties with explicit deserializer instances\nKafkaConsumer(Properties props,\n              Deserializer<K> keyDeserializer,\n              Deserializer<V> valueDeserializer)\n```\n\n**Notes:**\n- If deserializer instances are provided they take precedence over class names in config.\n- The constructor validates all configuration keys and throws `KafkaException` on invalid values.\n- Network connections are established lazily on the first `poll()`, not during construction.\n- `group.id` is **required** when using `subscribe()`; it is optional (but recommended) for `assign()`.\n\n---\n\n### ConsumerRebalanceListener\n\n**Package:** `org.apache.kafka.clients.consumer`\n\nA callback interface invoked when partition assignments change during a group rebalance. Implement this interface and pass it to `subscribe()`.\n\n```java\npublic interface ConsumerRebalanceListener {\n\n    /**\n     * Called when partitions are revoked from this consumer before a rebalance.\n     * At this point the consumer still owns the partitions.\n     *\n     * Typical use: commit offsets so the next consumer starts from the right position.\n     *\n     * @param partitions partitions being taken away from this consumer\n     */\n    void onPartitionsRevoked(Collection<TopicPartition> partitions);\n\n    /**\n     * Called after partitions have been assigned to this consumer.\n     * At this point the consumer owns the partitions.\n     *\n     * Typical use: initialize position (seek), load state, reset caches.\n     *\n     * @param partitions newly assigned partitions\n     */\n    void onPartitionsAssigned(Collection<TopicPartition> partitions);\n\n    /**\n     * Called when the consumer lost its partition assignment without a clean revocation\n     * (e.g., due to a session timeout or fencing). Any pending uncommitted offsets\n     * may be re-processed by another consumer.\n     *\n     * Default implementation delegates to onPartitionsRevoked().\n     * Override to skip committing (offsets can't be committed after a fence).\n     *\n     * @param partitions partitions that were lost\n     */\n    default void onPartitionsLost(Collection<TopicPartition> partitions) {\n        onPartitionsRevoked(partitions);\n    }\n}\n```\n\n**Callback ordering and timing:**\n\n1. Callbacks are always invoked **inside `poll()`**, never from a background thread.\n2. For a clean rebalance: `onPartitionsRevoked()` \u2192 assignment completes \u2192 `onPartitionsAssigned()`.\n3. For a lost assignment (timeout/fence): `onPartitionsLost()` \u2192 (re-)assignment \u2192 `onPartitionsAssigned()`.\n4. `onPartitionsRevoked()` is called with the **full set** of previously owned partitions, not just the ones being taken away (unless using cooperative rebalancing, where it receives only the delta).\n5. Calling `commitSync()` inside `onPartitionsRevoked()` is safe and common practice.\n6. **poll() may block longer than its timeout** when rebalance callbacks are running.\n\n---\n\n### ConsumerRecords\n\n**Package:** `org.apache.kafka.clients.consumer`\n\nContainer for a batch of records returned by a single `poll()` call. Partition assignment is automatically managed by the consumer group coordinator. The actual assignment is not made until the next `poll()` call triggers a rebalance.\n\n**Behavior:**\n- Replaces any existing topic subscription (idempotent if called with the same set).\n- Previous `assign()` call is cleared; the two modes are mutually exclusive.\n- Triggers a group rebalance to distribute partitions.\n\n**Throws:** `IllegalStateException` if called after `assign()`.\n\n---\n\n### subscribe(Collection\\<String\\> topics, ConsumerRebalanceListener listener)\n\n```java\nvoid subscribe(Collection<String> topics, ConsumerRebalanceListener listener)\n```\n\nSubscribe with a rebalance listener. Callbacks are invoked synchronously inside `poll()`.\n\n---\n\n### subscribe(Pattern pattern)\n\n```java\nvoid subscribe(Pattern pattern)\nvoid subscribe(Pattern pattern, ConsumerRebalanceListener listener)\n```\n\nSubscribe to all topics whose names match the given regular expression. The consumer will dynamically pick up newly created topics that match the pattern \u2014 useful for consuming from a growing set of topics without restart.\n\n**Notes:**\n- Pattern matching is evaluated periodically (controlled by `metadata.max.age.ms`).\n- A new topic matching the pattern triggers a rebalance.\n\n---\n\n### assign(Collection\\<TopicPartition\\> partitions)\n\n```java\nvoid assign(Collection<TopicPartition> partitions)\n```\n\nManually assign specific topic partitions to this consumer. The consumer becomes a **standalone consumer** \u2014 no group membership, no rebalancing, no automatic offset reset unless explicitly seeked.\n\n**Behavior:**\n- Replaces the entire current assignment atomically.\n- Does NOT trigger `ConsumerRebalanceListener` callbacks (there is no rebalance).\n- `group.id` is not required, but if provided, offsets can still be committed to `__consumer_offsets`.\n- Calling `assign()` after `subscribe()` clears the subscription.\n\n**Throws:** `IllegalStateException` if called after `subscribe()`.\n\n---\n\n### unsubscribe()\n\n```java\nvoid unsubscribe()\n```\n\nUnsubscribe from all topics (if using `subscribe()`) or clear all assignments (if using `assign()`). Also causes the consumer to leave the consumer group if it was a member, triggering a rebalance for remaining members.\n\n---\n\n### subscription() and assignment()\n\n```java\nSet<String> subscription()          // topics subscribed via subscribe()\nSet<TopicPartition> assignment()    // partitions currently assigned (subscribe OR assign)\n```\n\n`assignment()` returns the live partition set \u2014 may differ from what was passed to `subscribe()` because the broker controls assignment in group mode.\n\n---\n\n## Polling and Data Fetching\n\n### poll(Duration timeout)\n\n```java\nConsumerRecords<K, V> poll(Duration timeout)\n```\n\n**The central method of KafkaConsumer.** Fetches records from all assigned partitions and drives the consumer's internal machinery.\n\n**What poll() does in addition to fetching records:**\n1. Sends heartbeats to the group coordinator (if in a group).\n2. Detects and processes join/sync group responses triggering rebalances.\n3. Invokes `ConsumerRebalanceListener` callbacks synchronously.\n4. Sends and receives `commitAsync()` responses.\n5. Auto-commits offsets if `enable.auto.commit=true` and the interval has elapsed.\n6. Refreshes topic metadata if needed.\n\n**Timeout semantics:**\n- `timeout` specifies the **maximum time** to block waiting for records.\n- Returns immediately if records are already available in the fetch buffer.\n- Returns early (possibly empty) if a rebalance completes, an error occurs, or `wakeup()` is called.\n- **May block longer than timeout** if rebalance callbacks take time to execute. The timeout is measured from when `poll()` was called, not when callbacks return.\n- Passing `Duration.ZERO` performs a non-blocking poll (returns whatever is buffered immediately).\n\n**Return value:**\n- Never returns `null`. Returns an empty `ConsumerRecords` when no records are available.\n\n**Throws:**\n- `WakeupException` \u2014 if `wakeup()` was called from another thread.\n- `InterruptException` \u2014 if the calling thread was interrupted.\n- `AuthenticationException` \u2014 on SASL/SSL authentication failure.\n- `AuthorizationException` \u2014 on ACL permission failure.\n- `KafkaException` \u2014 for unrecoverable protocol errors.\n- `FencedInstanceIdException` \u2014 if the `group.instance.id` is fenced by a newer consumer.\n\n**max.poll.interval.ms enforcement:**\nIf `poll()` is not called within `max.poll.interval.ms` (default 300,000 ms) after the previous call, the consumer is considered failed. The broker removes it from the group and triggers a rebalance. On the next `poll()`, the consumer will attempt to rejoin. This guards against consumers that are alive but stuck processing a large batch.\n\n---\n\n## Offset Management Methods\n\n### Offset Concepts\n\nUnderstanding the different offset concepts is critical to correct consumer behavior:\n\n| Term | Definition |\n|---|---|\n| **position** | The offset of the **next record** to be fetched for a partition. Advances after each `poll()`. |\n| **committed offset** | The last durably acknowledged offset (stored in `__consumer_offsets`). Should equal position after committing. |\n| **beginning offset** | The earliest available offset (may be > 0 if log compaction or retention has deleted older records). |\n| **end offset** (high-watermark) | The offset of the next record that will be written. Records at or beyond this offset are not yet visible. |\n| **LSO (Last Stable Offset)** | For `read_committed` isolation: the highest offset of committed transactions. Records from in-flight transactions are invisible below LSO. |\n\n**The off-by-one rule:** Committed offset = last successfully processed offset + 1. When the consumer restarts, it fetches from the committed offset, meaning it starts at the record *after* the one that was last committed.\n\n---\n\n### commitSync()\n\n```java\nvoid commitSync()\nvoid commitSync(Duration timeout)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)\n```\n\nSynchronously commits offsets, blocking until the broker acknowledges or the timeout expires.\n\n**Behavior:**\n- With no arguments: commits the current position for all assigned partitions (i.e., one past the last fetched record).\n- With a map argument: commits only the specified partitions and offsets.\n- Retries automatically on retriable failures (e.g., coordinator not available).\n- Does NOT retry on `CommitFailedException` (group has been rebalanced; this consumer is no longer the owner).\n\n**When to use:** At-least-once delivery with predictable consistency. Use inside `onPartitionsRevoked()` to ensure offsets are committed before partitions are handed off.\n\n**Throws:**\n- `CommitFailedException` \u2014 group rebalanced and this consumer lost ownership of the partition.\n- `TimeoutException` \u2014 commit did not complete within the timeout.\n- `InterruptException` \u2014 calling thread was interrupted.\n\n---\n\n### commitAsync()\n\n```java\nvoid commitAsync()\nvoid commitAsync(OffsetCommitCallback callback)\nvoid commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets, OffsetCommitCallback callback)\n```\n\nAsynchronously commits offsets. Returns immediately; the result is delivered to the callback on a future `poll()` call.\n\n**Behavior:**\n- Does NOT retry on failure (retrying an async commit can commit a stale offset over a more recent one that succeeded asynchronously).\n- Multiple outstanding async commits are queued; they execute in order.\n- The callback is invoked from the `poll()` thread (not a background thread).\n- If the consumer shuts down before the callback fires, the callback may never be invoked.\n\n**OffsetCommitCallback:**\n```java\npublic interface OffsetCommitCallback {\n    void onComplete(Map<TopicPartition, OffsetAndMetadata> offsets, Exception exception);\n}\n```\n`exception` is `null` on success, or a `CommitFailedException`/other exception on failure.\n\n**When to use:** High-throughput scenarios where blocking on every commit is too expensive. Typically combined with a final `commitSync()` on shutdown.\n\n---\n\n### committed(Set\\<TopicPartition\\> partitions)\n\n```java\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions)\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions, Duration timeout)\n\n// Single-partition variants (deprecated in newer versions, prefer Set variants)\nOffsetAndMetadata committed(TopicPartition partition)\nOffsetAndMetadata committed(TopicPartition partition, Duration timeout)\n```\n\nFetches the last committed offset for the specified partitions from the broker. Returns `null` for a partition if no offset has ever been committed.\n\n**Notes:**\n- This is a **blocking network call** to the group coordinator.\n- Does not return the current `position()` \u2014 only what has been committed to the broker.\n- Useful for diagnostics, monitoring, and initializing seek positions.\n\n---\n\n### position(TopicPartition partition)\n\n```java\nlong position(TopicPartition partition)\nlong position(TopicPartition partition, Duration timeout)\n```\n\nReturns the offset of the **next record** to be fetched for the partition. This is the consumer's in-memory cursor, which advances with each `poll()` call.\n\n**Notes:**\n- May trigger a metadata fetch or offset lookup if the position is not yet known (e.g., just after subscription but before the first successful poll).\n- Differs from `committed()`: position advances as records are fetched, committed advances only when you call `commitSync()`/`commitAsync()`.\n\n---\n\n### seek(TopicPartition partition, long offset)\n\n```java\nvoid seek(TopicPartition partition, long offset)\nvoid seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)\n```\n\nOverride the position for a partition, causing the next `poll()` to fetch from the specified offset.\n\n**Notes:**\n- The partition must be in the current assignment.\n- Does not communicate with the broker immediately; takes effect on the next fetch.\n- `seek()` with an `OffsetAndMetadata` also sets the leader epoch, enabling fencing of stale replicas.\n- Seeking to an offset before the beginning offset causes an `OffsetOutOfRangeException` on the next fetch (or is silently reset according to `auto.offset.reset`).\n\n---\n\n### seekToBeginning(Collection\\<TopicPartition\\> partitions)\n\n```java\nvoid seekToBeginning(Collection<TopicPartition> partitions)\n```\n\nSeek to the earliest available offset for each specified partition. The broker resolves the actual beginning offset lazily on the next `poll()`.\n\n---\n\n### seekToEnd(Collection\\<TopicPartition\\> partitions)\n\n```java\nvoid seekToEnd(Collection<TopicPartition> partitions)\n```\n\nSeek to the end of each specified partition (i.e., to the high-watermark). The consumer will only receive records produced after the seek. Also resolved lazily.\n\n---\n\n### offsetsForTimes(Map\\<TopicPartition, Long\\> timestampsToSearch)\n\n```java\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch)\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch, Duration timeout)\n```\n\nLook up the earliest offset whose timestamp is greater than or equal to the given timestamp for each partition.\n\n**Returns:** Map from partition to `OffsetAndTimestamp`. The value is `null` if no record with a timestamp >= the query was found (e.g., the queried time is in the future).\n\n**Use case:** Replaying events from a specific point in time without knowing the exact offset.\n\n---\n\n### beginningOffsets and endOffsets\n\n```java\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions)\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions, Duration timeout)\n\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions)\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions, Duration timeout)\n```\n\nFetch the first and last+1 available offsets for specified partitions. These are blocking network calls to the leader replicas.\n\n**Notes:**\n- `endOffsets` returns the high-watermark (next offset to be written), not the LSO.\n- For `read_committed` consumers, records up to LSO are visible; records between LSO and HWM belong to in-flight transactions.\n- Useful for computing lag: `lag = endOffset - currentPosition`.\n\n---\n\n## Flow Control and Position Queries\n\n### pause(Collection\\<TopicPartition\\> partitions)\n\n```java\nvoid pause(Collection<TopicPartition> partitions)\n```\n\nSuspend fetching for the specified partitions. Subsequent `poll()` calls return no records for paused partitions, but heartbeats and other group activity continue normally.\n\n**Use case:** Back-pressure management \u2014 when downstream processing is slow, pause to prevent unbounded buffering. Use with a worker thread pool:\n\n1. Consumer thread fetches records and dispatches to workers.\n2. When worker queue is full, consumer calls `pause()`.\n3. Workers complete; consumer calls `resume()`.\n4. Consumer thread calls `poll()` which returns empty for paused partitions (keeping heartbeat alive).\n\n**Notes:**\n- Partitions that are paused but then reassigned during a rebalance are NOT automatically paused after reassignment.\n- Does not prevent the consumer from being considered alive by the coordinator.\n\n---\n\n### resume(Collection\\<TopicPartition\\> partitions)\n\n```java\nvoid resume(Collection<TopicPartition> partitions)\n```\n\nResume fetching for partitions previously paused with `pause()`. Commits any pending async offsets.\n2. If a member of a consumer group, sends a `LeaveGroup` request to the coordinator so the broker can immediately trigger a rebalance (rather than waiting for a session timeout).\n3. Closes all network connections.\n\n**If `timeout` is specified:** The consumer waits up to that duration for pending operations to complete before forcibly closing.\n\n**Calling `close()` without first calling `wakeup()` from another thread:** If the consumer is blocked in `poll()`, `close()` blocks until `poll()` returns. For multi-threaded shutdown, call `wakeup()` to unblock `poll()`, catch `WakeupException`, then call `close()`.\n\n---\n\n### wakeup()\n\n```java\nvoid wakeup()\n```\n\nThe only thread-safe method on `KafkaConsumer`. Interrupts a blocked `poll()` call on another thread, causing it to throw `WakeupException`. If `poll()` is not currently running, the next call to `poll()` will throw immediately.\n\n**Pattern for shutdown:**\n```java\n// From a shutdown hook or another thread:\nconsumer.wakeup();\n\n// From the consumer thread:\ntry {\n    while (running) {\n        ConsumerRecords<K, V> records = consumer.poll(Duration.ofMillis(100));\n        // process...\n    }\n} catch (WakeupException e) {\n    // expected during shutdown; ignore\n} finally {\n    consumer.close();\n}\n```\n\n---\n\n## Exception Handling Guide\n\n### CommitFailedException\n\n**Package:** `org.apache.kafka.clients.consumer`\n**Thrown by:** `commitSync()`\n\nThrown when a commit cannot be completed because the consumer is no longer a member of the group or the group generation has changed (rebalance occurred while the consumer was processing).\n\n**Recovery:** The consumer should re-subscribe and start consuming from the newly committed offset. Do not retry the commit \u2014 the partition is owned by another consumer.\n\n**Prevention:**\n- Reduce per-batch processing time so `max.poll.interval.ms` is not exceeded.\n- Increase `max.poll.interval.ms` if processing is inherently slow.\n- Decrease `max.poll.records` to reduce the amount processed per batch.\n\n---\n\n### WakeupException\n\n**Package:** `org.apache.kafka.clients.consumer`\n**Thrown by:** `poll()`, `commitSync()`, `position()`, `committed()`, and other blocking calls\n\nThrown when `wakeup()` is called from another thread. It signals the consumer loop to exit cleanly.\n\n**Recovery:** Catch at the top of the event loop, stop the loop, then call `close()`.\n\n---\n\n### InterruptException\n\n**Package:** `org.apache.kafka.common`\n**Thrown by:** All blocking methods\n\nThrown when the calling thread's interrupt flag is set. Refreshes topic metadata if needed.\n\n**Timeout semantics:**\n- `timeout` specifies the **maximum time** to block waiting for records.\n- Returns immediately if records are already available in the fetch buffer.\n- Returns early (possibly empty) if a rebalance completes, an error occurs, or `wakeup()` is called.\n- **May block longer than timeout** if rebalance callbacks take time to execute. The timeout is measured from when `poll()` was called, not when callbacks return.\n- Passing `Duration.ZERO` performs a non-blocking poll (returns whatever is buffered immediately).\n\n**Return value:**\n- Never returns `null`. Returns an empty `ConsumerRecords` when no records are available.\n\n**Throws:**\n- `WakeupException` \u2014 if `wakeup()` was called from another thread.\n- `InterruptException` \u2014 if the calling thread was interrupted.\n- `AuthenticationException` \u2014 on SASL/SSL authentication failure.\n- `AuthorizationException` \u2014 on ACL permission failure.\n- `KafkaException` \u2014 for unrecoverable protocol errors.\n- `FencedInstanceIdException` \u2014 if the `group.instance.id` is fenced by a newer consumer.\n\n**max.poll.interval.ms enforcement:**\nIf `poll()` is not called within `max.poll.interval.ms` (default 300,000 ms) after the previous call, the consumer is considered failed. The broker removes it from the group and triggers a rebalance. On the next `poll()`, the consumer will attempt to rejoin. This guards against consumers that are alive but stuck processing a large batch.\n\n---\n\n## Offset Management Methods\n\n### Offset Concepts\n\nUnderstanding the different offset concepts is critical to correct consumer behavior:\n\n| Term | Definition |\n|---|---|\n| **position** | The offset of the **next record** to be fetched for a partition. Advances after each `poll()`. |\n| **committed offset** | The last durably acknowledged offset (stored in `__consumer_offsets`). Should equal position after committing. |\n| **beginning offset** | The earliest available offset (may be > 0 if log compaction or retention has deleted older records). |\n| **end offset** (high-watermark) | The offset of the next record that will be written. Records at or beyond this offset are not yet visible. |\n| **LSO (Last Stable Offset)** | For `read_committed` isolation: the highest offset of committed transactions. Records from in-flight transactions are invisible below LSO. |\n\n**The off-by-one rule:** Committed offset = last successfully processed offset + 1. When the consumer restarts, it fetches from the committed offset, meaning it starts at the record *after* the one that was last committed.\n\n---\n\n### commitSync()\n\n```java\nvoid commitSync()\nvoid commitSync(Duration timeout)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)\n```\n\nSynchronously commits offsets, blocking until the broker acknowledges or the timeout expires.\n\n**Behavior:**\n- With no arguments: commits the current position for all assigned partitions (i.e., one past the last fetched record).\n- With a map argument: commits only the specified partitions and offsets.\n- Retries automatically on retriable failures (e.g., coordinator not available).\n- Does NOT retry on `CommitFailedException` (group has been rebalanced; this consumer is no longer the owner).\n\n**When to use:** At-least-once delivery with predictable consistency. Use inside `onPartitionsRevoked()` to ensure offsets are committed before partitions are handed off.\n\n**Throws:**\n- `CommitFailedException` \u2014 group rebalanced and this consumer lost ownership of the partition.\n- `TimeoutException` \u2014 commit did not complete within the timeout.\n- `InterruptException` \u2014 calling thread was interrupted.\n\n---\n\n### commitAsync()\n\n```java\nvoid commitAsync()\nvoid commitAsync(OffsetCommitCallback callback)\nvoid commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets, OffsetCommitCallback callback)\n```\n\nAsynchronously commits offsets. Returns immediately; the result is delivered to the callback on a future `poll()` call.\n\n**Behavior:**\n- Does NOT retry on failure (retrying an async commit can commit a stale offset over a more recent one that succeeded asynchronously).\n- Multiple outstanding async commits are queued; they execute in order.\n- The callback is invoked from the `poll()` thread (not a background thread).\n- If the consumer shuts down before the callback fires, the callback may never be invoked.\n\n**OffsetCommitCallback:**\n```java\npublic interface OffsetCommitCallback {\n    void onComplete(Map<TopicPartition, OffsetAndMetadata> offsets, Exception exception);\n}\n```\n`exception` is `null` on success, or a `CommitFailedException`/other exception on failure.\n\n**When to use:** High-throughput scenarios where blocking on every commit is too expensive. Typically combined with a final `commitSync()` on shutdown.\n\n---\n\n### committed(Set\\<TopicPartition\\> partitions)\n\n```java\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions)\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions, Duration timeout)\n\n// Single-partition variants (deprecated in newer versions, prefer Set variants)\nOffsetAndMetadata committed(TopicPartition partition)\nOffsetAndMetadata committed(TopicPartition partition, Duration timeout)\n```\n\nFetches the last committed offset for the specified partitions from the broker. Returns `null` for a partition if no offset has ever been committed.\n\n**Notes:**\n- This is a **blocking network call** to the group coordinator.\n- Does not return the current `position()` \u2014 only what has been committed to the broker.\n- Useful for diagnostics, monitoring, and initializing seek positions.\n\n---\n\n### position(TopicPartition partition)\n\n```java\nlong position(TopicPartition partition)\nlong position(TopicPartition partition, Duration timeout)\n```\n\nReturns the offset of the **next record** to be fetched for the partition. This is the consumer's in-memory cursor, which advances with each `poll()` call.\n\n**Notes:**\n- May trigger a metadata fetch or offset lookup if the position is not yet known (e.g., just after subscription but before the first successful poll).\n- Differs from `committed()`: position advances as records are fetched, committed advances only when you call `commitSync()`/`commitAsync()`.\n\n---\n\n### seek(TopicPartition partition, long offset)\n\n```java\nvoid seek(TopicPartition partition, long offset)\nvoid seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)\n```\n\nOverride the position for a partition, causing the next `poll()` to fetch from the specified offset.\n\n**Notes:**\n- The partition must be in the current assignment.\n- Does not communicate with the broker immediately; takes effect on the next fetch.\n- `seek()` with an `OffsetAndMetadata` also sets the leader epoch, enabling fencing of stale replicas.\n- Seeking to an offset before the beginning offset causes an `OffsetOutOfRangeException` on the next fetch (or is silently reset according to `auto.offset.reset`).\n\n---\n\n### seekToBeginning(Collection\\<TopicPartition\\> partitions)\n\n```java\nvoid seekToBeginning(Collection<TopicPartition> partitions)\n```\n\nSeek to the earliest available offset for each specified partition. The broker resolves the actual beginning offset lazily on the next `poll()`.\n\n---\n\n### seekToEnd(Collection\\<TopicPartition\\> partitions)\n\n```java\nvoid seekToEnd(Collection<TopicPartition> partitions)\n```\n\nSeek to the end of each specified partition (i.e., to the high-watermark). The consumer will only receive records produced after the seek. Also resolved lazily.\n\n---\n\n### offsetsForTimes(Map\\<TopicPartition, Long\\> timestampsToSearch)\n\n```java\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch)\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch, Duration timeout)\n```\n\nLook up the earliest offset whose timestamp is greater than or equal to the given timestamp for each partition.\n\n**Returns:** Map from partition to `OffsetAndTimestamp`. The value is `null` if no record with a timestamp >= the query was found (e.g., the queried time is in the future).\n\n**Use case:** Replaying events from a specific point in time without knowing the exact offset.\n\n---\n\n### beginningOffsets and endOffsets\n\n```java\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions)\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions, Duration timeout)\n\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions)\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions, Duration timeout)\n```\n\nFetch the first and last+1 available offsets for specified partitions. These are blocking network calls to the leader replicas.\n\n**Notes:**\n- `endOffsets` returns the high-watermark (next offset to be written), not the LSO.\n- For `read_committed` consumers, records up to LSO are visible; records between LSO and HWM belong to in-flight transactions.\n- Useful for computing lag: `lag = endOffset - currentPosition`.\n\n---\n\n## Flow Control and Position Queries\n\n### pause(Collection\\<TopicPartition\\> partitions)\n\n```java\nvoid pause(Collection<TopicPartition> partitions)\n```\n\nSuspend fetching for the specified partitions. Subsequent `poll()` calls return no records for paused partitions, but heartbeats and other group activity continue normally.\n\n**Use case:** Back-pressure management \u2014 when downstream processing is slow, pause to prevent unbounded buffering. Use with a worker thread pool:\n\n1. Consumer thread fetches records and dispatches to workers.\n2. When worker queue is full, consumer calls `pause()`.\n3. Workers complete; consumer calls `resume()`.\n4. Consumer thread calls `poll()` which returns empty for paused partitions (keeping heartbeat alive).\n\n**Notes:**\n- Partitions that are paused but then reassigned during a rebalance are NOT automatically paused after reassignment.\n- Does not prevent the consumer from being considered alive by the coordinator.\n\n---\n\n### resume(Collection\\<TopicPartition\\> partitions)\n\n```java\nvoid resume(Collection<TopicPartition> partitions)\n```\n\nResume fetching for partitions previously paused with `pause()`. Fetching resumes on the next `poll()`.\n\n---\n\n### paused()\n\n```java\nSet<TopicPartition> paused()\n```\n\nReturns the set of currently paused partitions.\n\n---\n\n## Metadata and Monitoring\n\n### partitionsFor(String topic)\n\n```java\nList<PartitionInfo> partitionsFor(String topic)\nList<PartitionInfo> partitionsFor(String topic, Duration timeout)\n```\n\nFetch partition metadata (leader, replicas, ISR) for the specified topic.\n\n**PartitionInfo fields:**\n```java\nString topic()\nint partition()\nNode leader()\nNode[] replicas()\nNode[] inSyncReplicas()\nNode[] offlineReplicas()\n```\n\n---\n\n### listTopics()\n\n```java\nMap<String, List<PartitionInfo>> listTopics()\nMap<String, List<PartitionInfo>> listTopics(Duration timeout)\n```\n\nFetch metadata for all topics visible to this consumer. Commits any pending async offsets.\n2. If a member of a consumer group, sends a `LeaveGroup` request to the coordinator so the broker can immediately trigger a rebalance (rather than waiting for a session timeout).\n3. Closes all network connections.\n\n**If `timeout` is specified:** The consumer waits up to that duration for pending operations to complete before forcibly closing.\n\n**Calling `close()` without first calling `wakeup()` from another thread:** If the consumer is blocked in `poll()`, `close()` blocks until `poll()` returns. For multi-threaded shutdown, call `wakeup()` to unblock `poll()`, catch `WakeupException`, then call `close()`.\n\n---\n\n### wakeup()\n\n```java\nvoid wakeup()\n```\n\nThe only thread-safe method on `KafkaConsumer`. Interrupts a blocked `poll()` call on another thread, causing it to throw `WakeupException`. If `poll()` is not currently running, the next call to `poll()` will throw immediately.\n\n**Pattern for shutdown:**\n```java\n// From a shutdown hook or another thread:\nconsumer.wakeup();\n\n// From the consumer thread:\ntry {\n    while (running) {\n        ConsumerRecords<K, V> records = consumer.poll(Duration.ofMillis(100));\n        // process...\n    }\n} catch (WakeupException e) {\n    // expected during shutdown; ignore\n} finally {\n    consumer.close();\n}\n```\n\n---\n\n## Exception Handling Guide\n\n### CommitFailedException\n\n**Package:** `org.apache.kafka.clients.consumer`\n**Thrown by:** `commitSync()`\n\nThrown when a commit cannot be completed because the consumer is no longer a member of the group or the group generation has changed (rebalance occurred while the consumer was processing).\n\n**Recovery:** The consumer should re-subscribe and start consuming from the newly committed offset. Do not retry the commit \u2014 the partition is owned by another consumer.\n\n**Prevention:**\n- Reduce per-batch processing time so `max.poll.interval.ms` is not exceeded.\n- Increase `max.poll.interval.ms` if processing is inherently slow.\n- Decrease `max.poll.records` to reduce the amount processed per batch.\n\n---\n\n### WakeupException\n\n**Package:** `org.apache.kafka.clients.consumer`\n**Thrown by:** `poll()`, `commitSync()`, `position()`, `committed()`, and other blocking calls\n\nThrown when `wakeup()` is called from another thread. It signals the consumer loop to exit cleanly.\n\n**Recovery:** Catch at the top of the event loop, stop the loop, then call `close()`.\n\n---\n\n### InterruptException\n\n**Package:** `org.apache.kafka.common`\n**Thrown by:** All blocking methods\n\nThrown when the calling thread's interrupt flag is set. Wraps `InterruptedException`.\n\n**Recovery:** Restore the thread interrupt flag and shut down: `Thread.currentThread().interrupt()`.\n\n---\n\n### OffsetOutOfRangeException\n\n**Thrown by:** `poll()`\n**Condition:** The consumer's position for a partition is before the beginning offset (log was compacted or old segments were deleted) and `auto.offset.reset=none`.\n\n**Recovery:** Seek to `beginningOffsets()` or `endOffsets()`, or change `auto.offset.reset` to `earliest` or `latest`.\n\n---\n\n### NoOffsetForPartitionException\n\n**Thrown by:** `poll()`\n**Condition:** No committed offset exists for a newly assigned partition and `auto.offset.reset=none`.\n\n**Recovery:** Explicitly `seek()` the consumer to a known offset, or change `auto.offset.reset`.\n\n---\n\n### RecordDeserializationException\n\n**Package:** `org.apache.kafka.clients.consumer`\n**Thrown by:** `poll()`\n**Condition:** The key or value deserializer threw an exception.\n\n**Recovery:** Catch the exception, call `consumer.seek()` to skip past the poison record (`exception.offset() + 1`), log or dead-letter the bad record, then continue polling.\n\n```java\n} catch (RecordDeserializationException e) {\n    log.error(\"Deserialization error at offset {}\", e.offset(), e);\n    consumer.seek(e.topicPartition(), e.offset() + 1);\n}\n```\n\n---\n\n### AuthenticationException / AuthorizationException\n\n**Thrown by:** `poll()` and network operations\n**Condition:** SASL authentication failure or missing ACL permissions.\n\n**Behavior:** These are **fatal** \u2014 the consumer will not retry automatically. Close and reconfigure.\n\n---\n\n### FencedInstanceIdException\n\n**Thrown by:** `poll()`\n**Condition:** Another consumer with the same `group.instance.id` has joined (static membership fencing).\n\n**Recovery:** This is a configuration error \u2014 two consumers sharing the same instance ID. Fix the deployment.\n\n---\n\n### TimeoutException\n\n**Package:** `org.apache.kafka.common.errors`\n**Thrown by:** Methods with an explicit `Duration timeout` parameter when the operation does not complete in time.\n\n**Recovery:** Retry with a longer timeout or investigate broker availability.\n\n---\n\n## Configuration-Driven Behaviors\n\nKey configuration properties and their effect on API behavior:\n\n### Group and Membership\n\n| Property | Default | Effect |\n|---|---|---|\n| `group.id` | (none) | Required for subscribe(). Properties();\nprops.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, \"broker1:9092,broker2:9092\");\nprops.put(ConsumerConfig.GROUP_ID_CONFIG, \"my-consumer-group\");\nprops.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());\nprops.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());\nprops.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, \"earliest\");\n// Leave enable.auto.commit=true (the default) for simplest at-least-once delivery\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\nconsumer.subscribe(List.of(\"orders\", \"payments\"));\n\ntry {\n    while (true) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(200));\n        for (ConsumerRecord<String, String> record : records) {\n            System.out.printf(\"topic=%s partition=%d offset=%d key=%s value=%s%n\",\n                record.topic(), record.partition(), record.offset(),\n                record.key(), record.value());\n        }\n    }\n} catch (WakeupException e) {\n    // expected on shutdown\n} finally {\n    consumer.close();\n}\n```\n\n**Notes:** With `enable.auto.commit=true`, offsets are committed every `auto.commit.interval.ms`. This provides at-least-once delivery: if the process dies between a `poll()` and the next auto-commit, some records will be reprocessed.\n\n---\n\n### Pattern 2: Manual Offset Commit (At-Least-Once)\n\n```java\nprops.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, \"false\");\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\nconsumer.subscribe(List.of(\"orders\"));\n\ntry {\n    while (true) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(200));\n        for (ConsumerRecord<String, String> record : records) {\n            processRecord(record);  // your processing logic\n        }\n        // Commit after processing the entire batch\n        // commitAsync for high throughput; commitSync for strong guarantees\n        consumer.commitAsync((offsets, exception) -> {\n            if (exception != null) {\n                log.error(\"Commit failed for offsets {}\", offsets, exception);\n            }\n        });\n    }\n} catch (WakeupException e) {\n    // shutdown signal; fall through to finally\n} finally {\n    try {\n        // Final synchronous commit to flush any pending async commits\n        consumer.commitSync();\n    } finally {\n        consumer.close();\n    }\n}\n```\n\n**Why the hybrid commitAsync + final commitSync?** `commitAsync()` doesn't retry (to avoid committing stale offsets), so some commits might silently fail. The final `commitSync()` ensures the last batch is reliably committed before shutdown.\n\n---\n\n### Pattern 3: Rebalance Listener with Manual Commit\n\n```java\nprops.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, \"false\");\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\n\nconsumer.subscribe(List.of(\"orders\"), new ConsumerRebalanceListener() {\n\n    @Override\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        // Commit offsets synchronously BEFORE partitions are handed off.\n        // This ensures the new owner starts from the correct position.\n        try {\n            consumer.commitSync();\n            log.info(\"Committed offsets before rebalance for {}\", partitions);\n        } catch (CommitFailedException e) {\n            log.warn(\"Commit failed during revocation (may be already fenced)\", e);\n        }\n    }\n\n    @Override\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Optional: initialize per-partition state, metrics, or seek positions.\n        log.info(\"Assigned partitions: {}\", partitions);\n        // Example: seek to a specific position loaded from an external store\n        for (TopicPartition tp : partitions) {\n            Long checkpoint = loadCheckpointFromDatabase(tp);\n            if (checkpoint != null) {\n                consumer.seek(tp, checkpoint);\n            }\n        }\n    }\n\n    @Override\n    public void onPartitionsLost(Collection<TopicPartition> partitions) {\n        // Called on unexpected loss (timeout/fence). Do NOT commit here \u2014\n        // the group has already moved on. Just clean up local state.\n        log.warn(\"Partitions lost unexpectedly: {}\", partitions);\n        clearLocalState(partitions);\n    }\n});\n```\n\n---\n\n### Pattern 4: Seek Operations\n\n```java\n// Replay from the beginning\nconsumer.seekToBeginning(consumer.assignment());\n\n// Skip to the end (only process new records)\nconsumer.seekToEnd(consumer.assignment());\n\n// Seek to a specific offset\nconsumer.seek(new TopicPartition(\"orders\", 2), 1000L);\n\n// Seek to a timestamp (e.g., replay from 1 hour ago)\nlong oneHourAgo = System.currentTimeMillis() - Duration.ofHours(1).toMillis();\nMap<TopicPartition, Long> timestampsToSearch = new HashMap<>();\nfor (TopicPartition tp : consumer.assignment()) {\n    timestampsToSearch.put(tp, oneHourAgo);\n}\nMap<TopicPartition, OffsetAndTimestamp> offsets = consumer.offsetsForTimes(timestampsToSearch);\nfor (Map.Entry<TopicPartition, OffsetAndTimestamp> entry : offsets.entrySet()) {\n    if (entry.getValue() != null) {\n        consumer.seek(entry.getKey(), entry.getValue().offset());\n    } else {\n        // No records at or after the timestamp \u2014 seek to end\n        consumer.seekToEnd(List.of(entry.getKey()));\n    }\n}\n```\n\n---\n\n### Pattern 5: Multi-Threaded Processing with Pause/Resume\n\nThe consumer thread remains single-threaded; it dispatches to a worker pool and uses `pause()`/`resume()` to apply back-pressure.\n\n```java\nExecutorService workers = Executors.newFixedThreadPool(NUM_WORKERS);\nBlockingQueue<ConsumerRecord<String, String>> queue = new LinkedBlockingQueue<>(MAX_QUEUE_SIZE);\n\n// Worker thread: drain the queue and process\nRunnable worker = () -> {\n    while (!Thread.currentThread().isInterrupted()) {\n        try {\n            ConsumerRecord<String, String> record = queue.take();\n            processRecord(record);\n        } catch (InterruptedException e) {\n            Thread.currentThread().interrupt();\n        }\n    }\n};\nfor (int i = 0; i < NUM_WORKERS; i++) workers.submit(worker);\n\n// Consumer event loop (single thread)\nSet<TopicPartition> pausedPartitions = new HashSet<>();\n\nwhile (running) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n    for (ConsumerRecord<String, String> record : records) {\n        if (!queue.offer(record)) {\n            // Queue is full \u2014 pause this partition to prevent overrun.\n            TopicPartition tp = new TopicPartition(record.topic(), record.partition());\n            consumer.pause(List.of(tp));\n            pausedPartitions.add(tp);\n            // Block until we can enqueue (consumer loop is idle; heartbeats\n            // are NOT sent. Keep this brief or increase max.poll.interval.ms)\n            queue.put(record);\n        }\n    }\n\n    // Resume partitions when the queue has drained below threshold\n    if (!pausedPartitions.isEmpty() && queue.size() < MAX_QUEUE_SIZE / 2) {\n        consumer.resume(pausedPartitions);\n        pausedPartitions.clear();\n    }\n}\n```\n\n**Warning:** While blocked in `queue.put()`, the consumer is NOT calling `poll()`, so heartbeats are not sent. Keep the blocking time well within `max.poll.interval.ms`, or redesign to use `queue.offer()` with a timeout and skip records rather than blocking.\n\n---\n\n### Pattern 6: Exactly-Once with Transactions (EOS)\n\nTo achieve exactly-once semantics in a consume-transform-produce pipeline, use a `KafkaProducer` with transactions and fence commits to the consumer group generation:\n\n```java\nproducer.initTransactions();\n\nwhile (true) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(200));\n    if (records.isEmpty()) continue;\n\n    producer.beginTransaction();\n    try {\n        for (ConsumerRecord<String, String> record : records) {\n            ProducerRecord<String, String> output = transform(record);\n            producer.send(output);\n        }\n        // Commit consumer offsets within the transaction.\n        // groupMetadata() provides the generation/member ID for fencing.\n        Map<TopicPartition, OffsetAndMetadata> offsets = currentOffsets(records);\n        producer.sendOffsetsToTransaction(offsets, consumer.groupMetadata());\n        producer.commitTransaction();\n    } catch (ProducerFencedException | OutOfOrderSequenceException e) {\n        // Fatal \u2014 another producer has taken over; ## Overview and Threading Model

`KafkaConsumer<K, V>` is the primary client for consuming records from Apache Kafka topics. It resides in the `org.apache.kafka.clients.consumer` package and implements the `Consumer<K, V>` interface.

### Threading Model

**KafkaConsumer is NOT thread-safe.** All methods except `wakeup()` must be called from a single thread. Concurrent access from multiple threads will throw `ConcurrentModificationException`.

The intended usage model is an **event loop** running on a single dedicated thread:

```
single thread:
  subscribe / assign
  loop:
    poll()          ← fetches records AND drives heartbeats, rebalances, offset commits
