  "content": "# Terraform Plan/Apply Pipeline Architecture\n\n## Overview\n\nTerraform's plan/apply pipeline is the core execution engine that transforms infrastructure-as-code declarations into real infrastructure changes. The pipeline follows a multi-stage process: configuration parsing, resource graph construction, provider interaction, state reconciliation, and finally execution with comprehensive tracking and error handling.\n\nThe architecture emphasizes:\n- **Dependency-driven execution**: All operations follow a directed acyclic graph (DAG) of dependencies\n- **Provider abstraction**: Infrastructure operations are delegated to pluggable providers\n- **State-centric design**: All infrastructure decisions are based on comparing desired state (configuration) with actual state\n- **Extensibility**: Multiple hook points allow integration with external systems\n- **Safe execution**: Lock-based concurrency control and transaction-like semantics for state mutations\n\n---\n\n## 1. Graph Builder System\n\n### 1.1 Graph Construction Architecture\n\nThe Graph Builder System is responsible for transforming Terraform configuration into an executable dependency graph. This is a multi-phase process:\n\n```\nConfiguration \u2192 Raw Graph \u2192 Transformed Graph \u2192 Executable Graph\n                  (nodes)      (optimized)      (ready to walk)\n```\n\n#### Phase 1: Raw Graph Construction\nThe raw graph phase creates nodes for each resource, data source, and module in the configuration:\n\n- **Resource Nodes** (`ResourceNode`): Represent managed infrastructure resources defined in configuration\n  - Contain the resource configuration as an HCL2 body\n  - Store metadata about the resource type and address (e.g., `aws_instance.web`)\n  - Track dependencies declared via explicit `depends_on` blocks\n\n- **Data Source Nodes** (`DataSourceNode`): Represent data sources for reading external information\n  - Similar structure to resource nodes but read-only\n  - Used for querying existing infrastructure or external systems\n  - Must complete before any dependent resources can plan/apply\n\n- **Module Nodes** (`ModuleNode`): Represent module instantiations\n  - Create a namespace for contained resources\n  - May be instantiated multiple times via `count` or `for_each`\n  - Establish hierarchical naming conventions (e.g., `module.vpc.aws_subnet.private`)\n\n- **Provider Nodes** (`ProviderNode`): Represent provider configurations\n  - Store provider configuration blocks\n  - Track provider version constraints\n  - Establish which provider manages which resources\n\n- **Variable Nodes**: Represent input variables and local values\n  - Store variable definitions and default values\n  - Track which resources depend on variable values\n\n- **Root Node**: A synthetic node that depends on all root-level graph nodes\n  - Ensures proper sequencing of operations\n  - Allows the walker to start from a single entry point\n\n#### Phase 2: Graph Transformation\n\nGraph transformers modify the raw graph to handle special semantics and optimize execution:\n\n**Key Transformers:**\n\n1. **Count/ForEach Expander** (`DynamicExpand`)\n   - Expands resources with `count` or `for_each` into multiple concrete instances\n   - Transforms `aws_instance.web[*]` into `aws_instance.web[0]`, `aws_instance.web[1]`, etc.\n   - Creates dependencies between instances (implicit ordering)\n   - Runs multiple times during graph evaluation to handle dynamic dependencies\n\n2. **Dependency Mapper** (`ReferenceTransform`)\n   - Analyzes HCL expressions to discover implicit dependencies\n   - Detects references like `aws_instance.web.id` and creates corresponding edges\n   - Identifies both resource-to-resource and resource-to-data-source dependencies\n   - Handles transitive dependencies through local values and variables\n\n3. **Provider Injector** (`ProviderTransform`)\n   - Connects resource and data source nodes to their provider nodes\n   - Ensures providers are initialized before any resources they manage\n   - Handles implicit provider selection based on resource type prefix\n\n4. **Module Expander** (`ModuleExpand`)\n   - Expands module instantiations into their constituent resources\n   - Creates artificial module-output nodes for inter-module communication\n   - Manages the module output values graph\n   - Handles module dependencies and ordering\n\n5. **State Node Mapper** (`StateRefresh`)\n   - Creates auxiliary nodes for state-dependent operations\n   - Enables refreshing existing resources before planning changes\n   - Tracks which state values are required for planning\n\n6. **Orphan Resource Handler**\n   - Identifies resources that exist in state but not in configuration\n   - Creates destroy nodes for orphaned resources\n   - Ensures proper destruction ordering\n\n7. **Destroy Ordering** (`DestroyEdgeTransform`)\n   - For destroy operations, reverses and modifies the dependency graph\n   - Ensures resources are destroyed in reverse creation order\n   - Creates explicit destroy-time dependencies\n\n#### Phase 3: Graph Validation and Compilation\n\nAfter transformation, the graph is validated:\n\n- **Cycle Detection**: Ensures the graph is acyclic (no circular dependencies)\n- **Node Validation**: Checks that all nodes have required fields\n- **Edge Validation**: Verifies that all edges point to existing nodes\n- **Completeness Check**: Ensures all resources are reachable from the root node\n\nThe compiled graph is then ready for execution by the walker.\n\n### 1.2 Graph Node Representation\n\nEach node in the graph implements the `GraphNode` interface:\n\n```go\ntype GraphNode interface {\n    Name() string                              // Unique identifier\n    Dependencies() []string                    // Explicit dependencies\n}\n\ntype GraphNodeReferenceable interface {\n    // Types that can be referenced in expressions\n    ReferenceableAddrs() []addrs.Referenceable\n}\n\ntype GraphNodeEvalable interface {\n    // Nodes that can be evaluated (have side effects)\n    EvalBlock(ctx context.Context, ...) (*EvalResult, error)\n}\n```\n\n**Node Lifecycle:**\n\n1. **Creation**: Nodes are created during graph construction from configuration\n2. **Transformation**: Transformers modify nodes and graph structure\n3. **Evaluation**: Walker evaluates nodes in dependency order\n4. **Cleanup**: Resources are deallocated after graph evaluation completes\n\n### 1.3 Dependency Edge Types\n\nThe graph represents different types of dependencies with specific semantics:\n\n1. **Configuration Dependencies**\n   - Source: Explicit `depends_on` declarations\n   - Semantics: Hard dependency; dependent node cannot proceed until dependency completes\n   - Example: `depends_on = [aws_security_group.default]`\n\n2. **Reference Dependencies**\n   - Source: Implicit references in HCL expressions\n   - Semantics: Expression evaluation depends on referenced resource state\n   - Example: `security_group_ids = [aws_security_group.default.id]`\n\n3. **Module Dependencies**\n   - Source: Resource membership in modules\n   - Semantics: Module outputs depend on contained resources\n   - Example: Module declares `output \"subnet_ids\"`\n\n4. **Provider Dependencies**\n   - Source: Resource type prefixes map to providers\n   - Semantics: Resources depend on provider initialization\n   - Example: `aws_*` resources depend on `provider.aws`\n\n5. **State Dependencies**\n   - Source: Existing state records\n   - Semantics: In refresh operations, state determines read order\n   - Example: Reading data source before using its output\n\n---\n\n## 2. Provider Interface and Lifecycle\n\n### 2.1 Provider Plugin Architecture\n\nTerraform delegates all infrastructure operations to providers through a plugin interface. This architecture enables:\n\n- **Vendor independence**: Any cloud provider can implement the interface\n- **Version flexibility**: Providers can be developed and released independently\n- **Isolation**: Provider bugs or crashes don't crash Terraform core\n- **Security**: Providers run as separate processes with limited privileges\n\n#### Plugin Discovery and Loading\n\n**Discovery Phase:**\n1. Terraform searches standard plugin directories for provider binaries\n   - `.terraform/plugins/` (local cache)\n   - User's plugin directory (`~/.terraform.plugins/` on Unix)\n   - System-wide plugin directories\n\n2. Terraform verifies plugin signatures and checksums\n   - Ensures provider authenticity\n   - Prevents accidental loading of corrupted binaries\n\n3. Plugin metadata is extracted and cached\n   - Available resources and data sources\n   - Version information\n   - Schema definitions\n\n**Loading Phase:**\n1. The provider executable is launched as a subprocess\n2. RPC communication is established over stdin/stdout\n3. The provider declares its schema and capabilities\n4. The provider is added to the provider registry\n\n#### Provider Schema\n\nEach provider exposes a schema describing its capabilities:\n\n```go\ntype ProviderSchema struct {\n    Provider       *Schema          // Schema for provider config block\n    ResourceTypes  map[string]*Schema\n    DataSourceTypes map[string]*Schema\n}\n\ntype Schema struct {\n    Version      int64\n    Block        *configschema.Block  // HCL schema\n    ImpliedType  cty.Type             // Go type representation\n}\n```\n\nThe schema is used for:\n- **Configuration validation**: Ensuring user configuration matches the schema\n- **Type inference**: Determining types of resource attributes\n- **Documentation generation**: Building provider documentation\n- **Code completion**: Supporting IDE integrations\n\n### 2.2 Provider Configuration and Initialization\n\n#### Configuration Processing\n\nProvider configurations in Terraform code are processed in the following order:\n\n1. **Raw Configuration Parsing**\n   - HCL configuration blocks are parsed into a `configschema.Block` representation\n   - Configuration values are extracted as `cty.Value` objects\n\n2. **Variable Interpolation**\n   - Variables and local values referenced in provider configuration are evaluated\n   - Expression evaluation happens in the context of a scope containing:\n     - Declared variables and their values\n     - Local values\n     - Previously-evaluated data source outputs\n\n3. **Provider Aliasing**\n   - Provider blocks can have `alias` meta-arguments for multiple instances\n   - Resources select a provider via the `provider` meta-argument\n   - Default provider is selected based on resource type prefix\n\n4. **Configuration Validation**\n   - Each provider configuration is validated against the provider's schema\n   - Type mismatches and missing required fields are reported\n\n#### Provider Instance Management\n\n**Context Initialization:**\nEach provider instance is initialized with a context that includes:\n\n```go\ntype ProviderMeta struct {\n    Type          string              // e.g., \"aws\"\n    Configuration cty.Value           // Evaluated config\n    Features      map[string]bool      // Feature flags\n    Client        clients.Client      // RPC client to plugin process\n    Schema        *ProviderSchema     // Cached schema\n}\n```\n\n**Lifecycle:**\n1. **Configure**: Provider receives its configuration and initializes internal state\n   - Database connections established\n   - API credentials configured\n   - Regional endpoints set\n   - Returns either success or diagnostics explaining configuration problems\n\n2. **Active**: Provider is ready to handle resource operations\n   - Maintains connection to cloud APIs\n   - Caches frequently-used configuration\n   - Manages internal resource pools (connection pools, session tokens)\n\n3. **Stop**: Provider is shut down and resources cleaned up\n   - Network connections closed\n   - Cached credentials cleared\n   - Temporary files deleted\n   - Child processes terminated\n\n**Provider Lifecycle Hooks:**\n\nTerraform exposes provider lifecycle hooks that allow providers to customize initialization:\n\n- `ValidateConfig`: Provider validates user configuration before full initialization\n- `Configure`: Provider initializes with configuration values\n- `GetProviderSchema`: Plugin returns schema definitions\n- `Stop`: Provider performs cleanup on shutdown\n\n### 2.3 Resource Operation Delegation\n\nWhen Terraform needs to perform an operation on a resource, it delegates to the provider:\n\n#### Plan Phase Operations\n\n**For each resource in the graph:**\n\n1. **Read current state** (if resource exists in state)\n   - Provider queries the actual infrastructure\n   - Detects and reports any external changes\n   - Returns current attribute values\n\n2. **Plan changes**\n   - Provider compares configuration against current state\n   - Determines what changes are needed\n   - Returns a change summary specifying:\n     - `Create`: Resource doesn't exist, will be created\n     - `Update`: Resource exists, will be modified\n     - `Delete`: Resource exists, will be destroyed\n     - `NoOp`: No changes needed\n     - `DestroyThenCreate`: Resource must be replaced\n\n3. **Validate planned changes**\n   - Provider checks that the proposed changes are valid\n   - Validates that state transitions are possible\n   - Reports any incompatibilities\n\n**Change Representation:**\n\n```go\ntype ResourceChange struct {\n    Type         ChangeType          // Create, Update, Delete, etc.\n    Before       cty.Value          // State before change\n    After        cty.Value          // State after change\n    BeforeSensitive []bool          // Which fields contain sensitive data\n    AfterSensitive  []bool\n}\n```\n\n#### Apply Phase Operations\n\n**For each resource change in the graph:**\n\n1. **Create Operation**\n   - Provider creates the resource in the target infrastructure\n   - Configuration values are passed as the desired state\n   - Provider returns the actual created resource attributes\n   - State is updated with provider-assigned values (IDs, timestamps, etc.)\n\n2. **Update Operation**\n   - Provider modifies the existing resource\n   - Only changed attributes are passed to the provider\n   - Provider may reject updates that require replacement\n   - Updated attributes are returned and state is modified\n\n3. **Delete Operation**\n   - Provider destroys the resource\n   - Resource is removed from infrastructure\n   - Dependent resources can then be deleted\n\n4. **Replace Operation**\n   - Provider destroys the old resource (if deletable)\n   - Provider creates the new resource\n   - State is updated to remove old resource and add new one\n   - Atomic from Terraform's perspective (may not be atomic on provider side)\n\n**Error Handling in Operations:**\n\nIf a resource operation fails:\n- The resource state is rolled back to what it was before the operation\n- The error is reported with detailed diagnostics\n- Dependent operations are not attempted\n- User can retry the apply after fixing the underlying issue\n\n---\n\n## 3. State Management\n\n### 3.1 State Structure and Representation\n\nTerraform's state is the source of truth for the current infrastructure. It records:\n\n```go\ntype State struct {\n    Version      int                 // Schema version\n    TerraformVersion string          // Terraform version that created state\n    Serial       int64               // Increment on each modification\n    Lineage      string              // Unique ID for state lineage\n\n    RootModule   *ModuleState        // Root module resources\n}\n\ntype ModuleState struct {\n    Resources    map[string]*ResourceState\n    Outputs      map[string]*OutputState\n    ModuleCalls  map[string]*ModuleCallState\n}\n\ntype ResourceState struct {\n    Type         string              // Resource type (e.g., \"aws_instance\")\n    Primary      *InstanceState      // Main instance\n    Deposed      []*InstanceState    // Instances awaiting deletion\n}\n\ntype InstanceState struct {\n    ID           string              // Unique provider-assigned ID\n    Attributes   map[string]string   // All resource attributes\n    Meta         map[string]interface{} // Terraform metadata\n    Ephemeral    *EphemeralState     // Transient data (not persisted)\n}\n```\n\n**State Metadata:**\n- `Serial`: Version number incremented on each write; allows detection of concurrent modifications\n- `Lineage`: Unique identifier ensuring state files form a coherent history\n- `TerraformVersion`: Records which Terraform version last modified the state\n\n### 3.2 State Lifecycle\n\n#### State Initialization\n\n**At pipeline start:**\n\n1. **Workspace Selection**\n   - User specifies which workspace (default: \"default\")\n   - Terraform determines which state backend to use\n   - Multiple workspaces allow independent state for different environments\n\n2. **State Reading**\n   - Backend loads state from persistent storage (local file, S3, etc.)\n   - State is deserialized from JSON or other format\n   - Checksum is verified to detect corruption\n   - If state doesn't exist, an empty state is created\n\n3. **State Locking**\n   - If backend supports locking, a lock is acquired\n   - Lock prevents concurrent modifications\n   - Lock includes metadata: who locked it, when, and why\n   - Timeout prevents eternal locks from hung processes\n\n#### State Modification During Plan\n\nDuring planning, state is **not modified** but is used to:\n- Compare configuration against actual infrastructure state\n- Detect external changes to infrastructure\n- Calculate what changes need to be made\n\nA **snapshot** of state is created for:\n- Evaluating resource references\n- Detecting state changes from external sources\n- Calculating proposed changes\n\n#### State Modification During Apply\n\n**For each successful resource operation:**\n\n1. **Pre-apply state**: Resource state before the operation\n2. **Operation execution**: Provider performs the actual change\n3. **Post-apply state**: Provider returns the new resource attributes\n4. **State update**: State is modified to record the new attributes\n5. **State persistence**: Updated state is written back to storage\n\n**State Update Semantics:**\n\n```go\nfunc (s *State) SetResourceInstanceCurrent(\n    addr addrs.AbsResourceInstance,\n    obj *states.ResourceInstanceObject,\n    provider addrs.AbsProviderConfig,\n) {\n    // 1. Either all planned changes apply successfully\n  - Or state is left in a consistent state for retry\n\n### 3.3 State Persistence and Backends\n\n#### Backend Interface\n\nBackends implement the state storage protocol:\n\n```go\ntype Backend interface {\n    // Acquire exclusive access to state\n    Lock(ctx context.Context, metadata *LockMetadata) (string, error)\n    Unlock(lockID string) error\n\n    // State operations\n    StateMgr(workspace string) (StateManager, error)\n    DeleteWorkspace(workspace string) error\n    Workspaces() ([]string, error)\n}\n\ntype StateManager interface {\n    // Persistent storage operations\n    State() *State\n    WriteState(*State) error\n    RefreshState() error\n\n    // Enables detection of concurrent modifications\n    PersistentMeta() *PersistentMeta\n}\n```\n\n#### Backend Types\n\n1. **Local Backend** (file-based)\n   - Stores state in a local file (default: `terraform.tfstate`)\n   - Locking via local filesystem (Unix: flock)\n   - Suitable for single-user, local development\n   - Risk: File corruption, accidental deletion\n\n2. **Remote Backends** (cloud storage)\n   - Supports concurrent access with distributed locking\n   - Enables team collaboration\n   - Provides state versioning and audit logs\n   - May enforce policy and cost controls\n\n3. **Enhanced Backends**\n   - Terraform Cloud/Enterprise\n   - Provides remote execution (runs happen on cloud servers)\n   - Enables plan/apply in a standardized environment\n   - Integrates with VCS for plan on pull requests\n\n#### State Persistence Flow\n\n**Writing State:**\n```\nUpdated State (in-memory)\n  \u2192 Serialize to JSON\n  \u2192 Encrypt (if configured)\n  \u2192 Write to backend\n  \u2192 Update local cache\n  \u2192 Increment serial number\n```\n\n**Reading State:**\n```\nBackend storage\n  \u2192 Decrypt (if encrypted)\n  \u2192 Deserialize from JSON\n  \u2192 Validate structure\n  \u2192 Store in-memory representation\n```\n\n#### State Synchronization\n\nTerraform ensures state consistency across concurrent operations:\n\n**Serial Number Tracking:**\n- Each state modification increments the serial number\n- Before writing state, current serial is checked\n- If serial has changed, state was modified externally\n- User is prompted to handle the conflict (refresh/force)\n\n**Lock Files:**\n```json\n{\n  \"ID\": \"unique-lock-id\",\n  \"Operation\": \"apply\",\n  \"Info\": \"terraform apply 1.0.0\",\n  \"Who\": \"user@hostname\",\n  \"Version\": \"1.0.0\",\n  \"Created\": \"2024-01-15T10:30:00Z\",\n  \"Path\": \"module.vpc\"\n}\n```\n\n### 3.4 State Snapshots for Evaluation\n\nDuring graph evaluation, state snapshots enable:\n\n**Consistent View:**\n- Resources reference the state snapshot, not live state\n- Avoids issues where earlier resources modify state that later resources need\n- Ensures deterministic evaluation\n\n**Filtering and Scoping:**\n- Different graph evaluations use different state snapshots\n- `terraform plan` uses one snapshot\n- `terraform apply` uses the pre-apply snapshot for planning\n- Refresh operations create updated snapshots\n\n**Example Flow:**\n```\n1. Evaluation\n\n```\nterraform plan evaluation:\n  \u251c\u2500 Initialize providers\n  \u2502  \u251c\u2500 Load provider schema\n  \u2502  \u251c\u2500 Validate provider config\n  \u2502  \u251c\u2500 Establish provider connection\n  \u2502  \u2514\u2500 Configure each provider instance\n  \u251c\u2500 Walk the graph in dependency order\n  \u2502  \u251c\u2500 For each node:\n  \u2502  \u2502  \u251c\u2500 Evaluate data sources\n  \u2502  \u2502  \u251c\u2500 Refresh resource state (optional)\n  \u2502  \u2502  \u251c\u2500 Compare configuration to state\n  \u2502  \u2502  \u251c\u2500 Determine required changes\n  \u2502  \u2502  \u251c\u2500 Validate compatibility\n  \u2502  \u2502  \u2514\u2500 Record in plan\n  \u2502  \u2514\u2500 Handle errors and skipped nodes\n  \u2514\u2500 Aggregate plan summary\n     \u251c\u2500 Count by operation type\n     \u251c\u2500 Identify affected resources\n     \u2514\u2500 Estimate costs (if provider supports)\n```\n\n**State**: Plan object contains all proposed changes\n\n#### Phase 5: Apply Evaluation\n\n```\nterraform apply execution:\n  \u251c\u2500 Load plan (if using saved plan)\n  \u251c\u2500 Confirm with user (if interactive)\n  \u251c\u2500 Acquire state lock\n  \u251c\u2500 Reload state (may have changed since plan)\n  \u251c\u2500 Walk graph executing planned changes\n  \u2502  \u251c\u2500 For each planned change:\n  \u2502  \u2502  \u251c\u2500 Acquire resource-level lock (if applicable)\n  \u2502  \u2502  \u251c\u2500 Execute provider operation\n  \u2502  \u2502  \u251c\u2500 Update state with result\n  \u2502  \u2502  \u251c\u2500 Release resource-level lock\n  \u2502  \u2502  \u2514\u2500 Execute post-operation hooks\n  \u2502  \u251c\u2500 On error: rollback and stop\n  \u2502  \u2514\u2500 On completion: record final state\n  \u251c\u2500 Release state lock\n  \u2514\u2500 Output final state and summaries\n```\n\n**State**: Infrastructure changes are applied and state is updated\n\n### 4.2 Graph Walker Implementation\n\nThe **Walker** is responsible for traversing the dependency graph and evaluating nodes:\n\n```go\ntype Walker struct {\n    Graph         *Graph\n    Evaluator     Evaluator\n    Hooks         []Hook\n    ErrorFunc     ErrorFunc\n}\n\nfunc (w *Walker) Walk(ctx context.Context) error {\n    // 1. Topologically sort graph nodes\n    sorted := w.Graph.TopologicalSort()\n\n    // 2. For each node in dependency order:\n    for _, node := range sorted {\n        // 3. Evaluate node (may be concurrent)\n        result, err := w.Evaluator.Eval(node)\n\n        // 4. Call hooks before and after evaluation\n        w.executeHooks(HookBeforeEval, node)\n        w.executeHooks(HookAfterEval, node, result, err)\n\n        // 5. Handle errors\n        if err != nil {\n            return w.ErrorFunc(node, err)\n        }\n    }\n\n    return nil\n}\n```\n\n#### Topological Sorting\n\nThe walker must visit nodes in an order that respects dependencies:\n\n**Algorithm:**\n1. Start with nodes that have no dependencies (root node)\n2. Add each node whose dependencies are satisfied\n3. Continue until all nodes are visited\n4. Return error if cycles are detected\n\n**Result Properties:**\n- Nodes are sorted so dependencies precede dependents\n- Multiple valid sort orders may exist (DAG property)\n- Walker can execute nodes in parallel if dependencies allow\n\n#### Concurrent Evaluation\n\nThe walker can evaluate multiple nodes concurrently if their dependencies don't overlap:\n\n```\nGraph structure:\n  root\n    \u251c\u2500 provider.aws (1)\n    \u251c\u2500 aws_vpc.main (1)\n    \u251c\u2500 aws_subnet.a (2) [depends on vpc]\n    \u2514\u2500 aws_subnet.b (2) [depends on vpc]\n\nEvaluation order:\n  Step 1: Evaluate provider (sequential)\n  Step 2: Evaluate vpc (sequential)\n  Step 3: Evaluate subnet.a AND subnet.b (parallel)\n```\n\n**Concurrency Limits:**\n- Default: Unlimited parallelism (respecting dependencies)\n- Can be limited with `-parallelism=N` flag\n- Prevents overwhelming remote APIs or provider plugin processes\n\n#### Node Evaluation\n\nFor each node, the evaluator executes a series of steps:\n\n**For Resource Nodes (Plan):**\n```go\nfunc (e *Evaluator) EvalResource(node *ResourceNode) {\n    1. Fetch current state (if exists)\n    2. Call provider RefreshResourceState()\n    3. Call provider PlanResourceChange()\n    4. Validate proposed changes\n    5. Record plan for this resource\n}\n```\n\n**For Resource Nodes (Apply):**\n```go\nfunc (e *Evaluator) EvalResource(node *ResourceNode) {\n    1. Fetch planned changes for this resource\n    2. Call provider ApplyResourceChange()\n    3. Update state with result\n    4. Handle error (rollback state)\n    5. Record apply result\n}\n```\n\n**For Data Source Nodes:**\n```go\nfunc (e *Evaluator) EvalDataSource(node *DataSourceNode) {\n    1. Evaluate data source configuration\n    2. Call provider ReadDataSource()\n    3. Store result in state (temporary)\n    4. Make available for reference by dependent resources\n}\n```\n\n### 4.3 Hook System\n\nTerraform's hook system allows observation and customization of execution:\n\n#### Hook Interface\n\n```go\ntype Hook interface {\n    PreApply(metadata *OperationMetadata) HookAction\n    PostApply(metadata *OperationMetadata, result *OperationResult) HookAction\n\n    PrePlan(metadata *OperationMetadata) HookAction\n    PostPlan(metadata *OperationMetadata, result *PlanResult) HookAction\n\n    // And many others...\n}\n\ntype HookAction int\nconst (\n    HookActionContinue HookAction = iota\n    HookActionStop     // Halt execution\n)\n```\n\n#### Hook Points\n\nHooks are invoked at these key execution points:\n\n1. **Initialization Hooks**\n   - `OnInit`: Terraform initializes\n   - `OnInitProvider`: Provider is initialized\n   - `OnInitModule`: Module is loaded\n\n2. **Pre-Operation Hooks**\n   - `PreApply`: Before applying a resource\n   - `PrePlan`: Before planning a resource\n   - `PreRefresh`: Before refreshing state\n   - `PreValidate`: Before validation\n\n3. **Post-Operation Hooks**\n   - `PostApply`: After applying a resource (success or failure)\n   - `PostPlan`: After planning a resource\n   - `PostRefresh`: After refreshing state\n   - `PostValidate`: After validation\n\n4. **Error Hooks**\n   - `OnError`: Error during any operation\n   - `OnApplyError`: Error during apply\n   - `OnPlanError`: Error during plan\n\n#### Built-in Hooks\n\nTerraform includes several built-in hooks:\n\n**UI Hook:**\n- Displays plan and apply progress to the user\n- Shows resource creation/modification/deletion operations\n- Provides real-time feedback\n\n**Logging Hook:**\n- Records all operations to log file\n- Includes timing information\n- Useful for debugging and auditing\n\n**State Hook:**\n- Persists state after each successful operation\n- Ensures state is saved even if apply fails partway through\n\n#### Custom Hook Usage\n\nExternal tools can implement hooks for:\n- **Auditing**: Record all infrastructure changes\n- **Policy Enforcement**: Block changes that violate policies\n- **Monitoring**: Alert external systems about changes\n- **Workflow Integration**: Integrate with CI/CD pipelines\n\n**Example Hook Implementation:**\n```go\ntype PolicyHook struct {\n    policies []Policy\n}\n\nfunc (h *PolicyHook) PreApply(meta *OperationMetadata) HookAction {\n    change := meta.Change\n    for _, policy := range h.policies {\n        if !policy.Evaluate(change) {\n            log.Printf(\"Policy violation: %s\", policy.Name)\n            return HookActionStop\n        }\n    }\n    return HookActionContinue\n}\n```\n\n### 4.4 Dynamic Expansion\n\nResources with `count` or `for_each` meta-arguments are expanded dynamically:\n\n#### Count Meta-Argument\n\n```hcl\nresource \"aws_instance\" \"web\" {\n  count = 3\n  ami   = \"ami-12345\"\n}\n```\n\nExpands to:\n```\naws_instance.web[0]\naws_instance.web[1]\naws_instance.web[2]\n```\n\n**Expansion Process:**\n1. Detect `count` expression in configuration\n2. Evaluate count expression to determine count\n3. Create N resource instance nodes\n4. Propagate dependencies to all N instances\n\n**Dependencies Between Instances:**\n- Instance N implicitly depends on instance N-1\n- Ensures sequential creation and deletion\n- Can be overridden with explicit `depends_on`\n\n#### ForEach Meta-Argument\n\n```hcl\nresource \"aws_subnet\" \"private\" {\n  for_each = {\n    \"az-a\" = \"subnet-az-a\"\n    \"az-b\" = \"subnet-az-b\"\n  }\n\n  availability_zone = each.key\n  cidr_block       = each.value\n}\n```\n\nExpands to:\n```\naws_subnet.private[\"az-a\"]\naws_subnet.private[\"az-b\"]\n```\n\n**Key Properties:**\n- Each instance is independent (no implicit dependencies)\n- Instances can be created/destroyed independently\n- Map key is available as `each.key`\n- Map value is available as `each.value`\n\n#### Dynamic Expansion Timing\n\nDynamic expansion happens **during graph evaluation**, not during graph construction:\n\n**Reasons:**\n1. Count/ForEach expressions may depend on resource outputs\n2. Resource outputs are only available during evaluation\n3. Expansion must handle circular dependencies\n\n**Process:**\n```\nGraph construction \u2192 Graph evaluation starts\n  \u251c\u2500 Evaluate nodes with static structure\n  \u251c\u2500 Detect count/for_each expressions\n  \u251c\u2500 Evaluate count/for_each expressions\n  \u251c\u2500 Create expanded instances\n  \u251c\u2500 Re-evaluate graph with expanded nodes\n  \u2514\u2500 Continue until no more expansions\n```\n\n#### Impact on State\n\nDynamic expansion affects state management:\n\n**State Tracking:**\n- State tracks each instance independently\n- `aws_instance.web[0]` and `aws_instance.web[1]` have separate state records\n- Changing count from 3 to 2 marks the 3rd instance for deletion\n\n**Scale Changes:**\n```\nPlan to change count from 3 to 4:\n  - Instance 0: No change (already exists)\n  - Instance 1: No change (already exists)\n  - Instance 2: No change (already exists)\n  - Instance 3: Create (new)\n\nPlan to change count from 4 to 2:\n  - Instance 0: No change\n  - Instance 1: No change\n  - Instance 2: Destroy\n  - Instance 3: Destroy\n```\n\n---\n\n## 5. Error Handling and Recovery\n\n### 5.1 Error Propagation\n\nErrors during execution are propagated through the evaluation stack:\n\n**Error Types:**\n\n1. **Configuration Errors**\n   - Syntax errors in HCL files\n   - Invalid variable references\n   - Type mismatches\n   - **Handling**: Reported immediately; execution stops\n\n2. **Validation Errors**\n   - Schema validation failures\n   - Invalid resource configurations\n   - Constraint violations\n   - **Handling**: Reported in plan; apply doesn't proceed\n\n3. **Provider Errors**\n   - Resource creation failed\n   - API errors from cloud provider\n   - Invalid provider configuration\n   - **Handling**: Depends on error type and timing\n\n4. **State Errors**\n   - State lock conflicts\n   - Corrupted state file\n   - Concurrent modification conflicts\n   - **Handling**: User intervention required\n\n### 5.2 Apply-Time Error Recovery\n\nWhen an error occurs during apply:\n\n**State Management:**\n```\nPre-apply state: A\nApply operation 1: Success \u2192 State B\nApply operation 2: Failure (error) \u2192 State rolls back to B\nApply operation 3: Never executed\nFinal state: B (consistent with successful operations)\n```\n\n**User Options:**\n1. **Retry**: Fix the issue and run apply again\n2. **Skip**: Remove the failing resource and replan\n3. **Manual Fix**: Manually fix the infrastructure and refresh state\n\n### 5.3 Locking and Concurrency Control\n\n#### State Locking\n\nState locks prevent concurrent modifications:\n\n**Lock Acquisition:**\n```\nterraform apply\n  \u251c\u2500 Request state lock from backend\n  \u251c\u2500 Wait for lock availability (with timeout)\n  \u251c\u2500 Verify lock metadata matches expected\n  \u2514\u2500 Acquire lock with metadata\n```\n\n**Lock Information:**\n```json\n{\n  \"ID\": \"unique-id-uuid4\",\n  \"Operation\": \"apply\",\n  \"Info\": \"terraform apply -auto-approve\",\n  \"Who\": \"username\",\n  \"Version\": \"1.5.0\",\n  \"Created\": \"2024-01-15T10:30:00Z\",\n  \"Path\": \"staging/us-west-2\"\n}\n```\n\n**Lock Release:**\n```\nNormally:   After apply completes (success or failure)\nOn timeout: Forcibly released with warning\nOn SIGINT:  Gracefully released after cleanup\n```\n\n**Detecting Stale Locks:**\n- Lock timeout: If lock exceeds max age (e.g., 24 hours)\n- Process detection: Check if process holding lock is alive\n- Force unlock: User can forcibly release lock with `-force-unlock`\n\n#### Resource-Level Locking\n\nSome backends support resource-level locks:\n\n**Use Cases:**\n- Preventing concurrent modifications to specific resources\n- Enabling finer-grained concurrency control\n- Supporting partial apply (apply specific resources)\n\n**Example:**\n```\nterraform apply -target=aws_instance.web\n\nLocks:\n  - State lock: Acquired\n  - aws_instance.web: Acquired (resource-specific)\n  - Other resources: Not locked (can be modified separately)\n```\n\n---\n\n## 6. Extension Points and Customization\n\n### 6.1 Provider Extension Points\n\nProviders can extend Terraform's behavior through:\n\n1. **Custom Variable Types**\n   - Providers can define custom types for validation\n   - # Terraform Plan/Apply Pipeline Architecture

## Overview

Terraform's plan/apply pipeline is the core execution engine that transforms infrastructure-as-code declarations into real infrastructure changes. The pipeline follows a multi-stage process: configuration parsing, resource graph construction, provider interaction, state reconciliation, and finally execution with comprehensive tracking and error handling.

The architecture emphasizes:
- **Dependency-driven execution**: All operations follow a directed acyclic graph (DAG) of dependencies
- **Provider abstraction**: Infrastructure operations are delegated to pluggable providers
- **State-centric design**: All infrastructure decisions are based on comparing desired state (configuration) with actual state
- **Extensibility**: Multiple hook points allow integration with external systems
- **Safe execution**: Lock-based concurrency control and transaction-like semantics for state mutations

---
