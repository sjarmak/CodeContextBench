They can be associated with quick fixes through the Code Actions API.\n\n## Core Types\n\n### DiagnosticSeverity\n\nEnum representing the severity level of a diagnostic.\n\n**Signature:**\n```typescript\nexport enum DiagnosticSeverity {\n  Error = 0,\n  Warning = 1,\n  Information = 2,\n  Hint = 3\n}\n```\n\n**Members:**\n\n| Member | Value | Description |\n|--------|-------|-------------|\n| `Error` | 0 | Something not allowed by the rules of a language or other means. Used for errors that prevent compilation or execution. |\n| `Warning` | 1 | Something suspicious but allowed. Typically used for issues that won't prevent compilation but should be reviewed. |\n| `Information` | 2 | Something to inform about but not a problem. General informational messages. |\n| `Hint` | 3 | Something to hint at a better way of doing it, like proposing a refactoring. Used for code improvement suggestions. |\n\n**Behavior Notes:**\n- Default severity when creating a `Diagnostic` without specifying severity is `Error`\n- Severity levels are used by VS Code for visualization (color, icon) and filtering in the Problems panel\n- Filtering priority when too many diagnostics are present: diagnostics are kept in order of severity (Error > Warning > Information > Hint)\n\n### DiagnosticTag\n\nEnum representing additional metadata about the type of a diagnostic.\n\n**Signature:**\n```typescript\nexport enum DiagnosticTag {\n  Unnecessary = 1,\n  Deprecated = 2\n}\n```\n\n**Members:**\n\n| Member | Value | Description |\n|--------|-------|-------------|\n| `Unnecessary` | 1 | Unused or unnecessary code. Diagnostics with this tag are rendered faded out. The opacity is controlled by the `\"editorUnnecessaryCode.opacity\"` theme color (e.g., `\"editorUnnecessaryCode.opacity\": \"#000000c0\"` renders at 75% opacity). For high contrast themes, the `\"editorUnnecessaryCode.border\"` theme color underlines unnecessary code instead. |\n| `Deprecated` | 2 | Deprecated or obsolete code. Diagnostics with this tag are rendered with a strike-through. |\n\n**Behavior Notes:**\n- Tags are optional and can be combined in an array\n- Tags provide visual feedback to users about the nature of the issue beyond just severity\n- Tags affect rendering but not the core behavior or filtering of diagnostics\n\n### DiagnosticRelatedInformation\n\nClass representing additional context for a diagnostic, pointing to related code locations that caused or are related to the diagnostic.\n\n**Signature:**\n```typescript\nexport class DiagnosticRelatedInformation {\n  constructor(location: Location, message: string);\n  location: Location;\n  message: string;\n}\n```\n\n**Properties:**\n\n| Property | Type | Description |\n|----------|------|-------------|\n| `location` | `Location` | The location of the related diagnostic information. Contains a URI and a Range pointing to the relevant code location. |\n| `message` | `string` | The human-readable message describing the related information. |\n\n**Behavior Notes:**\n- Used to point to code locations that cause or relate to a diagnostic (e.g., when symbol names collide, all definitions can be marked)\n- Can be accessed by clicking on related information markers in the editor\n- Multiple related information items can be associated with a single diagnostic\n- Related information is particularly useful for complex diagnostics with multiple contributing factors\n\n**Example:**\n```typescript\n// Pointing to duplicate symbol definitions\ndiagnostic.relatedInformation = [\n  new vscode.DiagnosticRelatedInformation(\n    new vscode.Location(uri, new vscode.Range(5, 0, 5, 10)),\n    'First declaration of symbol'\n  ),\n  new vscode.DiagnosticRelatedInformation(\n    new vscode.Location(uri, new vscode.Range(15, 0, 15, 10)),\n    'Second declaration of symbol'\n  )\n];\n```\n\n### Diagnostic\n\nClass representing a diagnostic, such as a compiler error or warning. Diagnostic objects are only valid in the scope of a file.\n\n**Signature:**\n```typescript\nexport class Diagnostic {\n  constructor(range: Range, message: string, severity?: DiagnosticSeverity);\n  range: Range;\n  message: string;\n  severity: DiagnosticSeverity;\n  source?: string;\n  code?: string | number | { value: string | number; target: Uri };\n  relatedInformation?: DiagnosticRelatedInformation[];\n  tags?: DiagnosticTag[];\n}\n```\n\n**Properties:**\n\n| Property | Type | Required | Description |\n|----------|------|----------|-------------|\n| `range` | `Range` | Yes | The range in the document to which this diagnostic applies. Diagnostics are always scoped to a collection and a resource.\n\n**Signature:**\n```typescript\nexport interface DiagnosticCollection extends Iterable<[uri: Uri, diagnostics: readonly Diagnostic[]]> {\n  readonly name: string;\n  set(uri: Uri, diagnostics: readonly Diagnostic[] | undefined): void;\n  set(entries: ReadonlyArray<[Uri, readonly Diagnostic[] | undefined]>): void;\n  delete(uri: Uri): void;\n  clear(): void;\n  forEach(callback: (uri: Uri, diagnostics: readonly Diagnostic[], collection: DiagnosticCollection) => any, thisArg?: any): void;\n  get(uri: Uri): readonly Diagnostic[] | undefined;\n  has(uri: Uri): boolean;\n  dispose(): void;\n}\n```\n\n**Properties:**\n\n| Property | Type | Description |\n|----------|------|-------------|\n| `name` | `string` (readonly) | The name of this diagnostic collection (e.g., 'typescript', 'eslint'). Every diagnostic from this collection is associated with this name. The name is used by the task framework when defining problem matchers. |\n\n**Methods:**\n\n#### set(uri: Uri, diagnostics: readonly Diagnostic[] | undefined): void\n**Description:** Assign diagnostics for the given resource. Will replace existing diagnostics for that resource.\n\n**Parameters:**\n- `uri: Uri` - A resource identifier (URI).\n- `diagnostics: readonly Diagnostic[] | undefined` - Array of diagnostics or `undefined` to clear diagnostics for this URI.\n\n**Returns:** void\n\n**Behavior Notes:**\n- Replaces all previous diagnostics for the given URI\n- Passing `undefined` is equivalent to calling `delete(uri)`\n- Fires the `onDidChangeDiagnostics` event after setting diagnostics\n- Setting an empty array effectively clears diagnostics for the URI\n\n#### set(entries: ReadonlyArray<[Uri, readonly Diagnostic[] | undefined]>): void\n**Description:** Replace diagnostics for multiple resources in this collection.\n\n**Parameters:**\n- `entries: ReadonlyArray<[Uri, readonly Diagnostic[] | undefined]>` - Array of tuples like `[[file1, [d1, d2]], [file2, [d3, d4]]]`, or `undefined`.\n\n**Returns:** void\n\n**Behavior Notes:**\n- Multiple tuples of the same URI are merged (e.g., `[[file1, [d1]], [file1, [d2]]]` is equivalent to `[[file1, [d1, d2]]]`)\n- If a diagnostics item is `undefined` as in `[file1, undefined]`, all previous diagnostics for that file are removed (but not subsequent ones)\n- More efficient than calling `set(uri, diagnostics)` multiple times as it batches updates\n- Fires a single `onDidChangeDiagnostics` event for all changed URIs\n- Results in a single synchronization with the main thread\n\n**Example:**\n```typescript\ncollection.set([\n  [uri1, [diag1, diag2]],\n  [uri2, [diag3]],\n  [uri3, undefined] // Clear diagnostics for uri3\n]);\n```\n\n#### delete(uri: Uri): void\n**Description:** Remove all diagnostics from this collection that belong to the provided URI. Equivalent to `set(uri, undefined)`.\n\n**Parameters:**\n- `uri: Uri` - A resource identifier.\n\n**Returns:** void\n\n**Behavior Notes:**\n- Removes all diagnostics for the given URI\n- Fires the `onDidChangeDiagnostics` event\n- If the URI has no diagnostics, this is a no-op\n\n#### clear(): void\n**Description:** Remove all diagnostics from this collection. Equivalent to calling `set(undefined)`.\n\n**Returns:** void\n\n**Behavior Notes:**\n- Removes diagnostics for all resources in the collection\n- Fires the `onDidChangeDiagnostics` event with all previously diagnosed URIs\n- After clearing, `forEach` will not iterate any entries\n- Useful for bulk cleanup when an extension no longer needs to report diagnostics\n\n#### forEach(callback: (uri: Uri, diagnostics: readonly Diagnostic[], collection: DiagnosticCollection) => any, thisArg?: any): void\n**Description:** Iterate over each entry in this collection.\n\n**Parameters:**\n- `callback: (uri: Uri, diagnostics: readonly Diagnostic[], collection: DiagnosticCollection) => any` - Function to execute for each entry. Receives the URI, the diagnostics array for that URI, and the collection itself.\n- `thisArg?: any` - Optional `this` context used when invoking the callback function.\n\n**Returns:** void\n\n**Behavior Notes:**\n- The diagnostics array passed to the callback is immutable (readonly)\n- Iteration order is not guaranteed\n- If the collection is modified during iteration, behavior is undefined\n- Useful for debugging or when you need to access all diagnostics at once\n\n**Example:**\n```typescript\ncollection.forEach((uri, diagnostics) => {\n  console.log(`${uri.fsPath}: ${diagnostics.length} diagnostics`);\n});\n```\n\n#### get(uri: Uri): readonly Diagnostic[] | undefined\n**Description:** Get the diagnostics for a given resource. Note that you cannot modify the diagnostics array returned from this call.\n\n**Parameters:**\n- `uri: Uri` - A resource identifier.\n\n**Returns:** `readonly Diagnostic[] | undefined` - An immutable array of diagnostics or `undefined` if no diagnostics exist for this URI.\n\n**Behavior Notes:**\n- Returns an immutable copy of the diagnostics array\n- Returns `undefined` (not an empty array) if the URI has no diagnostics\n- The returned array cannot be modified directly\n- Changes to diagnostics must be done through `set()` or `delete()`\n\n**Example:**\n```typescript\nconst diagnostics = collection.get(uri);\nif (diagnostics) {\n  console.log(`Found ${diagnostics.length} diagnostics`);\n}\n```\n\n#### has(uri: Uri): boolean\n**Description:** Check if this collection contains diagnostics for a given resource.\n\n**Parameters:**\n- `uri: Uri` - A resource identifier.\n\n**Returns:** `boolean` - `true` if this collection has diagnostics for the given resource, `false` otherwise.\n\n**Behavior Notes:**\n- Returns `true` only if `get(uri)` would return a non-undefined value\n- More efficient than calling `get(uri) !== undefined`\n\n#### dispose(): void\n**Description:** Dispose and free associated resources. Calls `clear()` internally.\n\n**Returns:** void\n\n**Behavior Notes:**\n- Should be called when the extension no longer needs the collection\n- Automatically clears all diagnostics\n- After disposal, all methods will throw an error when called\n- Collections passed to `context.subscriptions.push()` are automatically disposed when the extension is deactivated\n- Failing to dispose can lead to memory leaks and continued reporting of diagnostics\n\n**Example:**\n```typescript\nconst collection = vscode.languages.createDiagnosticCollection('my-extension');\n// ... use collection ...\ncollection.dispose(); // Clean up resources\n```\n\n## Functions\n\n### createDiagnosticCollection(name?: string): DiagnosticCollection\n\n**Description:** Create a new diagnostic collection.\n\n**Signature:**\n```typescript\nexport function createDiagnosticCollection(name?: string): DiagnosticCollection;\n```\n\n**Parameters:**\n\n| Parameter | Type | Required | Description |\n|-----------|------|----------|-------------|\n| `name` | `string?` | No | The name of the collection. # VS Code Diagnostic API Reference

## Overview

The Diagnostic API provides a way for VS Code extensions to report code issues (errors, warnings, information, hints) to users within the editor. The API allows extensions to:

- Create diagnostic collections to organize diagnostics by source (e.g., 'typescript', 'eslint')
- Report diagnostics for specific files or documents
- Track and manage diagnostics across the workspace
- Subscribe to diagnostic changes globally
- Provide detailed diagnostic information with related locations and additional metadata

Diagnostics are displayed in the editor gutter, problems panel, and inline as squiggles. They can be associated with quick fixes through the Code Actions API.
