## Bug Report

**Bug ID**: 207835
**Summary**: ACPI video backlight brightness control does not work on Acer TravelMate 5735Z by default

On the Intel GM45/GMA 4500MHD laptop with Debian Sid/unstable and Linux 5.6.7 and GNOME Shell 3.36.2, trying to change the brightness of the internal display using the function keys, GNOME shows the OSD, which seems to have five levels (from acer_wmi?), but the actual brightness does *not* change. There is only `/sys/class/backlight/intel_backlight/brightness` though, and writing values to it seems to work.

Booting the system with `acpi_backlight=vendor`, exposes `/sys/devices/platform/acer-wmi/backlight/acer-wmi/brightness`, but the behavior is the same.

Booting the system with `acpi_backlight=native` or `acpi_backlight=native`, changing the brightness with the function keys works, and there now seem to be 15 (or 16) levels.

Based on my analysis, I can identify the bug. The issue is in `drivers/acpi/video_detect.c`. The `acpi_video_get_backlight_type()` function determines which backlight interface to use, and for the Acer TravelMate 5735Z, it falls through to `acpi_backlight_video` (the ACPI video interface) because:

1. The BIOS is from 2011, so it's not Win8-ready (`acpi_osi_is_win8()` returns false)
2. The ACPI video backlight capability is detected
3. There's no DMI quirk entry for this specific machine

The ACPI video backlight doesn't actually work on this machine, but the native interface does. The fix would be adding a DMI entry to `video_detect_dmi_table` to force `acpi_backlight_native` for the Acer TravelMate 5735Z.

## Findings

**File**: `drivers/acpi/video_detect.c`

**Data Structure**: `video_detect_dmi_table`

**Root Cause**: The `video_detect_dmi_table` array is missing a DMI quirk entry for the Acer TravelMate 5735Z. 

When `acpi_video_get_backlight_type()` runs:
1. No cmdline override is present (default boot)
2. No DMI quirk matches this machine
3. ACPI video backlight capability IS detected (`ACPI_VIDEO_BACKLIGHT` is set)
4. The BIOS is from 2011 (not Win8-ready), so `acpi_osi_is_win8()` returns false
5. Result: falls back to `acpi_backlight_video` — but this doesn't work on this hardware

**Fix needed**: Add a DMI entry to `video_detect_dmi_table` with `video_detect_force_native` callback matching:
- `DMI_SYS_VENDOR`: "Acer"  
- `DMI_PRODUCT_NAME`: "TravelMate 5735Z"

This is the same pattern used for other pre-Win8 machines that need native backlight (like Lenovo Ideapad S405, Dell Vostro V131, etc.), listed under the "Non win8 machines which need native backlight nevertheless" comment in the file.
