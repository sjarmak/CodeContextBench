**Reported by:** Frontend Integration Team
**Severity:** High
**Ticket:** FORMS-2847

### Reproduction

A form field with a widget configured to include an attribute like `data-label='Value with "quotes"'` renders the quotes unescaped in the final HTML, breaking the DOM structure.

# Fix for Form Field Rendering with Custom Widget Attributes (FORMS-2847)

## Problem
When a form field widget specifies custom HTML attributes with special characters (quotes, ampersands, etc.), the attribute values were not being properly escaped in the rendered HTML, resulting in malformed DOM structures.

Example:
```python
widget = TextInput(attrs={'data-label': 'Value with "quotes"'})
# Before fix: <input ... data-label="Value with "quotes"">  ❌ Invalid HTML
# After fix:  <input ... data-label="Value with &quot;quotes&quot;">  ✓ Valid HTML
```

## Root Cause
The form templates were using the `stringformat` filter to render attribute values. This filter was marked with `is_safe=True` in Django's default filters, which bypassed HTML autoescape. This prevented proper escaping of special characters in attribute values.

## Solution
Remove the `stringformat` filter from all form templates and rely on Django's/Jinja2's built-in autoescape mechanism to properly escape attribute values.

### Key Changes:

1. **Django Templates** (`django/forms/templates/`):
   - `attrs.html`: Removed `|stringformat:'s'` filter, added `and value is not None` check
   - `widgets/attrs.html`: Removed `|stringformat:'s'` filter, added `and value is not None` check
   - `widgets/input.html`: Removed `|stringformat:'s'` filter from widget.value
   - `widgets/select_option.html`: Removed `|stringformat:'s'` filter from widget.value

2. **Jinja2 Templates** (`django/forms/jinja2/`):
   - `attrs.html`: Added `and value is not None` check (already didn't use stringformat)
   - `widgets/attrs.html`: Added `and value is not None` check (already didn't use stringformat)

## How It Works
- Django and Jinja2 template engines have autoescape enabled by default
- When `stringformat` filter was removed, autoescape now properly escapes:
  - Double quotes → `&quot;`
  - Ampersands → `&amp;`
  - Angle brackets → `&lt;` and `&gt;`
- SafeString values (marked with `mark_safe()`) are still not escaped, allowing JavaScript handlers to work correctly

## Testing Impact
The fix:

- ✓ Preserves SafeString values without re-escaping
- ✓ Handles boolean attributes correctly (True = no value, False = skip, other = render)
- ✓ Handles None values correctly (skipped with new `value is not None` check)
