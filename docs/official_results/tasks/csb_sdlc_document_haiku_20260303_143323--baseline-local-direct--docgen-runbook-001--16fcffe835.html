## Objective

Produce an operational runbook for Prometheus's TSDB (Time Series Database) compaction process. TSDB compaction merges overlapping time blocks, removes tombstoned series, and maintains query performance over time.

## Scope

Explore the TSDB compaction implementation in:
- `tsdb/compact.go` — core compaction logic
- `tsdb/head_read.go` and `tsdb/head.go` — head block behavior
- `tsdb/db.go` — database lifecycle and compaction triggering

## Required Sections

Your runbook at `/workspace/documentation.md` must include:

### 1. Overview
- What compaction does and why it is needed
- Compaction levels and block structure
- When compaction is triggered (time-based, size-based)

### 2. Monitoring Indicators
- Key Prometheus metrics to watch during compaction (e.g., `prometheus_tsdb_compactions_total`, `prometheus_tsdb_compaction_duration_seconds`)
- Alert thresholds and what they indicate
- How to distinguish healthy vs. stuck compaction

### 3. Failure Modes
- At least 4 failure modes with symptoms and causes:
  - Disk space exhaustion
  - Compaction stuck / not progressing
  - Block corruption
  - OOM during compaction
- For each: symptom, likely cause, and diagnostic steps

### 4. Recovery Procedures
- Step-by-step recovery for each failure mode
- When to restart Prometheus vs. manual block manipulation
- How to use `promtool tsdb` commands for inspection/repair

### 5. Code Reference
- Key functions in the codebase relevant to each failure mode

# Prometheus TSDB Compaction Operational Runbook

## 1. Overview

### What is TSDB Compaction?

Prometheus TSDB compaction is the process of merging overlapping time-series database (TSDB) blocks into fewer, larger blocks while removing data marked for deletion (tombstones). Compaction serves several critical purposes:

- **Reduces disk I/O**: Fewer blocks to scan during queries means fewer file descriptor operations
- **Merges overlapping blocks**: Handles cases where multiple blocks have overlapping time ranges
- **Removes deleted data**: Eliminates tombstoned series to reclaim disk space
- **Maintains query performance**: Keeps the block structure optimized for query execution

### TSDB Architecture

Prometheus stores metrics in a hierarchical block structure:

- **Head Block**: In-memory data buffer that accumulates new samples. When it reaches 1.5x the configured chunk range in duration, it becomes eligible for compaction to disk
- **Persisted Blocks**: Time-series data on disk organized by time range, with levels determined by compaction progress
- **Block Levels**: Compaction creates progressively larger blocks:
  - Level 0: Single head block compaction (~2 hours, default)
  - Level 1: Compaction of 2-3 level-0 blocks
  - Level 2: Compaction of 2-3 level-1 blocks
  - And so on...

### Default Block Ranges

By default, Prometheus uses exponential block ranges starting at `MinBlockDuration` (2 hours):
- Level 0: 2 hours
- Level 1: 10 hours
- Level 2: 50 hours
- Level 3: 250 hours
- And more levels based on `MaxBlockDuration`

### When Compaction Triggers

Compaction is triggered in the following scenarios:

1. **Time-based trigger** (most common):
   - Head block age exceeds 1.5 × chunk range
