Gradle Build System\n\n### Project Structure\nThe Kafka codebase is organized as a **Gradle multi-project build** with the following key modules:\n\n**Core Modules:**\n- `clients` - Kafka client libraries (producer, consumer)\n- `core` - Kafka broker implementation\n- `server` - Server-specific components\n- `server-common` - Shared server utilities\n- `streams` - Kafka Streams\n- `connect` - Kafka Connect (with sub-modules for API, runtime, connectors)\n- `metadata` - KRaft metadata functionality\n- `raft` - KRaft consensus protocol\n\n**Supporting Modules:**\n- `storage` - Tiered storage implementation\n- `group-coordinator` - Group coordination protocol\n- `examples` - Example applications\n- `jmh-benchmarks` - JMH microbenchmarks\n- `generator` - Code generation utilities\n- `shell` - Interactive shell tools\n\nSee `settings.gradle` for the complete list of modules.\n\n### Essential Gradle Commands\n\n#### Build JAR\n```bash\n# Build all JARs\n./gradlew jar\n\n# Build JAR for a specific module\n./gradlew core:jar\n./gradlew clients:jar\n./gradlew streams:jar\n```\n\n#### Code Quality Checks\n```bash\n# Run checkstyle (enforces consistent coding style)\n./gradlew checkstyleMain checkstyleTest\n\n# Run spotbugs (static analysis for potential bugs)\n./gradlew spotbugsMain spotbugsTest -x test\n\n# Check and fix import ordering (spotless)\n./gradlew spotlessCheck\n./gradlew spotlessApply  # Uses build cache\n```\n\n---\n\n## 3. Running Tests\n\n### Test Types\n- **Unit Tests**: Fast, in-process tests (Gradle task: `unitTest`)\n- **Integration Tests**: Longer-running tests that may start services (Gradle task: `integrationTest`)\n- **System Tests**: Full-stack tests using ducktape framework in `tests/` directory\n\n### Running All Tests\n\n```bash\n# Run all unit and integration tests\n./gradlew test\n\n# Run only unit tests\n./gradlew unitTest\n\n# Run only integration tests\n./gradlew integrationTest\n\n# Run tests for a specific module\n./gradlew core:test\n./gradlew clients:test\n./gradlew streams:testAll  # For Streams multi-project\n```\n\n### Running Specific Tests\n\n```bash\n# Run a specific test class\n./gradlew clients:test --tests RequestResponseTest\n./gradlew core:test --tests kafka.api.ProducerFailureHandlingTest\n\n# Run a specific test method\n./gradlew clients:test --tests org.apache.kafka.clients.MetadataTest.testTimeToNextUpdate\n./gradlew core:test --tests kafka.api.ProducerFailureHandlingTest.testCannotSendToInternalTopic\n\n# Run tests matching a pattern\n./gradlew test --tests \"*MetadataTest*\"\n```\n\n### Running Tests with Options\n\n```bash\n# Force re-running tests (ignore cache)\n./gradlew test --rerun\n./gradlew clients:test --tests RequestResponseTest --rerun\n\n# Run with fail-fast (stop on first failure)\n./gradlew test --fail-fast\n\n# Run repeatedly until failure\nI=0; while ./gradlew clients:test --tests RequestResponseTest --rerun --fail-fast; do\n  (( I=$I+1 )); echo \"Completed run: $I\"; sleep 1;\ndone\n\n# CI Pipeline\n\n### CI Systems Used\n\n**Primary CI:**\n- **Jenkins** - Main CI system that runs on pull requests and commits\n- Configuration: `Jenkinsfile` (in repository root)\n- PR Triggering: Whitelisted users can trigger builds (see `.asf.yaml`)\n\n**Secondary CI:**\n- **GitHub Actions** - Used for Docker-related tasks and releases\n- Configuration: `.github/workflows/*.yml`\n  - `docker_build_and_test.yml` - Build and test Docker images\n  - `docker_official_image_build_and_test.yml` - Official image tests\n  - `docker_promote.yml` - Docker image promotion\n  - `docker_rc_release.yml` - Release candidate Docker builds\n  - `docker_scan.yml` - Security scanning\n\n### Jenkins Build Pipeline (`Jenkinsfile`)\n\nThe Jenkinsfile defines the CI pipeline stages:\n\n1. **Validation** (`doValidation()`)\n   - Runs `check` task (excluding tests)\n   - Code quality checks: checkstyle, spotbugs, import validation\n   - Uses `spotlessCheck`, SpotBugs reports\n\n2. **Testing** (`doTest()`)\n   - Runs full test suite: unit and integration tests\n   - Configuration:\n     - `maxParallelForks=2` - Runs 2 test processes in parallel\n     - `maxTestRetries=1` - Retries failed tests once\n     - `maxTestRetryFailures=10` - Max 10 retry failures per run\n   - Collects JUnit XML results: `**/build/test-results/**/TEST-*.xml`\n\n3. **Streams Archetype**\n   - Verifies Kafka Streams quickstart archetype compiles\n\n4. **Multiple Scala Versions**\n   - Runs builds with both Scala 2.12 and 2.13\n\n### PR Triggering\n\nPR builds are triggered on:\n- New PRs from whitelisted users (see `.asf.yaml`)\n- Commits to PRs\n- Comments on PRs\n\n**Whitelisted users** (can trigger CI):\nLocated in `.asf.yaml`:\n- FrankYang0529, kamalcph, apoorvmittal10, lianetm, brandboat\n- kirktrue, nizhikov, OmniaGM, dongnuo123, frankvicky\n\nFor non-whitelisted users, a whitelisted user must trigger the build.\n\n### CI Configuration Files\n\n**Key Files:**\n- `/Jenkinsfile` - Main Jenkins pipeline configuration\n- `/.asf.yaml` - GitHub integration and Jenkins whitelisting\n- `/.github/workflows/` - GitHub Actions workflows\n- `/gradle.properties` - Version and Gradle configuration\n\n### Required Checks Before Merging\n\nBefore a PR can be merged, these checks must pass:\n\n1. \u2705 Jenkins build succeeds\n2. \u2705 All tests pass\n3. \u2705 Code quality checks pass (checkstyle, spotbugs, spotless)\n4. \u2705 Code review approval from committer\n5. \u2705 CI build status shows green\n\n### Local CI Emulation\n\nTo catch CI failures locally before pushing:\n\n```bash\n# Run the same validation checks as CI\n./gradlew clean check -x test --profile --continue\n\n# Run full test suite\n./gradlew test --profile --continue\n\n# Check code style\n./gradlew checkstyleMain checkstyleTest spotlessCheck\n\n# Run spotbugs analysis\n./gradlew spotbugsMain spotbugsTest -x test\n\n# Full CI-like build (takes longer)\n./gradlew clean check --profile --continue\n```\n\n---\n\n## 5. Visit https://issues.apache.org/jira/browse/KAFKA\n2. Filter by status: Open, In Progress (unclaimed)\n3. Look for issues marked:\n   - \"good first issue\" - Suitable for new contributors\n   - \"help-wanted\" - Actively seeking contributions\n   - Lower priority/complexity for easier problems\n\n4. Comment on the issue: \"I'd like to work on this\"\n5. A committer may assign it to you\n\n**Note**: You don't need to be assigned, but commenting prevents duplicate work.\n\n### Branch Naming Conventions\n\nWhile not strictly enforced, follow this convention:\n\n```bash\n# For JIRA issues\ngit checkout -b KAFKA-XXXXX\n\n# For feature branches (if not tied to JIRA)\ngit checkout -b feature/short-description\ngit checkout -b bugfix/short-description\n\n# Example\ngit checkout -b KAFKA-15123\ngit checkout -b feature/improve-consumer-lag\n```\n\n### Development Workflow\n\n#### Step 1: Clone and Setup\n```bash\ngit clone https://github.com/apache/kafka.git\ncd kafka\ngit checkout main\ngit pull origin main  # Ensure latest\n```\n\n#### Step 2: Create and Checkout Branch\n```bash\ngit checkout -b KAFKA-XXXXX\n# Or your branch name\n```\n\n#### Step 3: Make Changes\n- Edit source files\n- Add/update tests\n- Follow code style (see section below)\n\n#### Step 4: Test Locally\n```bash\n# Run affected module tests\n./gradlew core:test --tests YourTestClass\n\n# Run code quality checks\n./gradlew checkstyleMain checkstyleTest spotlessCheck\n./gradlew spotlessApply  # Auto-fix imports\n\n# Run full check suite\n./gradlew check -x test\n```\n\n#### Step 5: Commit Changes\n```bash\ngit add <files>\ngit commit -m \"KAFKA-XXXXX: Brief description of change\n\nLonger explanation of what the change does and why.\nCan span multiple lines.\"\n```\n\n**Commit Message Format:**\n- Line 1: `KAFKA-XXXXX: Short description` (50 chars max)\n- Blank line\n- Detailed explanation (wrapped at 72 chars)\n- Reference issues: `Fixes KAFKA-XXXXX` or `Related to KAFKA-XXXXX`\n\n#### Step 6: Push and Create PR\n```bash\ngit push origin KAFKA-XXXXX\n```\n\nGitHub will show a prompt to create a PR. # Kafka Contributor Guide

A comprehensive guide for contributing to the Apache Kafka project. This document covers the tools, processes, and workflows needed to build, test, and submit code changes.

---

## 1. Build Prerequisites

### Supported Java Versions
- **Supported**: Java 8, 11, 17, and 21
- **Default/Recommended**: Java 11 or 17 (Java 8 and 11 support for broker/tools are deprecated and will be removed in Kafka 4.0)
- **Note**: Spotless code formatter has issues with Java 21, so use JDK 11 or 17 for `spotlessCheck` and `spotlessApply`

Install Java:
```bash
# Check current Java version
java -version

# On Ubuntu/Debian
sudo apt-get install openjdk-17-jdk

# On macOS with Homebrew
brew install openjdk@17
```

### Scala Version
- **Supported**: Scala 2.12 and 2.13
- **Default**: Scala 2.13.14 (as of version 3.9.0)
- **Note**: Scala 2.12 support is deprecated and will be removed in Kafka 4.0

You don't need to install Scala manually; Gradle manages it through dependencies.

### Gradle Wrapper
The project includes a **Gradle wrapper** (`gradlew`), so you don't need to install Gradle separately:
- `./gradlew` - Automatically downloads and uses the correct Gradle version
- `./gradlewAll` - Runs tasks with all supported Scala versions

### Docker (Optional, for system tests)
- Docker 1.12.3 or higher is required to run system integration tests locally
- System tests use the [ducktape](https://github.com/confluentinc/ducktape) framework

### No Other Dependencies Required
The build system will automatically download all other dependencies (Maven Central).
