## Summary

I've analyzed the Prometheus metrics exposition pattern across both Kubernetes and Grafana. Here are the key findings:

**Kubernetes** implements a sophisticated, version-aware metrics infrastructure:
- Central registry: `KubeRegistry` interface (staging/src/k8s.io/component-base/metrics/registry.go)
- HTTP exposition: `HandlerFor()` and `HandlerWithReset()` in staging/src/k8s.io/component-base/metrics/http.go

- Organized collectors by domain: workqueue, version, controllers, client-go integration
- StableCollector interface for custom collectors with deprecation tracking

**Grafana** takes a simpler, dependency-injection based approach:
- HTTP endpoint: `metricsEndpoint()` middleware in pkg/api/http_server.go
- Metrics service: pkg/infra/metrics/service.go with ProvideRegisterer/ProvideGatherer
- Centralized metrics: pkg/infra/metrics/metrics.go with single initMetricVars() registration
- Multi-registry composition: merges Kubernetes legacyregistry with prometheus.DefaultGatherer
- Automatic metric prefixing with "grafana_" namespace

The key difference is that Kubernetes prioritizes version-aware, distributed metric organization with custom collector patterns, while Grafana prioritizes simplicity with centralized registration and composition of multiple metric sources.
