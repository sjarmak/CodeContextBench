# Bug Investigation: Application Crashes When Ad-Blocker Cache Is Corrupted

**Repository:** qutebrowser/qutebrowser
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

When qutebrowser starts up and attempts to load the ad-blocker filter cache, the application crashes if the cache file is corrupted or contains invalid data.

The expected behavior is that the application handles corrupted cache files gracefully — it should log an error, discard the invalid cache, and continue running normally (allowing the user to browse without ad blocking until the cache is rebuilt).

Instead, the deserialization error propagates as an unhandled exception, causing a crash before the browser window even appears. Users who encounter a corrupted cache file cannot use qutebrowser at all until they manually delete the cache file.

This is particularly problematic because cache corruption can happen for reasons outside the user's control, such as a system crash or disk error during a cache write.

# Bug Analysis: Ad-Blocker Cache Corruption Crash

## Summary
When qutebrowser starts and attempts to load a corrupted ad-blocker filter cache, the application crashes with an unhandled exception instead of gracefully handling the error.

## Root Cause
**File:** `/app/qutebrowser/components/braveadblock.py`  
**Method:** `read_cache()` (lines 204-230)  
**Issue:** Exception type mismatch in error handling

### The Bug
The code attempts to catch `ValueError` exceptions:
```python
try:
    self._engine.deserialize_from_file(str(self._cache_path))
except ValueError as e:
    if str(e) != "DeserializationError":
        raise
    message.error("Reading adblock filter data failed (corrupted data?). "
                  "Please run :adblock-update.")
```

However, the adblock library raises `DeserializationError` (not `ValueError`):
- `DeserializationError` is NOT a subclass of `ValueError`
- The exception handler does not catch it
- The exception propagates unhandled, causing a crash

### Exception Hierarchy
When a corrupted cache is deserialized:
```
adblock.DeserializationError
  └─ adblock.BlockerException
       └─ adblock.AdblockException
            └─ Exception
                 └─ BaseException
```

This is NOT in the `ValueError` inheritance chain.

## Impact
**Severity:** High  
**When:** At application startup when ad-blocker cache loading is attempted  
**User Impact:** Application completely fails to start; users must manually delete the cache file to use qutebrowser again

## Expected Behavior
When a corrupted cache is encountered:
1. Catch the `DeserializationError` exception
2. Display error message: "Reading adblock filter data failed (corrupted data?). Please run :adblock-update."
3. Continue running normally without ad-blocking until the cache is rebuilt

## Current Behavior
The exception is not caught, propagates unhandled, and crashes the application before the browser window appears.

## The Fix
Change the exception handler to catch the actual exception type raised by the adblock library. Options:
1. Catch `adblock.DeserializationError` directly
2. Catch a broader exception type that includes it
3. Use a try-except that catches all exceptions and filters by type name

3. Verifies that `DeserializationError` is raised unhandled (confirming the bug)
