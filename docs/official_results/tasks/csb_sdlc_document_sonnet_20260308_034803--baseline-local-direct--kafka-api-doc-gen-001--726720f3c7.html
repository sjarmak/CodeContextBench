The commit happens at the start of a `poll()` call if the interval has elapsed.\n\n**Pitfalls of auto-commit:**\n- Records may be re-processed after a crash if they were fetched but not processed before the next auto-commit window.\n- The consumer commits the fetch position, not the last record the application confirmed it processed.\n- With auto-commit, there is no guarantee of at-least-once processing unless `poll()` is called frequently.\n\nSet `enable.auto.commit=false` for exactly-once or at-least-once processing with manual control.\n\n### Synchronous Commit\n\n```java\nvoid commitSync()\nvoid commitSync(Duration timeout)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)\n```\n\nCommits offsets and **blocks** until the broker acknowledges or an error occurs.\n\n**Parameters:**\n\n- No-arg / timeout-only variants \u2014 Commits the current position for **all** assigned partitions.\n- `offsets` \u2014 A specific map of `TopicPartition` to `OffsetAndMetadata` to commit. Only the specified partitions are committed.\n- `timeout` \u2014 Maximum time to wait for acknowledgment. If omitted, uses `default.api.timeout.ms`.\n\n**Behavior:**\n\n- Retries on retriable errors (e.g., coordinator changes) until timeout.\n- Throws `CommitFailedException` on fatal commit failures.\n- Safe to call after processing each record for maximum durability, but high-frequency calls add latency.\n\n**Exceptions:**\n\n- `CommitFailedException` \u2014 Commit was rejected because the consumer was fenced out of the group (e.g., due to `max.poll.interval.ms` exceeded). The consumer must rejoin. **Do not retry after this exception.**\n- `TimeoutException` \u2014 Broker did not respond within the timeout.\n- `WakeupException` \u2014 `wakeup()` interrupted the blocking wait.\n- `InterruptException` \u2014 Thread was interrupted.\n- `AuthorizationException` \u2014 Missing `WRITE` permission on the `__consumer_offsets` topic.\n- `KafkaException` \u2014 Unrecoverable error.\n\n### Asynchronous Commit\n\n```java\nvoid commitAsync()\nvoid commitAsync(OffsetCommitCallback callback)\nvoid commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets, OffsetCommitCallback callback)\n```\n\nSends a commit request and returns immediately. The optional callback is invoked (on the polling thread, during a subsequent `poll()`) when the commit completes or fails.\n\n**Parameters:**\n\n- No-arg variant \u2014 Commits current positions, no notification on completion.\n- `callback` \u2014 `OffsetCommitCallback.onComplete(Map<TopicPartition, OffsetAndMetadata> offsets, Exception exception)` called on completion. `exception` is `null` on success.\n- `offsets` \u2014 Explicit partition-to-offset map.\n\n**Behavior:**\n\n- Does **not** retry on failure. Unlike `commitSync`, if the commit request fails transiently, it is lost.\n- Multiple in-flight async commits can coexist. Callbacks are delivered in the order requests were sent.\n- **Do not retry** in the callback: a later async commit may have already succeeded with a higher offset, and retrying an older commit with a lower offset would regress the committed position.\n- For a clean shutdown: call `commitAsync()` in the loop and `commitSync()` once after the loop exits to ensure the final commit is durable.\n\n**`OffsetCommitCallback` interface:**\n\n```java\npublic interface OffsetCommitCallback {\n    void onComplete(Map<TopicPartition, OffsetAndMetadata> offsets, Exception exception);\n}\n```\n\n### Querying Committed Offsets\n\n```java\nOffsetAndMetadata committed(TopicPartition partition)\nOffsetAndMetadata committed(TopicPartition partition, Duration timeout)\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions)\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions, Duration timeout)\n```\n\nFetches the last committed offset for the given partition(s) **from the broker**. This is a synchronous, blocking network call.\n\n**Return value:** `null` (single-partition variants) or a map entry with `null` value for partitions that have no committed offset. This is distinct from offset 0; it means the partition has never been committed for this group.\n\n**Use cases:**\n- Inspecting lag from outside the polling loop.\n- Initializing positions in `onPartitionsAssigned` without using auto-reset.\n- Verifying that commits succeeded.\n\n**Exceptions:** `TimeoutException`, `AuthorizationException`, `KafkaException`.\n\n---\n\n## Flow Control and Position Queries\n\n### Current Position\n\n```java\nlong position(TopicPartition partition)\nlong position(TopicPartition partition, Duration timeout)\n```\n\nReturns the offset of the **next record** to be fetched for the partition. This reflects how far the consumer has fetched, not how far it has committed.\n\n**Note:** After a rebalance, if no seek is performed, the position is determined by the committed offset (if any) or `auto.offset.reset`. Calling `position()` before any `poll()` triggers a fetch of the committed offset or application of the reset policy.\n\n**Exceptions:** `IllegalStateException` (partition not assigned), `TimeoutException`, `InvalidOffsetException` (on `auto.offset.reset=none` with no committed offset).\n\n### Seek Operations\n\n```java\nvoid seek(TopicPartition partition, long offset)\nvoid seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)\nvoid seekToBeginning(Collection<TopicPartition> partitions)\nvoid seekToEnd(Collection<TopicPartition> partitions)\n```\n\nManually repositions the fetch position for one or more partitions. Signals an intentional interruption, typically from a shutdown hook. The consumer remains functional after this exception.\n\n**Recovery:** Catch at the outer loop level, do not re-throw into the processing path. Perform a final `commitSync()` if needed, then `close()`.\n\n### `CommitFailedException`\n\n**Package:** `org.apache.kafka.clients.consumer`\n\nThrown by `commitSync()` when the commit is rejected by the broker. Common causes:\n\n1. **Poll timeout exceeded:** `max.poll.interval.ms` elapsed between two consecutive `poll()` calls. The broker removed the consumer from the group. Offsets from the old generation cannot be committed. **Solution:** Increase `max.poll.interval.ms`, reduce processing time per batch, or reduce `max.poll.records`.\n2. **Concurrent consumer on same partition:** Another consumer (e.g., a zombie) committed to the same partition with a higher generation id. **Solution:** Ensure only one consumer instance per `group.id`.\n3. **Authorization error:** Consumer lacks permission to commit. **Solution:** Grant `WRITE` on `__consumer_offsets`.\n\nAfter `CommitFailedException`, the consumer must re-subscribe and rejoin the group. Do not retry the commit with the same consumer instance.\n\n### `TimeoutException`\n\n**Package:** `org.apache.kafka.common.errors`\n\nThrown when a blocking call (metadata fetch, commit, offset query) does not receive a response within the configured timeout. Retriable\u2014the operation can be attempted again.\n\n### `AuthenticationException`\n\nThrown when the broker rejects the client's credentials. Check `security.protocol`, SASL configuration, and credential validity.\n\n### `AuthorizationException`\n\nThrown when the client is authenticated but lacks permission for the operation. Common scenarios:\n- Missing `READ` on the topic (`poll()`).\n- Missing `READ` on the group resource (`subscribe()`).\n- Missing `WRITE` on `__consumer_offsets` (commit).\n- Missing `DESCRIBE` on the topic (metadata calls).\n\n### `InterruptException`\n\n**Package:** `org.apache.kafka.common.errors`\n\nWraps `java.lang.InterruptedException`. Thrown when the calling thread is interrupted while a blocking call is waiting. The consumer's interrupt flag is cleared. Re-interrupt the thread if appropriate.\n\n### `InvalidOffsetException` and `NoOffsetForPartitionException`\n\nThrown by `position()` or internally during `poll()` when `auto.offset.reset=none` and there is no committed offset for an assigned partition. **Resolution:** Commit an initial offset or change the reset policy.\n\n### `KafkaException`\n\nBase class for most Kafka client exceptions. Wraps internal errors. Check the message and `getCause()` for details.\n\n### `TopicAuthorizationException`\n\nSpecialization of `AuthorizationException` for topic-level authorization failures.\n\n### `GroupAuthorizationException`\n\nSpecialization for group-level authorization failures (e.g., missing permission to join group).\n\n---\n\n## Configuration-Driven Behaviors\n\n### Auto-Reset Policy (`auto.offset.reset`)\n\n| Value | Behavior when no committed offset exists |\n|---|---|\n| `earliest` | Start from the first available record in the partition |\n| `latest` | Start from the end of the partition (skip all existing records) |\n| `none` | Throw `NoOffsetForPartitionException` |\n\nApplies only when the consumer starts without a committed offset (first run for a group, or partition added to topic after group was created). Once a committed offset exists, this setting is ignored.\n\n### Isolation Level (`isolation.level`)\n\n| Value | Behavior |\n|---|---|\n| `read_uncommitted` (default) | Reads all records, including those from open and aborted transactions |\n| `read_committed` | Reads only records from committed transactions and non-transactional records; blocks at LSO |\n\nWith `read_committed`:\n- Records from open transactions are invisible until the transaction commits.\n- Records from aborted transactions are permanently filtered out (control records signal the abort to the consumer).\n- The consumer's position may lag the actual log end offset by the duration of the longest open transaction.\n- `endOffsets()` returns the LSO, not the actual log end offset.\n\n### Static Group Membership (`group.instance.id`)\n\nWhen `group.instance.id` is set, the consumer uses **static membership**:\n- The member ID is stable across restarts (no group rejoin on reconnect within `session.timeout.ms`).\n- Partition assignment is preserved across consumer restarts as long as the instance reconnects within `session.timeout.ms`.\n- Reduces rebalance frequency for deployments with predictable rolling restarts.\n\n### Partition Assignment Strategy\n\n| Strategy | Behavior |\n|---|---|\n| `RangeAssignor` | Assigns contiguous ranges of partitions per topic; may be uneven across consumers |\n| `RoundRobinAssignor` | Distributes partitions evenly in round-robin order |\n| `StickyAssignor` | Minimizes partition movement on rebalance while maintaining balance |\n| `CooperativeStickyAssignor` | Like `StickyAssignor` but uses incremental (cooperative) rebalancing\u2014revokes only partitions that need to move |\n\n`CooperativeStickyAssignor` enables incremental rebalances where only partitions actually being transferred are revoked. All other partitions continue to be consumed without interruption. This is the recommended default for most applications.\n\n---\n\n## Common Patterns and Best Practices\n\n### Pattern 1: Basic Subscription and Polling Loop\n\n```java\nProperties props = new Properties();\nprops.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, \"broker1:9092,broker2:9092\");\nprops.put(ConsumerConfig.GROUP_ID_CONFIG, \"my-group\");\nprops.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());\nprops.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class.getName());\nprops.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, \"earliest\");\n// auto-commit is enabled by default\n\ntry (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {\n    consumer.subscribe(List.of(\"orders\", \"payments\"));\n\n    while (!Thread.currentThread().isInterrupted()) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(500));\n        for (ConsumerRecord<String, String> record : records) {\n            System.out.printf(\"topic=%s partition=%d offset=%d key=%s value=%s%n\",\n                record.topic(), record.partition(), record.offset(),\n                record.key(), record.value());\n        }\n    }\n}\n```\n\n### Pattern 2: Manual Offset Commit (At-Least-Once)\n\nDisable auto-commit and commit explicitly after processing. If a rebalance is triggered, the `ConsumerRebalanceListener` callbacks run inside `poll()`. A slow `onPartitionsRevoked` (e.g., flushing state to a database) will delay `poll()`'s return. Design callbacks to complete quickly, or implement incremental work with periodic checks.\n\n### `seekToBeginning` / `seekToEnd` Are Lazy\n\nThese methods do not perform network lookups immediately. They record the intent, which is evaluated on the next `poll()`. If you call `seekToBeginning(partitions)` and then `position(partition)` before the next `poll()`, the position query will trigger the evaluation synchronously.\n\n### Committed Offset `null` vs Offset 0\n\n`committed()` returning `null` means the group has **never committed** for that partition. ## Overview and Threading Model

`KafkaConsumer` (package `org.apache.kafka.clients.consumer`) is the primary client for consuming records from Apache Kafka topics. It implements a **single-threaded, event-loop model**: all API calls must be made from the same thread that owns the consumer instance, with the sole exception of `wakeup()`.

### Threading Rules

- **Not thread-safe.** Concurrent calls from multiple threads will produce `ConcurrentModificationException`.

- The standard pattern is one consumer per thread, or one thread dedicated solely to calling `poll()` and dispatching work to a thread pool.
