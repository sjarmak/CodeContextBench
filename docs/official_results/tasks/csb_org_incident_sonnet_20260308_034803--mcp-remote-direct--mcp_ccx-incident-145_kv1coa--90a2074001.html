## Grafana Alert Evaluation Cycle Overrun

The implementation spans three files:

### 1. `pkg/services/ngalert/schedule/ticker/ticker.go`
Implements the custom `T` ticker that drives the scheduler. Unlike `time.Ticker`, it **never drops ticks** — it queues them in a channel so the consumer can detect when it's falling behind. The `run()` goroutine emits ticks at fixed interval marks.

### 2. `pkg/services/ngalert/schedule/schedule.go`
The main scheduler implementation:
- **`schedulePeriodic()`** — reads from the ticker channel, records `BehindSeconds` metric (how far behind wall clock vs tick time), and calls `processTick()` on each tick
- **`processTick()`** — determines which rules are due to run on the current tick
- **`runJobFn()`** — calls `alertRule.Eval()` and **detects + logs overruns**: when `Eval()` returns a non-nil `dropped` message, it logs `"Tick dropped because alert rule evaluation is too slow"` and increments the `EvaluationMissed` metric

### 3. `pkg/services/ngalert/schedule/alert_rule.go`
**`alertRule.Eval()`** — uses a single-slot channel. Before sending a new tick, it does a non-blocking drain of the channel. If a previous tick is still pending (the evaluation goroutine hasn't consumed it), that tick is returned as `droppedMsg`, signaling the overrun to the scheduler.
