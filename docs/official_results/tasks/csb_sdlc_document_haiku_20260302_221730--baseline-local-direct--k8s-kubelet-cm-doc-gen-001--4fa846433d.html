Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### **Cgroup Subsystem Unavailable**\n- **Detection:** `container_manager_linux.go:151-202` (validateSystemRequirements)\n- **Consequence:**\n  - If CPU quota missing: CPU hardcapping disabled (SoftRequirement set, not fatal)\n  - If required subsystems missing: Fatal error, kubelet fails to start\n- **Mitigation:** Requires root, cgroup mounts on system; documented in release notes\n\n#### **Device Plugin Allocation Failure**\n- **Detection:** Device manager checkpoint corruption or plugin RPC failure\n- **Consequence:** Pod rejected at admission time if device allocation fails\n- **Recovery:** Checkpoint file reset on kubelet restart (`devicemanager/checkpoint/`)\n- **Design Choice:** Aggressive fail-fast to prevent resource deadlock\n\n#### **CPU Manager State Corruption**\n- **Detection:** Checkpointed state file unreadable or malformed\n- **Consequence:** CPU manager falls back to None policy (no pinning)\n- **Reference:** `cpumanager/cpu_manager.go:550+` (state file handling)\n- **Mitigation:** Periodic reconciliation (default 1 second) corrects drift\n\n#### **Memory Manager NUMA Mismatch**\n- **Detection:** Pod request changes at runtime (impossible, but handled)\n- **Consequence:** Container placed on wrong NUMA node; performance degradation\n- **Acceptance Criteria:** Memory manager validates pod specs at allocation time\n\n#### **QoS Cgroup Update Race**\n- **Detection:** Pod added/removed during QoS shares recalculation\n- **Consequence:** CPU shares slightly inaccurate for 1 minute (until next update)\n- **Design Choice:** Acceptable; cgroup updates are eventually consistent\n- **Reference:** Runs every 1 minute (`qos_container_manager_linux.go:45`)\n\n### 4.2 Resource Limits and Capacity Constraints\n\n#### **Maximum Containers per Node**\n- **Constraint:** Cgroup filesystem inode limit (filesystem-dependent)\n- **Typical Limit:** ~10,000 containers (Linux ext4 inode exhaustion)\n- **Mitigation:** Plan node sizing; monitor inode usage\n- **Code:** No explicit limit in container manager\n\n#### **CPU Manager Reconciliation Period**\n- **Default:** 10 seconds (configured via `--cpu-manager-reconcile-period`)\n- **Tradeoff:**\n  - Shorter period: Faster convergence to desired CPU state, higher CPU cost\n  - Longer period: Less overhead, but takes longer to enforce pins\n- **Reference:** `cpumanager/cpu_manager.go:109-110`\n\n#### **Device Allocation Overhead**\n- **Bottleneck:** Device plugin RPC calls during pod admission\n- **Limit:** Serial execution (one device allocation at a time)\n- **Mitigation:** Async device operations in future; currently blocking\n- **Reference:** `devicemanager/manager.go:200+` (serial Allocate calls)\n\n#### **QoS Cgroup Share Recalculation Cost**\n- **Frequency:** Every 1 minute\n- **Cost:** Iterates all active pods, sums CPU requests\n- **Scale Limit:** Negligible unless 10,000+ pods per node\n- **Reference:** `qos_container_manager_linux.go:84-149`\n\n#### **Node Allocatable Enforcement Retry**\n- **Design:** Retry every 1 minute if memory usage exceeds limit\n- **Rationale:** Pod evictions take time; enforcement is eventual\n- **Reference:** `node_container_manager_linux.go:95-106`\n\n### 4.3 Design Tradeoffs\n\n#### **Cgroup Version Compatibility**\n\n**Choice:** Support cgroup v1 and v2; deprecating v1\n\n**Cgroup v1 Characteristics:**\n- Per-subsystem hierarchies (separate cpu, memory, cpuset trees)\n- Manual CPU share calculations\n- Well-tested, widespread\n\n**Cgroup v2 Characteristics:**\n- Unified hierarchy (single tree)\n- Simpler CPU weight/limit model\n- Memory QoS features (memory.high, memory.min)\n- Preferred; v1 deprecated with warning (`container_manager.go:52-58`)\n\n**Tradeoff:** v1 support increases code complexity; deprecation path provides transition time\n\n**Affected Code:**\n- `cgroup_v1_manager_linux.go` (v1-specific CPU share calculations)\n- `cgroup_v2_manager_linux.go` (v2-specific unified controls)\n- `cgroup_manager_linux.go` (common abstraction)\n\n#### **CPU Manager: Static vs. Best-Effort**\n\n**Choice:** Two policies (None, Static); no hybrid\n\n**None Policy:**\n- All CPUs shared equally via CPU shares\n- Lightweight, works on any system\n- No NUMA awareness\n\n**Static Policy:**\n- Exclusive CPUs for guaranteed pods with integer requests\n- Higher performance for latency-sensitive workloads\n- Requires NUMA topology discovery\n\n**Tradeoff:** Two policies minimize complexity while covering common cases\n\n**No Middle Ground:**\n- Could have partial pinning, but increases complexity without clear benefit\n- Users choose at node level via `--cpu-manager-policy` (applied to all pods)\n\n#### **Device Allocation: Checkpoint vs. Online Discovery**\n\n**Choice:** Persistent checkpoint of allocations; device plugins optional re-discovery\n\n**Checkpoint Benefits:**\n- Survives kubelet restart\n- Device plugin crash doesn't lose allocations\n- Enables device plugin health checks\n\n**Online Discovery Limitation:**\n- Can't discover new devices after node boot without restart\n- Plugin listens on `/var/lib/kubelet/device-plugins/` socket\n\n**Tradeoff:** Checkpoint adds complexity but solves critical recovery scenario\n\n**Reference:** `devicemanager/checkpoint/checkpoint.go`\n\n#### **Topology Manager: Pod vs. Container Scope**\n\n**Choice:** Both scopes supported; configurable via `--topology-manager-scope`\n\n**Pod Scope:**\n- Single topology alignment for entire pod\n- Simpler; all containers share same NUMA nodes\n- Limits flexibility\n\n**Container Scope:**\n- Per-container topology alignment\n- More flexible; allows heterogeneous pods\n- More complex resource tracking\n\n**Tradeoff:** Pod scope is default (simpler); container scope available for advanced use\n\n**Reference:** `topologymanager/scope.go`, `scope_pod.go`, `scope_container.go`\n\n#### **QoS Hierarchy: Three-Tier vs. Dynamic**\n\n**Choice:** Fixed three tiers (Guaranteed, Burstable, BestEffort); no dynamic\n\n**Three-Tier Benefits:**\n- Stable, predictable resource governance\n- Simple to understand and debug\n- Matches Kubernetes QoS concept\n\n**Dynamic Alternative (Not Chosen):**\n- Could create per-app tiers or priority levels\n- Increases complexity significantly\n- No clear benefit without scheduler support\n\n**Tradeoff:** Three tiers sufficient for Kubernetes use cases\n\n## 5. Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### **Cgroup Subsystem Unavailable**\n- **Detection:** `container_manager_linux.go:151-202` (validateSystemRequirements)\n- **Consequence:**\n  - If CPU quota missing: CPU hardcapping disabled (SoftRequirement set, not fatal)\n  - If required subsystems missing: Fatal error, kubelet fails to start\n- **Mitigation:** Requires root, cgroup mounts on system; documented in release notes\n\n#### **Device Plugin Allocation Failure**\n- **Detection:** Device manager checkpoint corruption or plugin RPC failure\n- **Consequence:** Pod rejected at admission time if device allocation fails\n- **Recovery:** Checkpoint file reset on kubelet restart (`devicemanager/checkpoint/`)\n- **Design Choice:** Aggressive fail-fast to prevent resource deadlock\n\n#### **CPU Manager State Corruption**\n- **Detection:** Checkpointed state file unreadable or malformed\n- **Consequence:** CPU manager falls back to None policy (no pinning)\n- **Reference:** `cpumanager/cpu_manager.go:550+` (state file handling)\n- **Mitigation:** Periodic reconciliation (default 1 second) corrects drift\n\n#### **Memory Manager NUMA Mismatch**\n- **Detection:** Pod request changes at runtime (impossible, but handled)\n- **Consequence:** Container placed on wrong NUMA node; performance degradation\n- **Acceptance Criteria:** Memory manager validates pod specs at allocation time\n\n#### **QoS Cgroup Update Race**\n- **Detection:** Pod added/removed during QoS shares recalculation\n- **Consequence:** CPU shares slightly inaccurate for 1 minute (until next update)\n- **Design Choice:** Acceptable; cgroup updates are eventually consistent\n- **Reference:** Runs every 1 minute (`qos_container_manager_linux.go:45`)\n\n### 4.2 Resource Limits and Capacity Constraints\n\n#### **Maximum Containers per Node**\n- **Constraint:** Cgroup filesystem inode limit (filesystem-dependent)\n- **Typical Limit:** ~10,000 containers (Linux ext4 inode exhaustion)\n- **Mitigation:** Plan node sizing; monitor inode usage\n- **Code:** No explicit limit in container manager\n\n#### **CPU Manager Reconciliation Period**\n- **Default:** 10 seconds (configured via `--cpu-manager-reconcile-period`)\n- **Tradeoff:**\n  - Shorter period: Faster convergence to desired CPU state, higher CPU cost\n  - Longer period: Less overhead, but takes longer to enforce pins\n- **Reference:** `cpumanager/cpu_manager.go:109-110`\n\n#### **Device Allocation Overhead**\n- **Bottleneck:** Device plugin RPC calls during pod admission\n- **Limit:** Serial execution (one device allocation at a time)\n- **Mitigation:** Async device operations in future; currently blocking\n- **Reference:** `devicemanager/manager.go:200+` (serial Allocate calls)\n\n#### **QoS Cgroup Share Recalculation Cost**\n- **Frequency:** Every 1 minute\n- **Cost:** Iterates all active pods, sums CPU requests\n- **Scale Limit:** Negligible unless 10,000+ pods per node\n- **Reference:** `qos_container_manager_linux.go:84-149`\n\n#### **Node Allocatable Enforcement Retry**\n- **Design:** Retry every 1 minute if memory usage exceeds limit\n- **Rationale:** Pod evictions take time; enforcement is eventual\n- **Reference:** `node_container_manager_linux.go:95-106`\n\n### 4.3 Design Tradeoffs\n\n#### **Cgroup Version Compatibility**\n\n**Choice:** Support cgroup v1 and v2; deprecating v1\n\n**Cgroup v1 Characteristics:**\n- Per-subsystem hierarchies (separate cpu, memory, cpuset trees)\n- Manual CPU share calculations\n- Well-tested, widespread\n\n**Cgroup v2 Characteristics:**\n- Unified hierarchy (single tree)\n- Simpler CPU weight/limit model\n- Memory QoS features (memory.high, memory.min)\n- Preferred; v1 deprecated with warning (`container_manager.go:52-58`)\n\n**Tradeoff:** v1 support increases code complexity; deprecation path provides transition time\n\n**Affected Code:**\n- `cgroup_v1_manager_linux.go` (v1-specific CPU share calculations)\n- `cgroup_v2_manager_linux.go` (v2-specific unified controls)\n- `cgroup_manager_linux.go` (common abstraction)\n\n#### **CPU Manager: Static vs. Best-Effort**\n\n**Choice:** Two policies (None, Static); no hybrid\n\n**None Policy:**\n- All CPUs shared equally via CPU shares\n- Lightweight, works on any system\n- No NUMA awareness\n\n**Static Policy:**\n- Exclusive CPUs for guaranteed pods with integer requests\n- Higher performance for latency-sensitive workloads\n- Requires NUMA topology discovery\n\n**Tradeoff:** Two policies minimize complexity while covering common cases\n\n**No Middle Ground:**\n- Could have partial pinning, but increases complexity without clear benefit\n- Users choose at node level via `--cpu-manager-policy` (applied to all pods)\n\n#### **Device Allocation: Checkpoint vs. Online Discovery**\n\n**Choice:** Persistent checkpoint of allocations; device plugins optional re-discovery\n\n**Checkpoint Benefits:**\n- Survives kubelet restart\n- Device plugin crash doesn't lose allocations\n- Enables device plugin health checks\n\n**Online Discovery Limitation:**\n- Can't discover new devices after node boot without restart\n- Plugin listens on `/var/lib/kubelet/device-plugins/` socket\n\n**Tradeoff:** Checkpoint adds complexity but solves critical recovery scenario\n\n**Reference:** `devicemanager/checkpoint/checkpoint.go`\n\n#### **Topology Manager: Pod vs. Container Scope**\n\n**Choice:** Both scopes supported; configurable via `--topology-manager-scope`\n\n**Pod Scope:**\n- Single topology alignment for entire pod\n- Simpler; all containers share same NUMA nodes\n- Limits flexibility\n\n**Container Scope:**\n- Per-container topology alignment\n- More flexible; allows heterogeneous pods\n- More complex resource tracking\n\n**Tradeoff:** Pod scope is default (simpler); container scope available for advanced use\n\n**Reference:** `topologymanager/scope.go`, `scope_pod.go`, `scope_container.go`\n\n#### **QoS Hierarchy: Three-Tier vs. Dynamic**\n\n**Choice:** Fixed three tiers (Guaranteed, Burstable, BestEffort); no dynamic\n\n**Three-Tier Benefits:**\n- Stable, predictable resource governance\n- Simple to understand and debug\n- Matches Kubernetes QoS concept\n\n**Dynamic Alternative (Not Chosen):**\n- Could create per-app tiers or priority levels\n- Increases complexity significantly\n- No clear benefit without scheduler support\n\n**Tradeoff:** Three tiers sufficient for Kubernetes use cases\n\n## 5. The kubelet container manager subsystem is responsible for enforcing resource constraints, managing container resource allocation, and coordinating resource sharing across pods on a Kubernetes node.
