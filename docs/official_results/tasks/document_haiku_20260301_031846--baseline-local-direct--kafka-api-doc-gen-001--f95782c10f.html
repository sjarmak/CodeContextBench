# Task: Generate API Reference Documentation for Kafka KafkaConsumer

## Objective

## Scope

Your documentation should cover the **KafkaConsumer API** from the `org.apache.kafka.clients.consumer` package, including:

1. **Core Consumer Lifecycle**
   - Constructor variants and configuration
   - Subscription methods (dynamic and manual assignment)
   - Polling and data fetching semantics
   - Resource cleanup and closing

2. **Offset Management**
   - Synchronous and asynchronous commit strategies
   - Offset queries and position control
   - Seek operations and offset discovery
   - Committed offset semantics

3. **Consumer Group Mechanics**
   - ConsumerRebalanceListener interface and callbacks
   - Rebalance triggers and timing
   - Group membership and heartbeat behavior
   - Partition assignment vs subscription models

4. **Flow Control and Position Management**
   - Pause and resume functionality
   - Position queries and manipulation
   - Offset-to-timestamp lookups
   - Beginning and end offset discovery

5. **Error Handling**
   - Exception types and recovery strategies
   - CommitFailedException and group fencing

   - Timeout and authentication errors

## Overview

The `KafkaConsumer` is a Kafka client that consumes records from a Kafka cluster. It transparently handles broker failures and adapts to partition migration within the cluster. The consumer interacts with brokers to enable groups of consumers to load balance consumption using consumer groups.

### Key Characteristics

- **Connection Management**: The consumer maintains TCP connections to necessary brokers. Failure to close the consumer will leak these connections.
- **Thread Safety**: The consumer is NOT thread-safe except for the `wakeup()` method.
- **Cross-Version Compatibility**: Communicates with brokers version 0.10.0 or newer. Some features may not be available on older brokers.

### Architecture Model

The Kafka consumer operates on an **event loop model**:
- A single consumer thread owns all I/O operations
- Rebalance callbacks execute within the `poll()` call (not asynchronously)
- The consumer maintains heartbeat, fetch, and group coordination operations

It performs multiple operations:\n\n1. **Heartbeat and Group Management**: Maintains liveness in consumer group\n2. **Rebalance Coordination**: Executes rebalances if needed\n3. **Data Fetching**: Retrieves records from assigned partitions\n4. **Offset Tracking**: Updates internal position tracking\n\n**Timeout Behavior**:\n\n- `timeout` specifies how long to wait for data to become available\n- Returns immediately if:\n  - Records are available for fetching\n  - Rebalance completes\n  - Fetch completes within timeout\n- Returns `ConsumerRecords.empty()` if timeout expires without data\n- **Important**: Rebalance callbacks execute DURING poll(), potentially blocking longer than timeout\n\n**Actual Blocking Duration**:\n\n```\nActual wait = min(timeout, time to fetch or rebalance complete)\n             + time to execute rebalance callbacks\n```\n\nDuring rebalances, `poll()` may block significantly longer than the timeout while executing user callbacks.\n\n**Partition Position Advancement**:\n\n- Consumer maintains internal position per partition\n- Position automatically advances after records are returned\n- `poll()` returns records starting from current position\n- Position reflects the offset of the NEXT record to fetch\n\n**max.poll.records Configuration**:\n\n- Limits records returned per `poll()` call\n- Prevents unbounded batch sizes\n- Helps bound processing time per poll interval\n\n**Rebalance Behavior**:\n\nRebalances only occur during `poll()` calls. If you don't call `poll()` at least as frequently as `max.poll.interval.ms`, the consumer proactively leaves the group.\n\n### Records Access and Iteration\n\n```java\nConsumerRecords<K, V> records = consumer.poll(Duration.ofMillis(100));\n\n// Iterate all records\nfor (ConsumerRecord<K, V> record : records) {\n    // Process record\n}\n\n// Get records by partition\nfor (TopicPartition partition : records.partitions()) {\n    List<ConsumerRecord<K, V>> partitionRecords = records.records(partition);\n}\n\n// Get records by topic\nfor (ConsumerRecord<K, V> record : records.records(\"topic-name\")) {\n    // Process records from specific topic\n}\n\n// Get next offsets (v4.0+)\nMap<TopicPartition, OffsetAndMetadata> nextOffsets = records.nextOffsets();\n```\n\n---\n\n## Offset Management\n\n### Core Offset Concepts\n\n**Position**: The offset of the next record that will be fetched.\n- Automatically advances as records are returned\n- Advances to offset N when record with offset N-1 is consumed\n- Can be manually set with `seek()`\n\n**Committed Offset**: The last offset securely stored.\n- Used on restart to resume from where consumption left off\n- Updated via `commitSync()`, `commitAsync()`, or auto-commit\n- Retrieved via `committed()`\n\n**LSO (Last Stable Offset)**: In `read_committed` isolation, the first offset of an open transaction.\n- Acts as end boundary for `read_committed` consumers\n- Messages after LSO are filtered out\n- Affects `seekToEnd()` and `endOffsets()` behavior\n\n### Automatic Offset Commits\n\n**Configuration**:\n- `enable.auto.commit=true` (default)\n- `auto.commit.interval.ms=5000` (default)\n\n**Behavior**:\n- Offsets committed automatically at fixed intervals\n- Commits happen asynchronously between `poll()` calls\n- Commits current position for all subscribed/assigned partitions\n- If error occurs before commit completes, records may be reprocessed\n\n**Semantics**: \"at-least-once\" - records may be reprocessed on failure.\n\n**Important**: You MUST consume all data from each `poll()` call before the next `poll()`, or closing the consumer. Otherwise, the committed offset can advance ahead of the consumed position, causing records to be skipped.\n\n### Manual Offset Commits (Synchronous)\n\n```java\npublic void commitSync()\npublic void commitSync(Duration timeout)\npublic void commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)\npublic void commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)\n```\n\n**Behavior**:\n- Blocks until commit succeeds or unrecoverable error occurs\n- Commits offsets returned from most recent `poll()`\n- Or commits explicitly specified offsets\n\n**Exception Handling**:\n- Throws `CommitFailedException` if:\n  - Partition no longer assigned (group rebalanced)\n  - Consumer no longer part of group\n  - Consumer was fenced by another instance\n- Throws `TimeoutException` if timeout expires\n- Throws `WakeupException` if `wakeup()` called concurrently\n\n**Use Cases**:\n- Process batch of records, then commit\n- Process records with external side effects (database inserts)\n- Atomic processing with offset commit\n\n### Manual Offset Commits (Asynchronous)\n\n```java\npublic void commitAsync()\npublic void commitAsync(OffsetCommitCallback callback)\npublic void commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets,\n                        OffsetCommitCallback callback)\n```\n\n**Behavior**:\n- Returns immediately without blocking\n- Performs commit in background\n- Callback invoked when commit completes\n- Multiple async commits can be in flight\n\n**Ordering Guarantees**:\n- Asynchronous commits are queued in order\n- Callbacks are invoked in order they were registered\n- `commitSync()` waits for previous async callbacks to complete\n\n**Callback Invocation**:\n- Called from the thread running `poll()` (during next poll)\n- Receives `OffsetCommitCallback` with exception if failed\n\n**Use Cases**:\n- Non-blocking offset commits for high throughput\n- Deferred error handling\n\n### Offset Queries\n\n#### position()\n\n```java\npublic long position(TopicPartition partition)\npublic long position(TopicPartition partition, Duration timeout)\n```\n\n**Returns**: The offset of the next record to be fetched.\n\n**Blocking**: May issue remote call to determine position.\n\n**Use Cases**:\n- Track current consumption progress\n- Calculate lag (compared to `endOffsets()`)\n\n#### committed()\n\n```java\npublic Map<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions)\npublic Map<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions, Duration timeout)\n```\n\n**Returns**: Last committed offsets for given partitions.\n\n**Returns null**: For partitions with no committed offset.\n\n**Blocking**: Fetches from broker, may block until timeout.\n\n**Use Cases**:\n- Verify offsets were committed\n- Resume from committed position on restart\n\n### Seek Operations\n\n#### seek()\n\n```java\npublic void seek(TopicPartition partition, long offset)\npublic void seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)\n```\n\n**Behavior**:\n- Overrides next fetch offset for a partition\n- Takes effect on next `poll()` call\n- If called multiple times for same partition, latest offset wins\n- Can be called at any time\n\n**Caution**: May cause data loss if used incorrectly. Seeking forward skips messages.\n\n**Use Cases**:\n- Restore from external offset store on startup\n- Skip to latest on slow consumers\n- Replay from specific offset\n\n#### seekToBeginning()\n\n```java\npublic void seekToBeginning(Collection<TopicPartition> partitions)\n```\n\n**Behavior**:\n- Evaluates lazily - actual seek happens on next `poll()`\n- Seeks to first seekToEnd()\n\n```java\npublic void seekToEnd(Collection<TopicPartition> partitions)\n```\n\n**Behavior**:\n- Evaluates lazily - actual seek happens on next `poll()`\n- Seeks to last offset in partition\n- For `read_committed` consumers: seeks to Last Stable Offset (LSO)\n- Empty collection seeks all currently assigned partitions\n\n**Use Cases**:\n- Skip backlog and process only new Long> timestampsToSearch)\npublic Map<TopicPartition, OffsetAndTimestamp> offsetsForTimes(\n    Map<TopicPartition, Long> timestampsToSearch, Duration timeout)\n```\n\n**Behavior**:\n- Finds offset operation\n\n**Use Cases**:\n- Replay records from specific time\n- Time-based recovery\n- Timestamp-based log reprocessing\n\n---\n\n## Rebalance Mechanics\n\n### Rebalance Triggers\n\nRebalances occur only during `poll()` calls when:\n\n1. **Group Membership Changes**:\n   - New consumer joins group\n   - Consumer fails (detected via heartbeat timeout)\n   - Consumer leaves group via `unsubscribe()`\n\n2. **Subscription Changes**:\n   - Topic list changes via `subscribe()`\n   - Regex pattern matches new topics\n   - New partitions added to subscribed topics\n\n3. **Proactive Departure**:\n   - Consumer exceeds `max.poll.interval.ms` without calling `poll()`\n   - Consumer proactively leaves to avoid blocking partitions\n\n### Rebalance Protocols\n\n**Classic/Eager Protocol**:\n- All partitions revoked\n- All consumers rejoin and request new assignments\n- Partition assignment computed\n- All assignments delivered at once\n- `onPartitionsRevoked` always called with all partitions\n- High latency, longer unavailability window\n\n**Cooperative Protocol**:\n- Only affected partitions revoked\n- Partitions remain assigned while rebalancing\n- `onPartitionsRevoked` called only for partitions being revoked\n- Lower latency, reduced stop-the-world time\n\n### Callback Ordering\n\nDuring a rebalance:\n\n1. **Consumer loses partitions** (if any):\n   ```\n   onPartitionsRevoked(revoked_partitions) -> save state/offsets\n   ```\n\n2. **Rebalance computation** (server-side):\n   ```\n   Assignor assigns partitions based on current members\n   ```\n\n3. **Consumer gains partitions** (if any):\n   ```\n   onPartitionsAssigned(new_partitions) -> load state/seek to offsets\n   ```\n\n4. **Generation ID incremented**:\n   ```\n   Consumer now has new generation in group\n   ```\n\n**Ordering Guarantee**: The old consumer's `onPartitionsRevoked` is always called before the new consumer's `onPartitionsAssigned` for the same partition (under normal conditions).\n\n### Rebalance Exceptions\n\n#### CommitFailedException\n\nThrown when `commitSync()` is called and:\n- Partitions were revoked since last successful commit\n- Consumer is no longer in group\n- Consumer was fenced by another instance\n\nThis indicates consumer lost its group membership.\n\n#### RebalanceInProgressException\n\nThrown when calling certain APIs while rebalance is in progress:\n- Rebalance detected, assignment is changing\n- Call `poll()` to complete rebalance, then retry\n\n### Partition Loss and Session Timeout\n\n**Session Timeout** (`session.timeout.ms`, default 10s):\n- Consumer fails to send heartbeat for this duration\n- Broker removes consumer from group\n- Partitions reassigned to other consumers\n- Consumer is fenced and cannot commit offsets\n\n**Max Poll Interval** (`max.poll.interval.ms`, default 5 minutes):\n- Consumer doesn't call `poll()` for this duration\n- Consumer proactively leaves group to prevent livelock\n- Partitions reassigned to other consumers\n\nWhen partitions are lost unexpectedly:\n- `onPartitionsLost()` is called\n- `onPartitionsRevoked()` was NOT called for these partitions\n- Don't attempt to commit offsets (you're no longer the owner)\n\n---\n\n## Flow Control and Position Management\n\n### Pause and Resume\n\n```java\npublic void pause(Collection<TopicPartition> partitions)\npublic void resume(Collection<TopicPartition> partitions)\npublic Set<TopicPartition> paused()\n```\n\n**Behavior**:\n- `pause()`: Stop fetching from specified partitions\n- `resume()`: Resume fetching from paused partitions\n- `poll()` still executes heartbeat and group management\n- `poll()` does NOT return records from paused partitions\n\n**Semantics**:\n- Paused partitions retain their position\n- Resuming continues from same position\n- Pause state survives across rebalances (but not assignment changes)\n\n**Use Cases**:\n- Backpressure: Pause when processing queue is full\n- Prioritization: Fetch from some partitions before others\n- Flow control: Decouple consumption from processing\n\n### Current Lag Calculation\n\n```java\npublic OptionalLong currentLag(TopicPartition partition)\n```\n\n**Returns**: Approximate lag in number of records.\n\n```\nlag = endOffsets() - position()\n```\n\n**Use Cases**:\n- Monitor consumer lag\n- Trigger alerts\n- Decide when to backoff\n\n### Beginning and End Offsets\n\n```java\npublic Map<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions)\npublic Map<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions, Duration timeout)\npublic Map<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions)\npublic Map<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions, Duration timeout)\n```\n\n**Returns**: Offset range for given partitions.\n\n**Semantics**:\n- `beginningOffsets()`: First offset available (based on retention policy)\n- `endOffsets()`: Offset where next write would go (position to seek for latest)\n- For `read_committed`: `endOffsets()` returns Last Stable Offset\n\n**Use Cases**:\n- Calculate lag\n- Determine range to reprocess\n- Reset to beginning or end\n\n---\n\n## Exception Handling\n\n### CommitFailedException\n\n**When Thrown**:\n- `commitSync()` called after consumer lost group membership\n- Partition no longer assigned due to rebalance\n- Consumer fenced by another instance\n\n**Handling**:\n```java\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n    // Partitions were revoked during commit\n    // Consumer is no longer in group\n    // Cannot recover from this error\n    log.error(\"Commit failed, consumer fenced\", e);\n    break;  // Exit polling loop\n}\n```\n\n**Recovery**: Must rejoin group via `poll()`. Check if partitions are still assigned.\n\n### WakeupException\n\n**When Thrown**:\n- `wakeup()` called from another thread during blocking operation\n- Typically in `poll()`, `commitSync()`, `position()`, etc.\n\n**Handling Pattern**:\n```java\nprivate final AtomicBoolean closed = new AtomicBoolean(false);\n\nvoid run() {\n    try {\n        while (!closed.get()) {\n            ConsumerRecords records = consumer.poll(Duration.ofSeconds(1));\n            // Process records\n        }\n    } catch (WakeupException e) {\n        if (!closed.get()) throw e;  // Rethrow if not closing\n    } finally {\n        consumer.close();\n    }\n}\n\npublic void shutdown() {\n    closed.set(true);\n    consumer.wakeup();  // Called from another thread\n}\n```\n\n**Semantics**: `wakeup()` is the only thread-safe consumer method. Use it to interrupt blocking operations from external threads.\n\n### InterruptException\n\n**When Thrown**:\n- Thread calling consumer is interrupted (Thread.interrupt() called)\n- Raised from blocking operations\n\n**Handling**: Similar to `WakeupException`. Generally avoid thread interrupts and use `wakeup()` instead.\n\n### TimeoutException\n\n**When Thrown**:\n- Operation exceeds specified timeout\n- Returned from `poll()`, `commitSync()`, `position()`, etc.\n\n**Handling**:\n```java\ntry {\n    consumer.commitSync(Duration.ofSeconds(30));\n} catch (TimeoutException e) {\n    // Could not complete within timeout\n    // Consumer state is unknown\n    log.warn(\"Commit timeout\", e);\n    // Consider closing consumer and restarting\n}\n```\n\n### AuthenticationException / AuthorizationException\n\n**When Thrown**:\n- Invalid credentials provided\n- Insufficient permissions for operation\n- Credentials expired\n\n**Handling**: These are fatal errors. Close consumer and fix configuration.\n\n### InvalidOffsetException\n\n**When Thrown**:\n- Offset doesn't exist (out of range)\n- Called from `position()`, `seek()`, etc.\n\n**Cause**: Usually log truncation deleted the offset.\n\n**Recovery**:\n```java\ntry {\n    long pos = consumer.position(partition);\n} catch (InvalidOffsetException e) {\n    // Offset was deleted, reset to beginning or end\n    consumer.seekToBeginning(Collections.singleton(partition));\n}\n```\n\n### Transactional Read Semantics Exceptions\n\n**LogTruncationException**:\n- Thrown when `read_committed` consumer detects log truncation\n- Offset specified in commit was deleted\n- Data loss detected\n\n**Recovery**: Reset position and reprocess from earlier checkpoint.\n\n---\n\n## Threading Model and Concurrency\n\n### Single-Threaded Consumer\n\nThe Kafka consumer is **NOT thread-safe**. Only one thread can safely call consumer methods at a time.\n\n**Thread-Unsafe Example**:\n```java\n// INCORRECT - Will cause ConcurrentModificationException or data corruption\nExecutorService executor = Executors.newFixedThreadPool(2);\nexecutor.submit(() -> consumer.poll(Duration.ofMillis(100)));\nexecutor.submit(() -> consumer.commitSync());  // UNSAFE!\n```\n\n### wakeup() - The Exception\n\nThe **ONLY** thread-safe method is `wakeup()`.\n\n**Guarantees**:\n- Can be called from any thread\n- Safely interrupts blocking operations\n- Throws `WakeupException` in target thread\n\n**Pattern**:\n```java\n// Consumer thread\ntry {\n    while (running) {\n        records = consumer.poll(Duration.ofSeconds(1));\n        // Process records\n    }\n} catch (WakeupException e) {\n    if (!running) {\n        // Normal shutdown\n    } else {\n        throw e;  // Unexpected wakeup\n    }\n} finally {\n    consumer.close();\n}\n\n// Shutdown thread (different thread)\npublic void shutdown() {\n    running = false;\n    consumer.wakeup();  // Thread-safe!\n}\n```\n\n### Multi-Threaded Consumption Pattern 1: One Consumer Per Thread\n\n**Architecture**:\n```\n[Consumer 1] -> [Partition 0, 2, 4]\n[Consumer 2] -> [Partition 1, 3, 5]\n[Consumer 3] -> [Partition 7, 8]\n```\n\n**Pros**:\n- Simplest implementation\n- Fastest, no inter-thread coordination\n- Easy in-order processing per partition\n- Thread affinity for data locality\n\n**Cons**:\n- More TCP connections to cluster\n- Less batching efficiency\n- Limited by partition count\n\n### Multi-Threaded Consumption Pattern 2: Decouple Consumption and Processing\n\n**Architecture**:\n```\n[Consumer Thread] -> [BlockingQueue] -> [Processor Pool]\n   polls()           (ConsumerRecords)   (executes work)\n```\n\n**Implementation**:\n```java\nclass ConsumerThread implements Runnable {\n    private final KafkaConsumer<String, String> consumer;\n    private final BlockingQueue<ConsumerRecord<String, String>> queue;\n\n    public void run() {\n        try {\n            consumer.subscribe(Arrays.asList(\"topic\"));\n            while (running) {\n                ConsumerRecords<String, String> records =\n                    consumer.poll(Duration.ofMillis(100));\n\n                for (ConsumerRecord<String, String> record : records) {\n                    queue.put(record);  // Blocks if queue full (backpressure)\n                }\n\n                // Commit after enqueuing\n                consumer.commitAsync();\n            }\n        } catch (WakeupException e) {\n            if (running) throw e;\n        } finally {\n            consumer.close();\n        }\n    }\n}\n\n// Processor threads\nclass ProcessorThread implements Runnable {\n    public void run() {\n        while (true) {\n            ConsumerRecord<String, String> record = queue.take();\n            processRecord(record);\n        }\n    }\n}\n```\n\n**Pros**:\n- Independent scaling of consumers and processors\n- Handles slow processing without blocking fetch\n- Throughput optimization\n\n**Cons**:\n- Ordering guarantees require careful design\n- Manual offset commit coordination\n- Queue memory overhead\n\n### Handling Concurrent Exceptions\n\n**ConcurrentModificationException**:\n```java\n// WRONG - multiple threads calling poll()\nExecutorService executor = Executors.newFixedThreadPool(2);\nexecutor.submit(() -> consumer.poll(Duration.ofMillis(100)));\nexecutor.submit(() -> consumer.poll(Duration.ofMillis(100)));  // ConcurrentModificationException!\n```\n\n**Proper Pattern**:\n```java\n// Only one thread calls poll()\nThread pollThread = new Thread(() -> {\n    try {\n        while (!closed.get()) {\n            ConsumerRecords<K, V> records = consumer.poll(Duration.ofSeconds(1));\n            queue.put(records);  // Hand off to other threads\n        }\n    } catch (WakeupException e) {\n        if (!closed.get()) throw e;\n    } finally {\n        consumer.close();\n    }\n});\npollThread.start();\n\n// Other threads can only call wakeup()\nThread shutdownThread = new Thread(() -> {\n    closed.set(true);\n    consumer.wakeup();  // Thread-safe!\n});\n```\n\n### Thread Interrupts vs wakeup()\n\n**Thread Interrupts** (`Thread.interrupt()`):\n- Supported but not recommended\n- May abort clean shutdown\n- Raises `InterruptException`\n- Mainly for cases where `wakeup()` unavailable\n\n**Best Practice**: Use `wakeup()` instead.\n\n---\n\n## Configuration-Driven Behaviors\n\n### Group Management and Rebalancing\n\n**`group.id`** (required for group management):\n- Identifies consumer group\n- Empty/null disables group coordination (use `assign()` only)\n\n**`session.timeout.ms`** (default: 10000ms):\n- Heartbeat timeout\n- Broker assumes consumer dead if no heartbeat for this duration\n- Triggers rebalance\n- Trade-off: Lower = faster failure detection, higher = more lenient network jitter\n\n**`heartbeat.interval.ms`** (default: 3000ms):\n- Frequency of heartbeat messages\n- Should be 1/3 of `session.timeout.ms`\n\n**`max.poll.interval.ms`** (default: 300000ms = 5 minutes):\n- Max time between `poll()` calls before proactive leave\n- Prevents livelock (heartbeats sent but no progress)\n- Increase for slow processing, decrease for faster rebalance\n\n**`max.poll.records`** (default: 500):\n- Limits records returned per `poll()` call\n- Smaller value = more frequent polls = faster rebalance\n- Larger value = more efficient but delays rebalance\n\n### Offset Management\n\n**`enable.auto.commit`** (default: true):\n- Enable automatic offset commits\n- Commits every `auto.commit.interval.ms`\n- Async, best-effort (may fail silently)\n\n**`auto.commit.interval.ms`** (default: 5000ms):\n- Frequency of auto-commits\n- Actual commits happen between `poll()` calls\n\n**`isolation.level`** (default: read_uncommitted):\n- `read_uncommitted`: Read all messages\n- `read_committed`: Skip aborted transactional messages, stop at LSO\n\n**`default.api.timeout.ms`** (default: 60000ms):\n- Default timeout for `commitSync()`, `position()`, etc.\n- Can be overridden per method call\n\n### Fetching and Performance\n\n**`fetch.min.bytes`** (default: 1):\n- Minimum data broker must accumulate before responding to fetch\n- Larger value = more latency but better batching\n\n**`fetch.max.wait.ms`** (default: 500ms):\n- Max time broker waits to accumulate `fetch.min.bytes`\n- Affects latency when traffic is light\n\n**`max.partition.fetch.bytes`** (default: 1048576 = 1MB):\n- Max bytes per partition per fetch response\n- Must be <= broker `replica.fetch.max.bytes`\n\n**`bootstrap.servers`**:\n- Initial brokers to connect to\n- Used to discover cluster topology\n- Can specify multiple for redundancy\n\n### Deserialization\n\n**`key.deserializer`** and **`value.deserializer`**:\n- Fully qualified class names of deserializer implementations\n- Implements `Deserializer<K>` and `Deserializer<V>`\n- `configure()` method called with configs unless passed to constructor\n\n### Transactional Reads\n\n**`isolation.level=read_committed`** enables:\n- Filtering of aborted transactional messages\n- Consumption stops at LSO (Last Stable Offset)\n- Gaps in offsets due to filtered messages\n- Affects `endOffsets()` and `seekToEnd()`\n\n### Offset Storage\n\n**`enable.auto.commit=false`** with manual commits:\n- Provides exact control over committed offset\n- Allows atomic commit with external side effects\n- Enables \"exactly-once\" with external store\n\n---\n\n## Usage Examples\n\n### Example 1: Basic Subscription with Auto-Commit\n\n```java\nProperties props = new occur only during `poll()` calls when:\n\n1. **Group Membership Changes**:\n   - New consumer joins group\n   - Consumer fails (detected via heartbeat timeout)\n   - Consumer leaves group via `unsubscribe()`\n\n2. **Subscription Changes**:\n   - Topic list changes via `subscribe()`\n   - Regex pattern matches new topics\n   - New partitions added to subscribed topics\n\n3. **Proactive Departure**:\n   - Consumer exceeds `max.poll.interval.ms` without calling `poll()`\n   - Consumer proactively leaves to avoid blocking partitions\n\n### Rebalance Protocols\n\n**Classic/Eager Protocol**:\n- All partitions revoked\n- All consumers rejoin and request new assignments\n- Partition assignment computed\n- All assignments delivered at once\n- `onPartitionsRevoked` always called with all partitions\n- High latency, longer unavailability window\n\n**Cooperative Protocol**:\n- Only affected partitions revoked\n- Partitions remain assigned while rebalancing\n- `onPartitionsRevoked` called only for partitions being revoked\n- Lower latency, reduced stop-the-world time\n\n### Callback Ordering\n\nDuring a rebalance:\n\n1. **Consumer loses partitions** (if any):\n   ```\n   onPartitionsRevoked(revoked_partitions) -> save state/offsets\n   ```\n\n2. **Rebalance computation** (server-side):\n   ```\n   Assignor assigns partitions based on current members\n   ```\n\n3. **Consumer gains partitions** (if any):\n   ```\n   onPartitionsAssigned(new_partitions) -> load state/seek to offsets\n   ```\n\n4. **Generation ID incremented**:\n   ```\n   Consumer now has new generation in group\n   ```\n\n**Ordering Guarantee**: The old consumer's `onPartitionsRevoked` is always called before the new consumer's `onPartitionsAssigned` for the same partition (under normal conditions).\n\n### Rebalance Exceptions\n\n#### CommitFailedException\n\nThrown when `commitSync()` is called and:\n- Partitions were revoked since last successful commit\n- Consumer is no longer in group\n- Consumer was fenced by another instance\n\nThis indicates consumer lost its group membership.\n\n#### RebalanceInProgressException\n\nThrown when calling certain APIs while rebalance is in progress:\n- Rebalance detected, assignment is changing\n- Call `poll()` to complete rebalance, then retry\n\n### Partition Loss and Session Timeout\n\n**Session Timeout** (`session.timeout.ms`, default 10s):\n- Consumer fails to send heartbeat for this duration\n- Broker removes consumer from group\n- Partitions reassigned to other consumers\n- Consumer is fenced and cannot commit offsets\n\n**Max Poll Interval** (`max.poll.interval.ms`, default 5 minutes):\n- Consumer doesn't call `poll()` for this duration\n- Consumer proactively leaves group to prevent livelock\n- Partitions reassigned to other consumers\n\nWhen partitions are lost unexpectedly:\n- `onPartitionsLost()` is called\n- `onPartitionsRevoked()` was NOT called for these partitions\n- Don't attempt to commit offsets (you're no longer the owner)\n\n---\n\n## Flow Control and Position Management\n\n### Pause and Resume\n\n```java\npublic void pause(Collection<TopicPartition> partitions)\npublic void resume(Collection<TopicPartition> partitions)\npublic Set<TopicPartition> paused()\n```\n\n**Behavior**:\n- `pause()`: Stop fetching from specified partitions\n- `resume()`: Resume fetching from paused partitions\n- `poll()` still executes heartbeat and group management\n- `poll()` does NOT return records from paused partitions\n\n**Semantics**:\n- Paused partitions retain their position\n- Resuming continues from same position\n- Pause state survives across rebalances (but not assignment changes)\n\n**Use Cases**:\n- Backpressure: Pause when processing queue is full\n- Prioritization: Fetch from some partitions before others\n- Flow control: Decouple consumption from processing\n\n### Current Lag Calculation\n\n```java\npublic OptionalLong currentLag(TopicPartition partition)\n```\n\n**Returns**: Approximate lag in number of records.\n\n```\nlag = endOffsets() - position()\n```\n\n**Use Cases**:\n- Monitor consumer lag\n- Trigger alerts\n- Decide when to backoff\n\n### Beginning and End Offsets\n\n```java\npublic Map<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions)\npublic Map<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions, Duration timeout)\npublic Map<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions)\npublic Map<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions, Duration timeout)\n```\n\n**Returns**: Offset range for given partitions.\n\n**Semantics**:\n- `beginningOffsets()`: First offset available (based on lag\n- Determine range to reprocess\n- Reset to beginning or end\n\n---\n\n## Exception Handling\n\n### CommitFailedException\n\n**When Thrown**:\n- `commitSync()` called after consumer lost group membership\n- Partition no longer assigned due to rebalance\n- Consumer fenced by another instance\n\n**Handling**:\n```java\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n    // Partitions were revoked during commit\n    // Consumer is no longer in group\n    // Cannot recover from this error\n    log.error(\"Commit failed, consumer fenced\", e);\n    break;  // Exit polling loop\n}\n```\n\n**Recovery**: Must rejoin group via `poll()`. Check if partitions are still assigned.\n\n### WakeupException\n\n**When Thrown**:\n- `wakeup()` called from another thread during blocking operation\n- Typically in `poll()`, `commitSync()`, `position()`, etc.\n\n**Handling Pattern**:\n```java\nprivate final AtomicBoolean closed = new AtomicBoolean(false);\n\nvoid run() {\n    try {\n        while (!closed.get()) {\n            ConsumerRecords records = consumer.poll(Duration.ofSeconds(1));\n            // Process records\n        }\n    } catch (WakeupException e) {\n        if (!closed.get()) throw e;  // Rethrow if not closing\n    } finally {\n        consumer.close();\n    }\n}\n\npublic void shutdown() {\n    closed.set(true);\n    consumer.wakeup();  // Called from another thread\n}\n```\n\n**Semantics**: `wakeup()` is the only thread-safe consumer method. Use it to interrupt blocking operations from external threads.\n\n### InterruptException\n\n**When Thrown**:\n- Thread calling consumer is interrupted (Thread.interrupt() called)\n- Raised from blocking operations\n\n**Handling**: Similar to `WakeupException`. Generally avoid thread interrupts and use `wakeup()` instead.\n\n### TimeoutException\n\n**When Thrown**:\n- Operation exceeds specified timeout\n- Returned from `poll()`, `commitSync()`, `position()`, etc.\n\n**Handling**:\n```java\ntry {\n    consumer.commitSync(Duration.ofSeconds(30));\n} catch (TimeoutException e) {\n    // Could not complete within timeout\n    // Consumer state is unknown\n    log.warn(\"Commit timeout\", e);\n    // Consider closing consumer and restarting\n}\n```\n\n### AuthenticationException / AuthorizationException\n\n**When Thrown**:\n- Invalid credentials provided\n- Insufficient permissions for operation\n- Credentials expired\n\n**Handling**: These are fatal errors. Close consumer and fix configuration.\n\n### InvalidOffsetException\n\n**When Thrown**:\n- Offset doesn't exist (out of range)\n- Called from `position()`, `seek()`, etc.\n\n**Cause**: Usually log truncation deleted the offset.\n\n**Recovery**:\n```java\ntry {\n    long pos = consumer.position(partition);\n} catch (InvalidOffsetException e) {\n    // Offset was deleted, reset to beginning or end\n    consumer.seekToBeginning(Collections.singleton(partition));\n}\n```\n\n### Transactional Read Semantics Exceptions\n\n**LogTruncationException**:\n- Thrown when `read_committed` consumer detects log truncation\n- Offset specified in commit was deleted\n- Data loss detected\n\n**Recovery**: Reset position and reprocess from earlier checkpoint.\n\n---\n\n## Threading Model and Concurrency\n\n### Single-Threaded Consumer\n\nThe Kafka consumer is **NOT thread-safe**. Only one thread can safely call consumer methods at a time.\n\n**Thread-Unsafe Example**:\n```java\n// INCORRECT - Will cause ConcurrentModificationException or data corruption\nExecutorService Management and Rebalancing\n\n**`group.id`** (required for group management):\n- Identifies consumer group\n- Empty/null disables group coordination (use `assign()` only)\n\n**`session.timeout.ms`** (default: 10000ms):\n- Heartbeat timeout\n- Broker assumes consumer dead if no heartbeat for this duration\n- Triggers rebalance\n- Trade-off: Lower = faster failure detection, higher = more lenient network jitter\n\n**`heartbeat.interval.ms`** (default: 3000ms):\n- Frequency of heartbeat messages\n- Should be 1/3 of `session.timeout.ms`\n\n**`max.poll.interval.ms`** (default: 300000ms = 5 minutes):\n- Max time between `poll()` calls before proactive leave\n- Prevents livelock (heartbeats sent but no progress)\n- Increase for slow processing, decrease for faster rebalance\n\n**`max.poll.records`** (default: 500):\n- Limits records returned per `poll()` call\n- Smaller value = more frequent polls = faster rebalance\n- Larger value = more efficient but delays rebalance\n\n### Offset Management\n\n**`enable.auto.commit`** (default: true):\n- Enable automatic offset commits\n- Commits every `auto.commit.interval.ms`\n- Async, best-effort (may fail silently)\n\n**`auto.commit.interval.ms`** (default: 5000ms):\n- Frequency of auto-commits\n- Actual commits happen between `poll()` calls\n\n**`isolation.level`** (default: read_uncommitted):\n- `read_uncommitted`: Read all messages\n- `read_committed`: Skip aborted transactional messages, stop at LSO\n\n**`default.api.timeout.ms`** (default: 60000ms):\n- Default timeout for `commitSync()`, `position()`, etc.\n- Can be overridden per method call\n\n### Fetching and Performance\n\n**`fetch.min.bytes`** (default: 1):\n- Minimum data broker must accumulate before responding to fetch\n- Larger value = more latency but better batching\n\n**`fetch.max.wait.ms`** (default: 500ms):\n- Max time broker waits to accumulate `fetch.min.bytes`\n- Affects latency when traffic is light\n\n**`max.partition.fetch.bytes`** (default: 1048576 = 1MB):\n- Max bytes per partition per fetch response\n- Must be <= broker `replica.fetch.max.bytes`\n\n**`bootstrap.servers`**:\n- Initial brokers to connect to\n- Used to discover cluster topology\n- Can specify multiple for redundancy\n\n### Deserialization\n\n**`key.deserializer`** and **`value.deserializer`**:\n- Fully qualified class names of deserializer implementations\n- Implements `Deserializer<K>` and `Deserializer<V>`\n- `configure()` method called with configs unless passed to constructor\n\n### Transactional Reads\n\n**`isolation.level=read_committed`** enables:\n- Filtering of aborted transactional messages\n- Consumption stops at LSO (Last Stable Offset)\n- Gaps in offsets due to filtered messages\n- Affects `endOffsets()` and `seekToEnd()`\n\n### Offset Storage\n\n**`enable.auto.commit=false`** with manual commits:\n- Provides exact control over committed offset\n- Allows atomic commit with ## Overview

The `KafkaConsumer` is a Kafka client that consumes records from a Kafka cluster. It transparently handles broker failures and adapts to partition migration within the cluster. The consumer interacts with brokers to enable groups of consumers to load balance consumption using consumer groups.

### Key Characteristics

- **Connection Management**: The consumer maintains TCP connections to necessary brokers. Failure to close the consumer will leak these connections.
- **Thread Safety**: The consumer is NOT thread-safe except for the `wakeup()` method.
- **Cross-Version Compatibility**: Communicates with brokers version 0.10.0 or newer. Some features may not be available on older brokers.

### Architecture Model

The Kafka consumer operates on an **event loop model**:
- A single consumer thread owns all I/O operations
- Rebalance callbacks execute within the `poll()` call (not asynchronously)
- The consumer maintains heartbeat, fetch, and group coordination operations
