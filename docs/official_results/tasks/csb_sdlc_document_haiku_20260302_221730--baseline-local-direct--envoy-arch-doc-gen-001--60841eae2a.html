# Architecture Document: Envoy HTTP Connection Manager

**Repository:** envoyproxy/envoy
**Output:** Write your document to `/workspace/documentation.md`

## Task

Produce an architecture document for Envoy's **HTTP Connection Manager** (HCM) subsystem. The document must explain how the major components interact to process an HTTP request from arrival on a downstream connection through to upstream dispatch. Do not simply list APIs — explain the **design**, **data flow**, and **extension points**.

## Scope

Your document must cover these four components and how they work together:

### 1. ConnectionManagerImpl
The central network filter that owns the connection lifecycle. Explain:
- How it implements `Network::ReadFilter` to receive raw bytes
- How it creates and manages `ActiveStream` objects for each HTTP request
- Its role in codec creation (lazy codec instantiation for H1/H2 vs. H3)
- Connection-level concerns: drain decisions, overload management, watermark-based flow control

### 2. FilterManager and the HTTP Filter Chain
The per-stream filter chain execution engine. Explain:
- The decoder filter chain (request path) and encoder filter chain (response path)
- How `FilterChainFactory` creates filters for each stream
- Filter iteration: how `decodeHeaders`/`decodeData`/`decodeTrailers` propagate through decoder filters
- How filters can stop iteration, modify headers, or send local replies
- The distinction between `StreamDecoderFilter`, `StreamEncoderFilter`, and `StreamFilter` (dual)

### 3. Router Filter
The terminal decoder filter that forwards requests upstream. Explain:
- Route selection: how the router uses `RouteConfiguration` to pick a cluster
- How it obtains an HTTP connection pool from `ClusterManager`
- Upstream request lifecycle: `UpstreamRequest` creation, retry logic, timeout handling
- Shadow routing and hedged request support (if applicable)

### 4. Cluster Manager and Upstream Connectivity
The upstream connection pool and cluster management layer. Explain:
- How `ClusterManagerImpl` provides connection pools per cluster
- Load balancing: how the router obtains a host from the cluster's load balancer
- Connection pool mechanics: the relationship between logical connection pools and physical connections
- How health checking and outlier detection feed back into the load balancer

## Document Requirements

1. **Component Responsibilities** — what each component owns
2. **Data Flow** — the path of an HTTP request from downstream bytes to upstream dispatch, and the response path back
3. **Extension Points** — where users extend Envoy (HTTP filters, cluster extensions, load balancers, access loggers)
4. **Error Handling** — how errors at each stage (codec errors, filter errors, upstream failures) are handled
5. **relevant source files** — reference the actual source files in the envoyproxy/envoy repository

## Anti-Requirements

- Do NOT generate a simple API listing or header-file dump
- Do NOT fabricate class names or file paths that don't exist in the repository
- Do NOT cover Envoy components outside the HCM request path (e.g., xDS config delivery, listener management)

}\n  Iterator end() { return entries_.end(); }\n};\n\n// Encoder filters: reverse iteration\nstruct StreamEncoderFilters {\n  std::vector<ActiveStreamEncoderFilterPtr> entries_;\n  Iterator begin() { return entries_.rbegin(); }  // reverse begin\n  Iterator end() { return entries_.rend(); }      // reverse end\n};\n```\n\n#### Filter Lifecycle and Iteration Control\n\nEach filter wrapper (ActiveStreamDecoderFilter/ActiveStreamEncoderFilter) manages iteration state:\n\n```cpp\nstruct ActiveStreamFilterBase {\n  enum class IterationState {\n    Continue,        // Continue to next filter\n    StopIteration,   // Stop, but continue processing later events\n    StopAllIteration // Buffer data, stop processing until conditions change\n  };\n\n  bool commonHandleAfterHeadersCallback(FilterHeadersStatus status, bool& end_stream);\n  bool commonHandleAfterDataCallback(FilterDataStatus status, Buffer::Instance& provided_data,\n                                     bool& buffer_was_streaming);\n};\n```\n\nWhen a filter returns:\n- **FilterHeadersStatus::Continue**: The next filter is invoked immediately\n- **FilterHeadersStatus::StopIteration**: Iteration halts; the filter must explicitly call `continueDecoding()` or `continueEncoding()`\n- **FilterHeadersStatus::StopAllIteration**: In addition to stopping iteration, data buffering is enabled\n\n#### Filter Chain Factory\n\nThe `FilterChainFactory` interface allows configuration of which filters are instantiated for each stream:\n\n```cpp\nclass FilterChainFactory {\npublic:\n  virtual void createFilterChain(FilterChainManager& manager) PURE;\n};\n```\n\nAt stream creation, the factory is invoked with the FilterManager to install decoder/encoder filters in the appropriate order. Filters are instantiated in order and may be:\n- **Decoder-only**: Implement `StreamDecoderFilter`\n- **Encoder-only**: Implement `StreamEncoderFilter`\n- **Dual filters**: Implement `StreamFilter` (both decoder and encoder)\n\n#### Local Reply Mechanism\n\nIf a filter (or any upstream component) detects an error, it can issue a local reply:\n\n```cpp\nfilter_manager_.sendLocalReply(\n  Http::Code::BadRequest,\n  \"Invalid header\",\n  [](ResponseHeaderMap& headers) { /* modify headers */ },\n  absl::nullopt,  // grpc_status\n  \"filter_validation_failed\"\n);\n```\n\nThe FilterManager:\n1. Stops further filter iteration\n2. Creates response headers (with the provided status code)\n3. Invokes encoder filters in reverse order\n4. Sends the response directly to the downstream codec\n5. Resets the stream\n\n---\n\n### 3. Router Filter\n\n**Location:** `source/common/router/router.h` and `source/common/router/router.cc`\n\nThe router filter is the **terminal decoder filter** in the standard HCM filter chain. Its responsibility is to:\n1. Select a route based on request properties\n2. Determine the target cluster/upstream service\n3. Create an `UpstreamRequest` and forward the request to the cluster manager\n4. Handle retries, timeouts, and upstream response processing\n\n#### Route Selection\n\nThe router uses the `RouteConfiguration` (obtained from the route provider) to match the incoming request:\n\n```cpp\n// From router_filter_interface.h\nclass Router {\n  virtual void setRoute(Router::RouteConstSharedPtr route) override {\n    // Cache the selected route\n  }\n};\n```\n\nRoute matching considers:\n- **Virtual hosts**: Hostname-based grouping of routes (from `:authority` header or SNI)\n- **Routes**: Ordered list of route entries matched by:\n  - Path prefix/regex\n  - Methods (GET, POST, etc.)\n  - Headers and header patterns\n  - Query parameters\n  - Traffic policy (canary, shadow)\n\nOnce a route is selected, the router extracts the **cluster name** from the route entry and queries the `ClusterManager` for an HTTP connection pool.\n\n#### Upstream Request Lifecycle\n\nWhen a route is selected, the router creates an `UpstreamRequest`:\n\n```cpp\nclass UpstreamRequest : public GenericConnectionPoolCallbacks,\n                        public UpstreamToDownstreamImplBase {\n  void acceptHeadersFromRouter(bool end_stream);\n  void acceptDataFromRouter(Buffer::Instance& data, bool end_stream);\n  void onPoolReady(std::unique_ptr<GenericUpstream>&& upstream,\n                   Upstream::HostDescriptionConstSharedPtr host,\n                   const Network::ConnectionInfoProvider& address_provider,\n                   StreamInfo::StreamInfo& info,\n                   absl::optional<Http::Protocol> protocol) override;\n};\n```\n\nThe flow:\n\n1. **Request Pools**: The router calls `cluster_manager_->getPool()` passing:\n   - Cluster name\n   - Load balancing context (for host selection)\n   - Protocol preferences (H1, H2, H3)\n\n2. **Pool Acquisition**: The ClusterManager returns a connection pool. If no connection is available:\n   - **Pool Pending**: The UpstreamRequest's `onPoolReady()` callback is invoked once a connection is obtained\n   - **Pool Failure**: `onPoolFailure()` is invoked; the router issues a local reply with 503 or 504\n\n3. **Request Forwarding**: Once `onPoolReady()` is called with an upstream connection:\n   - An `UpstreamCodecFilter` is instantiated within an `UpstreamFilterManager`\n   - Request headers/body/trailers are forwarded to the upstream codec\n   - The upstream codec encodes and sends the request on the wire\n\n4. **Response Reception**: The upstream codec decodes response bytes and invokes:\n   - `decodeHeaders()`: Response status and headers\n   - `decodeData()`: Response body chunks\n   - `decodeTrailers()`: Optional trailers\n   - These flow backwards through the encoder filter chain to the downstream encoder\n\n#### Retry Logic\n\nThe router implements sophisticated retry handling via `RetryStateImpl`:\n\n```cpp\nclass RetryStateImpl {\n  void onUpstreamReset(Upstream::UpstreamResetReason reason);\n  bool shouldRetry(absl::optional<Http::Code> code, bool reset_reason_is_timeout);\n};\n```\n\nRetries are triggered by:\n- **Connection failures**: Pool timeout, connection refused, connection reset\n- **Specific response codes**: 5xx errors (configurable via retry policies)\n- **Timeout**: Per-try timeout exceeded\n\nConfiguration specifies:\n- Maximum retry count\n- Backoff policy (exponential with jitter)\n- Retriable response codes\n- Host selection strategy (use same host, alternate host)\n\n#### Timeout Management\n\nThe router maintains two timeout levels:\n\n- **Request timeout**: Total time allowed for the request (including retries)\n- **Per-try timeout**: Time limit for each individual attempt\n\nThese are set via:\n- Route configuration\n- `x-envoy-upstream-rq-timeout-ms` request header\n- `x-envoy-upstream-rq-per-try-timeout-ms` header\n\nWhen a timeout occurs:\n1. The current upstream request is reset\n2. If retries remain, a new attempt is initiated\n3. Otherwise, a 504 response is sent\n\n#### Hedged Requests (Per-Try Timeout)\n\nIf enabled, the router can initiate a secondary request before the primary completes:\n\n```cpp\n// When per_try_timeout triggers before the full request timeout:\n// 1. A new upstream request is started\n// 2. The first response to complete is used\n// 3. The slower request is abandoned\n```\n\n#### Shadow Routing\n\nThe router supports shadowing requests to a shadow cluster for testing:\n\n```cpp\nif (FilterUtility::shouldShadow(shadow_policy, runtime, stable_random)) {\n  auto shadow_request = std::make_unique<UpstreamRequest>(...);\n  shadow_request->acceptHeadersFromRouter(end_stream);\n  // Response is silently discarded; no feedback to client\n}\n```\n\n---\n\n### 4. Cluster Manager and Upstream Connectivity\n\n**Location:** `source/common/upstream/cluster_manager_impl.h` and related files\n\nThe ClusterManager is the upstream connection pool and cluster management layer. It:\n1. Owns a registry of Cluster objects (one per configured cluster)\n2. Provides HTTP and TCP connection pools for each cluster\n3. Coordinates health checking and outlier detection\n4. Implements load balancing across hosts within a cluster\n\n#### ClusterManager Structure\n\n```cpp\nclass ClusterManagerImpl : public ClusterManager {\n  Cluster& cluster(const std::string& name);\n  Http::ConnectionPool::InstancePtr\n  getHttpConnectionPool(const ClusterConnectivityState& state,\n                        LoadBalancerContext* lb_context);\n};\n```\n\nEach `Cluster` object encapsulates:\n- A set of hosts/endpoints\n- A load balancer factory\n- Health checking configuration\n- Outlier detection rules\n- Circuit breaker limits\n\n#### Connection Pool Mechanics\n\nWhen the router requests a connection pool, the flow is:\n\n```\nRouter::getPool(cluster_name, load_balancing_context)\n  \u2193\nClusterManager::getPool()\n  \u2193\nCluster::getConnPool(dispatcher, load_balancer_context)\n  \u2193\nConnectionPoolGrid::getPool()  [per-protocol instance]\n  \u2193\nConnPoolBase::getHTTP1Connection() [if HTTP/1 pool requested]\n  \u2193\n{\n  if (connection available and idle) {\n    return existing connection\n  } else if (connection limit not reached) {\n    create new connection and return\n  } else {\n    queue pending request\n  }\n}\n```\n\nA connection pool manages:\n- **Active connections**: Connections with in-flight requests\n- **Idle connections**: Connections available for reuse\n- **Pending requests**: Requests waiting for an available connection\n- **Watermark buffers**: Per-connection write buffers with high/low water marks\n\n#### Host Selection (Load Balancing)\n\nThe load balancer selects a specific host (endpoint) for each request:\n\n```cpp\nclass LoadBalancer {\n  HostConstSharedPtr chooseHost(LoadBalancerContext* context);\n};\n```\n\nThe router provides a `LoadBalancerContext` containing:\n- Request metadata (headers, cookies, etc.) for hash-based selection\n- Current retry count (for retry host selection)\n- Upstream zone information (for zone-aware routing)\n\nLoad balancing algorithms include:\n- **Round Robin**: Distribute evenly across healthy hosts\n- **Least Connections**: Select the host with fewest active connections\n- **Ring Hash**: Consistent hashing for session affinity\n- **Random**: Random selection\n- **Maglev**: Fast, deterministic hash with minimal disruption on host changes\n- **Locality-aware**: Prefer hosts in the same zone/region as the client\n\n#### Health Checking and Outlier Detection\n\n**Health Checking**: Periodic health checks (HTTP, TCP, gRPC) are configured per cluster:\n- Determines which hosts are available for traffic\n- Failed hosts are marked as unhealthy and excluded from load balancing\n- Recovered hosts are re-enabled\n\n**Outlier Detection**: Monitors request failures at runtime:\n- Tracks failure rates per host\n- Ejects hosts that exceed failure thresholds (e.g., >50% failure rate over 10 requests)\n- Periodically attempts to re-include ejected hosts\n- Provides fast, adaptive failure response (complementing health checks)\n\nBoth feed into the load balancer's host selection, ensuring traffic only reaches healthy hosts.\n\n#### Dynamic Cluster Discovery (CDS)\n\nClusters can be discovered dynamically via Cluster Discovery Service (CDS):\n\n```cpp\nclass CdsApi : public Config::SubscriptionCallbacks {\n  void onConfigUpdate(const Protobuf::RepeatedPtrField<ProtobufWkt::Any>& resources) override;\n};\n```\n\nCDS allows:\n- Adding new clusters without restarting\n- Updating cluster configuration (endpoints, health check policies)\n- Removing clusters gracefully\n\n---\n\n## Data Flow: Request Processing Pipeline\n\n### Downstream Request Path (Request Arrival to Upstream Dispatch)\n\n```\n1. Network Layer: Bytes arrive on downstream socket\n   \u2514\u2500\u2192 ConnectionManager::onData(Buffer& data, bool end_stream)\n\n2. Codec Instantiation (first onData() call only)\n   \u2514\u2500\u2192 ConnectionManager::createCodec()\n   \u2514\u2500\u2192 HTTP1/2/3 CodecImpl selected\n   \u2514\u2500\u2192 Codec parses protocol framing\n\n3. Stream Creation: Codec detects request headers\n   \u2514\u2500\u2192 Codec::ServerConnection::newStream() callback\n   \u2514\u2500\u2192 ConnectionManager::newStream(ResponseEncoder& response_encoder)\n   \u2514\u2500\u2192 Creates ActiveStream with FilterManager\n\n4. Request Decoding: Codec delivers framing events\n   \u2514\u2500\u2192 ActiveStream::decodeHeaders(RequestHeaderMap& headers, bool end_stream)\n   \u2514\u2500\u2192 FilterManager::decodeHeaders(headers, end_stream)\n\n5. Filter Chain Execution (Decoder Path): A \u2192 B \u2192 C\n   \u251c\u2500\u2192 FilterA::decodeHeaders() \u2192 returns Continue\n   \u251c\u2500\u2192 FilterB::decodeHeaders() \u2192 returns Continue\n   \u251c\u2500\u2192 RouterFilter::decodeHeaders()\n   \u2502   \u251c\u2500\u2192 Route selection (virtual host + route matching)\n   \u2502   \u251c\u2500\u2192 Cluster selection from route\n   \u2502   \u2514\u2500\u2192 ClusterManager::getPool(cluster_name, lb_context)\n   \u2502\n   \u2514\u2500\u2192 Pool resolution callback invoked (synchronous or deferred)\n       \u251c\u2500\u2192 Host selection via load balancer\n       \u251c\u2500\u2192 Connection pool creates/reuses connection to host\n       \u2514\u2500\u2192 UpstreamRequest created\n\n6. Body Data Processing: If request has body\n   \u2514\u2500\u2192 ActiveStream::decodeData(Buffer& data, bool end_stream)\n   \u2514\u2500\u2192 FilterManager::decodeData(data, end_stream)\n   \u2514\u2500\u2192 Filters iterate in order (A \u2192 B \u2192 Router)\n   \u2514\u2500\u2192 Router accumulates or forwards to upstream via UpstreamRequest\n\n7. Trailers Processing: If request has trailers\n   \u2514\u2500\u2192 ActiveStream::decodeTrailers(RequestTrailerMap& trailers)\n   \u2514\u2500\u2192 FilterManager::decodeTrailers(trailers)\n   \u2514\u2500\u2192 Filters iterate in order\n   \u2514\u2500\u2192 Router forwards to upstream\n\n8. Upstream Request Forwarding (UpstreamRequest)\n   \u2514\u2500\u2192 UpstreamCodecFilter encodes headers/data/trailers\n   \u2514\u2500\u2192 Codec writes to upstream connection\n   \u2514\u2500\u2192 Request transmitted on wire\n```\n\n### Upstream Response Path (Response Reception to Downstream Delivery)\n\n```\n1. Upstream Response: Bytes received on upstream socket\n   \u2514\u2500\u2192 Upstream codec parses HTTP framing\n\n2. Response Decoding: Codec detects response headers\n   \u2514\u2500\u2192 Codec::ResponseDecoder::decode1xxHeaders() / decodeHeaders()\n   \u2514\u2500\u2192 UpstreamCodecFilter receives via CodecBridge\n   \u2514\u2500\u2192 UpstreamFilterManager::encodeHeaders() [confusingly named]\n\n3. Upstream Filter Chain: Encoder filters (reverse order C \u2192 B \u2192 A)\n   \u2514\u2500\u2192 Allow modification of response before sending to client\n\n4. Response Delivery to Router Filter\n   \u2514\u2500\u2192 UpstreamRequest::decode1xxHeaders() / decodeHeaders() [legacy API]\n   \u2514\u2500\u2192 Router notifies downstream filter chain via\n       RouterFilterInterface::onUpstream1xxHeaders() / onUpstreamHeaders()\n\n5. Response Encoding: Router injects response into downstream encoder chain\n   \u2514\u2500\u2192 FilterManager::encodeHeaders() [actual encoding operation]\n   \u2514\u2500\u2192 Encoder filters iterate in reverse (C \u2192 B \u2192 A)\n   \u2514\u2500\u2192 Each filter may modify response headers\n\n6. Codec Encoding: Final encoder (ConnectionManager's codec)\n   \u2514\u2500\u2192 ActiveStream::encodeHeaders(ResponseHeaderMap& headers, bool end_stream)\n   \u2514\u2500\u2192 Codec encodes headers into wire format\n\n7. Response Body: Upstream body data\n   \u2514\u2500\u2192 UpstreamRequest::decodeData() \u2192 Router \u2192 FilterManager::encodeData()\n   \u2514\u2500\u2192 Encoder chain processes data\n   \u2514\u2500\u2192 Codec encodes to wire\n\n8. Trailers and Completion\n   \u2514\u2500\u2192 UpstreamRequest::decodeTrailers() \u2192 Router \u2192 FilterManager::encodeTrailers()\n   \u2514\u2500\u2192 Downstream codec sends trailers\n   \u2514\u2500\u2192 Stream marked complete\n\n9. Cleanup\n   \u2514\u2500\u2192 ActiveStream marked for deferred deletion\n   \u2514\u2500\u2192 Stream removed from connection manager's stream list\n   \u2514\u2500\u2192 Upstream connection returned to pool\n```\n\n---\n\n## Extension Points\n\nEnvoy provides several extension points where users can inject custom logic:\n\n### 1. HTTP Filters\n\n**Interface:** `envoy::http::StreamFilter` (decoder + encoder) or `StreamDecoderFilter` / `StreamEncoderFilter`\n\n**Instantiation:** Via `FilterChainFactory` in HTTP connection manager configuration\n\n**Common patterns:**\n\n- **Authentication/Authorization**: Check credentials, modify headers\n- **Rate Limiting**: Enforce per-user or per-IP request limits\n- **Request/Response Transformation**: Modify headers, body, metadata\n- **Observability**: Extract metrics, trace spans\n- **Caching**: Return cached responses for GET requests\n\n**Example:** The router filter itself is instantiated as the terminal decoder filter via the standard factory mechanism.\n\n### 2. Cluster Extensions\n\n**Interfaces:**\n- `Cluster`: Load balancer factory, health checks, circuit breakers\n- `LoadBalancer`: Host selection algorithm\n- `HealthChecker`: Protocol-specific health check implementation\n\n**Extension points:**\n\n- **Load Balancer Implementations**: Custom algorithms (e.g., weighted round-robin with failures, geo-aware)\n- **Health Check Protocols**: Custom protocols beyond HTTP, TCP, gRPC\n- **Outlier Detection Algorithms**: Custom failure detection logic\n\n### 3. Route Configuration\n\n**Interface:** `RouteConfiguration` (typically from file or RDS)\n\n**Extension points:**\n\n- **Route matching**: Virtual hosts, path/method/header matching\n- **Cluster selection**: Static cluster, weighted clusters, header-based routing, regex path matching\n- **Per-route configuration**: Timeouts, retries, shadows, traffic policies\n\n### 4. Request/Response Lifecycle Hooks\n\n**Via FilterState and Callbacks:**\n\n- **FilterState**: Arbitrary key-value storage on `StreamInfo` for passing data between filters\n- **Access Logging**: Custom log sinks and formats\n- **Tracing**: Span creation and annotation at multiple stages\n\n### 5. Error Handling and Local Replies\n\nFilters can trigger local replies (responses generated by Envoy, not forwarded to upstream):\n\n```cpp\n// From any filter\nfilter_manager_.sendLocalReply(\n  Http::Code::Forbidden,\n  \"Access denied\",\n  modify_headers_fn,\n  grpc_status,\n  \"authz_denied\"\n);\n```\n\nThis allows:\n- **Early rejection** before routing\n- **Custom error responses** with appropriate status codes and bodies\n- **Request body discarding** (no upstream connection initiated)\n\n---\n\n## Error Handling\n\n### Codec Errors\n\nIf the incoming data is invalid HTTP:\n- The codec throws an exception or detects a protocol violation\n- ConnectionManager catches the error and issues a 400 Bad Request local reply\n- The connection may be closed (depending on severity)\n\n### Filter Errors\n\nIf a filter encounters an error:\n- It can issue a local reply via `sendLocalReply()`\n- This stops filter iteration and sends a response to the client\n- The upstream request (if any) is canceled\n\n### Upstream Failures\n\nSeveral upstream failure modes are handled:\n\n**Connection Failures:**\n- Router requests a pool; connection fails before headers sent\n- If retries remain, a new attempt is made to a different host\n- If retries exhausted, a local 503 (unavailable) or 504 (gateway timeout) reply is sent\n\n**Slow/Unresponsive Upstream:**\n- Per-try timeout is exceeded\n- Current upstream request is reset\n- New attempt initiated (if retries remain)\n- If hedging is enabled, secondary request may already be in flight\n\n**Upstream Reset/Protocol Violation:**\n- Upstream closes connection unexpectedly\n- Router detects the reset via `onResetStream()` callback\n- Request can be retried if safe (GET, HEAD, OPTIONS, etc.)\n\n**Response Too Large:**\n- FilterManager detects response body exceeds configured limit\n- Stream is reset with a local 413 Payload Too Large reply\n\n---\n\n## Request Flow Examples\n\n### Simple Request-Response\n\n```\nClient \u2192 [GET /api/users HTTP/1.1]\n  \u2193\nConnectionManager::onData()\n  \u2193\nCodec detects HTTP/1.1 request\n  \u2193\nActiveStream created, FilterManager instantiated\n  \u2193\nDecoder filters execute:\n  - Auth filter: validates credentials \u2713\n  - Rate limit filter: checks quota \u2713\n  - Router: selects route, gets pool to \"user-service\" cluster\n  \u2193\nLoad balancer selects host 10.0.1.5:8080\n  \u2193\nConnection pool returns connection\n  \u2193\nUpstreamRequest forwards headers to upstream\n  \u2193\nUpstream server processes, returns [200 OK, body: [...]]\n  \u2193\nUpstreamRequest receives response headers\n  \u2193\nEncoder filters execute:\n  - Router: processes upstream response\n  - Rate limit filter: records request \u2713\n  - Auth filter: clears audit token \u2713\n  \u2193\nActiveStream encodes response to downstream\n  \u2193\nClient receives response\n```\n\n### Request with Retry\n\n```\nClient \u2192 [POST /api/orders HTTP/1.1, body]\n  \u2193\n... # Envoy HTTP Connection Manager Architecture

## Overview

Envoy's HTTP Connection Manager (HCM) is a sophisticated subsystem that coordinates the processing of HTTP requests from downstream clients, through a configurable filter chain, to upstream services. This document explains the architecture, data flow, and interaction patterns between the key components that make up this system.

The HCM operates at the network layer (as a `Network::ReadFilter`) and is protocol-agnostic—it transparently handles HTTP/1.1, HTTP/2, and HTTP/3 through pluggable codec implementations. The request processing follows a well-defined pipeline: connection establishment → codec instantiation → per-stream filter chain execution → route selection → upstream forwarding.
