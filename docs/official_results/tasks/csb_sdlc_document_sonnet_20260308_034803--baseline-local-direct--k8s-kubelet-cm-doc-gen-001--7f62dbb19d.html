It implements `lifecycle.PodAdmitHandler`, so it is registered in the kubelet's pod-admit chain (`kubelet.go:1064`).\n\n**Scope** (per-container or per-pod, `scope.go`): determines whether alignment is computed per container or across all containers in a pod simultaneously.\n\n**Policies** (selected by `--topology-manager-policy`):\n\n| Policy | Behaviour |\n|---|---|\n| `none` | Calls `Allocate()` on each hint provider without checking alignment. |\n| `best-effort` | Attempts aligned allocation; admits the pod regardless. |\n| `restricted` | Requires at least one aligned hint; rejects if none found. |\n| `single-numa-node` | Requires all resources on a single NUMA node; rejects otherwise. |\n\nWhen a pod is admitted, the scope calls `GetTopologyHints()` on each registered `HintProvider` (CPU manager, memory manager, device manager \u2014 all implement `HintProvider`). Each provider returns a list of `TopologyHint{NUMANodeAffinity bitmask.BitMask, Preferred bool}`. Failure Modes and Design Tradeoffs\n\n### 5.1 Startup Failures\n\n- Missing cgroup subsystems (`cpu`, `cpuacct`, `cpuset`, `memory`) are hard errors on cgroupv1. The kubelet will not start (`validateSystemRequirements`, `container_manager_linux.go:151`).\n- Swap enabled with `--fail-swap-on=true` (default) aborts startup. With swap on and tmpfs noswap unsupported, the kubelet emits a `PossibleMemoryBackedVolumesOnDisk` warning event but continues.\n- Invalid `cgroup-root` (path must exist on the filesystem) is a hard startup error.\n\n### 5.2 CPU Manager Policy Conflicts\n\n- The `static` policy requires `kube-reserved.cpu + system-reserved.cpu > 0`. If reserved CPUs are zero, the manager refuses to start (`cpu_manager.go:194`).\n- SMT topology mismatches are surfaced as `SMTAlignmentError` during pod admission, not at startup.\n- Checkpoint corruption (e.g. after a policy name change) causes the CPU manager to log an error and drain the node (`cpu_manager.go:234`): _\"please drain node and remove policy state file\"_.\n\n### 5.3 Topology Manager Admission Failures\n\n- With `restricted` or `single-numa-node` policy, pods that cannot be NUMA-aligned will fail admission with a `TopologyAffinityError`. This is a hard rejection; the kubelet will never start the pod without operator intervention (policy change or pod removal).\n- The 8-NUMA-node limit is a deliberate algorithmic cap: hint intersection is exponential in node count, and beyond 8 nodes the state space is considered too large to enumerate safely.\n\n### 5.4 Device Plugin Failures\n\n- A device plugin that disconnects is removed from the active endpoint list. Pods already using those devices continue to run (allocations are in the checkpoint), but new pods cannot be admitted for that resource.\n- `ShouldResetExtendedResourceCapacity()` (`container_manager_linux.go:1068`) detects node recreation (by comparing a fingerprint file) and returns `true`, causing the kubelet to zero out device plugin resource capacities on the node object. This prevents scheduling pods to a node that has lost its device plugin state.\n\n### 5.5 Reconciliation Drift\n\nThe CPU manager's periodic `reconcileState()` covers the window between when a cpuset is assigned in memory and when `UpdateContainerResources` reaches the runtime. If the runtime is temporarily unavailable, the reconciler retries on the next period. During this window, a container's actual cpuset in the kernel may not match the desired assignment.\n\n### 5.6 Cgroup v1/v2 Behavioural Differences\n\n- **CPU shares vs. CPU weight**: cgroupv2 uses `cpu.weight` (range 1\u201310000); cgroupv1 uses `cpu.shares` (range 2\u2013262144). `getCPUWeight()` in `cgroup_manager_linux.go:267` converts between them with a linear mapping.\n- **Memory QoS**: cgroupv2 adds `memory.min` (guaranteed minimum) and `memory.high` (soft throttle) in addition to `memory.limit_in_bytes`. The `Cgroup2MemoryMin` and `Cgroup2MemoryHigh` constants (`cgroup_manager_linux.go:43`) are written as `Unified` resource parameters in `ResourceConfig`.\n\n---\n\n## 6. Extension Points and Customisation Hooks\n\n### 6.1 CPU Manager Policy Options\n\nThe `static` policy supports fine-grained options passed via `--cpu-manager-policy-options`:\n\n- `full-pcpus-only`: Require that exclusive allocations align to full physical CPU cores (prevents hyper-thread splitting).\n- `distribute-cpus-across-numa`: Spread exclusive CPUs across multiple NUMA nodes rather than packing them.\n- `align-by-socket`: Align allocations to CPU socket boundaries.\n\nOptions are validated at startup in `NewStaticPolicyOptions()` (`pkg/kubelet/cm/cpumanager/policy_options.go`). Unknown options cause a startup error, preventing silent misconfiguration.\n\n### 6.2 Topology Manager Policy and Scope\n\nBoth `--topology-manager-policy` and `--topology-manager-scope` are runtime-configurable. The scope determines whether the hint-intersection considers each container independently (`container` scope, `scope_container.go`) or treats all containers in a pod as a single allocation unit (`pod` scope, `scope_pod.go`). The `pod` scope is necessary when containers within a pod share devices and must be co-located on the same NUMA node.\n\n### 6.3 Device Plugin Registration\n\nThird-party device plugins register by creating a Unix socket at `/var/lib/kubelet/device-plugins/<name>.sock`. The plugin watcher (`pkg/kubelet/pluginmanager`) delivers registration events to `deviceManager.GetWatcherHandler()`. This is the primary hardware extension point: any resource that can be represented as a set of opaque device IDs with NUMA affinity can be integrated without modifying kubelet source.\n\n### 6.4 DRA Drivers\n\nDRA drivers register via a separate socket path and implement the `dra.v1` gRPC protocol. Unlike device plugins, DRA allocations are API-server-mediated: the scheduler allocates `ResourceClaims`, and the kubelet's DRA manager calls `NodePrepareResources` / `NodeUnprepareResources` on the driver to make devices available inside the pod's namespaces (via CDI device IDs). This decoupling means DRA can support more complex allocation models (e.g. sharing a device across pods, network fabric allocation) that device plugins cannot express.\n\n**Risk**: DRA drivers run as privileged processes on the node and have the ability to inject arbitrary CDI device specifications (`opts.CDIDevices`, `container_manager_linux.go:737`). A compromised driver can inject devices into any pod. Cluster operators must treat DRA driver deployment with the same trust level as node-level privilege.\n\n### 6.5 QOS Reserved Percentages\n\n`ParseQOSReserved()` (`container_manager.go:267`) allows operators to reserve a percentage of memory for the QoS cgroup tiers via `--qos-reserved`. Only `memory` is supported. This enables over-commitment headroom reservation at the cgroup level independent of pod requests.\n\n### 6.6 `InternalContainerLifecycle` as an Internal Hook\n\n`InternalContainerLifecycle` is not a public extension point \u2014 it is an internal interface (`pkg/kubelet/cm`) consumed only by `kuberuntime`. However, it is the mechanism by which any new resource manager that needs to inject configuration into CRI `ContainerConfig` should register. Adding a new resource manager that influences container startup requires:\n\n1. Implementing `HintProvider` and registering with the topology manager.\n2. Storing allocation state in the manager.\n3. Reading that state in `PreCreateContainer` to modify `containerConfig`.\n4. Registering/deregistering the container in `PreStartContainer` / `PostStopContainer`.\n\n---\n\n## 7. Operational Implications\n\n### 7.1 Performance\n\n**Reconcile period tuning** (`--cpu-manager-reconcile-period`, default 10 s): Each reconcile period triggers an `UpdateContainerResources` RPC for every active container that has a tracked CPU assignment. On nodes with many containers, a short reconcile period increases CRI load. Widening the period increases the drift window between assignment and enforcement after container restarts.\n\n**Topology hint computation cost**: The hint-intersection algorithm in `scope_container.go` / `scope_pod.go` is O(product of hint counts). With the default max of 8 NUMA nodes and multiple hint providers, this is bounded to a few thousand operations per admission and is not a practical bottleneck. However, device plugins that return a large number of fine-grained hints (one hint per device rather than per-NUMA-node aggregation) can expand the search space significantly.\n\n**Cgroup write latency**: Every pod creation issues at least one `cgroupManager.Create()` and one `cgroupManager.Update()` call. Under the systemd driver, these are mediated by D-Bus RPC to systemd, which serialises cgroup operations on slow nodes. Operators on high-density nodes (hundreds of pods/min) may see admission latency dominated by systemd D-Bus round-trips.\n\n### 7.2 Reliability\n\n**Checkpoint files** are the critical recovery artefact. The kubelet stores three checkpoint files under `<kubelet-root>`:\n- `cpu_manager_state`\n- `memory_manager_state`\n- `device-plugins/<checkpoint>`\n\nThese are written atomically via `checkpointmanager`. If they are deleted while the kubelet is stopped, the manager will reconstruct state by querying the CRI for live containers on next start. If the files are corrupted or represent a previous policy name, a hard startup error is surfaced. The recommended recovery procedure is to drain the node and remove the state file before changing policies.\n\n**Policy change**: Changing `--cpu-manager-policy` between `none` and `static` requires draining the node first. The checkpoint format encodes the policy name; a mismatch is detected at startup and blocks the node from accepting new pods until resolved.\n\n### 7.3 Maintainability\n\nThe subsystem uses the **strategy pattern** consistently: `CgroupManager`, `Policy` (CPU/memory), and `Scope` (topology) are all interfaces with multiple concrete implementations. This makes unit-testing straightforward \u2014 `fake_container_manager.go`, `fake_cpu_manager.go`, `fake_memory_manager.go`, and `fake_topology_manager.go` all provide stub implementations used by the kubelet's own test suite.\n\nPlatform variance is handled through Go build tags (`//go:build linux`, `//go:build windows`) rather than runtime switches. Each platform file declares its own `containerManagerImpl` struct with only the fields applicable to that OS. This eliminates nil-check guards throughout the codebase but requires that any new feature be explicitly ported to each platform or left as a no-op in the unsupported file.\n\n---\n\n## 8. # Kubelet Container Manager: Subsystem Architecture Guide

## 1. Subsystem Purpose and Boundaries

The **container manager** (`pkg/kubelet/cm`) is the kubelet's resource governance layer. Its core mandate is to enforce isolation and resource allocation for every process running on the node — from individual containers up to the node itself. It sits between the kubelet's pod-lifecycle orchestration and the Linux kernel's cgroup and CPU/memory topology subsystems.

### What the container manager owns

- **Cgroup hierarchy construction** — creates and maintains the `kubepods/`, `kubepods/burstable/`, and `kubepods/besteffort/` cgroups, plus per-pod sub-cgroups, mapping pod QoS classes to kernel resource controls.
- **Node-level reservations** — enforces `kube-reserved` and `system-reserved` limits by placing kubelet processes and system daemons into dedicated cgroups with bounded resources.
- **Extended resource allocation** — drives the device plugin protocol to advertise, allocate, and track hardware devices (GPUs, FPGAs, SR-IOV VFs, etc.) for containers.
- **CPU and memory affinity** — co-ordinates exclusive CPU pinning and NUMA-local memory placement for Guaranteed pods.
- **Topology-aware allocation** — ensures that CPUs, memory, and devices allocated to a container all reside on the same NUMA nodes.
- **Dynamic Resource Allocation (DRA)** — manages `ResourceClaim` lifecycle for the structured device allocation API.
- **Container lifecycle hooks** — injects resource assignments (cpusets, NUMA memsets) into the CRI container config immediately before container creation.
