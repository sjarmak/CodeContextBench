## Primary Functions

**Run()** - The main entry point that orchestrates the election loop. As documented: "Run starts the leader election loop. Run will not return before leader election loop is stopped by ctx or it has stopped holding the leader lease."

**acquire()** - Attempts to gain leadership by repeatedly calling `tryAcquireOrRenew()` until successful, using jittered retry intervals.

**renew()** - Maintains leadership by periodically refreshing the leader lease before it expires.

**tryAcquireOrRenew()** - Core logic that either creates a new lease record or updates an existing one, implementing both fast-path optimization for current leaders and slow-path fallback.

## Failover and Re-election Mechanisms

**Lease-Based Expiration**: The system doesn't rely on heartbeats alone. Instead, it monitors `LeaseDuration` settings. When a leader fails, other candidates wait for the lease to expire before attempting acquisition.

**Clock Skew Tolerance**: The implementation acknowledges timing challenges: "The tolerance expressed as a maximum tolerated ratio of time passed on the fastest node to time passed on the slowest node can be approximately achieved with a configuration that sets the same ratio of LeaseDuration to RenewDeadline."

**Callbacks for State Transitions**: 
- `OnStartedLeading` fires when a candidate becomes leader
- `OnStoppedLeading` fires when leadership is lost
- `OnNewLeader` triggers when observing leader changes

**Release Mechanism**: The `release()` method enables graceful handoff when the context is cancelled, preventing dual leadership during transitions.

# Leader Election: Acquisition and Failover Handling

## tryAcquireOrRenew Function

The `tryAcquireOrRenew` method implements a two-path strategy:

**Fast Path:** If the current instance is already leading and the lease remains valid, it attempts an optimistic update using the last observed record without fetching fresh data.

**Slow Path:** If the fast path fails or the instance isn't leading, it retrieves the current lease record, checks expiration, and attempts to acquire or renew accordingly. The method increments `LeaderTransitions` when taking leadership from another holder.

## Failover Callbacks

The code defines two critical lifecycle callbacks in `LeaderCallbacks`:

1. **OnStartedLeading**: "called when a LeaderElector client starts leading" - invoked asynchronously when leadership is successfully acquired.

2. **OnStoppedLeading**: "called when a LeaderElector client stops leading" - always invoked on exit, even if leadership was never obtained. Importantly, this callback can fire without a preceding `OnStartedLeading` call.

These callbacks allow applications to manage stateful resources during leadership transitions, enabling graceful failover by triggering cleanup in `OnStoppedLeading` before another instance assumes leadership.
