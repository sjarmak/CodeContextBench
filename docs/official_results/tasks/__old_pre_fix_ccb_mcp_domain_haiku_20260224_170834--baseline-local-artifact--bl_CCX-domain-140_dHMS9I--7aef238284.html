# ExecutionBlock Class Definition

## Location and Purpose

The `ExecutionBlock` class is defined in the ArangoDB query execution system as a fundamental component of the AQL (ArangoDB Query Language) pipeline. According to the header file, it serves as:

**"The main function to produce data in this ExecutionBlock...perform the requested amount of offset, data and fullcount."**

## Key Responsibilities

**Execution Control**: The class manages the lifecycle of query execution through methods like `initializeCursor()` and the primary `execute()` function, which processes data according to an `AqlCallStack`.

**Data Production**: ExecutionBlock instances generate and pass data blocks through the query pipeline, returning execution state (WAITING, HASMORE, or DONE), skip results, and data blocks.

**Dependency Management**: Blocks maintain connections to dependent nodes via `addDependency()`, forming the directed execution graph.

**Performance Monitoring**: The class tracks execution statistics and profiling information through `ExecutionNodeStats` and profiling-level monitoring.

## Design Pattern

ExecutionBlock follows a producer pattern where each block processes upstream data and produces results for downstream consumers, with batch processing using a configurable default size of 1,000 items.

# Optimizer Class Definition

The `Optimizer` class in ArangoDB's AQL module manages execution plan transformation through systematic rule application.

## Core Purpose
The class optimizes query execution plans by applying a series of rules that generate alternative plans, which are then estimated and ranked by cost efficiency.

## Primary Methods for Plan Transformation

**Plan Creation & Management:**
- `createPlans()` - "do the optimization, this does the optimization, the resulting plans are all estimated, sorted by that estimate" and can be retrieved via `getPlans()`
- `addPlan()` - registers a new plan variant produced by an optimizer rule
- `addPlanInternal()` - worker method handling plan addition with rule tracking

**Plan Retrieval:**
- `stealBest()` - transfers ownership of the optimal plan to the caller while deleting alternatives
- `getPlans()` - provides read-only access to all current plans

**Rule Management:**
- `initializeRules()` - sets up applicable optimizer rules based on query options
- `disableRules()` - prevents specified rules from executing using a predicate function
- `enableRule()` / `disableRule()` - control individual rules by index or name

## Key Data Structures

The class maintains two `PlanList` containers: `_plans` for current optimization candidates and `_newPlans` for newly generated variants. A `Stats` struct tracks metrics including rules executed, plans created, and optional per-rule execution timing data.
