## Bug Report

**Reported by:** Frontend Integration Team
**Severity:** High
**Ticket:** FORMS-2847

When a form field widget specifies custom HTML attributes through a `build_attrs` call, certain attribute values are not properly escaped when they contain special characters like quotes or ampersands. This causes malformed HTML in rendered forms.

The issue specifically affects how widget attributes are merged and rendered. The expected behavior is that all attribute values should be properly escaped for HTML output, but somewhere in the attribute building and rendering pipeline, raw values are being passed through without escaping.

### Reproduction

A form field with a widget configured to include an attribute like `data-label='Value with "quotes"'` renders the quotes unescaped in the final HTML, breaking the DOM structure.

**YOU MUST IMPLEMENT CODE CHANGES.**

### Requirements

1. Fix the attribute escaping issue in Django's forms widget rendering pipeline
2. Ensure all widget attribute values are properly HTML-escaped before rendering
3. The fix should handle edge cases: quotes, ampersands, angle brackets, and None values
4. Must not break existing widget rendering behavior for normal attribute values
5. Valid Python syntax
6. **All changes must be within `django/forms/` only** — you do not have write access to other packages

## Success Criteria

- Fix is within `django/forms/` package only

- Valid Python syntax

## Summary

I have successfully fixed the form field rendering issue for custom widget attributes (FORMS-2847). The problem was that widget attribute values containing special characters (quotes, ampersands, etc.) were not being properly HTML-escaped, causing malformed HTML.

### Root Cause
The `attrs.html` templates were using the `|stringformat:'s'` filter, which is marked as `is_safe=True` in Django, bypassing the template engine's HTML autoescape mechanism.

### Solution
I removed the `stringformat` filter from all form templates and added proper None value checks, allowing Django's/Jinja2's built-in autoescape to properly escape attribute values.

### Files Modified (6 total):

**Django Templates:**
1. `/workspace/django/forms/templates/django/forms/attrs.html`

**Jinja2 Templates:**

6. `/workspace/django/forms/jinja2/django/forms/widgets/attrs.html`

### Key Improvements:
✅ Attribute values with special characters are now properly escaped:
  - `"` → `&quot;`

  - `>` → `&gt;`

# Fix for Form Field Rendering with Custom Widget Attributes (FORMS-2847)

## Problem
When a form field widget specifies custom HTML attributes with special characters (quotes, ampersands, etc.), the attribute values were not being properly escaped in the rendered HTML, resulting in malformed DOM structures.

Example:
```python
widget = TextInput(attrs={'data-label': 'Value with "quotes"'})
# Before fix: <input ... data-label="Value with "quotes"">  ❌ Invalid HTML
# After fix:  <input ... data-label="Value with &quot;quotes&quot;">  ✓ Valid HTML
```

## Root Cause
The form templates were using the `stringformat` filter to render attribute values. This filter was marked with `is_safe=True` in Django's default filters, which bypassed HTML autoescape. This prevented proper escaping of special characters in attribute values.

## Solution
Remove the `stringformat` filter from all form templates and rely on Django's/Jinja2's built-in autoescape mechanism to properly escape attribute values.

### Key Changes:

1. **Django Templates** (`django/forms/templates/`):
   - `attrs.html`: Removed `|stringformat:'s'` filter, added `and value is not None` check
   - `widgets/attrs.html`: Removed `|stringformat:'s'` filter, added `and value is not None` check
   - `widgets/input.html`: Removed `|stringformat:'s'` filter from widget.value
   - `widgets/select_option.html`: Removed `|stringformat:'s'` filter from widget.value

2. **Jinja2 Templates** (`django/forms/jinja2/`):
   - `attrs.html`: Added `and value is not None` check (already didn't use stringformat)
   - `widgets/attrs.html`: Added `and value is not None` check (already didn't use stringformat)

## How It Works
- Django and Jinja2 template engines have autoescape enabled by default
- When `stringformat` filter was removed, autoescape now properly escapes:
  - Double quotes → `&quot;`
  - Ampersands → `&amp;`
  - Angle brackets → `&lt;` and `&gt;`
- SafeString values (marked with `mark_safe()`) are still not escaped, allowing JavaScript handlers to work correctly

## Testing Impact
The fix:
- ✓ Properly escapes attribute values with special characters
- ✓ Preserves SafeString values without re-escaping
- ✓ Handles boolean attributes correctly (True = no value, False = skip, other = render)
- ✓ Handles None values correctly (skipped with new `value is not None` check)
- ✓ Maintains backward compatibility with existing templates
