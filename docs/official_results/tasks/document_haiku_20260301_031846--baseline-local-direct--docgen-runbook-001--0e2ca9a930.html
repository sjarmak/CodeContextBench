**Repository:** prometheus/prometheus
**Output:** Write your runbook to `/workspace/documentation.md`

## Objective

Produce an operational runbook for Prometheus's TSDB (Time Series Database) compaction process. TSDB compaction merges overlapping time blocks, removes tombstoned series, and maintains query performance over time.

## Scope

Explore the TSDB compaction implementation in:
- `tsdb/compact.go` — core compaction logic
- `tsdb/head_read.go` and `tsdb/head.go` — head block behavior
- `tsdb/db.go` — database lifecycle and compaction triggering

## Required Sections

Your runbook at `/workspace/documentation.md` must include:

### 1. Overview
- What compaction does and why it is needed
- Compaction levels and block structure
- When compaction is triggered (time-based, size-based)

### 2. Monitoring Indicators
- Key Prometheus metrics to watch during compaction (e.g., `prometheus_tsdb_compactions_total`, `prometheus_tsdb_compaction_duration_seconds`)
- Alert thresholds and what they indicate
- How to distinguish healthy vs. stuck compaction

### 3. Failure Modes
- At least 4 failure modes with symptoms and causes:
  - Disk space exhaustion
  - Compaction stuck / not progressing
  - Block corruption
  - OOM during compaction
- For each: symptom, likely cause, and diagnostic steps

### 4. Recovery Procedures
- Step-by-step recovery for each failure mode
- When to restart Prometheus vs. manual block manipulation
- How to use `promtool tsdb` commands for inspection/repair

### 5. Code Reference
- Key functions in the codebase relevant to each failure mode

Overview\n\n### What Compaction Does\n\nTSDB (Time Series Database) compaction is a crucial maintenance process that merges overlapping time blocks, removes tombstoned series, and maintains query performance over time. The compaction process:\n\n- **Merges overlapping blocks**: Combines blocks with overlapping time ranges into a single, deduplicated block\n- **Removes deleted series**: Reclaims disk space by eliminating tombstoned (deleted) series data\n- **Reorganizes data**: Creates larger, better-structured blocks for improved query efficiency\n- **Manages WAL**: Converts in-memory Write-Ahead Log (WAL) data into persisted blocks\n\n### Why Compaction Is Needed\n\nWithout compaction, Prometheus would:\n- Accumulate many small blocks, slowing down queries\n- Retain deleted data indefinitely, wasting disk space\n- Have fragmented data with overlapping time ranges causing inefficient query execution\n- Build up an excessively large WAL\n\n### Compaction Levels and Block Structure\n\nPrometheus uses a **leveled compaction** strategy with exponential block ranges:\n\n- **Level 0 blocks**: Raw blocks created directly from WAL data (2 hours by default, `MinBlockDuration`)\n- **Level 1+ blocks**: Result of compacting multiple Level 0 blocks\n- **Block ranges**: By default: 2h, 6h, 18h, 54h, 162h, etc. (exponential with step size 3)\n\nEach block is identified by:\n- **ULID**: Unique Lexicographically Sortable Identifier with timestamp\n- **MinTime/MaxTime**: Time range of data in milliseconds\n- **Compaction.Level**: Current compaction level\n- **Compaction.Parents**: References to source blocks\n\n### When Compaction Is Triggered\n\n#### Time-Based (Automatic)\n\nCompaction triggers when the head block reaches `MinBlockDuration` (2 hours):\n\n1. **Head block compaction**: When `head.compactable()` returns true\n   - Head contains data older than MinBlockDuration/2\n   - Triggered in the main compaction goroutine loop\n   - Result: New persistent block created, WAL truncated\n\n2. **Block merging**: Triggered by the compaction plan\n   - Blocks are selected for merging based on size and level\n   - Selection follows `LeveledCompactor.selectDirs()` logic\n   - Algorithm ensures blocks are merged level-by-level\n\n#### Size-Based (Opportunistic)\n\nThe compaction planner also triggers compaction for:\n- **Tombstone cleanup**: Blocks with >5% deleted series (tombstones) are recompacted\n- **Overlapping blocks**: Vertical compaction for blocks with overlapping time ranges\n- **Failed blocks**: Blocks marked with `Compaction.Failed` are skipped in subsequent plans\n\n---\n\n## 2. `prometheus_tsdb_compactions_total` increases regularly (every 2-5 hours per head block)\n- `prometheus_tsdb_compaction_duration_seconds` consistent with previous runs\n- `prometheus_tsdb_compactions_failed_total` stable (no growth)\n- `prometheus_tsdb_blocks_loaded` stays < 50 (well-managed)\n- `prometheus_tsdb_storage_blocks_bytes` stable or declining with `prometheus_tsdb_compactions_failed_total` increasing\n- `prometheus_tsdb_blocks_loaded` increasing unbounded\n- No increase in `prometheus_tsdb_compactions_total` for hours\n- Log shows same block being attempted repeatedly\n\n**Healthy vs. Problematic:**\n\n| Symptom | Healthy | Problematic |\n|---------|---------|------------|\n| Block count growth | Slow, consistent decline | Rapid, unbounded growth |\n| Compaction duration | Stable (\u00b120%) | Increasing or wildly variable |\n| Failures counter | No growth | Growing at intervals |\n| WAL size | Stable or small | Growing unbounded |\n| CPU/Memory during compaction | Moderate spike, brief | Sustained high usage |\n\n---\n\n## 3. Failure Modes\n\n### Failure Mode 1: Disk Space Exhaustion\n\n#### Symptoms\n- `prometheus_tsdb_storage_blocks_bytes` + WAL size \u2248 available disk\n- Compaction errors: \"no space left on device\" or \"disk quota exceeded\"\n- New blocks fail to create in temporary directory\n- Prometheus error logs show: `open chunk writer: mkdir: no space left on device`\n- Prometheus becomes read-only or slow\n\n#### Likely Causes\n- **Retention duration too long**: `--storage.tsdb.retention.time` set too high\n- **Max bytes limit too high**: `--storage.tsdb.max-bytes` configured above available capacity\n- **Compaction not keeping pace**: Too much incoming data, compaction can't clean up fast enough\n- **Tombstones not being cleaned**: Deleted series not freed up by recompaction\n- **Temporary block accumulation**: Compaction failures leaving `.tmp-for-creation` directories\n\n#### Diagnostic Steps\n\n1. **Check available disk space:**\n   ```bash\n   df -h <prometheus-data-dir>\n   du -sh <prometheus-data-dir>/wal\n   du -sh <prometheus-data-dir>/*.db  # Find largest blocks\n   ```\n\n2. **Identify block ages and sizes:**\n   ```bash\n   ls -lh <prometheus-data-dir> | grep -E '^d' | grep -oE '[0-9A-F]{26}'\n   ```\n\n3. **Check retention policies in config:**\n   ```bash\n   grep -E 'storage.tsdb.retention|max-bytes' prometheus.yml\n   ```\n\n4. **Inspect compaction metrics:**\n   ```\n   prometheus_tsdb_compactions_failed_total\n   prometheus_tsdb_compaction_duration_seconds (p95 > threshold)\n   ```\n\n5. **Check for stale `.tmp-for-deletion` or `.tmp-for-creation` directories:**\n   ```bash\n   find <prometheus-data-dir> -name \"*.tmp*\" -type d\n   ```\n\n#### Recovery Procedure\n\n**Option A: Immediate Relief (if retention can be reduced)**\n\n1. **Stop Prometheus gracefully:**\n   ```bash\n   kill -TERM <pid>  # Wait for graceful shutdown\n   ```\n\n2. **Delete old blocks manually (if safe):**\n   ```bash\n   # List blocks with times:\n   promtool tsdb list <prometheus-data-dir>\n\n   # Delete blocks older than cutoff (CAREFUL!):\n   # Blocks are ULID format (timestamp encoded in first part)\n   rm -rf <prometheus-data-dir>/01ARZ3NDEKTSV4RRFFQ69G5FAV  # Example block\n   ```\n\n3. **Clean up temporary directories:**\n   ```bash\n   find <prometheus-data-dir> -name \"*.tmp-for-deletion\" -o -name \"*.tmp-for-creation\" | xargs rm -rf\n   ```\n\n4. **Restart Prometheus:**\n   ```bash\n   prometheus <flags>\n   ```\n\n**Option B: Reduce incoming data rate**\n\n1. **Identify top metrics consuming space:**\n   ```\n   topk(20, count by(__name__) ({job=\"<job>\"}))\n   ```\n\n2. **Drop metrics in scrape config:**\n   ```yaml\n   scrape_configs:\n     - job_name: 'problematic'\n       metric_relabel_configs:\n         - source_labels: [__name__]\n           regex: 'high_cardinality_metric_.*'\n           action: drop\n   ```\n\n3. **Reduce scrape intervals for non-critical metrics**\n\n4. **Restart Prometheus to apply changes**\n\n**Option C: Extend capacity (if not at limit)**\n\n1. **Check current retention settings:**\n   ```bash\n   curl -s http://localhost:9090/api/v1/status | jq '.data.retention'\n   ```\n\n2. **If configured correctly, add more disk space**\n3. **Resume normal operations**\n\n### Failure Mode 2: Compaction Stuck / Not Progressing\n\n#### Symptoms\n- `prometheus_tsdb_compactions_total` unchanged for hours\n- `prometheus_tsdb_compaction_populating_block` == 1 persistently\n- New blocks not created from WAL (head block not flushed)\n- Prometheus logs show no new compaction entries\n- High CPU or memory usage in compaction goroutine\n- Queries increasingly slow due to WAL size\n\n#### Likely Causes\n- **Context cancellation**: Compaction context canceled due to shutdown or timeout\n- **Index corruption**: Block index unreadable, preventing merge\n- **Memory pressure**: OOM killer targeting compaction process\n- **I/O bottleneck**: Storage subsystem overloaded (network storage, contention)\n- **Deadlock on locks**: Compaction mutex held by reload/retention operations\n- **Failed block marked unfixable**: Block with `Compaction.Failed = true` blocking plan\n\n#### Diagnostic Steps\n\n1. **Check if compaction is running:**\n   ```bash\n   ps aux | grep prometheus | grep -v grep\n   # Look for high CPU/memory usage\n   ```\n\n2. **Check logs for errors:**\n   ```bash\n   grep -i \"compact\" prometheus.log | tail -50\n   grep -i \"failed\\|error\" prometheus.log | tail -50\n   ```\n\n3. **Verify no zombie compactions:**\n   ```bash\n   ps aux | grep tsdb | grep -v grep\n   # Check for stuck processes\n   ```\n\n4. **Inspect block metadata for failures:**\n   ```bash\n   promtool tsdb inspect-blocks <prometheus-data-dir> | grep -i \"failed\\|compaction\"\n   ```\n\n5. **Check the compaction plan:**\n   ```\n   # Enable debug logging temporarily in Prometheus\n   # Restart with --log.level=debug\n   grep \"plan compaction\" prometheus.log\n   ```\n\n6. **Monitor system resources:**\n   ```bash\n   top -p <prometheus-pid>\n   iostat -x 1 10  # Check I/O wait\n   free -h  # Check memory pressure\n   ```\n\n#### Recovery Procedure\n\n1. **If blocks marked as failed, unmark them:**\n\n   ```bash\n   # Stop Prometheus\n   kill -TERM <pid>\n\n   # Read block meta\n   cat <prometheus-data-dir>/<BLOCK-ULID>/meta.json\n\n   # Edit meta.json and set \"failed\": false\n   # OR use promtool (if available in your version):\n   promtool tsdb dump-meta <prometheus-data-dir>/<BLOCK-ULID>/meta.json\n   ```\n\n2. **If index corruption suspected, delete and rebuild:**\n\n   ```bash\n   # Stop Prometheus\n   kill -TERM <pid>\n\n   # Identify corrupt block\n   promtool tsdb inspect-blocks <prometheus-data-dir>\n\n   # Delete corrupt block (if safe):\n   rm -rf <prometheus-data-dir>/<BLOCK-ULID>\n\n   # Restart Prometheus - it will re-compact head to replace missing block\n   prometheus <flags>\n   ```\n\n3. **If stuck due to I/O, restart with priority change:**\n\n   ```bash\n   # Stop Prometheus\n   kill -TERM <pid>\n   wait\n\n   # Restart with increased I/O priority (if supported):\n   nice -n -10 ionice -c2 -n0 prometheus <flags>\n   ```\n\n4. **If lock contention, increase retention check interval:**\n\n   Modify configuration to reduce concurrent operations:\n   ```yaml\n   global:\n     evaluation_interval: 5m  # Increase if very frequent\n   ```\n\n5. **Monitor compaction resume:**\n   ```\n   watch -n 5 'curl -s http://localhost:9090/api/v1/query?query=prometheus_tsdb_compactions_total'\n   ```\n\n### Failure Mode 3: Block Corruption\n\n#### Symptoms\n- `prometheus_tsdb_reloads_failures_total` increasing\n- Errors like: \"corrupted block <ULID>: invalid chunk data\"\n- \"index out of bounds\" or \"checksum mismatch\" in logs\n- Specific block consistently fails to load\n- Query errors on certain time ranges\n- Prometheus won't start if corruption in critical block\n\n#### Likely Causes\n- **Hardware failure**: Bit flip, bad sector, or disk corruption\n- **Unclean shutdown**: Process killed during compaction or write\n- **Filesystem corruption**: Ext4, XFS, or storage layer corruption\n- **Concurrent writes**: Multiple Prometheus instances writing to same directory\n- **Out-of-disk-space write**: Block write failed mid-operation, file truncated\n- **Index version mismatch**: Older Prometheus version reading newer index format\n\n#### Diagnostic Steps\n\n1. **Identify corrupt blocks:**\n   ```bash\n   promtool tsdb inspect-blocks <prometheus-data-dir>\n   # Look for blocks marked as \"corrupted\" or with read errors\n   ```\n\n2. **Check specific block validity:**\n   ```bash\n   promtool tsdb dump-blocks <prometheus-data-dir> <BLOCK-ULID>\n   # If it fails, block is corrupt\n   ```\n\n3. **Verify filesystem health:**\n   ```bash\n   sudo fsck -n <device>  # Non-destructive check\n   sudo badblocks -v <device>  # Find bad sectors\n   ```\n\n4. **Check hardware logs:**\n   ```bash\n   dmesg | grep -i \"error\\|corruption\\|io\"\n   journalctl -u prometheus -n 100  # Prometheus unit logs\n   ```\n\n5. **Inspect block directory manually:**\n   ```bash\n   ls -la <prometheus-data-dir>/<BLOCK-ULID>/\n   file <prometheus-data-dir>/<BLOCK-ULID>/{chunks,index,meta.json}\n   ```\n\n#### Recovery Procedure\n\n**Option A: Delete corrupt block (if it has backup/replacement)**\n\n1. **Stop Prometheus:**\n   ```bash\n   kill -TERM <pid>\n   ```\n\n2. **Verify the block is safely deletable:**\n   ```bash\n   # Check if a newer block covers the time range\n   promtool tsdb list <prometheus-data-dir>\n   ```\n\n3. **Delete the corrupt block:**\n   ```bash\n   rm -rf <prometheus-data-dir>/<CORRUPT-BLOCK-ULID>\n   ```\n\n4. **Restart Prometheus:**\n   ```bash\n   prometheus <flags>\n   ```\n\n5. **Monitor recovery:**\n   ```bash\n   # Verify block is no longer in list\n   promtool tsdb list <prometheus-data-dir>\n\n   # Check metrics reload success\n   curl -s http://localhost:9090/api/v1/query?query=prometheus_tsdb_reloads_failures_total\n   ```\n\n**Option B: Attempt repair with promtool (if available)**\n\n```bash\n# Some versions of Prometheus include repair utilities\npromtool tsdb bench <prometheus-data-dir>  # Test if readable\n\n# Or manually edit meta.json if only metadata is corrupted\nvi <prometheus-data-dir>/<BLOCK-ULID>/meta.json\n# Fix obvious issues (e.g., incorrect timestamps)\n```\n\n**Option C: Rebuild from WAL (if recent enough)**\n\n1. **Stop Prometheus**\n2. **Back up corrupted blocks:**\n   ```bash\n   mkdir backup\n   mv <prometheus-data-dir>/0* backup/  # Move all blocks\n   ```\n\n3. **Start Prometheus with empty blocks directory:**\n   ```bash\n   prometheus <flags>\n   # It will rebuild from WAL\n   ```\n\n4. **Monitor WAL replay:**\n   ```\n   grep \"replayed\" prometheus.log\n   ```\n\n### Failure Mode 4: Out-of-Memory (OOM) During Compaction\n\n#### Symptoms\n- Process killed with \"Killed\" or \"Segmentation fault\"\n- System logs show: \"Out of memory: Kill process\"\n- `dmesg` shows OOM killer invocation\n- Compaction fails midway without clean error\n- Memory usage spikes during `prometheus_tsdb_compaction_populating_block`\n- No error in Prometheus logs (process killed externally)\n\n#### Likely Causes\n- **Large block compaction**: Merging many or very large blocks simultaneously\n- **High cardinality data**: Too many unique metric/label combinations\n- **Insufficient memory**: System or container limit too low for data scale\n- **Memory leak in compaction**: Chunk pool not returning memory\n- **Concurrent operations**: Reload/retention cleanup + compaction both running\n\n#### Diagnostic Steps\n\n1. **Check system memory during compaction:**\n   ```bash\n   # In separate terminal, watch while compaction runs\n   watch -n 1 'free -h && echo \"---\" && ps aux | grep prometheus'\n   ```\n\n2. **Check container/cgroup limits:**\n   ```bash\n   cat /sys/fs/cgroup/memory.limit_in_bytes  # If using cgroups\n   docker stats <container>  # If in container\n   ```\n\n3. **Monitor memory at metric level:**\n   ```\n   process_resident_memory_bytes  # Watch during compaction\n   prometheus_tsdb_compaction_populating_block\n   ```\n\n4. **Check for memory pressure in logs:**\n   ```bash\n   dmesg | tail -20\n   journalctl --priority=warning\n   ```\n\n5. **Identify largest blocks:**\n   ```bash\n   du -h <prometheus-data-dir>/ | sort -h | tail -10\n   ```\n\n#### Recovery Procedure\n\n1. **Immediate: Increase system memory**\n   - Add more RAM to the system\n   - Or reduce container/Kubernetes memory limit restraint\n\n2. **Reduce block size (long-term solution):**\n\n   Stop Prometheus and adjust configuration:\n   ```yaml\n   global:\n     external_labels:\n       # Option A: Reduce maximum block duration\n       #tsdb.max-block-duration: 24h  # Down from default 36h\n   ```\n\n   Or in command line:\n   ```bash\n   prometheus --storage.tsdb.max-block-duration=24h\n   ```\n\n3. **Reduce cardinality:**\n\n   ```yaml\n   scrape_configs:\n     - job_name: 'high_cardinality'\n       metric_relabel_configs:\n         - source_labels: [__name__]\n           regex: 'bad_metric_.*'\n           action: drop\n   ```\n\n4. **Restart Prometheus with new settings:**\n   ```bash\n   prometheus <flags>\n   ```\n\n5. **Monitor compaction:**\n   - Should take longer but succeed with reduced memory\n   - Check `process_resident_memory_bytes` stays below limit\n\n6. **If recurring, consider splitting Prometheus instances:**\n   - Use separate instances for different jobs\n   - Use federation instead of single monolithic instance\n\n---\n\n## 4. Recovery Procedures\n\n### Pre-Recovery Checklist\n\nBefore attempting recovery:\n\n1. **Gather diagnostic information:**\n   ```bash\n   date > recovery.log\n   df -h >> recovery.log\n   ps aux | grep prometheus >> recovery.log\n   curl -s http://localhost:9090/api/v1/status >> recovery.log\n   promtool tsdb inspect-blocks <data-dir> >> recovery.log 2>&1\n   ```\n\n2. **Back up data directory:**\n   ```bash\n   cp -r <prometheus-data-dir> <prometheus-data-dir>.backup.$(date +%s)\n   ```\n\n3. **Document current state:**\n   ```bash\n   ls -la <prometheus-data-dir> > before_recovery.txt\n   ```\n\n### When to Restart Prometheus vs. Manual Intervention\n\n| Situation | Action | Rationale |\n|-----------|--------|-----------|\n| Compaction stuck but no errors | Restart | Clean restart often clears transient lock issues |\n| Disk full | Reduce retention first | Manual deletion required before restart |\n| Corrupted block, recent backup exists | Delete block | Prevents read errors on startup |\n| Failed block with `Compaction.Failed=true` | Edit meta.json | Allows planner to retry or skip block |\n| Memory exhaustion | Increase memory + restart | Restart with new limits |\n| I/O overload | Reduce concurrency | Adjust config before restart |\n\n### Step-by-Step Generic Recovery\n\n1. **Stop Prometheus:**\n   ```bash\n   # Graceful stop (preferred)\n   kill -TERM <pid>\n   sleep 30\n\n   # If not stopped, force\n   kill -9 <pid>\n   ```\n\n2. **Verify all processes stopped:**\n   ```bash\n   ps aux | grep prometheus | grep -v grep\n   ```\n\n3. **Backup current state:**\n   ```bash\n   tar czf prometheus-data-$(date +%s).tar.gz <data-dir>\n   ```\n\n4. **Apply failure-specific recovery** (see sections above)\n\n5. **Start Prometheus with clean state:**\n   ```bash\n   prometheus \\\n     --storage.tsdb.path=<data-dir> \\\n     --log.level=info \\\n     <other flags>\n   ```\n\n6. **Monitor startup and recovery:**\n   ```bash\n   # Watch logs\n   tail -f prometheus.log\n\n   # Or in separate terminal\n   curl -s http://localhost:9090/-/healthy\n   ```\n\n7. **Verify metrics health:**\n   ```bash\n   curl -s http://localhost:9090/api/v1/query?query=prometheus_tsdb_blocks_loaded\n   curl -s http://localhost:9090/api/v1/query?query=prometheus_tsdb_compactions_total\n   ```\n\n### Using `promtool` for Inspection and Repair\n\n#### Inspect All Blocks\n\n```bash\npromtool tsdb inspect-blocks <data-dir>\n\n# Output example:\n# Block ID: 01ARZ3NDEKTSV4RRFFQ69G5FAV\n#   Compaction level: 1\n#   Created: <timestamp>\n#   Min time: 1609459200000\n#   Max time: 1609545600000\n#   Series: 10000\n#   Samples: 5000000\n```\n\n#### List Blocks\n\n```bash\npromtool tsdb list <data-dir>\n\n# Output example:\n# ULID                         MinTime                 MaxTime                 Duration  NumSeries  NumSamples  NumTombstones  Compaction\n# 01ARZ3NDEKTSV4RRFFQ69G5FAV   2021-01-01T00:00:00Z   2021-01-02T00:00:00Z   24h       10000      5000000     0              Level: 1\n```\n\n#### Dump Block Metadata\n\n```bash\npromtool tsdb dump-meta <data-dir>/<block-id>/meta.json\n\n# Shows full metadata including:\n# - Compaction level\n# - Parents (source blocks)\n# - Sources (original blocks)\n# - Stats (samples, series, tombstones)\n```\n\n#### Bench/Validate Block\n\n```bash\npromtool tsdb bench read <data-dir> --duration=1h\n\n# Validates all blocks can be read successfully\n```\n\n#### WAL Repair (if available)\n\n```bash\npromtool tsdb list-wbl <data-dir>/wal\n# Lists write-ahead log blocks\n\n# Or for WAL directory:\npromtool tsdb dump-wal <data-dir>/wal | head -100\n```\n\n---\n\n## 5. # Prometheus TSDB Compaction Operational Runbook

## 1. Overview

### What Compaction Does

TSDB (Time Series Database) compaction is a crucial maintenance process that merges overlapping time blocks, removes tombstoned series, and maintains query performance over time. The compaction process:

- **Merges overlapping blocks**: Combines blocks with overlapping time ranges into a single, deduplicated block
- **Removes deleted series**: Reclaims disk space by eliminating tombstoned (deleted) series data
- **Reorganizes data**: Creates larger, better-structured blocks for improved query efficiency
- **Manages WAL**: Converts in-memory Write-Ahead Log (WAL) data into persisted blocks

### Why Compaction Is Needed

Without compaction, Prometheus would:
- Accumulate many small blocks, slowing down queries
- Retain deleted data indefinitely, wasting disk space
- Have fragmented data with overlapping time ranges causing inefficient query execution
- Build up an excessively large WAL

### Compaction Levels and Block Structure

Prometheus uses a **leveled compaction** strategy with exponential block ranges:

- **Level 0 blocks**: Raw blocks created directly from WAL data (2 hours by default, `MinBlockDuration`)
- **Level 1+ blocks**: Result of compacting multiple Level 0 blocks
- **Block ranges**: By default: 2h, 6h, 18h, 54h, 162h, etc. (exponential with step size 3)

Each block is identified by:
- **ULID**: Unique Lexicographically Sortable Identifier with timestamp
- **MinTime/MaxTime**: Time range of data in milliseconds
- **Compaction.Level**: Current compaction level
- **Compaction.Parents**: References to source blocks
