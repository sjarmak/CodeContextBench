# big-code-cross-capmarkets-arch-001: Kafka-Flink Streaming Data Flow (Cross-Repo)

## Task

Trace how streaming data flows from Apache Kafka through Apache Flink's connector framework. Map the complete pipeline from Kafka's producer/consumer API, through the Flink connector base framework, to Flink's streaming runtime. Identify the serialization/deserialization boundary between the two systems and explain how Flink's checkpoint mechanism integrates with Kafka's consumer offset commits.

## Context

- **Repositories**: apache/kafka (Java, ~1M LOC) + apache/flink (Java, ~2M LOC)
- **Category**: Architectural Understanding (cross-repo, capital markets)
- **Difficulty**: hard
- **Domain**: Streaming data infrastructure for capital markets (trade ingestion, pricing, risk)

## Architecture Overview

The Kafka-Flink integration spans two independent projects:

1. **Apache Kafka** — Provides the producer/consumer client API, serialization interfaces, and consumer group coordination
2. **Apache Flink** — Provides the Source/Sink API (flink-core), the connector base framework (flink-connector-base), and the SourceOperator runtime integration (flink-runtime)

The Flink Kafka connector (in a separate repo: apache/flink-connector-kafka) extends Flink's connector-base framework and wraps Kafka's consumer/producer APIs. This task focuses on the API contracts in both main repos that the connector bridges.

## Requirements

1. Map Kafka's producer API: Producer interface, KafkaProducer, ProducerRecord, Serializer
2. Map Kafka's consumer API: Consumer interface, KafkaConsumer, ConsumerRecord, Deserializer, OffsetAndMetadata
3. Map Flink's Source API: Source, SourceReader, SplitEnumerator interfaces
4. Map Flink's connector-base framework: SourceReaderBase, SplitReader, RecordEmitter
5. Map Flink's serde boundary: DeserializationSchema, SerializationSchema
6. Trace the checkpoint-offset integration: SourceOperator.snapshotState() -> SourceReader -> KafkaConsumer.commitSync()

Kafka Producer API\n\n**Role in Architecture:** Source of data before Kafka cluster.\n\nThe Kafka producer API defines the boundary for sending data into Kafka:\n- **Producer<K, V> interface** (producer/Producer.java): Generic contract with `send(ProducerRecord)`, `flush()`, `commitTransaction()` methods\n- **ProducerRecord<K, V>** (producer/ProducerRecord.java): Immutable data container with:\n  - `topic`: target Kafka topic\n  - `partition`: optional partition assignment\n  - `key`: optional routing key for partition selection\n  - `value`: actual message payload\n  - `headers`: record metadata (can be used for tracing, schema versions)\n  - `timestamp`: event time or ingestion time\n\n- **KafkaProducer** (producer/KafkaProducer.java): Concrete implementation with:\n  - Internal `RecordAccumulator` (batches records for efficiency)\n  - Internal `Sender` thread (background I/O for non-blocking send)\n  - **First serialization boundary:** `Serializer<K>` and `Serializer<V>` convert typed objects to byte arrays before network transmission\n\n**Relevance to Kafka-Flink:** The producer path is typically used in reverse\u2014Flink SINKs write to Kafka. The focus of this analysis is the CONSUMER path (data flowing INTO Flink).\n\n---\n\n### 2. Kafka Consumer API\n\n**Role in Architecture:** Primary ingestion point for streaming data from Kafka into Flink.\n\nThe Kafka consumer API provides the fundamental offset management and record delivery semantics:\n\n- **Consumer<K, V> interface** (consumer/Consumer.java): Generic contract defining:\n  - Schema can apply additional transformations (schema registry lookup, field projection, type conversion)\n- Example: `AvroDeserializationSchema`, `JsonDeserializationSchema`, custom Checkpoint Mechanism Integration with Kafka Offset Commits\n\n**Role in Architecture:** Achieves exactly-once semantics by coordinating Flink checkpoints with Kafka offset commits.\n\nThis is the **core integration point** between Kafka and Flink:\n\n#### The Problem Being Solved\nKafka tracks offsets (which records have been consumed). Flink tracks snapshots (operator state at a point in time). Without coordination:\n- If Flink crashes after processing a record but before saving its state, Kafka will re-deliver the record (no exactly-once)\n- If Kafka commits an offset before Flink has saved state, offset commit is wasted\n\n#### The Solution: Deferred Offset Commits\n\nFlink uses the **semantic that offset commits only happen AFTER checkpoint completion**, ensuring the processed records are saved:\n\n1. **Snapshot Phase (while checkpoint is being taken):**\n   ```\n   JobManager.triggerCheckpoint(checkpointId) [barrier in stream]\n       \u2193\n   SourceOperator.snapshotState(StateSnapshotContext context)\n       \u251c\u2500\u2192 long checkpointId = context.getCheckpointId()\n       \u251c\u2500\u2192 sourceReader.snapshotState(checkpointId)\n       \u2502    [SourceReader returns List<SplitT> with current offsets]\n       \u2502\n       \u2514\u2500\u2192 readerState.update(splits)\n           [Saves to ListState<byte[]> via split serializer]\n   ```\n   - `snapshotState()` is called during checkpoint coordination (barriers in stream)\n   - Returns the **current position/offset** of each Kafka partition split\n   - Offset is NOT committed to Kafka yet (only saved in Flink state backend)\n   - This happens on every checkpoint (frequency: e.g., every 1 minute)\n\n2. **Completion Phase (after checkpoint is durable):**\n   ```\n   JobManager.notifyCheckpointComplete(checkpointId)\n       \u2193\n   SourceOperator.notifyCheckpointComplete(checkpointId)\n       \u251c\u2500\u2192 super.notifyCheckpointComplete(checkpointId)  [operator cleanup]\n       \u2502\n       \u2514\u2500\u2192 sourceReader.notifyCheckpointComplete(checkpointId)\n           [Critical: This triggers offset commit to Kafka]\n           \u2193\n           SourceReaderBase.notifyCheckpointComplete()\n               \u2193\n               SplitFetcherManager\n                   \u2193\n                   Fetcher Thread (non-task thread)\n                       \u2193\n                       KafkaConsumer.commitSync(\n                           Map<TopicPartition, OffsetAndMetadata> offsets\n                       )\n                       [Send OffsetCommit request to Broker]\n                       \u2193\n                       Broker OffsetManager\n                           [Stores offset in __consumer_offsets topic]\n   ```\n   -
