## Review Summary

### Defect 1 — Critical: Wrong frame property in controller (`comments-controller.js:370`)

`getCommentLikes` reads `frame.data.id` but the `browse` endpoint declares `id` in `options`, not `data`. This means `commentId` is always `undefined`, so the service query returns nothing (or everything). The sibling method `getCommentReporters` correctly uses `frame.options.id`.

**Fix:** Change `frame.data.id` → `frame.options.id`

---

### Defect 2 — High: Missing member relation in service query (`comments-service.js:266`)

`getCommentLikes` passes `withRelated: []`, so like records are returned without any associated member data. Since the entire feature is "display of members who have liked a comment", this renders the endpoint useless. The sibling method `getCommentReporters` correctly uses `withRelated: ['member']`.

**Fix:** Change `withRelated: []` → `withRelated: ['member']`

---

### Defect 3 — Medium: Missing `headers` declaration on browse endpoint (`comment-likes.js:8`)

Ghost's API convention requires all read-only `browse` actions to declare `headers: { cacheInvalidate: false }` to explicitly opt out of cache invalidation. Every other read-only browse endpoint in the codebase (e.g., `actions.js`, `announcements.js`) includes this. The missing declaration may cause the framework to incorrectly send cache-invalidation signals on GET requests.

**Fix:** Add `headers: { cacheInvalidate: false }` before the `options` array
