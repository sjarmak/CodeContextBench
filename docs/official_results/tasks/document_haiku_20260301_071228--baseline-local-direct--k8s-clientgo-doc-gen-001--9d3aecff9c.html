     1→# `client-go` Architecture
     2→
     3→This document explains the internal architecture of `client-go` for contributors. It describes the
     4→major components, how they interact, and the key design decisions that shape the library.
     5→
     6→## Client Configuration
     7→
     8→There is an architectural separation between loading client configuration and using it. The
     9→`rest.Config` object is the in-memory representation of this configuration. The
    10→`tools/clientcmd` package is the standard factory for producing it. `clientcmd` handles the
    11→complex logic of parsing `kubeconfig` files, merging contexts, and handling external
    12→authentication providers (e.g., OIDC).
    13→
    14→## REST Client
    15→
    16→The `rest.Client` is the foundational HTTP client that underpins all other clients. It separates
    17→the low-level concerns of HTTP transport, serialization, and error handling from higher-level,
    18→Kubernetes-specific object logic.
    19→
    20→The `rest.Config` object is used to build the underlying HTTP transport, which is typically a
    21→chain of `http.RoundTripper` objects. Each element in the chain is responsible for a specific
    22→task, such as adding an `Authorization` header. This is the mechanism by which all authentication
    23→is injected into requests.
    24→
    25→The client uses a builder pattern for requests (e.g., `.Verb()`, `.Resource()`), deferring
    26→response processing until a method like `.Into(&pod)` is called. This separation is key to
    27→supporting different client models from a common base.

    "content": "# `client-go` Architecture\n\nThis document explains the internal architecture of `client-go` for contributors. It describes the\nmajor components, how they interact, and the key design decisions that shape the library.\n\n## Client Configuration\n\nThere is an architectural separation between loading client configuration and using it. The\n`rest.Config` object is the in-memory representation of this configuration. The\n`tools/clientcmd` package is the standard factory for producing it. `clientcmd` handles the\ncomplex logic of parsing `kubeconfig` files, merging contexts, and handling external\nauthentication providers (e.g., OIDC).\n\n## REST Client\n\nThe `rest.Client` is the foundational HTTP client that underpins all other clients. It separates\nthe low-level concerns of HTTP transport, serialization, and error handling from higher-level,\nKubernetes-specific object logic.\n\nThe `rest.Config` object is used to build the underlying HTTP transport, which is typically a\nchain of `http.RoundTripper` objects. Each element in the chain is responsible for a specific\ntask, such as adding an `Authorization` header. This is the mechanism by which all authentication\nis injected into requests.\n\nThe client uses a builder pattern for requests (e.g., `.Verb()`, `.Resource()`), deferring\nresponse processing until a method like `.Into(&pod)` is called. This separation is key to\nsupporting different client models from a common base.\n\n### Endpoint Interactions\n\n*   **Content Negotiation:** The client uses HTTP `Accept` headers to negotiate the wire format\n    (JSON or Protobuf). A key performance optimization using this mechanism is the ability to\n    request metadata-only objects via the `as=PartialObjectMetadata;g=meta.k8s.io;v=v1` Accept custom parameter.\n    Also the `as=Table;g=meta.k8s.io;v=v1` Accept custom parameters may be used to request lists as tables. \n*   **Subresources:** The client can target standard subresources like `/status` or `/scale` for\n    object mutations, and it can also handle action-oriented subresources like `/logs` or\n    `/exec`, which often involve streaming data.\n*   **List Pagination:** For `LIST` requests, the client can specify a `limit`. The server will\n    return up to that many items and, if more exist, a `continue` token. The client is\n    responsible for passing this token in a subsequent request to retrieve the next page.\n    Higher-level tools like the `Reflector`'s `ListerWatcher` handle this logic automatically.\n*   **Streaming Watches:** A `WATCH` request returns a `watch.Interface` (from\n    `k8s.io/apimachinery/pkg/watch`), which provides a channel of structured `watch.Event`\n    objects (`ADDED`, `MODIFIED`, `DELETED`, `BOOKMARK`). This decouples the watch consumer from\n    the underlying streaming protocol.\n\n### Errors, Warnings, and Rate Limiting\n\n*   **Structured Errors:** The client deserializes non-2xx responses into a structured\n    `errors.StatusError`, enabling programmatic error handling (e.g., `errors.IsNotFound(err)`).\n*   **Warnings:** It processes non-fatal `Warning` headers from the API server via a\n    `WarningHandler`.\n*   **Client-Side Rate Limiting:** The `QPS` and `Burst` settings in `rest.Config` are the\n    client's half of the contract with the server's API Priority and Fairness system.\n*   **Server-Side Throttling:** The client's default transport automatically handles HTTP `429`\n    responses by reading the `Retry-After` header, waiting, and retrying the request.\n\n## Typed and Dynamic Clients\n\nTo handle the extensible nature of the Kubernetes API, `client-go` provides two primary client\nmodels.\n\nThe **`kubernetes.Clientset`** provides compile-time, type-safe access to core, built-in APIs.\n\nThe **`dynamic.DynamicClient`** represents all objects as `unstructured.Unstructured`, allowing it\nto interact with any API resource, including CRDs. It relies on two discovery mechanisms:\n1.  The **`discovery.DiscoveryClient`** determines *what* resources exist. The\n    **`CachedDiscoveryClient`** is an optimization that caches this data on disk to solve.\n2. This is essential for scalable controllers.\n\n**Problem solved**: Controllers need current state of many objects. Polling all objects continuously is expensive. Watching all objects consumes server resources. The solution: LIST once, WATCH incrementally, process events.\n\n#### Reflector: LIST/WATCH Orchestrator\n\nThe `Reflector` (`/workspace/staging/src/k8s.io/client-go/tools/cache/reflector.go`) is the data pump:\n\n```go\ntype Reflector struct {\n\tname string\n\tstore ReflectorStore           // Where to push changes\n\tlisterWatcher ListerWatcherWithContext\n\tlastSyncResourceVersion string // RV from last successful LIST\n\tresyncPeriod time.Duration     // How often to re-list everything\n\tminWatchTimeout time.Duration  // Random 5-10 min for load distribution\n\tShouldResync func() bool       // Periodic resync trigger\n}\n\n// ListAndWatch loop (simplified):\n// 1. LIST to get initial RV\n// 2. WATCH from that RV\n// 3. On \"too old\" error, re-LIST and restart WATCH\n// 4. Periodically call Resync() to catch missed deletes\n```\n\n**Resource Version**: A server-side opaque token indicating a point in time. Essential for consistency:\n- `LIST` with empty RV: latest (but potentially inconsistent with following WATCH)\n- `WATCH` with specific RV: stream changes after that RV\n- \"too old\" error: RV expired from server's watch cache (default 5 min); must re-list\n\n**Watch Bookmarks optimization**: Servers send periodic bookmark events with updated RV, allowing Reflector to restart from a recent point without re-listing.\n\n**Pagination via WatchList**: The `useWatchList` flag (`reflector.go` line 147) enables streaming paginated results instead of chunked LIST calls. Advantages:\n- Fewer server memory spikes\n- Lower network latency (streaming vs buffered)\n\n#### DeltaFIFO: Event Accumulation Queue\n\nThe `DeltaFIFO` (`/workspace/staging/src/k8s.io/client-go/tools/cache/delta_fifo.go`) is a producer-consumer queue that accumulates changes:\n\n```go\n// Delta types\nconst (\n\tAdded     DeltaType = \"Added\"\n\tUpdated   DeltaType = \"Updated\"\n\tDeleted   DeltaType = \"Deleted\"\n\tReplaced  DeltaType = \"Replaced\"    // re-list happened\n\tSync      DeltaType = \"Sync\"        // periodic resync\n\tSyncAll   DeltaType = \"SyncAll\"     // bulk resync\n\tReplacedAll DeltaType = \"ReplacedAll\" // bulk with atomicity\n)\n\ntype DeltaFIFO struct {\n\tlock sync.RWMutex\n\tcond sync.Cond\n\titems map[string]Deltas         // Key -> [Delta, Delta, ...]\n\tqueue []string                  // FIFO key ordering\n\tpopulated bool                  // First Replace() done?\n\tknownObjects KeyListerGetter    // Reference set for detecting deletes\n}\n```\n\n**Key insight**: A `Deltas` is a slice of changes for one object. When Pop() is called, all accumulated changes since the last Pop() are returned together. This solves the problem of rapid changes between processing cycles:\n\n```\nWatch events: Pod-A Added, Pod-A Updated, Pod-A Updated\nDeltaFIFO stores: Pod-A -> [Added, Updated, Updated]\nPop() returns: (Pod-A, [Added, Updated, Updated])\nController processes all three deltas and sees final state correctly\n```\n\n**Delete detection via KnownObjects**: When `Replace()` is called (full re-list), DeltaFIFO compares the new list against `knownObjects` (from Indexer). Objects in `knownObjects` but not in new list are emitted as Deleted deltas, catching deletions missed by watch.\n\n#### Indexer and SharedIndexInformer: The Cache and Processor\n\nThe `Indexer` (part of `cache.Store` interface) is the actual in-memory cache:\n\n```go\ntype Store interface {\n\tAdd(obj interface{}) error\n\tUpdate(obj interface{}) error\n\tDelete(obj interface{}) error\n\tList() []interface{}\n\tGet(obj interface{}) (interface{}, exists bool, err error)\n\tGetByKey(key string) (interface{}, exists bool, err error)\n}\n```\n\nThe `SharedIndexInformer` (`shared_index_informer.go`) coordinates everything:\n\n```go\ntype sharedIndexInformer struct {\n\tindexer Indexer                    // The cache store\n\tprocessor *sharedProcessor\n\tlisterWatcher ListerWatcherWithContext\n\tobjectType runtime.Object\n\tresyncCheckPeriod time.Duration\n\tstarted bool\n\tcontroller Controller               // Runs Reflector + DeltaFIFO\n\treflector *Reflector\n}\n\n// Start() launches:\n// 1. controller.Run() \u2192 Reflector.ListAndWatch()\n// 2. processor.run() \u2192 Event handler dispatch\n```\n\n**SharedIndexInformer responsibilities**:\n1. **Caching**: Indexer stores objects with multiple query indexes (by namespace, by owner, custom)\n2. **Watching**: Reflector maintains live sync via LIST/WATCH\n3. **Event dispatch**: Calls registered AddFunc, UpdateFunc, DeleteFunc handlers\n4. **Work isolation**: Handlers only add keys to workqueue; actual reconciliation is separate\n\n**Thread safety**: Indexer is thread-safe for reads (Listers can query without locks); updates come through controlled channels from processor.\n\n#### Listers: Read-Only Cache Access\n\nTyped Listers (generated, e.g., `corev1.PodLister`) provide read-only, {\n\tList(selector labels.Selector) ([]*v1.Pod, error)\n\tGet(name string) (*v1.Pod, error)\n}\n```\n\n**Why Listers over Indexer?**:\n- Enforce read-only contract (no accidental mutations)\n- Provide The algorithm balances load distribution via random timeouts.\n\n### Resource Version Expiration (\"too old\")\n\n**Problem**: Watch cache on API server expires old resourceVersions (typically after 5 minutes). If a controller is slow to catch up, it may request a watch from an expired RV.\n\n**Solution** (in Reflector):\n```go\ncase isExpiredError(err):\n    r.setLastSyncResourceVersionUnavailable()\n    // Causes next ListAndWatch to LIST from scratch\n```\n\n**Cost**: Re-listing is more expensive than resuming from RV, but necessary for correctness. Prevents a controller from silently missing events.\n\n**Watch Bookmarks optimization**: Server sends periodic BOOKMARK events with current RV, allowing Reflector to fast-forward without re-listing.\n\n### Eventual Consistency of Indexer\n\n**Problem**: Events arrive out of order, or an event is dropped. Indexer could diverge from API server state.\n\n**Solution** (via periodic Resync):\n- Reflector periodically calls DeltaFIFO.Resync()\n- Resync() emits Sync deltas for all objects in Indexer\n- Controller reprocesses all objects, achieving eventual consistency\n- Default resync period: 0 (disabled; custom controllers set as needed)\n\n**Tradeoff**: Resync increases API server load (triggers Lister calls even without changes). Many production controllers disable resync and rely on watch accuracy.\n\n### Watch List Pagination (new in Kubernetes 1.27)\n\n**Problem**: Large LIST responses can spike memory on API server. Older pattern: chunk the LIST, then open a watch (unpaginatedresult) or stream the watch (paginatedresult).\n\n**Solution**: The `WatchList` feature streams paginated results in a single watch-like connection.\n\n**Tradeoff**: Requires server support (Kubernetes 1.27+). Falls back to chunked LIST if server doesn't support. Operators must explicitly enable `reflector.useWatchList = true`.\n\n### Shared Indexer Concurrency\n\n**Problem**: Informer handlers run concurrently; multiple controllers may query Indexer simultaneously while it's being updated.\n\n**Solution**:\n- Indexer uses read-write locks for mutations\n- DeltaFIFO and processor use locks to serialize updates\n- Listers perform snapshot reads (acquire read lock, read, release)\n- Controllers see occasional stale reads, but never corrupted data\n\n**Limit**: Highly concurrent workloads (100s of controllers) may contend on the Indexer lock. Mitigation: each controller gets its own SharedInformerFactory, distributing lock contention.\n\n### Client-Side Rate Limiting vs. Server-Side APF\n\n**Tradeoff**:\n- Client-side (`QPS`, `Burst`) is coarse-grained: applies to all operations equally\n- Server-side (API Priority and Fairness) is fine-grained: per-priority-level, per-user, per-namespace\n\n**Design**: Both layers are necessary. Client-side prevents a single client from overwhelming the server. Server-side ensures fair allocation among multiple clients. They work together:\n```\nClient A respects QPS=5 \u2500\u2510\nClient B respects QPS=5 \u251c\u2500\u2192 API Server APF system\nClient C respects QPS=5 \u2500\u2518\n                        \u2193\n             APF allocates seats fairly across high/medium/low priority\n```\n\n### Memory Footprint of Caching\n\n**Problem**: Large clusters have many objects. Storing all Pods, Nodes, Secrets in-memory is expensive.\n\n**Mitigations**:\n1. **TransformFunc** (`/workspace/staging/src/k8s.io/client-go/tools/cache/`):\n   - Strip sensitive fields before caching (e.g., remove large Secret data)\n   - Reduce memory by dropping unnecessary fields\n   - Idempotent; safe to apply to objects during Replace()\n\n2. **Namespace filtering** (WithNamespace option in SharedInformerFactory):\n   - Only cache objects in specific namespace(s)\n   - Common pattern: each controller watches only its namespace\n\n3. **Field selection** (TweakListOptions):\n   - Server can filter fields in List response\n   - Further reduces network and memory\n\n---\n\n## Extension Points and Customization\n\n### Custom RoundTrippers\n\nInject middleware into the HTTP transport chain:\n\n```go\nconfig := rest.Config{...}\nconfig.WrapTransport = func(rt http.RoundTripper) http.RoundTripper {\n\treturn &CustomLoggingRoundTripper{rt: rt}\n}\n```\n\n**Use cases**:\n- Request/response logging (with sanitization)\n- Custom retry logic (e.g., jitter, circuit breakers)\n- Observability (distributed tracing, metrics)\n- Chaos injection for testing\n\n### Custom Rate Limiters\n\nReplace default token-bucket with custom backoff logic:\n\n```go\ntype CustomRateLimiter struct { ... }\nfunc (c *CustomRateLimiter) TryAccept() bool { ... }\n\nconfig.RateLimiter = CustomRateLimiter{...}\n```\n\n**Use cases**:\n- Exponential jitter (prevents thundering herd on server recovery)\n- Context-aware limiting (different rates based on priority)\n- Adaptive limiting (feedback loop from server response times)\n\n### Custom Indexers\n\nAdd secondary indexes for efficient queries:\n\n```go\nindexers := cache.Indexers{\n\t\"controllerRef\": func(obj interface{}) ([]string, error) {\n\t\tpod := obj.(*v1.Pod)\n\t\tif owner := metav1.GetControllerOf(pod); Large clusters (10k+ Pods) may take 10-30 seconds. Mitigate with:\n   - Namespace filtering\n   - Field selection (TweakListOptions)\n   - Parallel informers (separate factories per API group)\n\n2. **Watch reconnection storms**: If API server is under load, many controllers trying to reconnect simultaneously can amplify load. Mitigations:\n   - Randomized minWatchTimeout spreads reconnections\n   - Exponential backoff in WatchErrorHandler\n   - Monitor \"reflector_list_duration_seconds\" metric\n\n3. **Indexer contention**: High-volume event streams (e.g., logging from all Pods) can saturate Indexer lock. Mitigations:\n   - Multiple SharedInformerFactory instances (per logical component)\n   - Reduce event volume via label/field selectors\n   - Monitor \"workqueue_depth\" and handler latencies\n\n4. **Memory scaling**: Cache size scales with cluster size. For a 5000-node cluster:\n   - Typical node object: ~50KB\n   - 5000 nodes \u00d7 50KB = ~250MB per informer\n   - Plus Pods, Endpoints, Services, etc.\n   - Typical controller footprint: 500MB\u20132GB\n\n### Reliability and Failure Handling\n\n1. **API server outage**: Informer watch drops. Reflector retries with exponential backoff, periodically attempting LIST. Controller work queues local items. Once API server recovers, catch-up happens via Resync or watch re-establishment.\n\n2. **Network partition**: Long-lived watch connections timeout (minWatchTimeout). Reflector detects and reconnects. No events are lost if watch reconnects within resourceVersion TTL (default 5 min).\n\n3. **OOM on client**: If Indexer grows too large, process may OOM. No in-built circuit breaker. Mitigations:\n   - TransformFunc to reduce object size\n   - Namespace filtering\n   - External monitoring to alert on memory growth\n   - Increase memory limits for critical controllers\n\n4. **Slow reconciliation loop**: If controller.Reconcile() is slow, work queue backlog grows. Informer keeps pushing events. Risk of memory explosion if queue is unbounded. Mitigation: implement backpressure (e.g., wait for queue depth before starting more handlers).\n\n### Debugging and Observability\n\n**Metrics** (`tools/metrics/`):\n- `workqueue_adds_total` - Total # Client-go: Advanced Systems Guide

## Overview

**client-go** is the official Go client library for interacting with Kubernetes clusters. It provides a comprehensive toolkit for API access, event-driven caching, and controller implementation. client-go serves as the foundation for kubectl, controller-manager, and nearly all custom controllers running in production Kubernetes environments.

This guide explains the internal architecture, component responsibilities, request flow, and extension mechanisms that enable client-go to efficiently and reliably mediate between Go applications and Kubernetes API servers at scale.

---

## Subsystem Boundaries and Dependencies

### Purpose and Scope

client-go abstracts the complexity of Kubernetes API communication into layered interfaces, each solving specific problems:

1. **Configuration Management** - Loading and merging kubeconfig files, handling authentication providers

3. **API Interaction** - Type-safe (typed clients) and type-agnostic (dynamic client) interfaces for CRUD operations
4. **Event-Driven Caching** - Efficient, event-sourced cache population using LIST/WATCH primitives
