## Django Middleware Implementation

### Key Files

1. **`django/core/handlers/base.py`** — Core middleware machinery
   - `BaseHandler.load_middleware()`: Iterates `settings.MIDDLEWARE` in **reverse**, wrapping each middleware around a growing handler chain, ending with `_middleware_chain` pointing to the outermost middleware.
   - `BaseHandler._get_response()`: The innermost callable — resolves URL, calls view, applies `process_view` / `process_template_response` / exception hooks.
   - `BaseHandler.process_exception_by_middleware()`: Iterates `_exception_middleware` list.

2. **`django/utils/deprecation.py`** — `MiddlewareMixin`
   - `__call__`: Calls `process_request` → `get_response` (next middleware) → `process_response` — the "onion" model.
   - `__acall__`: Async variant using `sync_to_async`.

3. **`django/core/handlers/exception.py`** — `convert_exception_to_response`
   - Wraps every middleware layer so exceptions don't leak between layers.

4. **`django/core/handlers/wsgi.py`** — `WSGIHandler`
   - WSGI entry point; calls `load_middleware()` on init, then dispatches to `_middleware_chain`.

### Call Order vs. WSGI Stacking

Django's `MIDDLEWARE` list is iterated **in reverse** during `load_middleware`, so the first entry becomes the outermost wrapper — identical in call order to manually nesting WSGI middleware `app = MW1(MW2(MW3(app)))`. The key difference: Django's onion model ensures only middleware that saw the request will see the response (short-circuiting in `process_request` skips all inner layers for both request and response).
