# Operational Runbook: Prometheus TSDB Compaction

## 1. Overview

### Purpose of Compaction

TSDB (Time Series Database) compaction is a critical maintenance process in Prometheus that:

- **Merges overlapping time blocks** into consolidated blocks that reduce query overhead and improve read performance
- **Removes tombstoned (deleted) series** from blocks to reclaim disk space previously marked for deletion
- **Optimizes chunk layout** to improve query performance and reduce disk I/O
- **Manages block levels** using an exponential block range structure to balance read/write amplification

### Why Compaction is Needed

Without compaction, Prometheus would accumulate:
1. Multiple overlapping time blocks with duplicate series data → increased query time
2. Tombstones (deletion markers) for removed series → wasted disk space
3. Small, numerous blocks → excessive file descriptor usage and memory overhead

### Block Structure and Levels

Prometheus uses **leveled compaction** with exponential block ranges:

- **Level 0 (Head blocks)**: Recently ingested data, 2 hours by default (controlled by `--storage.tsdb.min-block-duration`)
- **Level 1**: 2-3 Level 0 blocks merged together
- **Level 2**: 2-3 Level 1 blocks merged together
- **Level N**: Progressive doubling of time ranges (2h → 10h → 50h → 250h → ...)

Each level increases compaction level in block metadata:
```
meta.Compaction.Level = N   # Higher = more compact
meta.Compaction.Parents = [list of source blocks]
meta.Compaction.Sources = [list of original series sources]
```

### Compaction Triggers

Compaction is triggered automatically by three mechanisms:

#### 1. **Time-Based Trigger (Head Compaction)**
- Occurs when head block spans > 1.5x its configured duration
- Formula: `MaxTime - MinTime > ChunkRange / 2 * 3`
