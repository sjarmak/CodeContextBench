Subsystem Purpose and Scope\n\n### Primary Responsibilities\n\nThe container manager subsystem is responsible for:\n\n1. **Resource Constraint Enforcement** - Translating Kubernetes resource requests/limits into kernel cgroup limits\n2. **Resource Allocation Coordination** - Ensuring that CPU affinity, memory NUMA locality, and device assignments are coordinated across multiple allocation strategies\n3. **Cgroup Hierarchy Management** - Creating and maintaining the cgroup directory tree reflecting QoS classes and pod/container boundaries\n4. **Policy-Driven Allocation** - Supporting pluggable policies for CPU and memory allocation strategies\n5. **Device Plugin Integration** - Allowing external device managers to allocate specialized hardware\n6. **Dynamic Resource Allocation** - Managing vendor-specific resource claims via DRA plugins\n\n### Boundaries and Scope\n\n**Within Scope:**\n- Cgroup creation, configuration, and destruction (both cgroup v1 and v2)\n- CPU affinity assignment (static CPU manager policy)\n- NUMA-aware memory allocation and pinning\n- Device plugin lifecycle and resource allocation\n- DRA claim preparation and resource injection\n- NUMA topology hint aggregation and consensus\n- System container isolation (kubelet, docker daemon)\n- QoS-based resource reservation and enforcement\n\n**Outside Scope:**\n- Container runtime interface (handled by kubelet's pod manager)\n- OOM killing decisions (handled by eviction manager)\n- Pod scheduling (handled by scheduler)\n- Container image management\n- Log collection and metrics (handled by cadvisor integration)\n\n### Upstream and Downstream Dependencies\n\n**Upstream Dependencies:**\n- **Kubelet** - Calls `Start()` during initialization, requests resources during pod admission and container startup\n- **Scheduler** - Relies on node capacity/allocatable from `GetCapacity()` and `GetNodeAllocatableReservation()`\n- **API Server** - Provides pod and node specs; DRA manager watches for ResourceClaim updates\n- **Container Runtime (CRI)** - Receives container configs with enforced resource limits; calls `UpdateContainerResources()` for runtime enforcement\n\n**Downstream Dependencies:**\n- **Linux Kernel (cgroups)** - Enforces CPU, memory, PID limits; tracks resource usage\n- **Device Plugins** - Provide device lists and handle allocation requests\n- **DRA Plugins** - Prepare resources and provide CDI configuration\n- **cAdvisor** - Provides node machine info (CPU count, memory, topology); collects resource metrics\n- **systemd/cgroupfs** - Used to translate cgroup names and manage cgroup filesystems\n\n---\n\n## 2. no per-container affinity\n- `static` - Exclusively allocates CPUs to containers requesting integer CPU values\n\n**Key Methods:**\n- `Allocate()` - Called at pod admission; assigns CPUs based on policy\n- `AddContainer()` - Called when container starts; tracks container-to-CPU mapping\n- `RemoveContainer()` - Called when container stops; frees CPUs\n- `GetTopologyHints()` / `GetPodTopologyHints()` - Provides NUMA-aware hints\n\n**State Management:** Checkpoints allocation state to `state_checkpoint.json` in kubelet root directory; survives kubelet restarts.\n\n### 2.6 Memory Manager\n\n**File:** `pkg/kubelet/cm/memorymanager/memory_manager.go`\n\n**Type:** `Manager` interface / `manager` struct\n\n**Responsibilities:**\n- Implements NUMA-aware memory allocation policies\n- Assigns memory to specific NUMA nodes based on topology hints\n- Informs container runtime which NUMA nodes to bind memory to (via cpuset.mems)\n- Tracks per-container memory allocations\n\n**Supported Policies:**\n- `none` - No NUMA awareness; containers can use all memory\n- `static` - Allocates memory from specified NUMA nodes based on hints\n- `best-effort` - Attempts NUMA affinity where possible; falls back to any available memory\n\n**Key Methods:**\n- `Allocate()` - Pre-allocates memory to NUMA nodes during pod admission\n- `AddContainer()` - Registers container; updates memory cgroup affinity\n- `GetMemoryNUMANodes()` - Returns NUMA nodes for container memory binding\n- `GetTopologyHints()` - Provides NUMA affinity preferences\n\n**State:** Checkpoints allocation state; tracks memory allocation per NUMA node.\n\n### 2.7 Device Manager\n\n**File:** `pkg/kubelet/cm/devicemanager/manager.go`\n\n**Type:** `Manager` interface / `ManagerImpl` struct\n\n**Responsibilities:**\n- Discovers and manages device plugins (GPUs, custom accelerators)\n- Handles device plugin lifecycle (registration, health checks)\n- Allocates devices to containers via plugin callbacks\n- Provides device information to kubelet for capacity/allocatable reporting\n- Supports topology hints for device-aware scheduling\n\n**Key Methods:**\n- `Start()` - Discovers and connects to device plugins\n- `Allocate()` - Calls device plugin to allocate devices to pod\n- `GetDeviceRunContainerOptions()` - Retrieves devices/mounts/env for container runtime\n- `UpdatePluginResources()` - Updates node info with discovered device capacity\n\n**Plugin Lifecycle:**\n1. Plugin registers via Unix socket under `/var/lib/kubelet/plugins`\n2. Manager discovers plugin; calls `GetDevicePluginOptions()`, `ListAndWatch()`\n3. On pod admission, calls `Allocate()` to assign devices\n4. Device info injected into container runtime config\n\n### 2.8 Dynamic Resource Allocation (DRA) Manager\n\n**File:** `pkg/kubelet/cm/dra/manager.go`\n\n**Type:** `Manager` struct\n\n**Responsibilities:**\n- Manages dynamic resource claims (structured, vendor-specific resources)\n- Coordinates with DRA plugins to prepare/unprepare resources\n- Injects CDI device configuration into containers\n- Tracks claim lifecycle (prepare \u2192 run pod \u2192 unprepare)\n- Handles plugin registration and health checks\n\n**Key Methods:**\n- `PrepareDynamicResources()` - Called before pod starts; asks plugins to prepare claims\n- `UnprepareDynamicResources()` - Called when pod terminates; cleanup\n- `GetResources()` - Retrieves prepared CDI configuration for container\n- `UpdateAllocatedResourcesStatus()` - Updates pod status with allocated resources\n\n**Feature Gate:** `DynamicResourceAllocation`; disabled by default.\n\n**State:** Checkpoints claim preparation state to allow recovery across kubelet restarts.\n\n### 2.9 Topology Manager\n\n**File:** `pkg/kubelet/cm/topologymanager/topology_manager.go`\n\n**Type:** `Manager` interface / `manager` struct\n\n**Responsibilities:**\n- Aggregates topology hints from all hint providers (CPU, memory, device managers)\n- Determines optimal NUMA node affinity based on pluggable policies\n- Makes topology hints available to hint providers for allocation decisions\n- Acts as a PodAdmitHandler to reject pods that cannot satisfy topology constraints\n- Tracks container-to-topology assignments\n\n**Supported Policies:**\n- `none` - No topology awareness; hints are collected but ignored\n- `best-effort` - Prefers aligned allocation if possible; allows unaligned\n- `restricted` - Requires all resources from same NUMA node(s); rejects if not possible\n- `single-numa-node` - Requires all resources from single NUMA node; strictest\n\n**Scopes:**\n- `container` - Topology hints evaluated per container (default)\n- `pod` - Topology hints evaluated per pod (all containers must align)\n\n**Key Methods:**\n- `Admit()` - PodAdmitHandler; called during pod admission\n- `AddHintProvider()` - Registers a resource manager as hint provider\n- `GetAffinity()` - Hint providers query assigned topology after admission\n\n**Hint Aggregation Algorithm:**\n1. Query each hint provider for topology hints for container\n2. Generate all possible NUMA node combinations\n3. Filter hints using policy (none policy skips filtering)\n4. Choose best hint (preferred > unpreferred; narrower > wider)\n5. Provide chosen hint back to hint providers via `Allocate()` call\n\n### 2.10 Internal Container Lifecycle\n\n**File:** `pkg/kubelet/cm/internal_container_lifecycle.go`, `pkg/kubelet/cm/internal_container_lifecycle_linux.go`\n\n**Type:** `InternalContainerLifecycle` interface / `internalContainerLifecycleImpl` struct\n\n**Responsibilities:**\n- Provides lifecycle hooks for kubelet to integrate resource managers with container startup/stop\n- Ensures CPU/memory affinity is set before container starts\n- Triggers manager cleanup when containers stop\n\n**Hooks:**\n- `PreCreateContainer()` - Called before container creation; injects CPU/memory affinity into runtime config\n- `PreStartContainer()` - Called after container creation; registers container with managers\n- `PostStopContainer()` - Called when container stops; notifies managers to cleanup\n\n**Integration Points:**\n- CPU manager: sets `containerConfig.Linux.Resources.CpusetCpus`\n- Memory manager: sets `containerConfig.Linux.Resources.CpusetMems`\n- Topology manager: records container\u2192topology mapping\n\n---\n\n## 3. Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### 1. Topology Affinity Failures (Restricted Policy)\n\n**Scenario:** Pod requests resources that cannot be satisfied from single/overlapping NUMA nodes.\n\n**Behavior:**\n- Topology manager rejects pod admission with `TopologyAffinityError`\n- Pod remains `Unschedulable` in API server\n- No resources are allocated; pod never starts\n\n**Recovery:** User must modify pod resource requests to be compatible with available topology, or change policy to `best-effort`.\n\n**Code Reference:** `pkg/kubelet/cm/topologymanager/topology_manager.go` (Admit implementations in policies)\n\n#### 2. Cgroup Creation Failures\n\n**Scenario:** Required cgroup subsystems not mounted; invalid cgroup paths; permission denied.\n\n**Behavior:**\n- Kubelet fails to start; node goes `NotReady`\n- Error logged: \"failed to create top level QOS cgroup\"\n\n**Recovery:** Ensure cgroup subsystems are mounted; check kernel version supports required cgroups; verify kubelet runs as root or with CAP_SYS_ADMIN.\n\n**Code Reference:** `pkg/kubelet/cm/container_manager_linux.go:278` (QOS manager creation)\n\n#### 3. Device Plugin Allocation Failures\n\n**Scenario:** Device plugin unregisters mid-operation; allocation request fails; plugin becomes unhealthy.\n\n**Behavior:**\n- Pod admission is rejected if devices cannot be allocated\n- If device plugin fails after allocation, kubelet logs error but container still starts (resources already configured)\n- Node capacity for that device type drops; scheduler won't schedule more pods requesting that device\n\n**Recovery:**\n- Device plugin must restart and re-register\n- Existing pods using devices may continue running\n- New pods can be scheduled once plugin re-registers\n\n**Code Reference:** `pkg/kubelet/cm/devicemanager/manager.go` (Allocate method and error handling)\n\n#### 4. CPU Manager State Corruption\n\n**Scenario:** Checkpoint file corrupted or inaccessible; inconsistent state from previous run.\n\n**Behavior:**\n- CPU manager falls back to default (none) policy if state unrecoverable\n- Container starts without CPU affinity\n- Potential double-allocation of CPUs in next restart\n\n**Recovery:** Kubelet auto-recovery; state file is best-effort. If corruption persists, delete state file and accept temporary loss of CPU pinning guarantees.\n\n**Code Reference:** `pkg/kubelet/cm/cpumanager/state/state_checkpoint.go` (checkpoint loading)\n\n#### 5. NUMA Topology Change\n\n**Scenario:** Hardware topology changes at runtime (rare; hot-add NUMA node); cAdvisor reports different topology.\n\n**Behavior:**\n- Topology manager continues using old topology until kubelet restarts\n- Misalignment between hints and actual NUMA layout possible\n- No runtime revalidation\n\n**Impact:** Low risk; requires hardware-level changes.\n\n### 4.2 Resource Limits and Constraints\n\n| Limit | Default | Configurable | Impact |\n|-------|---------|--------------|--------|\n| Max NUMA nodes | 8 | No (hardcoded) | Prevents state explosion in topology manager; larger systems require work-around |\n| Cgroup nesting depth | Unlimited | No | Deep pod cgroups consume memory; no enforced limit |\n| CPUs for allocation | # Physical CPUs | Yes (reserved CPUs) | Cannot allocate more than available; excess reserved CPUs reduce capacity |\n| Memory for allocation | Total - reserved | Yes (reserved memory) | Cannot allocate more than available |\n| Device plugin count | Unlimited | No | More plugins = slower admission; no built-in limit |\n| Pod PID limit | 0 (disabled) | Yes (`--pod-pids-limit`) | When set, kubelet enforces per-pod PID cgroup limit |\n\n**Code References:**\n- NUMA node limit: `pkg/kubelet/cm/topologymanager/topology_manager.go:41` (defaultMaxAllowableNUMANodes)\n- PID limits: `pkg/kubelet/cm/types.go` (PidsLimit in ResourceConfig)\n\n### 4.3 Design Tradeoffs\n\n#### 1. Cooperative Allocation vs. Enforcement\n\n**Tradeoff:** Container manager _allocates_ resources at admission time (CPU, memory, devices) via hints, but does not strictly _enforce_ allocation for all resources.\n\n**Design Choice:** Cooperative approach.\n\n**Rationale:**\n- CPU affinity (cgroup cpuset) is enforced; container must run on assigned CPUs\n- Memory NUMA binding is enforced; memory allocated from specified nodes\n- Device allocation is enforced; devices not accessible outside container\n- BUT: CPU shares (cgroup.shares) and memory limits (cgroup.memory.limit) may be overridden by QoS manager later\n- This allows dynamic adjustment of limits during pod lifetime\n\n**Implication:** Pre-admission reservation is approximate; not absolute guarantee.\n\n**Code Reference:** `pkg/kubelet/cm/qos_container_manager_linux.go:141` (periodic UpdateCgroups)\n\n#### 2. Per-Container vs. Per-Pod Topology Scope\n\n**Tradeoff:** Topology manager can operate at container scope (each container gets optimal hints independently) or pod scope (all containers must use same NUMA nodes).\n\n**Design Choice:** Container scope is default; pod scope available as option.\n\n**Rationale:**\n- Container scope allows heterogeneous resource needs within single pod\n- Pod scope is stricter; useful for tightly-coupled workloads\n- Pod scope simpler to reason about; smaller state space\n\n**Implication:** Default behavior allows containers in same pod to be on different NUMA nodes.\n\n**Code Reference:** `pkg/kubelet/cm/topologymanager/topology_manager.go:192` (scope selection)\n\n#### 3. ## Executive Summary

The kubelet container manager is a core subsystem responsible for enforcing resource constraints, coordinating resource allocation across containers, and managing the cgroup hierarchy on nodes. It acts as a mediator between the Kubernetes API, the container runtime, and the host kernel's resource management facilities (cgroups), enabling fine-grained CPU, memory, and device allocation with support for NUMA topology awareness.
