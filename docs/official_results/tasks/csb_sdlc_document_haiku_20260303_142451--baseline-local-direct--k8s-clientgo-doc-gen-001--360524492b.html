Resource Discovery and OpenAPI Schemas\n\n**Files:** `discovery/discovery.go`, `discovery/cached/`, `openapi/cached/`, `restmapper/`\n\n**Responsibilities:**\n- **`discovery.DiscoveryClient`** (discovery/discovery.go): Queries the API server's `/apis` and `/api` endpoints to enumerate available GroupVersions and resources\n- **`CachedDiscoveryClient`** (discovery/cached/): Caches discovery data on disk (`~/.kube/cache/discovery`) to reduce repeated API calls\n- **OpenAPI client** (openapi/cached/): Fetches and caches OpenAPI v3 schemas from `/openapi/v3` for schema-aware tooling\n- **`RESTMapper`** (restmapper/): Maps GroupVersionKind (GVK) to GroupVersionResource (GVR), supporting both discovery and cached lookup\n\n**Key design:**\n- Caching significantly reduces API load during repeated discovery queries\n- OpenAPI schemas enable validation and schema awareness without embedding API definitions in the client\n- RESTMapper allows plural/singular name translation and scoping (cluster vs. namespaced)\n\n### 6. Informers and Caching Infrastructure (`tools/cache` package)\n\n**Files:** `tools/cache/controller.go`, `tools/cache/delta_fifo.go`, `tools/cache/reflector.go`, `tools/cache/shared_informer.go`, `tools/cache/indexer.go`\n\nThis is the most complex and critical subsystem.\n\n#### The Reflector: LIST/WATCH Pump\n\n**Type:** `Reflector` (tools/cache/reflector.go)\n\n**Responsibilities:**\n1. Performs an initial `LIST` at a specific `resourceVersion` to get a consistent snapshot of all objects\n2. Starts a `WATCH` from that `resourceVersion` to receive continuous updates\n3. Handles the **\"too old\" `resourceVersion` error** by periodically re-listing (default interval is `FullResyncPeriod`)\n4. Consumes watch bookmarks to obtain updated `resourceVersion` for more efficient recovery on watch timeout\n\n**Key behavior** (tools/cache/reflector.go:50-100):\n```go\n// Pseudocode flow:\nfor {\n    list.ResourceVersion = GetResourceVersion(lastSeenVersion)\n    items, rv := ListAll(resource, list)          // LIST\n    Store.Replace(items, rv)\n\n    // After initial LIST\n    watch := Watch(resource, rv)                   // WATCH from rv\n    for event := range watch {\n        Store.Add/Update/Delete(event)             // Individual updates\n        if event.Type == BOOKMARK {\n            lastSeenVersion = event.ResourceVersion\n        }\n    }\n\n    // If WATCH times out or fails with \"too old\", re-list\n}\n```\n\n**Critical invariant:** The Reflector maintains sequential delivery\u2014it never goes backward in `resourceVersion`.\n\n#### The DeltaFIFO: Accumulation Queue\n\n**Type:** `DeltaFIFO` (tools/cache/delta_fifo.go:63-100)\n\n**Responsibilities:**\n- Acts as a producer-consumer queue between the Reflector (producer) and the controller (consumer)\n- Accumulates changes (deltas) for each object key rather than storing just the latest state\n- Tracks multiple delta types: `Added`, `Updated`, `Deleted`, `Replaced` (for bulk refresh), `Sync` (periodic resync)\n- Ensures at-most-once processing: if an object is updated multiple times before being popped, all deltas are bundled into a single pop\n\n**Key design** (tools/cache/delta_fifo.go):\n```go\n// Each object key maps to a Deltas slice\nDeltas[key string][]Delta\n\n// Each Delta tracks how the object changed\ntype Delta struct {\n    Type   DeltaType     // Added, Updated, Deleted, etc.\n    Object runtime.Object\n}\n```\n\n**Why this design?** If a Pod is created, then its CPU limit is updated, then it's deleted\u2014before the consumer even pops\u2014all three deltas are preserved. The consumer processes them sequentially, maintaining causal ordering.\n\n#### The Indexer: Thread-safe In-Memory Cache\n\n**Type:** `Indexer` (tools/cache/indexer.go, tools/cache/cache.go)\n\n**Responsibilities:**\n- Stores all objects in a thread-safe hash map (keyed by namespace/name or name for non-namespaced objects)\n- Supports pluggable index functions to enable queries like \"all Pods running on Node X\"\n- Provides atomic `Add`, `Update`, `Delete` operations synchronized by RWMutex\n- Used by listers and controller business logic to query the cache\n\n**Key interfaces:**\n- `Store`: Basic get/set operations\n- `Indexer`: Store plus `Index()` and `IndexKeys()` for querying by index\n- `MutationCache`: Detects untracked object mutations (for debugging)\n\n#### The SharedIndexInformer: Coordination Layer\n\n**Type:** `SharedIndexInformer` (tools/cache/shared_informer.go)\n\n**Responsibilities:**\n- Coordinates the Reflector \u2192 DeltaFIFO \u2192 Indexer pipeline\n- Manages event handler registration (AddEventHandler, AddEventHandlerWithResyncPeriod)\n- Implements `HasSynced()` to signal when the cache is initially populated\n- Handles periodic resyncs by calling all registered handlers with all cached objects\n\n**Typical usage:**\n```go\ninformer := sharedInformerFactory.Core().V1().Pods().Informer()\ninformer.AddEventHandler(cache.ResourceEventHandlerFuncs{\n    AddFunc:    onPodAdded,\n    UpdateFunc: onPodUpdated,\n    DeleteFunc: onPodDeleted,\n})\ninformer.Run(stopCh)\nif !cache.WaitForCacheSync(stopCh, informer.HasSynced) {\n    // Failed to sync\n}\n// Now the cache is populated\n```\n\n**Key design invariant:** Event handlers are always called in order, never missing events, but events might be reordered across multiple object keys.\n\n#### The Lister: Read-Only Cache Access\n\n**Type:** `Lister`, `NamespaceLister` (tools/cache/listers.go)\n\n**Responsibilities:**\n- Provides read-only, thread-safe access to the Indexer's cache\n- Used by controller business logic to query objects without holding locks during complex operations\n- Returns live references to cached objects (not deep copies) for performance\n\n**Generated listers:** Code generation creates typed listers (e.g., `corev1.PodLister`) that wrap the Indexer and provide type safety.\n\n### 7. Work Queue and Rate Limiting (`util/workqueue` package)\n\n**Files:** `util/workqueue/queue.go`, `util/workqueue/delaying_queue.go`, `util/workqueue/rate_limiting_queue.go`\n\n**Responsibilities:**\n- **`Queue`** (workqueue/queue.go): Basic FIFO queue with `Add()`, `Get()`, `Done()`, `Forget()` operations supporting safe parallel processing\n- **`DelayingInterface`**: Extends Queue with `AddAfter()` to delay item processing\n- **`RateLimitingInterface`**: Implements exponential backoff via `AddRateLimited()` and failure counting\n- **Rate limiters** (workqueue/default_rate_limiters.go): Pluggable implementations (exponential backoff, item exponential backoff, bucket, max failures)\n\n**Design philosophy:** The work queue decouples event detection (informer) from reconciliation (controller). Informer handlers simply add object keys; the work queue handles retry logic with backoff.\n\n**Example flow:**\n```\nInformer sees Pod update\n  \u2192 Handler calls queue.Add(\"namespace/pod-name\")\n  \u2192 Handler returns immediately (non-blocking)\n\nController worker:\n  \u2192 Pops \"namespace/pod-name\" from queue\n  \u2192 Looks up Pod in Lister cache\n  \u2192 Performs reconciliation (may take seconds)\n  \u2192 On error: queue.AddRateLimited() \u2192 exponential backoff\n  \u2192 On success: queue.Forget() \u2192 reset retry counter\n```\n\n### 8. Leader Election (`tools/leaderelection` package)\n\n**Files:** `tools/leaderelection/leaderelection.go`, `tools/leaderelection/resourcelock/`\n\n**Responsibilities:**\n- Manages distributed leader election via a shared Kubernetes resource (typically a Lease)\n- Multiple replica controllers compete to acquire the lock\n- Only the leader calls the provided `OnStartedLeading()` callback\n- Handles graceful loss of leadership via `OnStoppedLeading()`\n\n**Key design:**\n- Uses Lease objects (more efficient than ConfigMaps) with atomic compare-and-swap semantics\n- Implements renewal before deadline expiration to prevent split-brain\n- All communication goes through the Kubernetes API server (no separate consensus protocol)\n\n## End-to-End Request and Control Flow\n\n### A: Simple Read Operation (List Pods)\n\n```\nUser code calls: clientset.CoreV1().Pods(\"default\").List(ctx, metav1.ListOptions{})\n\n  1. kubernetes/typed/core/v1/pod.go - Generated client code\n       \u2193\n  2. RESTClient.Get().Resource(\"pods\").Namespace(\"default\").Do(ctx)\n       \u2193\n  3. rest/request.go: Request.Do() - Constructs URL, serializes params\n       \u2193\n  4. Transport chain - Adds auth headers, applies rate limiting\n       \u2193\n  5. HTTP GET /api/v1/namespaces/default/pods\n       \u2193\n  6. API server responds with PodList JSON\n       \u2193\n  7. rest/request.go: Deserializes into &v1.PodList\n       \u2193\nUser code receives []*v1.Pod\n```\n\n**Files involved:**\n- `kubernetes/typed/core/v1/pod.go`: Generated client methods\n- `rest/request.go:Do()`: Request execution\n- `transport/round_trippers.go`: Auth and rate limiting\n- `rest/client.go`: RESTClient implementation\n\n### B: Watch with Informer (Most Common Controller Pattern)\n\nThis is the critical flow for controllers.\n\n```\nStep 1: User code creates informer factory\n  factory := informers.NewSharedInformerFactory(clientset, 10*time.Minute)\n  podInformer := factory.Core().V1().Pods().Informer()\n\nStep 2: User registers event handler (non-blocking)\n  podInformer.AddEventHandler(cache.ResourceEventHandlerFuncs{\n    AddFunc: func(obj interface{}) {\n      pod := obj.(*v1.Pod)\n      queue.Add(fmt.Sprintf(\"%s/%s\", pod.Namespace, pod.Name))\n    },\n    ...\n  })\n\nStep 3: Informer starts (in goroutine)\n  go factory.Start(ctx)\n\n  Internal flow:\n\n  a) SharedIndexInformer creates Reflector\n  b) Reflector.ListAndWatch():\n\n     i) LIST /api/v1/pods (all namespaces)\n        - Gets resourceVersion (e.g., \"12345\")\n        - Creates Indexer cache entries for all returned Pods\n        - DeltaFIFO receives Replaced delta for each Pod\n        - Controller pops DeltaFIFO, processes each Replaced delta\n        - Calls AddFunc handlers \u2192 user's queue.Add() called\n\n     ii) WATCH /api/v1/pods?watch=true&resourceVersion=12345\n         - Stream of ADDED, MODIFIED, DELETED, BOOKMARK events\n\n         ADDED event:\n           \u2192 Reflector.Add() \u2192 DeltaFIFO receives Added delta\n           \u2192 Controller pops, processes Added delta\n           \u2192 Handler AddFunc called\n\n         MODIFIED event:\n           \u2192 Reflector.Update() \u2192 DeltaFIFO receives Updated delta\n           \u2192 Controller pops, processes Updated delta\n           \u2192 Handler UpdateFunc called\n\n         DELETED event:\n           \u2192 Reflector.Delete() \u2192 DeltaFIFO receives Deleted delta\n           \u2192 Controller pops, processes Deleted delta\n           \u2192 Handler DeleteFunc called\n\n         BOOKMARK event (periodic):\n           \u2192 Reflector saves resourceVersion\n           \u2192 (No delta generated; used for \"too old\" recovery)\n\n     iii) On watch timeout or \"too old\" error:\n          \u2192 Reflector re-lists (goto ii)\n          \u2192 Uses latest resourceVersion from bookmark\n\n     iv) On HasSynced() = true:\n         \u2192 DeltaFIFO has processed all initial LIST items\n         \u2192 Cache is now complete\n\nStep 4: User code processes work queue\n  go func() {\n    for {\n      key, quit := queue.Get()\n      if quit { return }\n      defer queue.Done(key)\n\n      obj, exists, err := indexer.GetByKey(key)\n      if err != nil { queue.AddRateLimited(key); continue }\n      if !exists { continue } // Deleted\n\n      err := reconcile(obj)\n      if err != nil {\n        queue.AddRateLimited(key)  // Retry with backoff\n      } else {\n        queue.Forget(key)           // Success, reset backoff\n      }\n    }\n  }()\n\nStep 5: When Pod is created in cluster\n  - API server broadcasts WATCH event to all connected clients\n  - Reflector receives ADDED event\n  - DeltaFIFO enqueues Added delta\n  - Controller pops, calls AddFunc handler\n  - Handler calls queue.Add(podKey)\n  - User's reconciliation loop gets the key\n  - Queries Lister: indexer.PodsLister().Pods(\"ns\").Get(\"podname\")\n  - Performs business logic\n  - If error, queue.AddRateLimited() with exponential backoff\n  - On success, queue.Forget()\n\nKey invariants:\n- Events are delivered at least once (may retry on network failures)\n- All deltas for one object are processed sequentially\n- DeltaFIFO preserves causal ordering within one object key\n- Cache is eventually consistent (may lag behind API server by seconds)\n- Lost WATCH events trigger re-list via \"too old\" resourceVersion error\n```\n\n**Files involved:**\n- `tools/cache/controller.go`: Run loop that coordinates Reflector and DeltaFIFO\n- `tools/cache/reflector.go`: ListAndWatch implementation\n- `tools/cache/delta_fifo.go`: Event accumulation\n- `tools/cache/shared_informer.go`: Handler dispatch\n- `tools/cache/indexer.go`: Cache storage\n- `util/workqueue/rate_limiting_queue.go`: Retry logic\n- `rest/request.go`: Actual HTTP LIST/WATCH calls\n\n### C: Server-Side Apply (Declarative Updates)\n\n```\nUser code:\n  applyConfig := &appsv1ac.DeploymentApplyConfiguration{...}\n  result, err := typedAppsV1.Deployments(\"default\").Apply(ctx, applyConfig)\n\nFlow:\n  1. applyConfig is converted to a JSON patch with metadata\n  2. rest/request.Patch(types.ApplyPatchType) called\n  3. HTTP PATCH /apis/apps/v1/namespaces/default/deployments/myapp\n       Content-Type: application/apply-patch+json\n       Body: {\"apiVersion\":\"...\",\"kind\":\"...\",\"metadata\":{...},...}\n  4. API server merges patch with existing object (based on managedFields)\n  5. Server-side validation and webhooks run\n  6. Server updates the object and returns new state\n  7. Result is deserialized to &v1.Deployment\n\nBenefits:\n  - Multiple controllers can safely manage disjoint fields of same object\n  - No read-modify-write race conditions\n  - Server merges and tracks ownership (managedFields)\n```\n\n**Files involved:**\n- `applyconfigurations/`: Generated apply configuration builders\n- `rest/request.go`: PATCH method with ApplyPatchType\n- `util/apply/`: Server-side apply utilities\n\n## Failure Modes, Limits, and Design Tradeoffs\n\n### 1. Cache Consistency\n\n**Failure mode:** The cache can become stale or lag the API server.\n\n**Cause:** Network delays, watch stream timeouts, or server-side filtering.\n\n**Mitigation:**\n- Informers have a `FullResyncPeriod` (default 10 minutes per factory) that periodically re-lists all objects and calls handlers again\n- Watch bookmarks provide updated `resourceVersion` without re-listing\n- The \"too old\" `resourceVersion` error triggers automatic re-list\n- Application code should handle the case where `indexer.Get()` returns nil (object may have been deleted)\n\n**Tradeoff:** Eventual consistency rather than strong consistency. This is acceptable because Kubernetes APIs are designed for eventual consistency (no transactions).\n\n**Operational implications:** Controllers must be idempotent (safe to reconcile same object multiple times).\n\n### 2. Watch Stream Timeouts\n\n**Failure mode:** WATCH connections close unexpectedly (network flaky, API server limits, client-side timeout).\n\n**Cause:**\n- Default watch timeout in server is 30 seconds per request (with random jitter)\n- Clients implement `MinWatchTimeout` (default 5 minutes) to spread load\n- Network proxies may timeout idle connections\n\n**Mitigation:**\n- `Reflector.ListAndWatch()` automatically re-lists and continues watching\n- Exponential backoff on repeated failures (`rest/request.go`: backoff logic)\n- Watch stream loss detection via klog warnings and metrics\n\n**Operational impact:**\n- Expect watch reconnects every 5-30 minutes in normal operation\n- Count of \"re-list\" operations in metrics indicates watch stability issues\n- High re-list frequency can indicate API server problems or network issues\n\n### 3. Client-side Rate Limiting\n\n**Default behavior:** `rest.Config` defaults to `QPS=5, Burst=10`.\n\n**Failure mode:** Client requests are throttled by client-side rate limiter before reaching the API server.\n\n**Mitigation:**\n- Applications can increase `QPS` and `Burst` in `rest.Config`\n- Server-side rate limiting (API Priority and Fairness) takes precedence\u2014if server responds with 429, client backs off even if `QPS` not yet exceeded\n- Metrics in `tools/metrics` track throttle latency (see `longThrottleLatency` in rest/request.go:57)\n\n**Operational implications:**\n- If many controllers access same API server, aggregate QPS can be high\n- Burst handling allows short spikes (up to 10 requests) without throttling\n\n### 4. Memory Usage in Caches\n\n**Failure mode:** Large clusters or selectors that match many objects cause unbounded cache growth.\n\n**Cause:** Informers keep all matching objects in memory in the Indexer.\n\n**Mitigation:**\n- Use label/field selectors to filter what informer watches (e.g., `metav1.ListOptions{LabelSelector: \"...\"}`)\n- Set appropriate `FullResyncPeriod` (shorter = more frequent re-list, more CPU; longer = staler cache)\n- Use `metadata` client (`metadata.Interface`) to watch only object metadata instead of full objects\n\n**Operational implications:**\n- For large clusters, controller should use selective informers rather than watching all Pods\n- Memory usage grows O(n) with number of watched objects\n\n### 5. Concurrent Modification Errors\n\n**Failure mode:** User code modifies cached objects returned by lister, causing inconsistency.\n\n**Cause:** Listers return references to cached objects, not deep copies.\n\n**Mitigation:**\n- Documentation strongly discourages modifying objects returned by listers\n- `mutation_detector.go` detects and logs mutations in dev/test\n- Use `.DeepCopy()` before modifying\n\n**Operational implications:**\n- Prefer immutable patterns; clone if mutation needed\n\n### 6. Oversized API Requests\n\n**Failure mode:** LIST requests with many results hit memory limits or timeouts.\n\n**Cause:** Very large result sets (e.g., millions of Pods).\n\n**Mitigation:**\n- `tools/pager/`: Implements LIST pagination with `limit` and `continue` tokens\n- Reflector uses pager automatically\n- `Reflector.WatchListPageSize` configures initial list chunk size (default from server)\n\n**Operational implications:**\n- Pagination is transparent to user code\n- First sync of informer is slower with large result sets\n\n### 7. Lost Objects on Garbage Collection\n\n**Failure mode:** Objects are deleted, but controller doesn't learn about it (finalizers not called).\n\n**Cause:** If watch stream is not running (controller not resyncing), deletion events may be missed if resourceVersion becomes \"too old.\"\n\n**Mitigation:**\n- Reflector's re-list detects missing objects and generates Deleted events (via `DeltaFIFO` with `KnownObjects`)\n- Finalizers on controller-owned objects ensure ownership tracking\n- Dependency tracking via `ownerReferences` in objects\n\n**Operational implications:**\n- Use finalizers if controller must know about object deletion\n\n### 8. Split-brain with Leader Election\n\n**Failure mode:** Two replicas of stateful controller both think they're the leader.\n\n**Cause:** Lease lock renewal fails but controller doesn't notice.\n\n**Mitigation:**\n- LeaderElector implements renewal before deadline (not just at deadline)\n- `OnStoppedLeading()` is called when lock is lost\n- Lease TTL is short (default 15 seconds, jitter applied)\n\n**Operational implications:**\n- Controller replicas must be configured with same `LeaseNamespace` and identity\n- Set `RetryPeriod` lower than `LeaseDuration` to maximize renewal attempts\n\n## Extension Points and Customization Hooks\n\n### 1. # Kubernetes Client-Go Systems Guide

## Overview

`client-go` is the official Go client library for interacting with Kubernetes APIs. It is the foundation for all Kubernetes control plane components, operators, and application code that needs programmatic access to Kubernetes resources. This guide explains its internal architecture, component responsibilities, request flows, and operational considerations.

## Subsystem Purpose and Boundaries

### What client-go Solves

The Kubernetes API is extensible and supports multiple API groups, versions, and Custom Resource Definitions (CRDs). Additionally, controller and application code needs low-latency, event-driven access to resources rather than polling. `client-go` abstracts away:

1. **HTTP transport and authentication**: Managing TLS, tokens, certificate rotation, bearer tokens, and external auth providers
2. **Serialization and content negotiation**: Converting between Go objects and JSON/Protobuf wire formats
3. **Watch streams and LIST pagination**: Handling continuous updates and large result sets
4. **Caching and event distribution**: Providing in-memory caches that stay synchronized with the API server
5. **Typed vs. untyped access**: Offering both compile-time type-safe access (via generated code) and dynamic access (via `unstructured.Unstructured`)

- HTTP REST communication with Kubernetes API servers
