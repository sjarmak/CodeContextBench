# Kubelet Container Manager Subsystem Architecture Guide

## Executive Summary

The kubelet container manager (cm) subsystem is responsible for managing container resource allocation, enforcement, and lifecycle coordination on individual nodes. It bridges the gap between Kubernetes pod specifications and Linux kernel resource management mechanisms (cgroups, CPUSets, memory affinity) while providing pluggable extension points for device plugins and dynamic resource allocation. The subsystem implements a coordinator pattern where independent resource managers (CPU, memory, topology, devices, DRA) negotiate optimal resource placement through topology hints before finalizing allocations.

---

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose

The container manager subsystem ensures that:
- Containers running on a node receive the resource allocations specified by their pod definitions
- System and kubelet processes are protected from pod resource contention through reserved capacity
- Quality of Service (QoS) guarantees are enforced via cgroup hierarchies
- NUMA topology and hardware constraints are respected during resource allocation
- Device and custom resource plugins can register and allocate their resources
- Dynamic resources can be prepared and unprepared across the pod lifecycle

The kubelet container manager (cm) subsystem is responsible for managing container resource allocation, enforcement, and lifecycle coordination on individual nodes. It bridges the gap between Kubernetes pod specifications and Linux kernel resource management mechanisms (cgroups, CPUSets, memory affinity) while providing pluggable extension points for device plugins and dynamic resource allocation. The subsystem implements a coordinator pattern where independent resource managers (CPU, memory, topology, devices, DRA) negotiate optimal resource placement through topology hints before finalizing allocations.
