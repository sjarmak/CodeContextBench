Architectural Analysis for a Serverless Game Replay System

## Description

LedgerQuest Engine is a sophisticated, serverless game framework that orchestrates its game loop using AWS Step Functions, with game state persisted in DynamoDB. This distributed, stateless architecture presents unique challenges compared to traditional monolithic game servers. The business requires a new 'game replay' feature, allowing players and administrators to watch recordings of past game sessions. This is critical for debugging, content creation, and potential e-sports integration. The task is to produce a technical design proposal for this feature, focusing on how to integrate it into the existing serverless architecture efficiently and robustly.

    - Outline the high-level architecture for a new 'Replay Service'.

## Expected Approach

An expert developer would begin by understanding the high-level architecture from the documentation, specifically `docs/architecture/overview.md`, `docs/architecture/adr/001-serverless-game-loop.md`, and `docs/architecture/adr/002-ecs-on-dynamodb.md`. This provides the context that the game loop is not a continuous process but a series of orchestrated, stateless function calls.

Next, they would examine the core of the loop's orchestration: `infra/step_functions/game_loop_statemachine.asl.json`. This file is critical to identify the sequence of operations: input processing, AI updates, physics simulation, state commit, etc.

Based on this, they would identify two primary candidates for data capture:
1.  **Input Capture**: After the `InputProcessor` step. This involves capturing the validated command queue for each tick. This is highly efficient but relies on the engine being fully deterministic.
2.  **State Capture**: During or after the `StateCommitter` step. This involves capturing the full game state as it's written to DynamoDB.

The developer would then analyze the trade-offs. Input capture (`ledgerquest/services/game_loop/input_processor.py`) is superior for storage efficiency and aligns well with deterministic engine design. State capture (`ledgerquest/services/game_loop/state_committer.py`) is simpler to implement for playback but results in massive data storage costs.

For storage, S3 would be the logical choice for its low cost and ability to store large objects (replay files). The developer would propose a schema like `s3://ledgerquest-replays/{game_id}/{tick_number}.json` or a single compressed file per game.

For playback, they would propose a new set of API Gateway endpoints and Lambda functions that retrieve the replay file from S3 and stream it to the client. The client would then use a local instance of the game engine logic, feeding it the recorded inputs tick-by-tick to deterministically reconstruct the game session.

Finally, they would assess the impact: adding an S3 write operation to the Step Function adds latency, which should be mitigated (e.g., by making it a non-blocking call). They would also flag the risk of engine versioning, where a change in game logic could break older replays, and suggest a versioning strategy.

The system will enable players and administrators to watch recordings of past game sessions, supporting use cases including debugging, content creation, and potential e-sports integration.\n\n### Proposed Solution Summary\n\nThe replay system will capture **validated player inputs (commands)** after the `InputProcessor` step of the game loop, storing them in **Amazon S3** organized by game session and tick. For playback, a dedicated **Replay Service** will stream this input data to clients, which will use a local instance of the deterministic game engine to reconstruct the session tick-by-tick. This approach leverages the engine's deterministic nature to minimize storage costs while ensuring accurate replay fidelity.\n\n**Key Design Decisions:**\n1. **Input Capture Strategy** \u2014 Capture validated commands rather than full state snapshots\n2. **S3-based Storage** \u2014 Cost-effective, scalable storage for replay data\n3. **Client-side Reconstruction** \u2014 Leverage determinism for accurate, bandwidth-efficient playback\n4. **Non-blocking Capture** \u2014 Asynchronous writes to minimize impact on live game loop\n\n---\n\n## 2. Data Capture Strategy\n\n### 2.1 Analysis of Capture Points\n\nThe LedgerQuest game loop, orchestrated by the AWS Step Function defined in `infra/step_functions/game_loop_statemachine.asl.json`, follows this sequence:\n\n```\nInitializeFrame \u2192 ProcessInput \u2192 UpdateAI \u2192 PhysicsStep \u2192 RenderFrame \u2192 PersistState \u2192 WaitForNextTick \u2192 LoopDecider\n```\n\nTwo primary capture points were identified:\n\n| Capture Point | Location | Data Type | Files Involved |\n|--------------|----------|-----------|----------------|\n| **Input Capture** | After `ProcessInput` step | Validated commands | `ledgerquest/services/game_loop/input_processor.py:279-288` |\n| **State Capture** | During/after `PersistState` step | Full game state or deltas | `ledgerquest/services/game_loop/state_committer.py:182-265` |\n\n### 2.2 Recommended Approach: Input Capture\n\n**We recommend capturing data after the `InputProcessor` step** (`ledgerquest/services/game_loop/input_processor.py`).\n\nAt this point in the game loop, raw player interactions have been:\n1. Validated via JSON schema (`input_processor.py:123-143`)\n2. Checked for idempotency (`input_processor.py:250-273`)\n3. Converted to canonical `Command` objects (`input_processor.py:279-288`)\n\nThe `Command` dataclass (defined at `input_processor.py:87-101`) contains:\n\n```python\n@dataclass(frozen=True)\nclass Command:\n    tenant_id: str\n    game_id: str\n    player_id: str\n    command: InputType  # MOVE, CAST_SKILL, CHAT, PAUSE\n    payload: Dict[str, Any]\n    interaction_id: str\n```\n\n### 2.3 Justification: Why Input Capture Over State Capture\n\n| Criterion | Input Capture | State Capture |\n|-----------|---------------|---------------|\n| **Storage Size** | ~100 bytes/command | ~10-100 KB/tick |\n| **Storage Cost (1hr session @ 30 tps)** | ~10 MB | ~1-10 GB |\n| **Replay Accuracy** | Exact (deterministic) | Exact |\n| **Implementation Complexity** | Lower | Higher |\n| **Dependency on Determinism** | **Yes (critical)** | No |\n| **Engine Version Sensitivity** | High | Low |\n\n**Key Trade-offs:**\n\n1. **Storage Efficiency**: Input capture is orders of magnitude more efficient. A 1-hour session at 30 ticks/second with ~10 commands per tick would produce:\n   - Input: `30 \u00d7 3600 \u00d7 10 \u00d7 100 bytes \u2248 108 MB`\n   - State: `30 \u00d7 3600 \u00d7 50 KB \u2248 5.4 GB`\n\n2. **Determinism Requirement**: Input capture *requires* the game engine to be fully deterministic. Based on analysis of the codebase:\n   - The physics simulator (`ledgerquest/engine/physics/simulator.py`) uses deterministic float math with configurable tolerance (`simulator.py:64-67`)\n   - ADR-001 explicitly mentions \"deterministic state\" as a requirement (`docs/architecture/adr/001-serverless-game-loop.md:27-28`)\n   - The data model documentation mentions \"EventStore acts as the single source of truth to rebuild every game state exactly as it unfolded\" (`docs/architecture/data_model.md:24-25`)\n\n3. **Fallback Strategy**: We recommend implementing a **hybrid approach** as a fallback:\n   - Capture inputs every tick (primary)\n   - Capture periodic state snapshots (e.g., every 60 seconds) as checkpoints\n   - This enables \"fast-forward\" seeking and recovery from any non-determinism bugs\n\n### 2.4 Capture Implementation Point\n\nThe optimal integration point is within the `_publish_command` method of `InputProcessor` (`input_processor.py:294-323`). After successful EventBridge publication, emit the command to the replay capture system:\n\n```\nLocation: ledgerquest/services/game_loop/input_processor.py:294-323\nMethod: _publish_command()\nTrigger: After successful eb.put_events() call\n```\n\nAlternatively, add a new Step Function state after `ProcessInput` that writes to S3 in parallel with the main game loop continuation.\n\n---\n\n## 3. self.tick_buffer = []\n        self.playback_speed = 1.0\n\n    async def play(self):\n        # Load initial state\n        initial_state = await self.fetch_initial_state()\n        self.engine.load_state(initial_state)\n\n        # Start streaming ticks\n        tick_stream = self.stream_ticks(start=0)\n\n        while self.current_tick < self.manifest.total_ticks:\n            tick_data = await tick_stream.next()\n\n            # Apply all commands for this tick\n            for cmd in tick_data.cmds:\n                self.engine.apply_command(cmd)\n\n            # Advance engine simulation\n            self.engine.step()\n\n            # Render frame\n            self.renderer.present(self.engine.get_state())\n\n            # Control playback speed\n            await sleep(1.0 / (self.manifest.tick_rate_hz * self.playback_speed))\n\n            self.current_tick += 1\n\n    async def seek(self, target_tick):\n        # Find nearest checkpoint before target\n        checkpoint = self.find_nearest_checkpoint(target_tick)\n\n        if checkpoint:\n            state = await self.fetch_checkpoint(checkpoint.tick)\n            self.engine.load_state(state)\n            self.current_tick = checkpoint.tick\n        else:\n            # Replay from beginning\n            initial_state = await self.fetch_initial_state()\n            self.engine.load_state(initial_state)\n            self.current_tick = 0\n\n        # Fast-forward to target tick (no rendering)\n        while self.current_tick < target_tick:\n            tick_data = await self.get_tick(self.current_tick)\n            for cmd in tick_data.cmds:\n                self.engine.apply_command(cmd)\n            self.engine.step()\n            self.current_tick += 1\n```\n\n### 4.4 Integration with Existing Engine\n\nThe replay system reuses existing engine components:\n\n| Component | File | Usage in Replay |\n|-----------|------|-----------------|\n| Command ## 1. Overview

This document proposes a technical design for a **Game Replay System** for the LedgerQuest Engine. The system will enable players and administrators to watch recordings of past game sessions, supporting use cases including debugging, content creation, and potential e-sports integration.

### Proposed Solution Summary

The replay system will capture **validated player inputs (commands)** after the `InputProcessor` step of the game loop, storing them in **Amazon S3** organized by game session and tick. For playback, a dedicated **Replay Service** will stream this input data to clients, which will use a local instance of the deterministic game engine to reconstruct the session tick-by-tick. This approach leverages the engine's deterministic nature to minimize storage costs while ensuring accurate replay fidelity.

**Key Design Decisions:**
1. **Input Capture Strategy** — Capture validated commands rather than full state snapshots
2. **S3-based Storage** — Cost-effective, scalable storage for replay data
3. **Client-side Reconstruction** — Leverage determinism for accurate, bandwidth-efficient playback
4. **Non-blocking Capture** — Asynchronous writes to minimize impact on live game loop
