not be called during `poll()` |\n\n### Wakeup Mechanism\n\nThe `wakeup()` method is thread-safe and designed to interrupt a blocking `poll()` call:\n\n```java\n// From another thread:\nconsumer.wakeup();\n\n// In the consumer thread:\ntry {\n    while (true) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n        // process records\n    }\n} catch (WakeupException e) {\n    // Handle graceful shutdown\n}\n```\n\n### Multi-Threaded Processing Pattern\n\nFor processing records in parallel while maintaining correct offset management:\n\n```java\n// Consumer thread: polls and submits work\nExecutorService executor = Executors.newFixedThreadPool(4);\nMap<TopicPartition, Long> partitionOffsets = new HashMap<>();\n\ntry {\n    while (true) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n        for (TopicPartition tp : records.partitions()) {\n            List<ConsumerRecord<String, String>> partitionRecords = records.records(tp);\n\n            // Submit to executor\n            executor.submit(() -> {\n                for (ConsumerRecord<String, String> record : partitionRecords) {\n                    processRecord(record);\n                }\n            });\n\n            // Track offset for commit\n            ConsumerRecord<String, String> last = partitionRecords.get(partitionRecords.size() - 1);\n            partitionOffsets.put(tp, last.offset() + 1);\n        }\n\n        // Periodically commit processed offsets\n        if (!partitionOffsets.isEmpty()) {\n            Map<TopicPartition, OffsetAndMetadata> offsetsToCommit = new HashMap<>();\n            partitionOffsets.forEach((tp, offset) ->\n                offsetsToCommit.put(tp, new OffsetAndMetadata(offset)));\n\n            // Only commit after ALL workers have processed\n            consumer.pause(records.partitions());\n            executor.awaitTermination(5, TimeUnit.SECONDS);\n            consumer.commitSync(offsetsToCommit);\n            consumer.resume(records.partitions());\n        }\n    }\n} finally {\n    executor.shutdown();\n    consumer.close();\n}\n```\n\n---\n\n## Core Types\n\n### KafkaConsumer<K, V>\n\nThe main consumer class. Properties();\nprops.put(\"bootstrap.servers\", \"localhost:9092\");\nprops.put(\"group.id\", \"my-group\");\nprops.put(\"key.deserializer\", StringDeserializer.class.getName());\nprops.put(\"value.deserializer\", StringDeserializer.class.getName());\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\n```\n\n**Step 2: Subscribe or assign**\n\n```java\n// Dynamic assignment with consumer group\nconsumer.subscribe(Arrays.asList(\"topic1\", \"topic2\"));\n\n// OR manual assignment (mutually exclusive)\nconsumer.assign(Arrays.asList(\n    new TopicPartition(\"topic1\", 0),\n    new TopicPartition(\"topic1\", 1)\n));\n```\n\n**Step 3: Poll (group join happens here)**\n\n```java\n// First poll() triggers group join if subscribed\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n```\n\n### Joining a Consumer Group\n\nWhen a consumer first calls `poll()` after `subscribe()`:\n\n1. Consumer sends JoinGroup request with group ID\n2. Broker selects a group leader (first member to join)\n3. Leader runs partition assignment strategy (e.g., RangeAssignor)\n4. Members receive assignment via SyncGroup\n5. If custom listener provided, `onPartitionsAssigned()` is called\n6. Consumer begins fetching from assigned partitions\n\n**Timing Constraints:**\n\n- Group join must complete within `session.timeout.ms` (default 45 seconds)\n- If no `poll()` call for longer than `max.poll.interval.ms` (default 5 minutes), consumer is expelled\n- Expulsion is proactive\u2014consumer will fail on next `commitSync()` with `CommitFailedException`\n\n### Graceful Shutdown\n\n```java\ntry {\n    while (true) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n        // process records\n    }\n} finally {\n    consumer.close()  // or consumer.close(Duration)\n}\n```\n\n**Close Behavior:**\n\n- Triggers `onPartitionsRevoked()` callback for all partitions\n- Sends LeaveGroup request to broker\n- Closes broker connections\n- Waits up to `default.api.timeout.ms` for graceful termination\n- Does NOT auto-commit offsets (unless explicitly configured)\n\n**Force Leave Option:**\n\n```java\nconsumer.close(new CloseOptions().timeout(Duration.ofSeconds(1)))\n```\n\n---\n\n## Subscription and Assignment\n\n### Dynamic Subscription (Consumer Groups)\n\n#### `subscribe(Collection<String> topics)`\n\nSubscribe to a fixed set of topics.\n\n```java\nconsumer.subscribe(Arrays.asList(\"topic1\", \"topic2\", \"topic3\"));\n```\n\n**Characteristics:**\n\n- Triggers rebalancing to distribute partitions\n- Responds dynamically to partition count changes\n- Incompatible with `assign()` (calling both raises exception)\n- New subscriptions override previous ones\n\n**Exceptions:**\n\n- `IllegalArgumentException`: if topics collection is empty or null\n- `ConcurrentModificationException`: if topics collection modified during call\n\n#### `subscribe(Collection<String> topics, ConsumerRebalanceListener listener)`\n\nSubscribe with rebalance callbacks.\n\n```java\nclass MyRebalanceListener implements ConsumerRebalanceListener {\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        // Commit offsets for revoked partitions\n        consumer.commitSync();\n    }\n\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Initialize state for new partitions\n        for (TopicPartition tp : partitions) {\n            long offset = loadOffsetFromDatabase(tp);\n            consumer.seek(tp, offset);\n        }\n    }\n}\n\nconsumer.subscribe(Arrays.asList(\"topic1\"), new MyRebalanceListener());\n```\n\n**Callback Timing:**\n\n- `onPartitionsRevoked()` called during rebalance, before new assignment\n- `onPartitionsAssigned()` called after rebalance completes, before polling resumes\n\n### Pattern-Based Subscription\n\n#### `subscribe(Pattern pattern)` and `subscribe(Pattern pattern, ConsumerRebalanceListener listener)`\n\nSubscribe to topics matching a regex pattern.\n\n```java\n// Subscribe to all topics starting with \"events-\"\nconsumer.subscribe(Pattern.compile(\"^events-.*\"), listener);\n```\n\n**Dynamic Topic Detection:**\n\n- Consumer periodically refreshes topic metadata\n- New topics matching pattern automatically assigned to members\n- Removed topics trigger rebalancing\n\n**Caveats:**\n\n- Pattern matching happens on all topics known to cluster\n- No way to un-subscribe from matched topics individually\n- Must unsubscribe and re-subscribe with different pattern\n\n### Manual Partition Assignment\n\n#### `assign(Collection<TopicPartition> partitions)`\n\nManually assign specific partitions.\n\n```java\nList<TopicPartition> partitions = Arrays.asList(\n    new TopicPartition(\"topic1\", 0),\n    new TopicPartition(\"topic1\", 1)\n);\nconsumer.assign(partitions);\n```\n\n**Characteristics:**\n\n- Disables consumer group coordination\n- No rebalancing occurs\n- No `onPartitionsAssigned()` callback\n- Partitions remain assigned until next `assign()` call\n- Can be called multiple times to change assignment\n\n**Behavior with Offsets:**\n\n- Consumer must explicitly seek to desired offset\n- No committed offset is used for position initialization\n- `commitSync()` still works but offsets stored only for bookkeeping\n\n### Subscription State Management\n\n#### `subscription()` - Get current subscription\n\n```java\nSet<String> topics = consumer.subscription();  // Returns empty set if assigned\n```\n\n#### `assignment()` - Get current assignment\n\n```java\nSet<TopicPartition> partitions = consumer.assignment();\n```\n\n#### `unsubscribe()`\n\nClear all subscriptions and assignments.\n\n```java\nconsumer.unsubscribe();  // Triggers onPartitionsRevoked() for all partitions\n```\n\n---\n\n## Polling and Data Fetching\n\n### The Poll Loop\n\n#### `poll(Duration timeout)`\n\nThe central method in Kafka consumer usage.\n\n```java\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\nfor (ConsumerRecord<String, String> record : records) {\n    System.out.println(record.key() + \" -> \" + record.value());\n}\n```\n\n**Behavioral Semantics:**\n\nThe `poll()` method is complex and performs multiple functions:\n\n1. **Sends heartbeats** to maintain group membership (if subscribed)\n2. **Executes rebalance callbacks** if a rebalance was triggered\n3. **Fetches records** from brokers\n4. **Advances position** for successfully fetched records\n5. **Returns records** to the caller\n\n**Blocking Behavior:**\n\n- `timeout` parameter specifies maximum time to wait for records\n- During normal operation: returns quickly if records available, waits up to `timeout` if empty\n- During rebalance: `timeout` is exceeded while rebalance completes\n- Poll **may block longer than timeout** if rebalance callbacks take time\n- Timeout can be overridden by internal timers (e.g., metadata refresh)\n\n**Important Constraints:**\n\n- Must be called frequently to maintain group membership (at least once per `max.poll.interval.ms`)\n- Calling poll while processing prior batch can cause group expulsion\n- Only one `poll()` call can be in progress at a time (not thread-safe)\n\n**Return Value:**\n\n- `ConsumerRecords` object containing all fetched records\n- May be empty if no new records available\n- Records grouped by partition for efficient batch processing\n- Records are in order within each partition\n\n### Record Fetching\n\n#### Record Availability\n\nA record is available to the consumer when:\n\n1. It has been written to the partition\n2. All in-sync replicas have acknowledged it (for durability)\n3. If using `isolation.level=read_committed`: the transaction containing it is committed\n4. Position is <= record's offset\n\n#### Max Fetch Size\n\n`max.poll.records` configuration controls batch size:\n\n```properties\nmax.poll.records=500  # Default: fetch at most 500 records per poll\n```\n\nTrade-offs:\n- Lower values: more frequent polls, better responsiveness, higher overhead\n- Higher values: fewer polls, better throughput, more memory usage, longer processing deadlines\n\n#### Fetch Offset Management\n\nThe consumer tracks position per partition:\n\n```\nCommitted Offset     Position      End Offset\n      10      <---------11-------->      20\n\nConsumer will fetch from offset 11\nIf position >= end offset, no records available\n```\n\n---\n\n## Offset Management\n\n### Offset Concepts\n\n#### Position\n\nThe offset of the **next record** to be fetched.\n\n```java\nlong offset = consumer.position(partition);\n// If last fetched record was offset 100, position is 101\n```\n\n**Characteristics:**\n\n- Automatically advances when records are fetched via `poll()`\n- Can be manually set with `seek()`\n- Not persisted; resets after consumer restart\n\n#### Committed Offset\n\nThe last offset successfully committed to Kafka.\n\n```java\nMap<TopicPartition, OffsetAndMetadata> committed = consumer.committed(\n    Collections.singleton(partition)\n);\nOffsetAndMetadata oam = committed.get(partition);\nlong offset = oam.offset();  // 0 if never committed\n```\n\n**Characteristics:**\n\n- Persisted in `__consumer_offsets` topic\n- Used to resume after consumer restart\n- Can be committed via `commitSync()` or `commitAsync()`\n- Can be auto-committed based on time interval\n\n#### Offset Semantics\n\n**Critical Rule: Committed offset = next offset to read**\n\n```java\n// WRONG:\nConsumerRecord record = getLastRecord();\nconsumer.commitSync(record.offset());  // Commits record itself\n\n// RIGHT:\nConsumerRecord record = getLastRecord();\nconsumer.commitSync(record.offset() + 1);  // Commits offset of NEXT record\n\n// Or:\nconsumer.commitSync(Collections.singletonMap(partition,\n    new OffsetAndMetadata(record.offset() + 1)));\n```\n\n### Automatic Offset Committing\n\nWhen enabled, offsets are committed periodically:\n\n```properties\nenable.auto.commit=true              # Enable auto-commit\nauto.commit.interval.ms=5000         # Commit every 5 seconds\n```\n\n**Behavior:**\n\n- Commits all offsets from most recent `poll()` call\n- Offset committed = last fetched offset + 1\n- Happens asynchronously before next `poll()` call\n- No exception if commit fails (logged but not raised)\n\n**Risks:**\n\n- Records lost if consumer crashes between fetch and commit\n- At-least-once semantics: records may be re-delivered on restart\n- Cannot distinguish between processed and unprocessed records\n\n**Recommendation:** Disable for exactly-once semantics\n\n### Manual Offset Committing\n\n#### Synchronous Commit: `commitSync()`\n\n```java\ntry {\n    consumer.commitSync();  // Commits all offsets from last poll()\n} catch (CommitFailedException e) {\n    // Consumer no longer part of group, cannot commit\n    logger.error(\"Commit failed\", e);\n}\n```\n\n**Behavior:**\n\n- Blocks until commit succeeds or fails\n- Uses timeout from `default.api.timeout.ms`\n- Commits offsets for all fetched records from last `poll()` call\n- Fails immediately if `max.poll.interval.ms` exceeded (proactive expulsion)\n\n**Common Exception:**\n\n```\nCommitFailedException: The consumer is not part of an active consumer group\n```\n\nCauses:\n- Consumer not yet subscribed/joined group\n- Exceeded `max.poll.interval.ms` without calling `poll()`\n- Another consumer in group claims ownership of partition\n- Consumer was explicitly removed from group\n\n#### Synchronous Commit with Timeout: `commitSync(Duration timeout)`\n\n```java\nconsumer.commitSync(Duration.ofSeconds(5));  // Custom timeout\n```\n\n#### Explicit Offset Commit: `commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)`\n\nCommit specific offsets (e.g., after processing only some records).\n\n```java\nMap<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();\nfor (TopicPartition tp : records.partitions()) {\n    List<ConsumerRecord<String, String>> partitionRecords = records.records(tp);\n\n    // Find last successfully processed record\n    ConsumerRecord<String, String> lastProcessed = findLastSuccessful(partitionRecords);\n\n    if (lastProcessed != null) {\n        offsets.put(tp, new OffsetAndMetadata(\n            lastProcessed.offset() + 1,\n            Optional.of(lastProcessed.leaderEpoch().orElse(-1)),\n            \"processed\"\n        ));\n    }\n}\n\nconsumer.commitSync(offsets);  // Commit only processed offsets\n```\n\n#### Asynchronous Commit: `commitAsync()`\n\n```java\nconsumer.commitAsync((offsets, exception) -> {\n    if (exception != null) {\n        logger.error(\"Commit failed for offsets: {}\", offsets, exception);\n        // Retry or store for later commit\n    } else {\n        logger.info(\"Commit succeeded\");\n    }\n});\n\n// Continue polling immediately, callback invoked later\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n```\n\n**Characteristics:**\n\n- Returns immediately without waiting\n- Callback invoked when commit result available\n- Multiple pending async commits: callbacks invoke in order\n- Failures typically retried automatically\n\n#### Asynchronous Commit with Callback: `commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets, OffsetCommitCallback callback)`\n\n```java\nOffsetCommitCallback callback = (offsets, exception) -> {\n    if (exception != null) {\n        logger.error(\"Async commit failed\", exception);\n    }\n};\n\nconsumer.commitAsync(Collections.singletonMap(\n    partition,\n    new OffsetAndMetadata(nextOffset)\n), callback);\n```\n\n### Offset Commit Failures\n\n#### Retryable Failures\n\nThe consumer automatically retries:\n- Temporary network failures\n- Broker temporarily unavailable\n- Broker leadership changes during commit\n\n#### Non-Retryable Failures\n\nThe consumer does NOT retry:\n- `CommitFailedException`: consumer expelled from group\n- `AuthorizationException`: no permission to commit\n- Offset metadata too large (> 1MB)\n\n### Offset Reset Strategies\n\nIf position cannot be determined (no committed offset, consumer reset):\n\n```properties\nauto.offset.reset=earliest    # Start from beginning\nauto.offset.reset=latest      # Start from end\nauto.offset.reset=none        # Raise NoOffsetForPartitionException\n```\n\n**Applicable When:**\n\n- Consumer first joins group (no prior committed offset)\n- Partition added after consumer started\n- Consumer explicitly calls `seekToBeginning()` or `seekToEnd()`\n\n---\n\n## Flow Control and Position Management\n\n### Pause and Resume\n\n#### `pause(Collection<TopicPartition> partitions)`\n\nStop fetching from specified partitions.\n\n```java\n// Pause high-latency partition\nconsumer.pause(Collections.singleton(slowPartition));\n\n// Continue polling for other partitions\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n// slowPartition not included even if assigned\n```\n\n**Use Cases:**\n\n1. **Backpressure**: Downstream system is slow\n2. **Priority partitions**: Focus on high-priority topics first\n3. **Load balancing**: Redirect processing capacity\n4. **Throttling**: Rate-limit consumption\n\n**Important Notes:**\n\n- Paused partitions still count as \"assigned\"\n- Partition state does NOT persist through rebalance\n- `pause()` called during rebalance callback affects new assignment\n- Multiple `pause()` calls are cumulative\n\n#### `resume(Collection<TopicPartition> partitions)`\n\nResume fetching from paused partitions.\n\n```java\nconsumer.resume(Collections.singleton(slowPartition));\n```\n\n#### `paused()`\n\nGet the set of currently paused partitions.\n\n```java\nSet<TopicPartition> paused = consumer.paused();\n```\n\n### Position Control\n\n#### `seek(TopicPartition partition, long offset)`\n\nSet the offset to fetch on next `poll()`.\n\n```java\n// Seek to specific offset\nconsumer.seek(partition, 12345);\n\n// Next poll() starts from offset 12345\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n// First record will be offset 12345 (or next available)\n```\n\n**Semantics:**\n\n- Position is set immediately\n- Fetching starts at this offset on next `poll()`\n- Offset must be non-negative\n- Partition must be assigned to consumer\n\n**Exceptions:**\n\n- `IllegalArgumentException`: offset < 0\n- `IllegalStateException`: partition not assigned\n\n#### `seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)`\n\nSeek with leader epoch information (for truncation detection).\n\n```java\nOffsetAndMetadata offsetWithEpoch = new OffsetAndMetadata(\n    12345,\n    Optional.of(leaderEpoch),  // Include leader epoch\n    \"\"\n);\nconsumer.seek(partition, offsetWithEpoch);\n```\n\n#### `position(TopicPartition partition)`\n\nGet the current position (next offset to fetch).\n\n```java\nlong nextOffset = consumer.position(partition);\n```\n\n**Behavior:**\n\n- Performs remote call if position not yet initialized\n- Blocks up to `default.api.timeout.ms`\n- Returns offset of next record to be fetched\n\n#### `position(TopicPartition partition, Duration timeout)`\n\nPosition with custom timeout.\n\n```java\nlong pos = consumer.position(partition, Duration.ofSeconds(10));\n```\n\n### Seeking to Beginning and End\n\n#### `seekToBeginning(Collection<TopicPartition> partitions)`\n\nSeek to the first available offset.\n\n```java\nconsumer.seekToBeginning(Collections.singleton(partition));\n// or all assigned partitions\nconsumer.seekToBeginning(Collections.emptyList());\n```\n\n**Behavior:**\n\n- Lazy evaluation: actual seek happens at next `poll()` call\n- Sets position to earliest offset in partition\n- Use `beginningOffsets()` to query without seeking\n\n#### `seekToEnd(Collection<TopicPartition> partitions)`\n\nSeek to the last offset.\n\n```java\nconsumer.seekToEnd(Collections.singleton(partition));\n```\n\n**Behavior:**\n\n- Lazy evaluation: actual seek happens at next `poll()` call\n- Sets position just past last message\n- If `isolation.level=read_committed`: seeks to Last Stable Offset (LSO)\n- Next `poll()` returns no records for this partition (unless new messages arrive)\n\n### Timestamp-Based Seeking\n\n#### `offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch)`\n\nFind offsets for specific timestamps.\n\n```java\n// Find offset for records >= 1 hour ago\nlong oneHourAgo = System.currentTimeMillis() - 3600_000;\nMap<TopicPartition, Long> timestampMap = Collections.singletonMap(\n    partition,\n    oneHourAgo\n);\n\nMap<TopicPartition, OffsetAndTimestamp> result = consumer.offsetsForTimes(timestampMap);\n\nOffsetAndTimestamp oat = result.get(partition);\nif (oat != null) {\n    consumer.seek(partition, oat.offset());\n}\n```\n\n**Behavior:**\n\n- Returns earliest offset whose timestamp >= requested timestamp\n- Returns `null` if no matching offset found\n- Requires topic configured with `message.format.version >= 0.10`\n- Blocking call up to `default.api.timeout.ms`\n\n**Timestamps:**\n\n- Created by producer: `CreateTime` (default)\n- Added by broker: `LogAppendTime` (if configured)\n- Broker can have log compaction without timestamps\n\n### Beginning and End Offset Queries\n\n#### `beginningOffsets(Collection<TopicPartition> partitions)`\n\nQuery first available offset (doesn't seek).\n\n```java\nMap<TopicPartition, Long> beginnings = consumer.beginningOffsets(\n    Collections.singleton(partition)\n);\nlong firstOffset = beginnings.get(partition);\n```\n\n#### `endOffsets(Collection<TopicPartition> partitions)`\n\nQuery offset after last message (doesn't seek).\n\n```java\nMap<TopicPartition, Long> ends = consumer.endOffsets(\n    Collections.singleton(partition)\n);\nlong nextOffset = ends.get(partition);  // Offset to append next message\n```\n\n**Difference from seekToEnd:**\n- These methods only query; don't change position\n- Can be called even if partition not assigned\n- Useful for monitoring lag or validating offsets\n\n### Current Lag Calculation\n\n#### `currentLag(TopicPartition topicPartition)`\n\nGet approximate consumer lag.\n\n```java\nOptionalLong lag = consumer.currentLag(partition);\nif (lag.isPresent()) {\n    System.out.println(\"Lag: \" + lag.getAsLong() + \" records\");\n}\n```\n\n**Behavior:**\n\n- Returns `OptionalLong.empty()` if lag cannot be determined\n- Lag = `endOffset - position`\n- Based on cached metadata; may be approximate\n\n---\n\n## Rebalancing and Consumer Groups\n\n### Rebalance Triggers\n\nA rebalance occurs when:\n\n1. **New consumer joins group** (first `poll()` after subscribe)\n2. **Consumer leaves group** (`close()`, `unsubscribe()`, session timeout)\n3. **Partition count changes** (in subscribed topics)\n4. **Topic added** (if using regex subscription)\n5. **Consumer explicitly requests** (`enforceRebalance()`)\n\n### Rebalance Protocol\n\n**Classic (Eager) Rebalancing:**\n\n1. Stop: All consumers stop fetching\n2. Revoke: All consumers lose all partitions (even unchanged ones)\n3. Rebalance: Broker runs assignment strategy\n4. Assign: Consumers receive new assignment\n5. executor.shutdown();\n    consumer.close();\n}\n```\n\n---\n\n## Core Types\n\n### KafkaConsumer<K, V>\n\nThe main consumer class. Properties();\nprops.put(\"bootstrap.servers\", \"localhost:9092\");\nprops.put(\"group.id\", \"my-group\");\nprops.put(\"key.deserializer\", StringDeserializer.class.getName());\nprops.put(\"value.deserializer\", StringDeserializer.class.getName());\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\n```\n\n**Step 2: Subscribe or assign**\n\n```java\n// Dynamic assignment with consumer group\nconsumer.subscribe(Arrays.asList(\"topic1\", \"topic2\"));\n\n// OR manual assignment (mutually exclusive)\nconsumer.assign(Arrays.asList(\n    new TopicPartition(\"topic1\", 0),\n    new TopicPartition(\"topic1\", 1)\n));\n```\n\n**Step 3: Poll (group join happens here)**\n\n```java\n// First poll() triggers group join if subscribed\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n```\n\n### Joining a Consumer Group\n\nWhen a consumer first calls `poll()` after `subscribe()`:\n\n1. Consumer sends JoinGroup request with group ID\n2. Broker selects a group leader (first member to join)\n3. Leader runs partition assignment strategy (e.g., RangeAssignor)\n4. Members receive assignment via SyncGroup\n5. If custom listener provided, `onPartitionsAssigned()` is called\n6. Consumer begins fetching from assigned partitions\n\n**Timing Constraints:**\n\n- Group join must complete within `session.timeout.ms` (default 45 seconds)\n- If no `poll()` call for longer than `max.poll.interval.ms` (default 5 minutes), consumer is expelled\n- Expulsion is proactive\u2014consumer will fail on next `commitSync()` with `CommitFailedException`\n\n### Graceful Shutdown\n\n```java\ntry {\n    while (true) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n        // process records\n    }\n} finally {\n    consumer.close()  // or consumer.close(Duration)\n}\n```\n\n**Close Behavior:**\n\n- Triggers `onPartitionsRevoked()` callback for all partitions\n- Sends LeaveGroup request to broker\n- Closes broker connections\n- Waits up to `default.api.timeout.ms` for graceful termination\n- Does NOT auto-commit offsets (unless explicitly configured)\n\n**Force Leave Option:**\n\n```java\nconsumer.close(new CloseOptions().timeout(Duration.ofSeconds(1)))\n```\n\n---\n\n## Subscription and Assignment\n\n### Dynamic Subscription (Consumer Groups)\n\n#### `subscribe(Collection<String> topics)`\n\nSubscribe to a fixed set of topics.\n\n```java\nconsumer.subscribe(Arrays.asList(\"topic1\", \"topic2\", \"topic3\"));\n```\n\n**Characteristics:**\n\n- Triggers rebalancing to distribute partitions\n- Responds dynamically to partition count changes\n- Incompatible with `assign()` (calling both raises exception)\n- New subscriptions override previous ones\n\n**Exceptions:**\n\n- `IllegalArgumentException`: if topics collection is empty or null\n- `ConcurrentModificationException`: if topics collection modified during call\n\n#### `subscribe(Collection<String> topics, ConsumerRebalanceListener listener)`\n\nSubscribe with rebalance callbacks.\n\n```java\nclass MyRebalanceListener implements ConsumerRebalanceListener {\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        // Commit offsets for revoked partitions\n        consumer.commitSync();\n    }\n\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Initialize state for new partitions\n        for (TopicPartition tp : partitions) {\n            long offset = loadOffsetFromDatabase(tp);\n            consumer.seek(tp, offset);\n        }\n    }\n}\n\nconsumer.subscribe(Arrays.asList(\"topic1\"), new MyRebalanceListener());\n```\n\n**Callback Timing:**\n\n- `onPartitionsRevoked()` called during rebalance, before new assignment\n- `onPartitionsAssigned()` called after rebalance completes, before polling resumes\n\n### Pattern-Based Subscription\n\n#### `subscribe(Pattern pattern)` and `subscribe(Pattern pattern, ConsumerRebalanceListener listener)`\n\nSubscribe to topics matching a regex pattern.\n\n```java\n// Subscribe to all topics starting with \"events-\"\nconsumer.subscribe(Pattern.compile(\"^events-.*\"), listener);\n```\n\n**Dynamic Topic Detection:**\n\n- Consumer periodically refreshes topic metadata\n- New topics matching pattern automatically assigned to members\n- Removed topics trigger rebalancing\n\n**Caveats:**\n\n- Pattern matching happens on all topics known to cluster\n- No way to un-subscribe from matched topics individually\n- Must unsubscribe and re-subscribe with different pattern\n\n### Manual Partition Assignment\n\n#### `assign(Collection<TopicPartition> partitions)`\n\nManually assign specific partitions.\n\n```java\nList<TopicPartition> partitions = Arrays.asList(\n    new TopicPartition(\"topic1\", 0),\n    new TopicPartition(\"topic1\", 1)\n);\nconsumer.assign(partitions);\n```\n\n**Characteristics:**\n\n- Disables consumer group coordination\n- No rebalancing occurs\n- No `onPartitionsAssigned()` callback\n- Partitions remain assigned until next `assign()` call\n- Can be called multiple times to change assignment\n\n**Behavior with Offsets:**\n\n- Consumer must explicitly seek to desired offset\n- No committed offset is used for position initialization\n- `commitSync()` still works but offsets stored only for bookkeeping\n\n### Subscription State Management\n\n#### `subscription()` - Get current subscription\n\n```java\nSet<String> topics = consumer.subscription();  // Returns empty set if assigned\n```\n\n#### `assignment()` - Get current assignment\n\n```java\nSet<TopicPartition> partitions = consumer.assignment();\n```\n\n#### `unsubscribe()`\n\nClear all subscriptions and assignments.\n\n```java\nconsumer.unsubscribe();  // Triggers onPartitionsRevoked() for all partitions\n```\n\n---\n\n## Polling and Data Fetching\n\n### The Poll Loop\n\n#### `poll(Duration timeout)`\n\nThe central method in Kafka consumer usage.\n\n```java\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\nfor (ConsumerRecord<String, String> record : records) {\n    System.out.println(record.key() + \" -> \" + record.value());\n}\n```\n\n**Behavioral Semantics:**\n\nThe `poll()` method is complex and performs multiple functions:\n\n1. **Sends heartbeats** to maintain group membership (if subscribed)\n2. **Executes rebalance callbacks** if a rebalance was triggered\n3. **Fetches records** from brokers\n4. **Advances position** for successfully fetched records\n5. **Returns records** to the caller\n\n**Blocking Behavior:**\n\n- `timeout` parameter specifies maximum time to wait for records\n- During normal operation: returns quickly if records available, waits up to `timeout` if empty\n- During rebalance: `timeout` is exceeded while rebalance completes\n- Poll **may block longer than timeout** if rebalance callbacks take time\n- Timeout can be overridden by internal timers (e.g., metadata refresh)\n\n**Important Constraints:**\n\n- Must be called frequently to maintain group membership (at least once per `max.poll.interval.ms`)\n- Calling poll while processing prior batch can cause group expulsion\n- Only one `poll()` call can be in progress at a time (not thread-safe)\n\n**Return Value:**\n\n- `ConsumerRecords` object containing all fetched records\n- May be empty if no new records available\n- Records grouped by partition for efficient batch processing\n- Records are in order within each partition\n\n### Record Fetching\n\n#### Record Availability\n\nA record is available to the consumer when:\n\n1. It has been written to the partition\n2. All in-sync replicas have acknowledged it (for durability)\n3. If using `isolation.level=read_committed`: the transaction containing it is committed\n4. Position is <= record's offset\n\n#### Max Fetch Size\n\n`max.poll.records` configuration controls batch size:\n\n```properties\nmax.poll.records=500  # Default: fetch at most 500 records per poll\n```\n\nTrade-offs:\n- Lower values: more frequent polls, better responsiveness, higher overhead\n- Higher values: fewer polls, better throughput, more memory usage, longer processing deadlines\n\n#### Fetch Offset Management\n\nThe consumer tracks position per partition:\n\n```\nCommitted Offset     Position      End Offset\n      10      <---------11-------->      20\n\nConsumer will fetch from offset 11\nIf position >= end offset, no records available\n```\n\n---\n\n## Offset Management\n\n### Offset Concepts\n\n#### Position\n\nThe offset of the **next record** to be fetched.\n\n```java\nlong offset = consumer.position(partition);\n// If last fetched record was offset 100, position is 101\n```\n\n**Characteristics:**\n\n- Automatically advances when records are fetched via `poll()`\n- Can be manually set with `seek()`\n- Not persisted; resets after consumer restart\n\n#### Committed Offset\n\nThe last offset successfully committed to Kafka.\n\n```java\nMap<TopicPartition, OffsetAndMetadata> committed = consumer.committed(\n    Collections.singleton(partition)\n);\nOffsetAndMetadata oam = committed.get(partition);\nlong offset = oam.offset();  // 0 if never committed\n```\n\n**Characteristics:**\n\n- Persisted in `__consumer_offsets` topic\n- Used to resume after consumer restart\n- Can be committed via `commitSync()` or `commitAsync()`\n- Can be auto-committed based on time interval\n\n#### Offset Semantics\n\n**Critical Rule: Committed offset = next offset to read**\n\n```java\n// WRONG:\nConsumerRecord record = getLastRecord();\nconsumer.commitSync(record.offset());  // Commits record itself\n\n// RIGHT:\nConsumerRecord record = getLastRecord();\nconsumer.commitSync(record.offset() + 1);  // Commits offset of NEXT record\n\n// Or:\nconsumer.commitSync(Collections.singletonMap(partition,\n    new OffsetAndMetadata(record.offset() + 1)));\n```\n\n### Automatic Offset Committing\n\nWhen enabled, offsets are committed periodically:\n\n```properties\nenable.auto.commit=true              # Enable auto-commit\nauto.commit.interval.ms=5000         # Commit every 5 seconds\n```\n\n**Behavior:**\n\n- Commits all offsets from most recent `poll()` call\n- Offset committed = last fetched offset + 1\n- Happens asynchronously before next `poll()` call\n- No exception if commit fails (logged but not raised)\n\n**Risks:**\n\n- Records lost if consumer crashes between fetch and commit\n- At-least-once semantics: records may be re-delivered on restart\n- Cannot distinguish between processed and unprocessed records\n\n**Recommendation:** Disable for exactly-once semantics\n\n### Manual Offset Committing\n\n#### Synchronous Commit: `commitSync()`\n\n```java\ntry {\n    consumer.commitSync();  // Commits all offsets from last poll()\n} catch (CommitFailedException e) {\n    // Consumer no longer part of group, cannot commit\n    logger.error(\"Commit failed\", e);\n}\n```\n\n**Behavior:**\n\n- Blocks until commit succeeds or fails\n- Uses timeout from `default.api.timeout.ms`\n- Commits offsets for all fetched records from last `poll()` call\n- Fails immediately if `max.poll.interval.ms` exceeded (proactive expulsion)\n\n**Common Exception:**\n\n```\nCommitFailedException: The consumer is not part of an active consumer group\n```\n\nCauses:\n- Consumer not yet subscribed/joined group\n- Exceeded `max.poll.interval.ms` without calling `poll()`\n- Another consumer in group claims ownership of partition\n- Consumer was explicitly removed from group\n\n#### Synchronous Commit with Timeout: `commitSync(Duration timeout)`\n\n```java\nconsumer.commitSync(Duration.ofSeconds(5));  // Custom timeout\n```\n\n#### Explicit Offset Commit: `commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)`\n\nCommit specific offsets (e.g., after processing only some records).\n\n```java\nMap<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();\nfor (TopicPartition tp : records.partitions()) {\n    List<ConsumerRecord<String, String>> partitionRecords = records.records(tp);\n\n    // Find last successfully processed record\n    ConsumerRecord<String, String> lastProcessed = findLastSuccessful(partitionRecords);\n\n    if (lastProcessed != null) {\n        offsets.put(tp, new OffsetAndMetadata(\n            lastProcessed.offset() + 1,\n            Optional.of(lastProcessed.leaderEpoch().orElse(-1)),\n            \"processed\"\n        ));\n    }\n}\n\nconsumer.commitSync(offsets);  // Commit only processed offsets\n```\n\n#### Asynchronous Commit: `commitAsync()`\n\n```java\nconsumer.commitAsync((offsets, exception) -> {\n    if (exception != null) {\n        logger.error(\"Commit failed for offsets: {}\", offsets, exception);\n        // Retry or store for later commit\n    } else {\n        logger.info(\"Commit succeeded\");\n    }\n});\n\n// Continue polling immediately, callback invoked later\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n```\n\n**Characteristics:**\n\n- Returns immediately without waiting\n- Callback invoked when commit result available\n- Multiple pending async commits: callbacks invoke in order\n- Failures typically retried automatically\n\n#### Asynchronous Commit with Callback: `commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets, OffsetCommitCallback callback)`\n\n```java\nOffsetCommitCallback callback = (offsets, exception) -> {\n    if (exception != null) {\n        logger.error(\"Async commit failed\", exception);\n    }\n};\n\nconsumer.commitAsync(Collections.singletonMap(\n    partition,\n    new OffsetAndMetadata(nextOffset)\n), callback);\n```\n\n### Offset Commit Failures\n\n#### Retryable Failures\n\nThe consumer automatically retries:\n- Temporary network failures\n- Broker temporarily unavailable\n- Broker leadership changes during commit\n\n#### Non-Retryable Failures\n\nThe consumer does NOT retry:\n- `CommitFailedException`: consumer expelled from group\n- `AuthorizationException`: no permission to commit\n- Offset metadata too large (> 1MB)\n\n### Offset Reset Strategies\n\nIf position cannot be determined (no committed offset, consumer reset):\n\n```properties\nauto.offset.reset=earliest    # Start from beginning\nauto.offset.reset=latest      # Start from end\nauto.offset.reset=none        # Raise NoOffsetForPartitionException\n```\n\n**Applicable When:**\n\n- Consumer first joins group (no prior committed offset)\n- Partition added after consumer started\n- Consumer explicitly calls `seekToBeginning()` or `seekToEnd()`\n\n---\n\n## Flow Control and Position Management\n\n### Pause and Resume\n\n#### `pause(Collection<TopicPartition> partitions)`\n\nStop fetching from specified partitions.\n\n```java\n// Pause high-latency partition\nconsumer.pause(Collections.singleton(slowPartition));\n\n// Continue polling for other partitions\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n// slowPartition not included even if assigned\n```\n\n**Use Cases:**\n\n1. **Backpressure**: Downstream system is slow\n2. **Priority partitions**: Focus on high-priority topics first\n3. **Load balancing**: Redirect processing capacity\n4. **Throttling**: Rate-limit consumption\n\n**Important Notes:**\n\n- Paused partitions still count as \"assigned\"\n- Partition state does NOT persist through rebalance\n- `pause()` called during rebalance callback affects new assignment\n- Multiple `pause()` calls are cumulative\n\n#### `resume(Collection<TopicPartition> partitions)`\n\nResume fetching from paused partitions.\n\n```java\nconsumer.resume(Collections.singleton(slowPartition));\n```\n\n#### `paused()`\n\nGet the set of currently paused partitions.\n\n```java\nSet<TopicPartition> paused = consumer.paused();\n```\n\n### Position Control\n\n#### `seek(TopicPartition partition, long offset)`\n\nSet the offset to fetch on next `poll()`.\n\n```java\n// Seek to specific offset\nconsumer.seek(partition, 12345);\n\n// Next poll() starts from offset 12345\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n// First record will be offset 12345 (or next available)\n```\n\n**Semantics:**\n\n- Position is set immediately\n- Fetching starts at this offset on next `poll()`\n- Offset must be non-negative\n- Partition must be assigned to consumer\n\n**Exceptions:**\n\n- `IllegalArgumentException`: offset < 0\n- `IllegalStateException`: partition not assigned\n\n#### `seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)`\n\nSeek with leader epoch information (for truncation detection).\n\n```java\nOffsetAndMetadata offsetWithEpoch = new OffsetAndMetadata(\n    12345,\n    Optional.of(leaderEpoch),  // Include leader epoch\n    \"\"\n);\nconsumer.seek(partition, offsetWithEpoch);\n```\n\n#### `position(TopicPartition partition)`\n\nGet the current position (next offset to fetch).\n\n```java\nlong nextOffset = consumer.position(partition);\n```\n\n**Behavior:**\n\n- Performs remote call if position not yet initialized\n- Blocks up to `default.api.timeout.ms`\n- Returns offset of next record to be fetched\n\n#### `position(TopicPartition partition, Duration timeout)`\n\nPosition with custom timeout.\n\n```java\nlong pos = consumer.position(partition, Duration.ofSeconds(10));\n```\n\n### Seeking to Beginning and End\n\n#### `seekToBeginning(Collection<TopicPartition> partitions)`\n\nSeek to the first available offset.\n\n```java\nconsumer.seekToBeginning(Collections.singleton(partition));\n// or all assigned partitions\nconsumer.seekToBeginning(Collections.emptyList());\n```\n\n**Behavior:**\n\n- Lazy evaluation: actual seek happens at next `poll()` call\n- Sets position to earliest offset in partition\n- Use `beginningOffsets()` to query without seeking\n\n#### `seekToEnd(Collection<TopicPartition> partitions)`\n\nSeek to the last offset.\n\n```java\nconsumer.seekToEnd(Collections.singleton(partition));\n```\n\n**Behavior:**\n\n- Lazy evaluation: actual seek happens at next `poll()` call\n- Sets position just past last message\n- If `isolation.level=read_committed`: seeks to Last Stable Offset (LSO)\n- Next `poll()` returns no records for this partition (unless new messages arrive)\n\n### Timestamp-Based Seeking\n\n#### `offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch)`\n\nFind offsets for specific timestamps.\n\n```java\n// Find offset for records >= 1 hour ago\nlong oneHourAgo = System.currentTimeMillis() - 3600_000;\nMap<TopicPartition, Long> timestampMap = Collections.singletonMap(\n    partition,\n    oneHourAgo\n);\n\nMap<TopicPartition, OffsetAndTimestamp> result = consumer.offsetsForTimes(timestampMap);\n\nOffsetAndTimestamp oat = result.get(partition);\nif (oat != null) {\n    consumer.seek(partition, oat.offset());\n}\n```\n\n**Behavior:**\n\n- Returns earliest offset whose timestamp >= requested timestamp\n- Returns `null` if no matching offset found\n- Requires topic configured with `message.format.version >= 0.10`\n- Blocking call up to `default.api.timeout.ms`\n\n**Timestamps:**\n\n- Created by producer: `CreateTime` (default)\n- Added by broker: `LogAppendTime` (if configured)\n- Broker can have log compaction without timestamps\n\n### Beginning and End Offset Queries\n\n#### `beginningOffsets(Collection<TopicPartition> partitions)`\n\nQuery first available offset (doesn't seek).\n\n```java\nMap<TopicPartition, Long> beginnings = consumer.beginningOffsets(\n    Collections.singleton(partition)\n);\nlong firstOffset = beginnings.get(partition);\n```\n\n#### `endOffsets(Collection<TopicPartition> partitions)`\n\nQuery offset after last message (doesn't seek).\n\n```java\nMap<TopicPartition, Long> ends = consumer.endOffsets(\n    Collections.singleton(partition)\n);\nlong nextOffset = ends.get(partition);  // Offset to append next message\n```\n\n**Difference from seekToEnd:**\n- These methods only query; don't change position\n- Can be called even if partition not assigned\n- Useful for monitoring lag or validating offsets\n\n### Current Lag Calculation\n\n#### `currentLag(TopicPartition topicPartition)`\n\nGet approximate consumer lag.\n\n```java\nOptionalLong lag = consumer.currentLag(partition);\nif (lag.isPresent()) {\n    System.out.println(\"Lag: \" + lag.getAsLong() + \" records\");\n}\n```\n\n**Behavior:**\n\n- Returns `OptionalLong.empty()` if lag cannot be determined\n- Lag = `endOffset - position`\n- Based on cached metadata; may be approximate\n\n---\n\n## Rebalancing and Consumer Groups\n\n### Rebalance Triggers\n\nA rebalance occurs when:\n\n1. **New consumer joins group** (first `poll()` after subscribe)\n2. **Consumer leaves group** (`close()`, `unsubscribe()`, session timeout)\n3. **Partition count changes** (in subscribed topics)\n4. **Topic added** (if using regex subscription)\n5. **Consumer explicitly requests** (`enforceRebalance()`)\n\n### Rebalance Protocol\n\n**Classic (Eager) Rebalancing:**\n\n1. Stop: All consumers stop fetching\n2. Revoke: All consumers lose all partitions (even unchanged ones)\n3. Rebalance: Broker runs assignment strategy\n4. Assign: Consumers receive new assignment\n5. Resume: Consumers begin fetching\n\n**Cooperative Rebalancing:**\n\n1. Revoke: Consumers revoke only partitions they're losing\n2. Rebalance: Broker runs assignment strategy\n3. Assign: Consumers receive only new partitions\n4. Resume: Seamless transition with minimal stop\n\n### Consumer Rebalance Listener\n\n#### Implementing the Interface\n\n```java\npublic class MyRebalanceListener implements ConsumerRebalanceListener {\n    private final KafkaConsumer<?, ?> consumer;\n    private final Database database;\n\n    public MyRebalanceListener(KafkaConsumer<?, ?> consumer, Database database) {\n        this.consumer = consumer;\n        this.database = database;\n    }\n\n    @Override\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        // CRITICAL: Commit offsets before losing partitions\n        try {\n            consumer.commitSync();\n        } catch (CommitFailedException e) {\n            logger.error(\"Failed to commit\", e);\n        }\n\n        // Clean up partition-local state\n        for (TopicPartition tp : partitions) {\n            flushPartitionCache(tp);\n        }\n    }\n\n    @Override\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Initialize state for new partitions\n        for (TopicPartition tp : partitions) {\n            long offset = database.getOffset(tp);\n            if (offset >= 0) {\n                consumer.seek(tp, offset);\n            }\n        }\n    }\n\n    @Override\n    public void onPartitionsLost(Collection<TopicPartition> partitions) {\n        // Partitions already reassigned; don't commit\n        logger.warn(\"Partitions lost without revoke callback: {}\", partitions);\n    }\n}\n\n// Register listener\nconsumer.subscribe(Arrays.asList(\"topic1\", \"topic2\"),\n    new MyRebalanceListener(consumer, database));\n```\n\n#### Callback Execution Context\n\n**When Called:**\n\n- Only within `poll()` calls\n- Before other operations resume\n- Synchronously (callback must return before `poll()` returns)\n\n**Exception Handling:**\n\n- Exceptions from callbacks propagate to the `poll()` call\n- Consumer is still notified of assignment (callback failure doesn't prevent assignment)\n- `WakeupException` can be raised from nested consumer calls\n\n**Callback Constraints:**\n\n- Cannot call `poll()` recursively\n- Can call `commitSync()`, `seek()`, `position()`, etc.\n- Cannot call `subscribe()`, `assign()`, `unsubscribe()`, `close()`\n\n### Proactive Consumer Expulsion\n\nIf consumer doesn't call `poll()` within `max.poll.interval.ms`:\n\n1.
