## Scope

Your document must cover these four components and how they work together:

### 1. ConnectionManagerImpl
The central network filter that owns the connection lifecycle. Explain:
- How it implements `Network::ReadFilter` to receive raw bytes

### 2. FilterManager and the HTTP Filter Chain

### 4. Cluster Manager and Upstream Connectivity
The upstream connection pool and cluster management layer. Explain:
- How `ClusterManagerImpl` provides connection pools per cluster

Network bytes arrive \u2192 `onData()` called\n2. Codec parses headers/data/trailers from the buffer\n3. For each logical HTTP message:\n   - Codec calls `ConnectionManagerImpl::newStream()`\n   - This creates an `ActiveStream` and returns its `RequestDecoder` interface\n   - Codec delivers headers, data, trailers to the `ActiveStream`\n4. `ActiveStream` passes data through the `FilterManager`'s decoder filter chain\n5. The Router filter (terminal decoder filter) forwards to upstream\n6. Upstream response arrives, flows through `FilterManager`'s encoder chain (in reverse)\n7. `ActiveStream` encodes response back to the codec via callbacks\n8. Codec writes to the network\n\n### Connection-Level Flow Control\n\n**Watermarks** propagate from the downstream connection to all active streams:\n\n```cpp\nvoid onAboveWriteBufferHighWatermark() {\n  codec_->onUnderlyingConnectionAboveWriteBufferHighWatermark();\n}\n```\n\nThe codec notifies each stream's watermark callbacks. Streams backpressure the router filter, which backpressures upstream connections.\n\n### Drain and Graceful Shutdown\n\nWhen draining (e.g., on server shutdown):\n- No new requests are accepted\n- Existing requests complete\n- The HCM listens to `Network::DrainDecision` and returns drain status to the network layer\n\n---\n\n## 2. FilterManager and the HTTP Filter Chain\n\n**Files**: `source/common/http/filter_manager.h`, `source/common/http/filter_manager.cc`\n\n### Role and Responsibilities\n\n`FilterManager` orchestrates the execution of HTTP filters on a per-stream basis. It:\n\n1. **Creates the filter chain** using `FilterChainFactory`\n2. **Manages decoder filters** (request path, executed in configured order)\n3. **Manages encoder filters** (response path, executed in reverse order)\n4. **Handles filter lifecycle**: iteration control, buffering, stopping/resuming\n5. **Implements local replies** (error responses from filters, not from upstream)\n\n### Filter Chain Architecture\n\nFilters in the HCM implement one of three patterns:\n\n```cpp\nclass StreamDecoderFilter {\n  FilterHeadersStatus decodeHeaders(RequestHeaderMap&, bool end_stream);\n  FilterDataStatus decodeData(Buffer::Instance&, bool end_stream);\n  FilterTrailersStatus decodeTrailers(RequestTrailerMap&);\n  FilterMetadataStatus decodeMetadata(MetadataMap&);\n};\n\nclass StreamEncoderFilter {\n  Filter1xxHeadersStatus encode1xxHeaders(ResponseHeaderMap&);\n  FilterHeadersStatus encodeHeaders(ResponseHeaderMap&, bool end_stream);\n  FilterDataStatus encodeData(Buffer::Instance&, bool end_stream);\n  FilterTrailersStatus encodeTrailers(ResponseTrailerMap&);\n};\n\nclass StreamFilter : public StreamDecoderFilter, public StreamEncoderFilter {\n  // Dual decoder/encoder filter\n};\n```\n\n### Filter Execution Model\n\n**Decoder Chain (Request)**: Filters execute in configuration order.\n\n```\nRequest Headers\n    \u2193\n[Filter A] decodeHeaders()\n    \u2193\n[Filter B] decodeHeaders()\n    \u2193\n[Filter C] decodeHeaders() \u2192 [Router Filter] decodeHeaders()\n    \u2193\nRequest Data\n    \u2193\n[Filter A] decodeData()\n    \u2193\n[Filter B] decodeData()\n    \u2193\n[Filter C] decodeData()\n```\n\nEach filter receives one of three status values:\n- `FilterHeadersStatus::Continue` (or `FilterDataStatus::Continue`): continue iteration\n- `FilterHeadersStatus::StopIteration`: stop iteration, await resume\n- `FilterHeadersStatus::StopAllIteration`: stop and buffer all subsequent frames\n\nWhen a filter returns `StopIteration`, the `FilterManager` buffers the next frame and waits for the filter to call `continueDecoding()`.\n\n**Encoder Chain (Response)**: Filters execute in **reverse** order.\n\n```\nResponse Headers (from upstream or local reply)\n    \u2193\n[Filter C] encodeHeaders()\n    \u2193\n[Filter B] encodeHeaders()\n    \u2193\n[Filter A] encodeHeaders()\n    \u2193\nSend to codec\n```\n\nThis reversal allows filters to \"undo\" transformations they made during decoding.\n\n### Filter Iteration: Key Classes\n\n**`ActiveStreamDecoderFilter` and `ActiveStreamEncoderFilter`**: Wrappers around the actual filter instances.\n\n```cpp\nstruct ActiveStreamDecoderFilter {\n  StreamDecoderFilterSharedPtr filter_;\n  FilterChainFactory::FilterFactoryFunc factory_func_;\n  // Called when filter returns StopIteration, buffer is full, etc.\n  bool commonHandleAfterHeadersCallback(FilterHeadersStatus status, bool& end_stream);\n};\n```\n\n**`StreamDecoderFilters` and `StreamEncoderFilters`**: Containers with begin/end iterators.\n\n```cpp\nstruct StreamDecoderFilters {\n  std::vector<ActiveStreamDecoderFilterPtr> entries_;\n  Iterator begin() { return entries_.begin(); }\n  Iterator end() { return entries_.end(); }\n};\n\nstruct StreamEncoderFilters {\n  std::vector<ActiveStreamEncoderFilterPtr> entries_;\n  Iterator begin() { return entries_.rbegin(); }  // Reverse!\n  Iterator end() { return entries_.rend(); }\n};\n```\n\n### Filter Modification and Local Replies\n\nFilters can:\n\n1. **Modify headers**:\n   ```cpp\n   FilterHeadersStatus decodeHeaders(RequestHeaderMap& headers, bool end_stream) {\n     headers.addCopy(\"x-my-header\", \"value\");\n     return FilterHeadersStatus::Continue;\n   }\n   ```\n\n2. **Send a local reply** (terminate the request without forwarding):\n   ```cpp\n   FilterHeadersStatus decodeHeaders(RequestHeaderMap&, bool) {\n     callbacks_->sendLocalReply(Code::BadRequest, \"Invalid\");\n     return FilterHeadersStatus::StopIteration;\n   }\n   ```\n   This triggers the encoder chain with a synthesized response.\n\n3. **Stop iteration and buffer**:\n   ```cpp\n   FilterHeadersStatus decodeHeaders(RequestHeaderMap&, bool) {\n     return FilterHeadersStatus::StopIteration; // Wait for async work\n   }\n   ```\n   Later, call `continueDecoding()` to resume.\n\n### Stream Information and Context\n\nEach stream maintains a `StreamInfo::StreamInfo` object (in `ActiveStream`):\n- Request/response headers and trailers\n- Upstream cluster selected\n- Upstream host selected\n- Response code, duration, and other metadata\n- Dynamic metadata (populated by filters)\n- Filter state\n\nFilters access this via their `StreamFilterCallbacks`:\n```cpp\nclass StreamFilterCallbacks {\n  StreamInfo::StreamInfo& streamInfo();\n  Router::RouteConstSharedPtr route();\n  void resetStream(StreamResetReason reason, absl::string_view details);\n  // ...\n};\n```\n\n---\n\n## 3. Router Filter: The Terminal Decoder Filter\n\n**Files**: `source/common/router/router.h`, `source/common/router/router.cc`\n\n### Role and Responsibilities\n\nThe Router filter is the **terminal decoder filter** (always executed last) that:\n\n1. **Selects an upstream route** using `RouteConfiguration`\n2. **Picks a cluster** based on the route\n3. **Load balances** to select an upstream host\n4. **Obtains a connection** from the cluster's connection pool\n5. **Creates an `UpstreamRequest`** to forward the request\n6. **Handles retries, timeouts, hedging**, and other upstream concerns\n\n### Route Selection and Cluster Picking\n\n```cpp\nclass Filter : public Http::StreamDecoderFilter {\n  FilterHeadersStatus decodeHeaders(Http::RequestHeaderMap& headers, bool end_stream);\n  // ...\n\nprivate:\n  Router::RouteConstSharedPtr route_;          // Selected via RouteConfiguration\n  Upstream::ThreadLocalCluster* cluster_;      // From route\n  Router::RouteEntry::UpstreamProtocol protocol_;\n};\n```\n\nWhen `decodeHeaders()` is called:\n\n1. Query the `Router::Config` (built from `RouteConfiguration` proto) to find a route matching the request\n2. From the route, get the cluster name\n3. Query `ClusterManager` to fetch the `ThreadLocalCluster` for that name\n4. The cluster contains the load balancer, connection pool factory, etc.\n\n### Load Balancing Context\n\nThe Router filter implements `Upstream::LoadBalancerContext`:\n\n```cpp\nclass Filter : public Upstream::LoadBalancerContextBase {\n  absl::optional<uint64_t> computeHashKey() override;\n  const Router::MetadataMatchCriteria* metadataMatchCriteria() override;\n  // ...\n};\n```\n\nWhen obtaining a host from the cluster's load balancer:\n\n```cpp\nUpstream::HostConstSharedPtr host = cluster->loadBalancer().chooseHost(this);\n```\n\nThe load balancer calls back into the filter to:\n- Compute a hash (for consistent hashing)\n- Get metadata match criteria (for subset load balancing)\n- Access the stream info and connection context\n\n### Upstream Request Creation\n\nOnce a host is selected, the router creates an `UpstreamRequest`:\n\n```cpp\nclass UpstreamRequest : public Http::StreamDecoder {\n  UpstreamRequest(RouterFilterInterface& parent,\n                  std::unique_ptr<GenericConnPool>&& conn_pool, ...);\n  void acceptHeadersFromRouter(bool end_stream);\n  void acceptDataFromRouter(Buffer::Instance& data, bool end_stream);\n  // ...\n};\n```\n\n**Lifecycle**:\n\n1. Router calls `conn_pool->newStream()` to obtain a connection (or create one)\n2. `GenericConnPool::newStream()` calls `GenericConnectionPoolCallbacks::onGenericPoolReady()` with a `StreamEncoder` for the upstream connection\n3. Router creates an `UpstreamRequest`, passing the connection\n4. Router calls `acceptHeadersFromRouter()`, which sends headers upstream (possibly modified by upstream filters)\n5. Data and trailers flow from router \u2192 `UpstreamRequest` \u2192 upstream connection\n\n### Per-Try Timeout and Retries\n\nThe router manages request **timeouts** and **retries**:\n\n```cpp\nstruct TimeoutData {\n  std::chrono::milliseconds timeout_;      // Global timeout\n  std::chrono::milliseconds per_try_timeout_;\n};\n```\n\nThe router:\n- Reads timeout config from the route\n- Overrides with `x-envoy-expected-request-timeout-ms` if present\n- Sets per-try timeout on the upstream request\n- On timeout, retries if the route allows (and conditions are met)\n\n**Retry conditions**:\n- `x-envoy-retry-on` header (or route config) specifies retriable response codes\n- Upstream has not sent a response body yet\n- Retry budget not exhausted\n\n**Hedging** (per route config):\n- Send a second upstream request if the first takes too long\n- Return whichever response arrives first\n\n### Shadow Requests\n\nIf the route has a shadow policy:\n```cpp\nconst ShadowPolicy& shadow_policy = route->shadowPolicy();\nif (shouldShadow(shadow_policy, runtime, stable_random)) {\n  // Create a second upstream request to a shadow cluster\n  // But do NOT wait for its response\n}\n```\n\nShadow requests are **fire-and-forget**; the response is discarded.\n\n---\n\n## 4. Cluster Manager and Upstream Connectivity\n\n**Files**: `source/common/upstream/cluster_manager_impl.h`, `source/common/upstream/cluster_manager_impl.cc`\n\n### Role and Responsibilities\n\n`ClusterManager` provides the **upstream cluster abstraction**: managing clusters, their connection pools, health checks, and load balancing. It:\n\n1. **Owns cluster configurations** (name, endpoints, health checks, outlier detection)\n2. **Provides connection pools** to the router\n3. **Manages load balancers** (round-robin, least request, ring hash, etc.)\n4. **Monitors endpoint health** and updates load balancer membership\n5. **Runs outlier detection** to temporarily eject unhealthy hosts\n\n### Cluster and Host Concepts\n\n```cpp\nclass Cluster {\n  const std::string& name();\n  LoadBalancer& loadBalancer();\n  ConnectionPool::InstancePtr allocConnPool(...);\n  // ...\n};\n\nclass HostDescription {\n  const std::string& hostname();\n  const Address::InstanceConstSharedPtr address();\n  Stats::StatsScope& stats();\n  bool canCreateConnection(...);\n  // ...\n};\n```\n\n### Connection Pool Mechanics\n\nWhen the router calls `conn_pool->newStream()`:\n\n```cpp\nclass HttpConnPool : public ConnectionPool::Instance {\n  ConnectionPool::Cancellable* newStream(\n      const Http::RequestEncoder& request_encoder,\n      Http::ConnectionPool::Callbacks& callbacks);\n};\n```\n\n**Two scenarios**:\n\n**A) Reuse existing connection**:\n- The pool has an idle connection to the host\n- Immediately calls `callbacks->onPoolReady(encoder, ...)` with the connection's encoder\n- Router creates `UpstreamRequest` and sends headers\n\n**B) Create new connection**:\n- No idle connection available (and below max connections)\n- Create a new connection to the host\n- Connection calls `onPoolReady(encoder, ...)` when established\n- If connection fails, calls `onPoolFailure(...)`\n\n**Pooling strategy** depends on protocol:\n- **HTTP/1.1**: One request per connection (no multiplexing). Pool maintains N connections per host.\n- **HTTP/2**: Multiplexes up to N streams per connection. Pool maintains M connections with N streams each.\n- **HTTP/3**: Similar to H2 but over QUIC.\n\n### Load Balancing\n\nEach cluster has a `LoadBalancer`:\n\n```cpp\nclass LoadBalancer {\n  HostConstSharedPtr chooseHost(LoadBalancerContext* context) {\n    // Use context->computeHashKey(), metadataMatchCriteria(), etc.\n    // Return a host based on policy (round-robin, consistent hash, etc.)\n  }\n};\n```\n\n**Common policies**:\n- **Round-robin**: Cycle through hosts\n- **Least request**: Pick the host with fewest active requests\n- **Ring hash**: Consistent hash of request property (header, source IP, etc.)\n- **Maglev**: High-performance consistent hash\n- **Random**: Random selection\n\n### Health Checking and Outlier Detection\n\n**Health checks**:\n- Background check (TCP, HTTP, gRPC) to each host\n- Mark host as degraded/unhealthy if checks fail\n- Load balancer excludes unhealthy hosts (unless fallback is required)\n\n**Outlier detection** (optional):\n- Monitors upstream response codes, timeouts, connection errors\n- Temporarily ejects a host if it exceeds error thresholds\n- Periodically re-includes it to test recovery\n\nBoth feed into the load balancer's **host set** \u2014 the available hosts for selection.\n\n### Thread-Local Clusters\n\n`ClusterManager` provides a **thread-local view** of clusters:\n\n```cpp\nclass ThreadLocalCluster {\n  LoadBalancer& loadBalancer();\n  ConnectionPool::InstancePtr connPool(\n      ResourcePriority priority, absl::optional<Protocol> protocol);\n  ClusterInfoConstSharedPtr info();\n  // ...\n};\n```\n\nEach worker thread has a copy of the cluster state, avoiding locks during load balancing and connection pooling.\n\n---\n\n## Data Flow: Request Path (Downstream \u2192 # Envoy HTTP Connection Manager: Architecture Document

## Overview

The HTTP Connection Manager (HCM) is Envoy's primary subsystem for handling HTTP requests on the proxy. It bridges the gap between raw network I/O (bytes on a socket) and high-level request routing and filtering logic. This document explains how four core components work together to process an HTTP request from arrival on a downstream connection through to upstream dispatch and back.

The key insight is **separation of concerns across multiple layers**:
- **Network I/O**: `ConnectionManagerImpl` handles raw bytes
- **Protocol Parsing**: HTTP codecs (H1, H2, H3) parse bytes into logical messages
- **Per-Stream Processing**: `FilterManager` orchestrates the filter chain for each request
- **Request Forwarding**: The Router filter selects an upstream cluster and creates connections via `ClusterManager`
- **Upstream Connectivity**: `ClusterManager` manages connection pools and load balancing

---

## 1. ConnectionManagerImpl: The Core Network Filter

**Files**: `source/common/http/conn_manager_impl.h`, `source/common/http/conn_manager_impl.cc`

### Role and Responsibilities

`ConnectionManagerImpl` is a `Network::ReadFilter` that owns the entire lifecycle of an HTTP connection. It:

1. **Receives raw bytes** from the network layer via the `onData()` callback
2. **Creates and owns the codec** — lazily instantiated on first data or at connection time (H3)
3. **Manages `ActiveStream` objects** — one per HTTP request/response
4. **Handles connection-level concerns**:
   - Drain decisions (graceful shutdown)
   - Overload management (rejecting new streams when overloaded)
   - Connection statistics and access logging
   - Watermark-based flow control (propagated to all streams)
