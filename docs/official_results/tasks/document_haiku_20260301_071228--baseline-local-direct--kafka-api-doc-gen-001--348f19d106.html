# Task: Generate API Reference Documentation for Kafka KafkaConsumer

## Objective

Generate comprehensive API reference documentation for the Apache Kafka `KafkaConsumer` Java API. The documentation should cover the complete API surface with emphasis on behavioral semantics, offset management strategies, rebalancing mechanics, and error handling patterns.

## Scope

Your documentation should cover the **KafkaConsumer API** from the `org.apache.kafka.clients.consumer` package, including:

1. **Core Consumer Lifecycle**
   - Constructor variants and configuration
   - Subscription methods (dynamic and manual assignment)
   - Polling and data fetching semantics
   - Resource cleanup and closing

2. **Offset Management**
   - Synchronous and asynchronous commit strategies
   - Offset queries and position control
   - Seek operations and offset discovery
   - Committed offset semantics

3. **Consumer Group Mechanics**
   - ConsumerRebalanceListener interface and callbacks
   - Rebalance triggers and timing
   - Group membership and heartbeat behavior
   - Partition assignment vs subscription models

4. **Flow Control and Position Management**
   - Pause and resume functionality
   - Position queries and manipulation
   - Offset-to-timestamp lookups
   - Beginning and end offset discovery

5. **Error Handling**
   - Exception types and recovery strategies
   - CommitFailedException and group fencing
   - WakeupException and thread interruption
   - Timeout and authentication errors

## Requirements

### API Methods Documentation (40%)

Document all public methods of the `KafkaConsumer` class with:
- Method signatures including all overloads
- Parameter semantics and validation rules
- Return types and their meanings
- Exception types thrown and conditions

### Behavioral Notes (30%)

Explain critical behavioral semantics:
- **poll() blocking behavior**: when it returns immediately vs when it blocks, timeout handling, rebalance callback execution during poll
- **Offset commit semantics**: difference between sync/async commits, retry behavior, commit failure handling
- **Rebalance coordination**: when rebalances occur (only during poll), callback ordering (revoked then assigned), partition ownership guarantees
- **Thread safety**: which methods are thread-safe, wakeup() special case, event loop model
- **Group membership**: max.poll.interval.ms enforcement, proactive leave behavior, session timeout vs poll timeout
- **Manual vs dynamic assignment**: mutually exclusive nature, use cases for each model
- **Position vs committed offset**: the off-by-one relationship ("committed should be next offset to read")
- **Transactional semantics**: read_committed isolation level, LSO boundary, filtered messages

### Usage Examples (20%)

Provide concrete code examples demonstrating:
- **Basic subscription and polling loop**: subscribe to topics, poll for records, process messages
- **Manual offset commit**: disable auto-commit, explicit commitSync/commitAsync after processing
- **Rebalance listener**: implement ConsumerRebalanceListener, commit offsets on revoke, initialize positions on assign
- **Seek operations**: seekToBeginning, seekToEnd, seek to specific offset, timestamp-based seeking
- **Multi-threaded processing pattern**: single consumer thread with pause/resume coordination and worker pool

### Documentation Structure (10%)

Organize documentation with clear sections:
- Overview and threading model

`WakeupException`: If `wakeup()` was called before/during execution\n- `InterruptException`: If thread was interrupted before/during execution\n- `AuthenticationException`: If authentication fails\n- `AuthorizationException`: If lacking read access to topics or group ID\n- `TimeoutException`: If fetch metadata cannot be retrieved within timeout\n- `InvalidTopicException`: If subscribed topics are invalid\n- `FencedInstanceIdException`: If consumer is fenced by broker (instance ID changed)\n\n**Reset Policy Interaction:**\nThe `auto.offset.reset` configuration controls behavior when offset is Offset Concepts\n\n#### Position\n- The **offset of the next record to be fetched**\n- Automatically advanced by `poll()` as records are returned\n- Can be manually set via `seek()`\n- Returned by `position(TopicPartition)`\n\n**Key Relationship:** If `position()` is N, the next `poll()` will fetch starting from offset N.\n\n#### Committed Offset\n- The **offset of the next record to read** after restart\n- Stored durably by Kafka broker\n- Retrieved by `committed(Set<TopicPartition>)`\n- Updated via `commitSync()`, `commitAsync()`, or automatic commits\n\n**Commitment Rule:** When committing, the offset should be the **next offset to read**, not the last offset processed.\n\n```java\n// Correct: commit the offset of the next message to read\nconsumer.commitSync(Collections.singletonMap(\n    partition,\n    new OffsetAndMetadata(lastProcessedRecord.offset() + 1)\n));\n\n// Or use offset **after** the last message\n- For `read_committed`: End offset is the LSO (Last Stable Offset)\n\n### Automatic Offset Commits\n\n#### Configuration\n\n- `enable.auto.commit`: Set to `true` to `poll()` must be processed before next `poll()` call to avoid committed offset advancing ahead of processed records\n\n#### Rebalance Behavior\n\n- Auto-commits are triggered before rebalance starts\n- A final commit happens during `close()`\n\n### Manual Synchronous Commits\n\n#### Commit All Consumed Offsets\n\n```java\npublic void commitSync()\npublic void commitSync(Duration partitions. Kafka automatically assigns partitions to group members.\n\n**Key Properties:**\n- Only one consumer group member receives each partition\n- Partitions distribute across group members\n- Instances can dynamically join/leave\n- Broker manages group coordination and partition rebalancing\n\n### Group Membership and Lifecycle\n\n#### Initial Join\n\n- Consumer joins group on first `poll()` call after `subscribe()`\n- Triggers a rebalance to redistribute partitions\n- Assignment delayed until rebalance completes\n\n#### Session Management\n\n**session.timeout.ms:** Time before broker considers consumer dead (default: 45000ms)\n- If no heartbeat received for this duration, consumer removed from group\n- Triggers rebalance with other members\n\n**heartbeat.interval.ms:** Frequency of group heartbeats (default: 3000ms)\n- Recommended: `session.timeout.ms / 3`\n- Sent periodically; failure to send triggers timeout\n\n**max.poll.interval.ms:** Maximum time between polls (default: 300000ms)\n- If consumer doesn't call `poll()` within this interval, proactively exits group\n- Prevents \"livelock\" where consumer sends heartbeats but makes no progress\n- Exceeding this causes `CommitFailedException` on commit attempts\n\n#### Proactive Group Exit\n\nWhen consumer exceeds `max.poll.interval.ms`:\n1. Consumer proactively leaves the group\n2. `onPartitionsRevoked()` triggered on next `poll()` call\n3. Partitions reassigned to other group members\n4. Subsequent commits fail with `CommitFailedException`\n\nThis is a **safety mechanism** ensuring only active members commit offsets.\n\n### Rebalance Triggering Events\n\nRebalances occur when:\n- New consumer joins group (detected during heartbeat)\n- Consumer leaves group (gracefully or timeout)\n- Partition count changes for subscribed topics\n- Subscribed topic created/deleted\n- New topic matching subscribed pattern created\n- Consumer fails to send heartbeat\n- `enforceRebalance()` called\n\n### Rebalance Protocols\n\n#### Classic (Eager) Protocol\n- **Name:** `GroupProtocol.CLASSIC`\n- `onPartitionsRevoked()` called with **full assignment** (all partitions revoked)\n- All members must participate before assignment completes\n- More disruptive, all members must rebalance simultaneously\n\n#### Cooperative (Incremental) Protocol\n- **Name:** `GroupProtocol.CONSUMER` or cooperative assignments\n- `onPartitionsRevoked()` called only with **revoked partitions** (if non-empty)\n- Members can continue consuming stable partitions during rebalance\n- Less disruptive, only revoked partitions stop temporarily\n\n**Configuration:** `group.protocol` (legacy: `partition.assignment.strategy`)\n\n### Rebalance Listener Callbacks\n\n#### Callback Execution Order\n\nUnder normal conditions:\n1. **onPartitionsRevoked()** - Revoked partitions leaving this consumer\n2. *Rebalance at broker*\n3. **onPartitionsAssigned()** - New partitions assigned to this consumer\n\n#### onPartitionsRevoked Semantics\n\n```java\npublic void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n    // Save offsets, flush caches, release resources\n    // Called with full assignment under eager rebalance\n    // Called with subset under cooperative rebalance\n}\n```\n\n**Recommended Actions:**\n- Commit offsets to Kafka or external store\n- Flush any accumulated state\n- Close external connections tied to partitions\n- Clear partition-specific caches\n\n#### onPartitionsAssigned Semantics\n\n```java\npublic void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n    // Initialize positions, load external state, warm caches\n    // Called even if partitions is empty (signals rebalance completion)\n}\n```\n\n**Recommended Actions:**\n- Initialize position for new partitions\n- Load state from external offset store\n- Warm up partition-specific caches\n- Do NOT start consuming yet (happens automatically after callback returns)\n\n**Guaranteed:** All revocations complete before any member's `onPartitionsAssigned()` executes.\n\n#### onPartitionsLost Semantics\n\n```java\npublic void onPartitionsLost(Collection<TopicPartition> partitions) {\n    // Handle unexpected loss, don't commit offsets\n    // Partitions may already belong to other members\n}\n```\n\n**When Invoked:**\n- Session timeout occurred\n- Fatal error received from broker\n- Not a graceful revocation\n\n**Action:** Transactional Message Consumption\n\n### Isolation Levels\n\nThe `isolation.level` configuration controls how transactional messages are read.\n\n#### read_uncommitted (Default)\n\n- Reads all messages including uncommitted transactional messages\n- No client-side buffering\n- Most throughput\n- Messages may roll back if transaction aborts\n\n#### read_committed\n\n- Reads only committed transactional messages\n- Skips aborted transaction messages\n- LSO (Last Stable Offset) boundaries\n- Better consistency guarantees\n\n### Reading in read_committed Mode\n\n#### Message Filtering\n\n- Consumer automatically filters out aborted transaction markers\n- Transaction commit/abort markers not returned to application (but consume offsets)\n- Gap in offsets indicates filtered messages\n\n**Example:**\n```\nOffsets: 1, 2, 3, [4-6 aborted and filtered], 7, 8, 9\nReturned: 1, 2, 3, 7, 8, 9\n```\n\n#### Last Stable Offset (LSO)\n\nThe LSO is the offset of the first message in an open transaction. Consumer blocks at LSO until:\n- Transaction commits (LSO advances)\n- Transaction aborts (transaction filtered, LSO advances)\n- Timeout occurs\n\n**Effect on Offset Queries:**\n- `position()` does not advance past LSO\n- `seekToEnd()` seeks to LSO, not absolute end\n- `endOffsets()` returns LSO, not absolute end\n\n#### No Client-Side Buffering\n\nUnlike `read_uncommitted`:\n- Messages are not buffered waiting for transaction commit\n- Consumer simply stops at LSO if transaction still open\n- Next `poll()` returns empty until LSO advances\n\n### Consuming Exactly-Once with Transactions\n\n```java\n// Consumer only reads committed messages\nProperties props = new Properties();\nprops.setProperty(\"isolation.level\", \"read_committed\");\n\n// Producer writes transactions\n// Consumer reads committed messages atomically\n// Coordinate via ConsumerGroupMetadata for exactly-once processing\n```\n\n---\n\n## Metadata and Monitoring\n\n### Partition Information\n\n#### Get Partition Metadata\n\n```java\npublic List<PartitionInfo> partitionsFor(String topic)\npublic List<PartitionInfo> partitionsFor(String topic, Duration timeout)\n```\n\n**Behavior:**\n- Returns metadata about all partitions for a topic\n- Makes remote call if metadata not cached\n- Returns empty list if topic doesn't exist\n\n**Return Value:** List of `PartitionInfo` objects containing:\n- Partition ID\n- Leader broker ID\n- Replica broker IDs\n- In-sync replica (ISR) broker IDs\n\n#### List All Topics\n\n```java\npublic Map<String, List<PartitionInfo>> listTopics()\npublic Map<String, List<PartitionInfo>> listTopics(Duration timeout)\n```\n\n**Behavior:**\n- Returns metadata for all topics in cluster\n- May issue remote call\n- Blocks until metadata retrieved or timeout expires\n\n**Return:** Map from topic name to partition list\n\n### Metrics\n\n#### Get Consumer Metrics\n\n```java\npublic Map<MetricName, ? Offset Concepts\n\n#### Position\n- The **offset of the next record to be fetched**\n- Automatically advanced by `poll()` as records are returned\n- Can be manually set via `seek()`\n- Returned by `position(TopicPartition)`\n\n**Key Relationship:** If `position()` is N, the next `poll()` will fetch starting from offset N.\n\n#### Committed Offset\n- The **offset of the next record to read** after restart\n- Stored durably by Kafka broker\n- Retrieved by `committed(Set<TopicPartition>)`\n- Updated via `commitSync()`, `commitAsync()`, or automatic commits\n\n**Commitment Rule:** When committing, the offset should be the **next offset to read**, not the last offset processed.\n\n```java\n// Correct: commit the offset of the next message to starts\n- A final commit happens during `close()`\n\n### Manual Synchronous Commits\n\n#### Commit All Consumed Offsets\n\n```java\npublic void commitSync()\npublic void commitSync(Duration partitions. Kafka automatically assigns partitions to group members.\n\n**Key Properties:**\n- Only one consumer group member receives each partition\n- Partitions distribute across group members\n- Instances can dynamically join/leave\n- Broker manages group coordination and partition rebalancing\n\n### Group Membership and Lifecycle\n\n#### Initial Join\n\n- Consumer joins group on first `poll()` call after `subscribe()`\n- Triggers a rebalance to redistribute partitions\n- Assignment delayed until rebalance completes\n\n#### Session Management\n\n**session.timeout.ms:** Time before broker considers consumer dead (default: 45000ms)\n- If no heartbeat received for this duration, consumer removed from group\n- Triggers rebalance with other members\n\n**heartbeat.interval.ms:** Frequency of group heartbeats (default: 3000ms)\n- Recommended: `session.timeout.ms / 3`\n- Sent periodically; failure to send triggers timeout\n\n**max.poll.interval.ms:** Maximum time between polls (default: 300000ms)\n- If consumer doesn't call `poll()` within this interval, proactively exits group\n- Prevents \"livelock\" where consumer sends heartbeats but makes no progress\n- Exceeding this causes `CommitFailedException` on commit attempts\n\n#### Proactive Group Exit\n\nWhen consumer exceeds `max.poll.interval.ms`:\n1. Consumer proactively leaves the group\n2. `onPartitionsRevoked()` triggered on next `poll()` call\n3. Partitions reassigned to other group members\n4. Subsequent commits fail with `CommitFailedException`\n\nThis is a **safety mechanism** ensuring only active members commit offsets.\n\n### Rebalance Triggering Events\n\nRebalances occur when:\n- New consumer joins group (detected during heartbeat)\n- Consumer leaves group (gracefully or timeout)\n- Partition count changes for subscribed topics\n- Subscribed topic created/deleted\n- New topic matching subscribed pattern created\n- Consumer fails to send heartbeat\n- `enforceRebalance()` called\n\n### Rebalance Protocols\n\n#### Classic (Eager) Protocol\n- **Name:** `GroupProtocol.CLASSIC`\n- `onPartitionsRevoked()` called with **full assignment** (all partitions revoked)\n- All members must participate before assignment completes\n- More disruptive, all members must rebalance simultaneously\n\n#### Cooperative (Incremental) Protocol\n- **Name:** `GroupProtocol.CONSUMER` or cooperative assignments\n- `onPartitionsRevoked()` called only with **revoked partitions** (if non-empty)\n- Members can continue consuming stable partitions during rebalance\n- Less disruptive, only revoked partitions stop temporarily\n\n**Configuration:** `group.protocol` (legacy: `partition.assignment.strategy`)\n\n### Rebalance Listener Callbacks\n\n#### Callback Execution Order\n\nUnder normal conditions:\n1. **onPartitionsRevoked()** - Revoked partitions leaving this consumer\n2. *Rebalance at broker*\n3. **onPartitionsAssigned()** - New partitions assigned to this consumer\n\n#### onPartitionsRevoked Semantics\n\n```java\npublic void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n    // Save offsets, flush caches, release resources\n    // Called with full assignment under eager rebalance\n    // Called with subset under cooperative rebalance\n}\n```\n\n**Recommended Actions:**\n- Commit offsets to Kafka or external store\n- Flush any accumulated state\n- Close external connections tied to partitions\n- Clear partition-specific caches\n\n#### onPartitionsAssigned Semantics\n\n```java\npublic void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n    // Initialize positions, load external state, warm caches\n    // Called even if partitions is empty (signals rebalance completion)\n}\n```\n\n**Recommended Actions:**\n- Initialize position for new partitions\n- Load state from external offset store\n- Warm up partition-specific caches\n- Do NOT start consuming yet (happens automatically after callback returns)\n\n**Guaranteed:** All revocations complete before any member's Transactional Message Consumption\n\n### Isolation Levels\n\nThe `isolation.level` configuration controls how transactional messages are read.\n\n#### read_uncommitted (Default)\n\n- Reads all messages including uncommitted transactional messages\n- No client-side buffering\n- Most throughput\n- Messages may roll back if transaction aborts\n\n#### read_committed\n\n- Reads only committed
