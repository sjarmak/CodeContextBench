# Terraform's DAG Implementation

Based on the source code, Terraform's DAG (Directed Acyclic Graph) is a specialized graph structure that prevents circular dependencies. Here's how it works:

## Core Structure

The implementation centers on `AcyclicGraph`, which extends a basic `Graph` type and enforces the constraint that "cycles cannot exist." This is critical for infrastructure-as-code, where resources depend on each other in a specific order.

## Key Methods

**Graph Traversal:**
- `DepthFirstWalk()` and `ReverseDepthFirstWalk()` traverse the graph systematically
- `BreadthFirstWalk()` and `ReverseBreadthFirstWalk()` offer alternative traversal patterns
- These methods accept a callback function executed at each vertex

**Ancestor/Descendant Analysis:**
- `Ancestors()` identifies all upstream dependencies
- `Descendants()` finds all downstream dependents
- `FirstAncestorsWith()` and `FirstDescendantsWith()` support conditional filtering
- `MatchAncestor()` and `MatchDescendant()` test for specific conditions

**Graph Operations:**
- `Root()` locates the entry point (vertices with no dependencies)
- `TransitiveReduction()` optimizes by removing redundant edges
- `TopologicalOrder()` arranges vertices for proper execution sequencing
- `Validate()` ensures the graph has exactly one root and no cycles

**Cycle Detection:**
The `Cycles()` method identifies problematic circular dependencies, essential for preventing infinite loops during resource provisioning.

This structure enables Terraform to determine the correct order for applying infrastructure changes while validating that dependencies form a valid directed acyclic graph.

# Terraform Dependency Graph: Internal Architecture

## Overview

Terraform constructs a dependency graph from configuration files to execute operations like planning and state refreshing. This advanced topic isn't necessary for basic Terraform usage but provides insight into how the system orchestrates infrastructure changes.

## Node Types

The dependency graph contains three node varieties:

1. **Resource Nodes** - Each represents an individual resource. When `count` is configured, separate nodes exist for each instance. Configuration, diff, and state information attach to these nodes.

2. **Provider Configuration Nodes** - These represent provider initialization timing, such as when AWS credentials are supplied to a provider.

3. **Resource Meta-Nodes** - These group multiple resources without performing independent actions. They appear only when `count` exceeds one, improving graph visualization.

## Graph Construction Process

Building follows nine sequential stages:

1. Resource nodes are instantiated from configuration, with diff and state data attached

## Overview

Terraform's plan/apply pipeline is built around a dependency graph that models relationships between infrastructure resources, providers, variables, and outputs. The core orchestration type is `Context` (in `internal/terraform/context.go`), which exposes `Plan()` and `Apply()` entry points. Both operations follow the same fundamental pattern:

1. **Build a dependency graph** from configuration, state, and planned changes.
2. **Walk the graph in parallel**, executing each node once all its dependencies have completed.
3. **Delegate resource operations** to provider plugins via a gRPC-based interface.
4. **Update state** incrementally as each resource operation completes.

The architecture deliberately separates graph construction from evaluation: graph builders transform a set of configuration and state inputs into a directed acyclic graph (DAG), and the walker evaluates that graph concurrently, with each node's execution isolated through the `EvalContext` interface.
