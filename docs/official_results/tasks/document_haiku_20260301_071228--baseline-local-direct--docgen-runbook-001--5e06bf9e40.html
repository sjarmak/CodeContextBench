# Task: Generate Operational Runbook for Prometheus TSDB Compaction

**Repository:** prometheus/prometheus
**Output:** Write your runbook to `/workspace/documentation.md`

## Objective

Produce an operational runbook for Prometheus's TSDB (Time Series Database) compaction process. TSDB compaction merges overlapping time blocks, removes tombstoned series, and maintains query performance over time.

## Scope

Explore the TSDB compaction implementation in:
- `tsdb/compact.go` — core compaction logic
- `tsdb/head_read.go` and `tsdb/head.go` — head block behavior
- `tsdb/db.go` — database lifecycle and compaction triggering

## Required Sections

Your runbook at `/workspace/documentation.md` must include:

### 1. Overview
- What compaction does and why it is needed
- Compaction levels and block structure
- When compaction is triggered (time-based, size-based)

### 2. Monitoring Indicators
- Key Prometheus metrics to watch during compaction (e.g., `prometheus_tsdb_compactions_total`, `prometheus_tsdb_compaction_duration_seconds`)
- Alert thresholds and what they indicate
- How to distinguish healthy vs. stuck compaction

### 3. Failure Modes
- At least 4 failure modes with symptoms and causes:
  - Disk space exhaustion
  - Compaction stuck / not progressing
  - Block corruption
  - OOM during compaction
- For each: symptom, likely cause, and diagnostic steps

### 4. Recovery Procedures
- Step-by-step recovery for each failure mode
- When to restart Prometheus vs. manual block manipulation
- How to use `promtool tsdb` commands for inspection/repair

### 5. Code Reference
- Key functions in the codebase relevant to each failure mode

- Recovery steps must be numbered and actionable
- Each failure mode must link to a specific code location

# Prometheus TSDB Compaction Operational Runbook

## 1. Overview

### What is TSDB Compaction?

TSDB (Time Series Database) compaction is the process of merging overlapping time-series data blocks to optimize storage and maintain query performance. Over time, Prometheus writes data to multiple blocks, which can create inefficiencies. Compaction consolidates these blocks, removes tombstoned (deleted) series, and maintains a hierarchical block structure.

### Why is Compaction Needed?

1. **Storage Efficiency**: Merging blocks reduces index overhead and chunk fragmentation
2. **Query Performance**: Fewer blocks = faster block iteration and reduced I/O
3. **Space Reclamation**: Tombstoned series are physically removed during compaction
4. **Overlapping Resolution**: Vertical compaction merges overlapping time ranges

### Compaction Levels and Block Structure

Prometheus uses a **leveled compaction** strategy with exponential block ranges:

- **Level 1**: Blocks written directly from the head (in-memory buffer), time range ~2h (default `MinBlockDuration`)
- **Level 2**: Blocks created by merging multiple level 1 blocks, time range ~8h (2h × 4)
- **Level 3+**: Progressively larger time ranges as blocks are merged upward

Block naming and format:
- Each block is a directory named with a ULID (Universally Unique Lexicographically Sortable Identifier)
- Contains: `index`, `chunks/`, `meta.json`, `tombstones`
- Block time ranges are half-open intervals: `[MinTime, MaxTime)`

Compaction is triggered by two mechanisms:

#### 1. Time-Based (Head Compaction)
- Head block becomes "compactable" when its time range exceeds 1.5× the configured chunk range
- Default: If head has 3+ hours of data (1.5 × 2h), it's eligible to be compacted to disk
