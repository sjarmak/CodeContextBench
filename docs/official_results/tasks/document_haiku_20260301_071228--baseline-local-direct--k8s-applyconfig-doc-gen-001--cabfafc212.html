If invalid, **reverts** to pre-admission state and adds warning.\n\n**Limitation**: Validation only catches structural corruption (invalid JSON in fieldsV1). Webhooks can still add fields that bypass field ownership checks.\n\n#### Unbounded Manager Count\n\n**Problem**: If many different clients apply to same resource, managed fields grows without bound.\n\n**Mitigation** (CapManagersManager): Limit to DefaultMaxUpdateManagers=10. Older Update entries are merged (combined).\n\n**Limitation**: Compressed history loses field ownership details for old managers. After compression, can't extract precise previous configurations.\n\n**Impact**: If >10 controllers touch a resource, some history is lost.\n\n#### Conflict on Apply with Small Patch\n\n**Problem**: User applies a partial patch that conflicts with another controller's full specification.\n\n**Example**:\n```yaml\n# Controller A owns full Pod spec\n# User tries to apply just image: \"v2\"\n# But controller A had image: \"v1\"\n# Conflict on image!\n```\n\n**Solution**: Force-apply, but only if intentional.\n\n**Limitation**: Force-apply is all-or-nothing. Can't force specific fields.\n\n#### Schema Mismatch with CRDs\n\n**Problem**: CRD schema changes (list field becomes map), but old managed fields still reference old structure.\n\n**Mitigation**: If schema can't be found during merge, falls back to scalar merge (no recursion).\n\n**Limitation**: Older managed fields entries for pre-schema-change structure become stale.\n\n#### Lost Updates via GuaranteedUpdate Retry\n\n**Problem**: HTTP request timeout during GuaranteedUpdate retry loop.\n\n**Mitigation** (patch.go:734-736): If timeout error received, server can't guarantee whether object was persisted.\n\n**Behavior**: Returns 504 Gateway Timeout. Client must retry with awareness that object may have been created.\n\n**Limitation**: No way to know if partial application succeeded. Requires idempotent apply semantics.\n\n### 8.2 Limits\n\n| Limit | Value | Location | Note |\n|-------|-------|----------|------|\n| Max JSON Patch operations | 10,000 | patch.go:64 | Per RFC 6902 |\n| Max Update managers | 10 | managedfields/internal/fieldmanager.go:35 | Older entries merged |\n| Max request body size | Configurable (default 3MB) | handlers/patch.go:110 | Via apiserver flag |\n| Max field path depth | Unbounded | structured-merge-diff | Theoretical limit depends on memory |\n| Managed fields JSON size | Unbounded | Storage | Large numbers of managers \u2192 larger objects |\n\n### 8.3 Design Tradeoffs\n\n#### Tradeoff 1: Field-Level vs. Resource-Level Ownership\n\n**Choice**: Kubernetes uses field-level tracking.\n\n**Pro**: Multiple controllers can co-manage same resource at different fields. HPA sets replicas, user manages selector.\n\n**Con**: Adds complexity, requires schema understanding, merge logic more intricate.\n\n**Alternative**: Resource-level (last-write-wins). Simpler but forces exclusive ownership.\n\n#### Tradeoff 2: Three-Way Merge vs. Two-Way\n\n**Choice**: Kubernetes uses two-way merge (current state + patch), reconstructs previous state from managed fields.\n\n**Pro**: Doesn't require client to send three-way state, reduces API complexity.\n\n**Con**: Reconstructed \"previous\" state may be incomplete (other managers may have modified fields).\n\n**Example**:\n```\nLive state:  replicas=5, owned by HPA\nPatch:       replicas=3, sent by user\nReconstructed \"previous\" from managed fields: HPA owns replicas (assumed value is 5)\nMerge result: Conflict (HPA disagrees with user's 3)\n```\n\n#### Tradeoff 3: Force-Apply Atomicity\n\n**Choice**: Force-apply claims all fields in patch, not selective.\n\n**Pro**: Simple to implement, clear semantics.\n\n**Con**: Can't force-apply specific fields, must accept all-or-nothing.\n\n**Workaround**: Extract, modify specific fields, reapply (avoids conflicts).\n\n#### Tradeoff 4: Manager String Format\n\n**Choice**: Opaque JSON string containing { manager, operation, apiVersion }.\n\n**Pro**: Extensible, can add fields for future needs.\n\n**Con**: Not human-readable, requires JSON parsing to inspect.\n\n**Alternative**: Structured tuple (manager, op, version).\n\n---\n\n## Part 9: Operational Implications\n\n### 9.1 Performance Impact\n\n#### Apply Patch Processing\n\n**Cost**: Structured-merge-diff three-way merge is O(n) in live object size and patch size.\n\n**Optimization 1** (ProbabilisticSkipNonAppliedManager): For non-Apply operations, skip merge if fields are likely untouched (configurable probability, default 100% skip).\n\n**Optimization 2** (CapManagersManager): Merge old Update entries to keep manager count low.\n\n**Optimization 3** (patch.go:713-726): If object too large, retry after removing managedFields. # Kubernetes Server-Side Apply and ApplyConfigurations: A Deep Systems Guide

## Executive Summary

Server-Side Apply (SSA) is a declarative configuration management system in Kubernetes that tracks field ownership through **managed fields**, enabling multiple controllers to safely co-manage resources without conflicts. The subsystem comprises three major parts: (1) **applyconfigurations**, auto-generated builder types that provide a fluent API for constructing apply patches; (2) **server-side field management**, which tracks field ownership and detects conflicts; and (3) **structured-merge-diff**, an external library implementing the merge algorithm that respects field ownership boundaries.

This guide explains how these components work together, the control flow from client through server, conflict resolution semantics, and operational considerations.

## Part 1: Subsystem Purpose and Boundaries

### Purpose

The Server-Side Apply subsystem solves a fundamental problem in declarative configuration management: **how do multiple parties (controllers, operators, users) safely modify the same resource without overwriting each other's changes?**

- **Server tracks** who owns what through managed fields entries
