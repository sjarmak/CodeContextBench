     3→This document explains the internal architecture of `client-go` for contributors. It describes the
     4→major components, how they interact, and the key design decisions that shape the library.

     6→## Client Configuration
     7→
     8→There is an architectural separation between loading client configuration and using it. The
     9→`rest.Config` object is the in-memory representation of this configuration. The
    10→`tools/clientcmd` package is the standard factory for producing it. `clientcmd` handles the
    11→complex logic of parsing `kubeconfig` files, merging contexts, and handling external
    12→authentication providers (e.g., OIDC).
    13→
    14→## REST Client
    15→
    16→The `rest.Client` is the foundational HTTP client that underpins all other clients. It separates
    17→the low-level concerns of HTTP transport, serialization, and error handling from higher-level,
    18→Kubernetes-specific object logic.
    19→
    20→The `rest.Config` object is used to build the underlying HTTP transport, which is typically a
    21→chain of `http.RoundTripper` objects. Each element in the chain is responsible for a specific
    22→task, such as adding an `Authorization` header. This is the mechanism by which all authentication
    23→is injected into requests.
    24→
    25→The client uses a builder pattern for requests (e.g., `.Verb()`, `.Resource()`), deferring
    26→response processing until a method like `.Into(&pod)` is called. This separation is key to
    27→supporting different client models from a common base.

controllers to process objects at their own pace without blocking the watch stream\n\n**Delta Types:**\n- `Added`, `Updated`, `Deleted`: From watch events\n- `Replaced`: From Replace() calls (initial LIST or resync)\n- `Sync`: Periodic resync events\n\n#### 5c: Indexer (`tools/cache/store.go`, `tools/cache/thread_safe_store.go`)\n\nA thread-safe, in-memory key-value store with optional secondary indices.\n\n**Key Responsibility:**\n- Store the authoritative local copy of object state\n- Support efficient lookups (by key, by index, by list)\n- Provide thread-safe read access for controllers\n\n**Indexing:**\n- Default key function: `namespace/name` (MetaNamespaceKeyFunc)\n- Custom indices: E.g., index by owner reference, by node affinity, by label\n- Index functions: `namespace`, `node`, `pvc`, `service`, `endpoint`, etc.\n\n#### 5d: SharedInformer (`tools/cache/shared_informer.go`)\n\nThe public API for controllers to access the cache and receive notifications of changes.\n\n**Key Responsibility:**\n- Vend read-only Lister and Indexer interfaces to controller logic\n- Distribute cache synchronization events to multiple event handlers\n- Provide resync notifications at configurable intervals\n- Report cache sync status via `HasSynced()`\n\n**Event Handler Contract:**\n```go\ntype ResourceEventHandler interface {\n    OnAdd(obj interface{}, isInInitialList bool)\n    OnUpdate(oldObj, newObj interface{})\n    OnDelete(obj interface{})\n}\n```\n\n**Resync Semantics:**\n- After initial LIST completes, `isInInitialList=true` is passed to OnAdd()\n- Periodic resync events trigger OnAdd() with `isInInitialList=false`\n- Controllers may ignore resync events or use them for periodic cleanup\n\n#### 5e: SharedInformerFactory (`informers/factory.go`)\n\nA factory that creates and manages multiple SharedInformers across different API groups and versions.\n\n**Key Responsibility:**\n- Coordinate a single Reflector per resource type\n- Share DeltaFIFO and Indexer instances among logically-independent informers\n- Manage the lifecycle of all informers (Start, Stop, WaitForCacheSync)\n\n**Efficiency Gains:**\n- Single LIST/WATCH stream per resource type (not per informer)\n- Shared cache reduces memory footprint\n- Synchronized resync cycles\n\n### Layer 6: Work Queue and Reconciliation (`util/workqueue`)\n\nDecouples event detection (informer) from object processing (controller).\n\n**Key Responsibility:**\n- Create a critical boundary between asynchronous events and synchronous reconciliation\n- Support rate limiting and exponential backoff per item\n- Enable controller retries without blocking the informer's watch stream\n\n**Queue Types:**\n- `Queue`: Simple FIFO\n- `DelayingQueue`: Delays items before re-queueing\n- `RateLimitingQueue`: Applies pluggable rate limiting strategies\n\n**Rate Limiting Strategies:**\n- `BucketRateLimiter`: Per-item token bucket\n- `ItemExponentialFailureRateLimiter`: Exponential backoff based on failure count\n- `ItemFastSlowRateLimiter`: Transition from fast to slow backoff\n- `MaxOfRateLimiter`: Apply maximum of multiple strategies\n\n**Controller Pattern:**\n```go\nfor {\n    key, _ := queue.Get()\n    // Controller business logic (reconcile)\n    queue.Done(key)\n    if err != nil {\n        queue.AddRateLimited(key) // Retry with backoff\n    }\n}\n```\n\n### Layer 7: Leader Election (`tools/leaderelection`)\n\nFor high-availability deployments, ensures only one controller replica is active at a time.\n\n**Key Responsibility:**\n- Compete for a distributed lock (stored as a Lease object)\n- Transition between leader and follower roles\n- Provide hooks (OnStartedLeading, OnStoppedLeading) for controller lifecycle\n\n**Lock Mechanisms:**\n- `Lease` (preferred): Lightweight distributed lock using the Lease API\n- `ConfigMap`, `Endpoints`: Legacy mechanisms\n- Custom `ResourceLock` interfaces for extension\n\n**Behavior:**\n1. All replicas attempt to acquire a lock by creating/updating a Lease\n2. Winner transitions to \"leader\" and calls `OnStartedLeading()`\n3. Leader renews the lease periodically\n4. If renewal fails (leader crash), followers compete for the lock\n5. Loser transitions to \"follower\" and calls `OnStoppedLeading()`\n\n### Layer 8: Server-Side Apply (`applyconfigurations/`)\n\nType-safe builder API for declarative object mutations. Enables multiple controllers to co-manage the same object safely.\n\n**Location:** `applyconfigurations/` (code-generated)\n\n**Key Responsibility:**\n- Construct PATCH requests with `managedFields` metadata\n- Allow controllers to declare \"this field is mine\" without conflicts\n\n**Apply Configuration Structure:**\n```go\ndeploymentApplyConfig := appsv1ac.Deployment(\"my-app\", \"default\").\n    WithSpec(appsv1ac.DeploymentSpec().\n        WithReplicas(2).\n        WithTemplate(...))\n\nresult, err := clientset.AppsV1().Deployments(\"default\").Apply(\n    ctx, deploymentApplyConfig, metav1.ApplyOptions{FieldManager: \"my-controller\"})\n```\n\n**Benefits:**\n- Declarative: Specify desired state, not mutations\n- Conflict-aware: Multiple controllers can manage different fields\n- Idempotent: Reapplying same config has no effect\n\n---\n\n## Part 3: End-to-End Control and Request Flows\n\n### Flow 1: Typed Client Request\n\nUser code \u2192 Typed Clientset \u2192 REST Client \u2192 HTTP Transport \u2192 API Server \u2192 Response Deserialization\n\n**Concrete Example: List Deployments**\n\n```go\n// User code\ndeployments, err := clientset.AppsV1().Deployments(\"default\").List(ctx, metav1.ListOptions{})\n```\n\n**Steps:**\n\n1. **Type-Safe Wrapper** (`kubernetes/typed/apps/v1/deployment.go`):\n   - Wraps a `rest.RESTClient` specific to AppsV1\n   - Routes to resource-specific implementation\n\n2. **REST Request Construction** (`rest/request.go`):\n   ```go\n   r := restClient.Get().\n       Namespace(\"default\").\n       Resource(\"deployments\")\n   ```\n   - Sets verb: `GET`\n   - Sets path: `/apis/apps/v1/namespaces/default/deployments`\n   - Applies query params: `limit`, `continue`, `labelSelector`, etc.\n\n3. **Content Negotiation**:\n   - Sets Accept header: `application/json` (or `application/vnd.kubernetes.protobuf`)\n   - Server responds with same format\n\n4. **HTTP Transport** (`rest/transport.go`):\n   - Constructs a chain of `http.RoundTripper` middleware:\n     - TLS setup (client cert auth)\n     - Bearer token injection (auth header)\n     - Rate limiting\n     - Retry with backoff\n   - Executes HTTP request\n\n5. **Response Processing**:\n   - HTTP status codes: 2xx \u2192 success, 3xx \u2192 redirect, 4xx/5xx \u2192 error\n   - Error responses: Deserialized to `apierrors.StatusError` with reason (NotFound, Conflict, etc.)\n   - Success responses: Decoded into `*unstructured.Unstructured` then converted to typed `DeploymentList`\n\n6. **Return to User**:\n   ```\n   deployments := {\n       TypeMeta: v1.TypeMeta{APIVersion: \"apps/v1\", Kind: \"DeploymentList\"},\n       ListMeta: v1.ListMeta{ResourceVersion: \"12345\"},\n       Items: [...]\n   }\n   ```\n\n**Error Handling:**\n- If 404: `apierrors.IsNotFound(err)` returns true\n- If 409 (conflict): `apierrors.IsConflict(err)` returns true\n- Other errors: Wrapped in `RequestConstructionError` or transport errors\n\n---\n\n### Flow 2: Informer-Driven Controller Loop\n\n```\nReflector LIST/WATCH\n    \u2193\nDeltaFIFO (event buffer)\n    \u2193\nIndexer (cache)\n    \u2193\nEvent Handlers (notify controllers)\n    \u2193\nWork Queue (buffer pending work)\n    \u2193\nController Reconcile Loop\n    \u2193\n[Lister access back to Indexer]\n```\n\n**Concrete Example: Pod Informer in a Controller**\n\n**Initialization:**\n```go\nconfig, _ := clientcmd.BuildConfigFromFlags(\"\", kubeconfig)\nclientset, _ := kubernetes.NewForConfig(config)\nfactory := informers.NewSharedInformerFactory(clientset, 10*time.Minute)\npodInformer := factory.Core().V1().Pods().Informer()\n\nworkqueue.NewRateLimitingQueue(workqueue.DefaultControllerRateLimiter())\n\npodInformer.AddEventHandler(cache.ResourceEventHandlerFuncs{\n    AddFunc: func(obj interface{}) {\n        key, _ := cache.MetaNamespaceKeyFunc(obj)\n        queue.Add(key) // \"default/my-pod\"\n    },\n    UpdateFunc: func(oldObj, newObj interface{}) {\n        key, _ := cache.MetaNamespaceKeyFunc(newObj)\n        queue.Add(key)\n    },\n    DeleteFunc: func(obj interface{}) {\n        key, _ := cache.DeletionHandlingMetaNamespaceKeyFunc(obj)\n        queue.Add(key)\n    },\n})\n\nfactory.Start(stopCh)\ncache.WaitForCacheSync(stopCh, podInformer.HasSynced)\n```\n\n**Runtime (Informer Side):**\n\n1. **Reflector Initialization**:\n   - Calls `podClient.List(ctx, metav1.ListOptions{ResourceVersion: \"\", Limit: 500})`\n   - Receives initial resourceVersion (e.g., \"12345\")\n   - Starts WATCH with `ResourceVersion: \"12345\"`\n\n2. **Initial LIST Response**:\n   - Receives PodList with 50 items, `Continue` token set\n   - Pushes all 50 to DeltaFIFO with delta type `Replaced`\n   - Fetches next page via `List(..., Continue: token)`\n\n3. **Watch Stream Begins**:\n   - Watches for events: Pod created, modified, deleted\n   - Each event pushed to DeltaFIFO as `Added`, `Updated`, `Deleted`\n\n4. **DeltaFIFO Processing**:\n   - Shared informer controller loop pops deltas\n   - For each delta, updates Indexer\n   - Triggers event handler callbacks\n\n5. **Event Handler Notification**:\n   - `OnAdd()`: Called for each pod in initial LIST + new pods\n   - `OnUpdate()`: Called for modified pods\n   - `OnDelete()`: Called for deleted pods\n   - Each call adds the pod's key to the work queue\n\n6. **Resync Cycle** (every 10 minutes):\n   - Reflector calls `Resync()` on DeltaFIFO\n   - Pushes all cached objects back as `Sync` deltas\n   - Event handlers called again (giving controller chance to reconcile even if watch was missed)\n\n**Runtime (Controller Side):**\n\n1. **Work Queue Processing Loop**:\n   ```go\n   for {\n       key, _ := queue.Get()\n       defer queue.Done(key)\n\n       namespace, name, _ := cache.SplitMetaNamespaceKey(key)\n       pod, _ := podLister.Pods(namespace).Get(name)\n\n       // Controller business logic\n       if err := reconcile(pod); err != nil {\n           queue.AddRateLimited(key) // Retry with backoff\n           continue\n       }\n\n       queue.Forget(key) // Success, reset backoff counter\n   }\n   ```\n\n2. **Lister Access**:\n   - Controller calls `podLister.Pods(\"default\").Get(\"my-pod\")`\n   - Lister queries the Indexer (cached, no API call)\n   - Returns object or \"NotFound\"\n\n3. **Reconciliation**:\n   - Business logic (may make updates via typed clientset)\n   - If reconciliation fails: Work queue re-adds with exponential backoff\n   - If successful: Item forgotten (backoff counter reset)\n\n**Key Invariants:**\n- Cache is **eventually consistent** with API server (may lag by watch latency)\n- Events are **ordered per object** but not across objects\n- Rapid changes may collapse into a single cache update\n- Resync ensures \"stuck\" objects are revisited even if watch is broken\n\n---\n\n### Flow 3: Dynamic Client with Discovery\n\nUsed when the client doesn't have type definitions for a resource (e.g., a CRD).\n\n```\nDynamic Client Request\n    \u2193\nDiscovery (if needed, discover resource schema)\n    \u2193\nREST Client (like typed client, but with unstructured.Unstructured)\n    \u2193\n[No code generation required]\n```\n\n**Example: Create a Custom Resource**\n\n```go\ndynamicClient := dynamic.NewForConfig(config)\ngvr := schema.GroupVersionResource{\n    Group:    \"mycompany.com\",\n \"Widget\",\n        \"metadata\": map[string]interface{}{\n            \"name\":      \"my-widget\",\n            \"namespace\": \"default\",\n        },\n        \"spec\": map[string]interface{}{\n            \"color\": \"blue\",\n        },\n    },\n}\n\nresult, err := dynamicClient.Resource(gvr).Namespace(\"default\").Create(ctx, obj, metav1.CreateOptions{})\n```\n\n**Steps:**\n\n1. **Request Construction**:\n   - No type definitions needed; works with maps\n   - REST client constructs request path: `/apis/mycompany.com/v1/namespaces/default/widgets`\n   - Serializes object to JSON\n\n2. **API Server Processing**:\n   - Server deserializes and validates against OpenAPI schema\n   - Returns object with `metadata.uid`, `metadata.resourceVersion` populated\n\n3. **Response Deserialization**:\n   - Result remains as `unstructured.Unstructured`\n   - Caller accesses fields via `obj.UnstructuredContent()[\"spec\"].(map[string]interface{})`\n\n---\n\n## Part 4: Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### Network Failures\n\n**Scenario:** API server becomes unreachable during WATCH.\n\n**Reflector Behavior:**\n- watch connection drops with an error\n- Exponential backoff (up to 1s) before retry\n- On retry, if resourceVersion is still valid: Resume from that resourceVersion\n- If resourceVersion is too old: Re-list (reset to version 0)\n\n**Controller Impact:**\n- Event stream pauses; no new changes reach work queue\n- Existing work queue items continue processing\n- On reconnection: Reflector pushes accumulated changes (via resync cycle if needed)\n- Controller eventually catches up (after backoff and re-list)\n\n**Recovery Time:** Typically < 30 seconds (default backoff + resync timeout)\n\n---\n\n#### API Server Unavailability\n\n**Scenario:** All replicas of API server are down (brief outage or maintenance).\n\n**Reflector Behavior:**\n- LIST request fails (all replicas down)\n- Exponential backoff with jitter (configured via envBackoffDuration, default 600s max)\n- When API server recovers: resumes from last known resourceVersion\n\n**Controller Impact:**\n- Work queue starves (no new events can be fetched)\n- Informer reports `HasSynced() == false` until first successful LIST\n- Controllers waiting on `WaitForCacheSync()` block indefinitely\n\n**Mitigation:**\n- Set timeout on `WaitForCacheSync()`: `WaitForCacheSync(ctx, informer.HasSynced)` where ctx has deadline\n- Multiple API server replicas with load balancer\n- Service account with sufficient RBAC to list resources (common misconfiguration)\n\n---\n\n#### Resource Version Expiration (\"too old\")\n\n**Scenario:** Watch stream lags; API server evicts the resource version.\n\n**Error Message:** `\"the object has been deleted and no further information is available\"`\n\n**Reflector Behavior:**\n- watch returns error with code `410 Gone`\n- Reflector detects this as `isLastSyncResourceVersionUnavailable = true`\n- Next reconciliation: Performs full re-list with `ResourceVersion: \"\"` (strong consistency)\n\n**Prevention:**\n- Reduce resync period (default 10 minutes; smaller = more frequent re-list)\n- Monitor API server `etcd` compaction and retention settings\n- Watch bookmarks help (reflect updates resourceVersion without re-list)\n\n---\n\n#### Cache Desynchronization\n\n**Scenario:** Informer cache falls behind API server state due to bugs or race conditions.\n\n**Root Causes:**\n1. Lost watch events (network hiccup during compression, etc.)\n2. Reflector crashes and restarts (clean state loss)\n3. DeltaFIFO overflow (should never happen; indicates architectural issue)\n4. Bug in event handler (e.g., handler panics)\n\n**Detection:**\n- `reflector_data_consistency_detector.go`: Periodic LIST to validate cache against API server\n- Enabled via feature gate `ObjectCountTracker`\n- Logs warnings if discrepancies found\n\n**Recovery:**\n- Resync cycle re-lists all objects\n- Controllers re-process all objects (idempotent reconciliation should tolerate this)\n\n---\n\n#### Informer Handler Panics\n\n**Scenario:** Event handler code panics (e.g., null pointer dereference).\n\n**Behavior:**\n- `shared_informer.go` has `defer handleCrash()` around handler invocation\n- Panic is logged and recovered (does not crash informer)\n- Event is skipped; next event is processed\n- **Controller misses the update** \u2192 object state diverges\n\n**Prevention:**\n- Robust event handler code (validate inputs, handle nil)\n- Wrap event handler in try-catch wrapper (if using a framework)\n- Thorough testing of event handler edge cases\n\n---\n\n#### Work Queue Starvation\n\n**Scenario:** Controller reconciliation is slower than event arrival rate.\n\n**Behavior:**\n- Work queue grows unbounded (all in-memory)\n- Memory usage increases linearly\n- CPU usage on controller increases (processing items)\n- No per-item timeout; items never expire\n\n**Prevention:**\n- Monitor work queue depth and controller processing latency\n- Scale controller replicas if using leader election\n- Optimize reconciliation logic (avoid nested loops, expensive API calls)\n- Use exponential backoff to reduce retry rate for hot items\n\n**Limits:**\n- Work queue is unbounded in memory (no max size)\n- Typical deployments: 1000-10000 items before memory becomes an issue\n- Large clusters (10k+ objects): Consider sharded controllers or leader-based routing\n\n---\n\n#### High Cardinality Labels\n\n**Scenario:** Custom index function returns many keys per object.\n\n**Example:** Indexing pods by all container image tags (a pod may have 5+ containers, each with unique tags).\n\n**Impact:**\n- Indexer memory usage multiplies\n- Index query latency degrades (linear scan of matching keys)\n- Cache consistency checker may produce false positives\n\n**Prevention:**\n- Index only low-cardinality fields\n- Avoid indices with cross-product keys\n\n---\n\n### 4.2 Design Limits\n\n| Limit | Value | Notes |\n|-------|-------|-------|\n| Watch stream timeout | 5min - 10min (random) | Randomized to spread reconnections |\n| Resync period | Configurable (default 10min) | Smaller = more LIST requests, larger = slower recovery |\n| Max in-flight requests (QPS \u00d7 Burst) | QPS \u00d7 Burst (default 5 \u00d7 10) | Configured per `rest.Config` |\n| Work queue max size | Unbounded (memory limit) | Typical safe limit: 10k items |\n| Informer handler timeout | None (unbounded) | Slow handlers block the informer |\n| Cache consistency window | Resync period | After resync: cache is consistent |\n| Lister lookups per object | O(1) by key, O(n) by index | Index efficiency depends on cardinality |\n\n---\n\n### 4.3 Design Tradeoffs\n\n#### Tradeoff 1: Consistency vs. Availability\n\n**Choice:** Favor availability and eventual consistency over strong consistency.\n\n**Rationale:**\n- Strong consistency requires synchronous client-server communication on every operation\n- Kubernetes at scale (10k+ nodes) cannot afford this cost\n- Eventual consistency with resync cycles and re-listing provides sufficient consistency for controllers\n\n**Implication:**\n- Controllers must be idempotent (can tolerate reprocessing the same object)\n- Controllers must tolerate missing intermediate states (e.g., a pod created and deleted before the controller sees it)\n\n---\n\n#### Tradeoff 2: Polling vs. Watching\n\n**Choice:** Favor WATCH over polling, with fallback to re-list on errors.\n\n**Rationale:**\n- WATCH is more efficient (server pushes changes, not client pulls)\n- Resync period ensures consistency without continuous polling\n- Hybrid approach recovers from watch failures gracefully\n\n**Implication:**\n- Controllers see changes with latency of watch network RTT (typically <100ms)\n- Controller re-processing not guaranteed to be in order across multiple objects\n- Controllers must tolerate watching the same state multiple times (resync)\n\n---\n\n#### Tradeoff 3: Code Generation vs. Dynamic\n\n**Choice:** Generate type-safe clients for built-in APIs; provide dynamic client for extensibility.\n\n**Rationale:**\n- Type safety catches errors at compile time (build-time safety)\n- Dynamic client enables CRDs without code generation (runtime flexibility)\n- Hybrid approach suits both operator developers and CRD authors\n\n**Implication:**\n- Custom resources require either code generation or dynamic client usage\n- Informers are type-safe for built-in types but less ergonomic for CRDs\n- Operators targeting multiple versions must either code-gen for each or use dynamic informers\n\n---\n\n#### Tradeoff 4: Caching Overhead\n\n**Choice:** Maintain a full local cache (memory overhead) to enable efficient reads.\n\n**Rationale:**\n- Controllers read from local cache frequently (no API latency)\n- API server is not bottleneck (no per-read request)\n- Memory cost is acceptable for typical cluster state sizes (10k-100k objects)\n\n**Implication:**\n- Every controller replica maintains a full copy of relevant objects in memory\n- Scaling to 10k controller replicas: each has a copy of cached state (memory \u00d7 10k = significant cost)\n- Large objects (e.g., 1MB ConfigMaps) amplify the cost\n\n**Optimization:**\n- Metadata-only informers (using `PartialObjectMetadata`) reduce memory\n- Shared informer factory reduces per-informer overhead\n- Custom indices enable efficient querying without full list iteration\n\n---\n\n#### Tradeoff 5: Event Ordering\n\n**Choice:** Guarantee ordering per object, not across objects.\n\n**Rationale:**\n- Per-object ordering is sufficient for idempotent reconciliation\n- Cross-object ordering would require distributed coordination (expensive)\n- Allows parallelism: multiple controllers can process different objects simultaneously\n\n**Implication:**\n- Controller cannot rely on seeing Pod-A update before Pod-B update (even if A updated first)\n- Controllers must use `resourceVersion` or timestamps to detect ordering issues if needed\n- Useful for detecting lost events (e.g., if Pod A was updated but cache shows old version, a watch event was lost)\n\n---\n\n## Part 5: Extension Points and Customization Hooks\n\n### 5.1 REST Transport Customization\n\n**Extension Point:** `rest.WrapperFunc` and `http.RoundTripper` chain.\n\n**Use Cases:**\n- Inject custom auth headers (e.g., API key passed through a secret service)\n- Add request/response logging\n- Metrics collection (latency, error rates)\n- Request tracing (OpenTelemetry integration)\n\n**Example:**\n\n```go\n// Custom transport middleware\nfunc MyAuthMiddleware(rt http.RoundTripper) http.RoundTripper {\n    return &myCustomRT{rt: rt}\n}\n\ntype myCustomRT struct{ rt http.RoundTripper }\nfunc (m *myCustomRT) RoundTrip(req *http.Request) (*http.Response, error) {\n    // Fetch auth token from external service\n    token, _ := fetchTokenFromService()\n    req.Header.Set(\"Authorization\", \"Bearer \" + token)\n    return m.rt.RoundTrip(req)\n}\n\n// Apply to rest.Config\nconfig, _ := clientcmd.BuildConfigFromFlags(\"\", kubeconfig)\nconfig.WrapTransport = MyAuthMiddleware\n```\n\n**Risks:**\n- Broken middleware can silently drop requests (ensure error handling)\n- Middleware overhead applies to every request registration.\n\n**Use Cases:**\n- Support custom wire formats (e.g., MessagePack instead of JSON)\n- Compression (gzip encoding in Accept header)\n- Custom error codes\n\n**Mechanism:**\n```go\nconfig.NegotiatedSerializer = myCustomSerializer\n```\n\n**Risks:**\n- Must implement full codec interface (both encode and decode)\n- Incompatibility with API server serialization breaks requests\n\n---\n\n### 5.3 Reflector Customization\n\n**Extension Point:** `ListerWatcher`, `TransformFunc`, `WatchErrorHandler`.\n\n**Use Cases:**\n- Filter watch events (transform function to strip sensitive data)\n- Custom error handling for watch failures\n- Alternative list/watch sources (e.g., cache server, proxy)\n\n**Example: Transform Function**\n\n```go\n// Strip sensitive fields before caching\nfunc stripSecrets(obj interface{}) (interface{}, error) {\n    secret := obj.(*v1.Secret)\n    // Remove Data field to avoid memory bloat\n    secret.Data = nil\n    return secret, nil\n}\n\nreflector := cache.NewReflector(\n    listerWatcher,\n    &v1.Secret{},\n    deltaFIFO,\n    reflector.NewReflectorOptions().WithTransformer(stripSecrets),\n)\n```\n\n**Risks:**\n- Transform function overhead applies to every object in initial LIST + every watch event\n- Transformed objects are what's cached; controllers see truncated state (may cause bugs if not carefully designed)\n\n---\n\n### 5.4 Cache Indexing Customization\n\n**Extension Point:** `SharedIndexInformer.AddIndexers()`, custom `IndexFunc`.\n\n**Use Cases:**\n- Enable efficient lookups by custom criteria (e.g., \"list pods by node\", \"list deployments by label key\")\n- Avoid O(n) scans for filtered queries\n\n**Example: Index by Node Affinity**\n\n```go\nindexers := cache.Indexers{\n    \"affinity\": func(obj interface{}) ([]string, error) {\n        pod := obj.(*v1.Pod)\n        var nodes []string\n        if pod.Spec.Affinity != nil && pod.Spec.Affinity.NodeAffinity != nil {\n            for _, term := range pod.Spec.Affinity.NodeAffinity.RequiredDuringSchedulingIgnoredDuringExecution.NodeSelectorTerms {\n                for _, expr := range term.MatchExpressions {\n                    for _, val := range expr.Values {\n                        nodes = append(nodes, val)\n                    }\n                }\n            }\n        }\n        return nodes, nil\n    },\n}\npodInformer.Indexer().AddIndexers(indexers)\n\n// Later, efficient lookup\npods, _ := podInformer.Indexer().ByIndex(\"affinity\", \"node-name\")\n```\n\n**Risks:**\n- Index cardinality explosion (if index function returns too many keys per object, memory usage explodes)\n- Index coherency bugs (if index function is non-deterministic or changes over time)\n\n---\n\n### 5.5 Event Handler Customization\n\n**Extension Point:** `ResourceEventHandler` and `ResourceEventHandlerFuncs`.\n\n**Use Cases:**\n- Custom event processing (filtering, aggregation)\n- Per-handler rate limiting (different resync periods per handler)\n- Context propagation (passing request ID through event handler)\n\n**Example: Custom Handler with Filtering**\n\n```go\ntype MyEventHandler struct {\n    filter func(obj interface{}) bool\n    queue  workqueue.RateLimitingInterface\n}\n\nfunc (h *MyEventHandler) OnAdd(obj interface{}, _ bool) {\n    if !h.filter(obj) {\n        return // Skip objects that don't match filter\n    }\n    key, _ := cache.MetaNamespaceKeyFunc(obj)\n    h.queue.Add(key)\n}\n\npodInformer.AddEventHandler(MyEventHandler{\n    filter: func(obj interface{}) bool {\n        pod := obj.(*v1.Pod)\n        return pod.Labels[\"app\"] == \"critical\"\n    },\n    queue: workqueue.NewRateLimitingQueue(...),\n})\n```\n\n**Risks:**\n- Blocking event handler blocks the informer LIST or resync)\n- `Sync`: Periodic resync events\n\n#### 5c: Indexer (`tools/cache/store.go`, `tools/cache/thread_safe_store.go`)\n\nA thread-safe, in-memory key-value store with optional secondary indices.\n\n**Key Responsibility:**\n- Store the authoritative local copy of object state\n- Support efficient lookups (by key, by index, by list)\n- Provide thread-safe read access for controllers\n\n**Indexing:**\n- Default key function: `namespace/name` (MetaNamespaceKeyFunc)\n- Custom indices: E.g., index by owner reference, by node affinity, by label\n- Index functions: `namespace`, `node`, `pvc`, `service`, `endpoint`, etc.\n\n#### 5d: SharedInformer (`tools/cache/shared_informer.go`)\n\nThe public API for controllers to access the cache and receive notifications of changes.\n\n**Key Responsibility:**\n- Vend read-only Lister and Indexer interfaces to controller logic\n- Distribute cache synchronization events to multiple event handlers\n- Provide resync notifications at configurable intervals\n- Report cache sync status via `HasSynced()`\n\n**Event Handler Contract:**\n```go\ntype ResourceEventHandler interface {\n    OnAdd(obj interface{}, isInInitialList bool)\n    OnUpdate(oldObj, newObj interface{})\n    OnDelete(obj interface{})\n}\n```\n\n**Resync Semantics:**\n- After initial LIST completes, `isInInitialList=true` is passed to OnAdd()\n- Periodic resync events trigger OnAdd() with `isInInitialList=false`\n- Controllers may ignore resync events or use them for periodic cleanup\n\n#### 5e: SharedInformerFactory (`informers/factory.go`)\n\nA factory that creates and manages multiple SharedInformers across different API groups and versions.\n\n**Key Responsibility:**\n- Coordinate a single Reflector per resource type\n- Share DeltaFIFO and Indexer instances among logically-independent informers\n- Manage the lifecycle of all informers (Start, Stop, WaitForCacheSync)\n\n**Efficiency Gains:**\n- Single LIST/WATCH stream per resource type (not per informer)\n- Shared cache reduces memory footprint\n- Synchronized resync cycles\n\n### Layer 6: Work Queue and Reconciliation (`util/workqueue`)\n\nDecouples event detection (informer) from object processing (controller).\n\n**Key Responsibility:**\n- Create a critical boundary between asynchronous events and synchronous reconciliation\n- Support rate limiting and exponential backoff per item\n- Enable controller retries without blocking the informer's watch stream\n\n**Queue Types:**\n- `Queue`: Simple FIFO\n- `DelayingQueue`: Delays items before re-queueing\n- `RateLimitingQueue`: Applies pluggable rate limiting strategies\n\n**Rate Limiting Strategies:**\n- `BucketRateLimiter`: Per-item token bucket\n- `ItemExponentialFailureRateLimiter`: Exponential backoff based on failure count\n- `ItemFastSlowRateLimiter`: Transition from fast to slow backoff\n- `MaxOfRateLimiter`: Apply maximum of multiple strategies\n\n**Controller Pattern:**\n```go\nfor {\n    key, _ := queue.Get()\n    // Controller business logic (reconcile)\n    queue.Done(key)\n    if err != nil {\n        queue.AddRateLimited(key) // Retry with backoff\n    }\n}\n```\n\n### Layer 7: Leader Election (`tools/leaderelection`)\n\nFor high-availability deployments, ensures only one controller replica is active at a time.\n\n**Key Responsibility:**\n- Compete for a distributed lock (stored as a Lease object)\n- Transition between leader and follower roles\n- Provide hooks (OnStartedLeading, OnStoppedLeading) for controller lifecycle\n\n**Lock Mechanisms:**\n- `Lease` (preferred): Lightweight distributed lock using the Lease API\n- `ConfigMap`, `Endpoints`: Legacy mechanisms\n- Custom `ResourceLock` interfaces for extension\n\n**Behavior:**\n1. All replicas attempt to acquire a lock by creating/updating a Lease\n2. Winner transitions to \"leader\" and calls `OnStartedLeading()`\n3. Leader renews the lease periodically\n4. If renewal fails (leader crash), followers compete for the lock\n5. Loser transitions to \"follower\" and calls `OnStoppedLeading()`\n\n### Layer 8: Server-Side Apply (`applyconfigurations/`)\n\nType-safe builder API for declarative object mutations. Enables multiple controllers to co-manage the same object safely.\n\n**Location:** `applyconfigurations/` (code-generated)\n\n**Key Responsibility:**\n- Construct PATCH requests with `managedFields` metadata\n- Allow controllers to declare \"this field is mine\" without conflicts\n\n**Apply Configuration Structure:**\n```go\ndeploymentApplyConfig := appsv1ac.Deployment(\"my-app\", \"default\").\n    WithSpec(appsv1ac.DeploymentSpec().\n        WithReplicas(2).\n        WithTemplate(...))\n\nresult, err := clientset.AppsV1().Deployments(\"default\").Apply(\n    ctx, deploymentApplyConfig, metav1.ApplyOptions{FieldManager: \"my-controller\"})\n```\n\n**Benefits:**\n- Declarative: Specify desired state, not mutations\n- Conflict-aware: Multiple controllers can manage different fields\n- Idempotent: Reapplying same config has no effect\n\n---\n\n## Part 3: End-to-End Control and Request Flows\n\n### Flow 1: Typed Client Request\n\nUser code \u2192 Typed Clientset \u2192 REST Client \u2192 HTTP Transport \u2192 API Server \u2192 Response Deserialization\n\n**Concrete Example: List Deployments**\n\n```go\n// User code\ndeployments, err := clientset.AppsV1().Deployments(\"default\").List(ctx, metav1.ListOptions{})\n```\n\n**Steps:**\n\n1. **Type-Safe Wrapper** (`kubernetes/typed/apps/v1/deployment.go`):\n   - Wraps a `rest.RESTClient` specific to AppsV1\n   - Routes to resource-specific implementation\n\n2. **REST Request Construction** (`rest/request.go`):\n   ```go\n   r := restClient.Get().\n       Namespace(\"default\").\n       Resource(\"deployments\")\n   ```\n   - Sets verb: `GET`\n   - Sets path: `/apis/apps/v1/namespaces/default/deployments`\n   - Applies query params: `limit`, `continue`, `labelSelector`, etc.\n\n3. **Content Negotiation**:\n   - Sets Accept header: `application/json` (or `application/vnd.kubernetes.protobuf`)\n   - Server responds with same format\n\n4. **HTTP Transport** (`rest/transport.go`):\n   - Constructs a chain of `http.RoundTripper` middleware:\n     - TLS setup (client cert auth)\n     - Bearer token injection (auth header)\n     - Rate limiting\n     - Retry with backoff\n   - Executes HTTP request\n\n5. **Response Processing**:\n   - HTTP status codes: 2xx \u2192 success, 3xx \u2192 redirect, 4xx/5xx \u2192 error\n   - Error responses: Deserialized to `apierrors.StatusError` with reason (NotFound, Conflict, etc.)\n   - Success responses: Decoded into `*unstructured.Unstructured` then converted to typed `DeploymentList`\n\n6. **Return to User**:\n   ```\n   deployments := {\n       TypeMeta: v1.TypeMeta{APIVersion: \"apps/v1\", Kind: \"DeploymentList\"},\n       ListMeta: v1.ListMeta{ResourceVersion: \"12345\"},\n       Items: [...]\n   }\n   ```\n\n**Error Handling:**\n- If 404: `apierrors.IsNotFound(err)` returns true\n- If 409 (conflict): `apierrors.IsConflict(err)` returns true\n- Other errors: Wrapped in `RequestConstructionError` or transport errors\n\n---\n\n### Flow 2: Informer-Driven Controller Loop\n\n```\nReflector LIST/WATCH\n    \u2193\nDeltaFIFO (event buffer)\n    \u2193\nIndexer (cache)\n    \u2193\nEvent Handlers (notify controllers)\n    \u2193\nWork Queue (buffer pending work)\n    \u2193\nController Reconcile Loop\n    \u2193\n[Lister access back to Indexer]\n```\n\n**Concrete Example: Pod Informer in a Controller**\n\n**Initialization:**\n```go\nconfig, _ := clientcmd.BuildConfigFromFlags(\"\", kubeconfig)\nclientset, _ := kubernetes.NewForConfig(config)\nfactory := informers.NewSharedInformerFactory(clientset, 10*time.Minute)\npodInformer := factory.Core().V1().Pods().Informer()\n\nworkqueue.NewRateLimitingQueue(workqueue.DefaultControllerRateLimiter())\n\npodInformer.AddEventHandler(cache.ResourceEventHandlerFuncs{\n    AddFunc: func(obj interface{}) {\n        key, _ := cache.MetaNamespaceKeyFunc(obj)\n        queue.Add(key) // \"default/my-pod\"\n    },\n    UpdateFunc: func(oldObj, newObj interface{}) {\n        key, _ := cache.MetaNamespaceKeyFunc(newObj)\n        queue.Add(key)\n    },\n    DeleteFunc: func(obj interface{}) {\n        key, _ := cache.DeletionHandlingMetaNamespaceKeyFunc(obj)\n        queue.Add(key)\n    },\n})\n\nfactory.Start(stopCh)\ncache.WaitForCacheSync(stopCh, podInformer.HasSynced)\n```\n\n**Runtime (Informer Side):**\n\n1. **Reflector Initialization**:\n   - Calls `podClient.List(ctx, metav1.ListOptions{ResourceVersion: \"\", Limit: 500})`\n   - Receives initial resourceVersion (e.g., \"12345\")\n   - Starts WATCH with `ResourceVersion: \"12345\"`\n\n2. **Initial LIST Response**:\n   - Receives PodList with 50 items, `Continue` token set\n   - Pushes all 50 to DeltaFIFO with delta type `Replaced`\n   - Fetches next page via `List(..., Continue: token)`\n\n3. **Watch Stream Begins**:\n   - Watches for events: Pod created, modified, deleted\n   - Each event pushed to DeltaFIFO as `Added`, `Updated`, `Deleted`\n\n4. **DeltaFIFO Processing**:\n   - Shared informer controller loop pops deltas\n   - For each delta, updates Indexer\n   - Triggers event handler callbacks\n\n5. **Event Handler Notification**:\n   - `OnAdd()`: Called for each pod in initial LIST + new pods\n   - `OnUpdate()`: Called for modified pods\n   - `OnDelete()`: Called for deleted pods\n   - Each call adds the pod's key to the work queue\n\n6. **Resync Cycle** (every 10 minutes):\n   - Reflector calls `Resync()` on DeltaFIFO\n   - Pushes all cached objects back as `Sync` deltas\n   - Event handlers called again (giving controller chance to reconcile even if watch was missed)\n\n**Runtime (Controller Side):**\n\n1. **Work Queue Processing Loop**:\n   ```go\n   for {\n       key, _ := queue.Get()\n       defer queue.Done(key)\n\n       namespace, name, _ := cache.SplitMetaNamespaceKey(key)\n       pod, _ := podLister.Pods(namespace).Get(name)\n\n       // Controller business logic\n       if err := reconcile(pod); err != nil {\n           queue.AddRateLimited(key) // Retry with backoff\n           continue\n       }\n\n       queue.Forget(key) // Success, reset backoff counter\n   }\n   ```\n\n2. **Lister Access**:\n   - Controller calls `podLister.Pods(\"default\").Get(\"my-pod\")`\n   - Lister queries the Indexer (cached, no API call)\n   - Returns object or \"NotFound\"\n\n3. **Reconciliation**:\n   - Business logic (may make updates via typed clientset)\n   - If reconciliation fails: Work queue re-adds with exponential backoff\n   - If successful: Item forgotten (backoff counter reset)\n\n**Key Invariants:**\n- Cache is **eventually consistent** with API server (may lag by watch latency)\n- Events are **ordered per object** but not across objects\n- Rapid changes may collapse into a single cache update\n- Resync ensures \"stuck\" objects are revisited even if watch is broken\n\n---\n\n### Flow 3: Dynamic Client with Discovery\n\nUsed when the client doesn't have type definitions for a resource (e.g., a CRD).\n\n```\nDynamic Client Request\n    \u2193\nDiscovery (if needed, discover resource schema)\n    \u2193\nREST Client (like typed client, but with unstructured.Unstructured)\n    \u2193\n[No code generation required]\n```\n\n**Example: Create a Custom Resource**\n\n```go\ndynamicClient := dynamic.NewForConfig(config)\ngvr := schema.GroupVersionResource{\n    Group:    \"mycompany.com\",\n \"Widget\",\n        \"metadata\": map[string]interface{}{\n            \"name\":      \"my-widget\",\n            \"namespace\": \"default\",\n        },\n        \"spec\": map[string]interface{}{\n            \"color\": \"blue\",\n        },\n    },\n}\n\nresult, err := dynamicClient.Resource(gvr).Namespace(\"default\").Create(ctx, obj, metav1.CreateOptions{})\n```\n\n**Steps:**\n\n1. **Request Construction**:\n   - No type definitions needed; works with maps\n   - REST client constructs request path: `/apis/mycompany.com/v1/namespaces/default/widgets`\n   - Serializes object to JSON\n\n2. **API Server Processing**:\n   - Server deserializes and validates against OpenAPI schema\n   - Returns object with `metadata.uid`, `metadata.resourceVersion` populated\n\n3. **Response Deserialization**:\n   - Result remains as `unstructured.Unstructured`\n   - Caller accesses fields via `obj.UnstructuredContent()[\"spec\"].(map[string]interface{})`\n\n---\n\n## Part 4: Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### Network Failures\n\n**Scenario:** API server becomes unreachable during WATCH.\n\n**Reflector Behavior:**\n- watch connection drops with an error\n- Exponential backoff (up to 1s) before retry\n- On retry, if resourceVersion is still valid: Resume from that resourceVersion\n- If resourceVersion is too old: Re-list (reset to version 0)\n\n**Controller Impact:**\n- Event stream pauses; no new changes reach work queue\n- Existing work queue items continue processing\n- On reconnection: Reflector pushes accumulated changes (via resync cycle if needed)\n- Controller eventually catches up (after backoff and re-list)\n\n**Recovery Time:** Typically < 30 seconds (default backoff + resync timeout)\n\n---\n\n#### API Server Unavailability\n\n**Scenario:** All replicas of API server are down (brief outage or maintenance).\n\n**Reflector Behavior:**\n- LIST request fails (all replicas down)\n- Exponential backoff with jitter (configured via envBackoffDuration, default 600s max)\n- When API server recovers: resumes from last known resourceVersion\n\n**Controller Impact:**\n- Work queue starves (no new events can be fetched)\n- Informer reports `HasSynced() == false` until first successful LIST\n- Controllers waiting on `WaitForCacheSync()` block indefinitely\n\n**Mitigation:**\n- Set timeout on `WaitForCacheSync()`: `WaitForCacheSync(ctx, informer.HasSynced)` where ctx has deadline\n- Multiple API server replicas with load balancer\n- Service account with sufficient RBAC to list resources (common misconfiguration)\n\n---\n\n#### Resource Version Expiration (\"too old\")\n\n**Scenario:** Watch stream lags; API server evicts the resource version.\n\n**Error Message:** `\"the object has been deleted and no further information is available\"`\n\n**Reflector Behavior:**\n- watch returns error with code `410 Gone`\n- Reflector detects this as `isLastSyncResourceVersionUnavailable = true`\n- Next reconciliation: Performs full re-list with `ResourceVersion: \"\"` (strong consistency)\n\n**Prevention:**\n- Reduce resync period (default 10 minutes; smaller = more frequent re-list)\n- Monitor API server `etcd` compaction and retention settings\n- Watch bookmarks help (reflect updates resourceVersion without re-list)\n\n---\n\n#### Cache Desynchronization\n\n**Scenario:** Informer cache falls behind API server state due to bugs or race conditions.\n\n**Root Causes:**\n1. Lost watch events (network hiccup during compression, etc.)\n2. Reflector crashes and restarts (clean state loss)\n3. DeltaFIFO overflow (should never happen; indicates architectural issue)\n4. Bug in event handler (e.g., handler panics)\n\n**Detection:**\n- `reflector_data_consistency_detector.go`: Periodic LIST to validate cache against API server\n- Enabled via feature gate `ObjectCountTracker`\n- Logs warnings if discrepancies found\n\n**Recovery:**\n- Resync cycle re-lists all objects\n- Controllers re-process all objects (idempotent reconciliation should tolerate this)\n\n---\n\n#### Informer Handler Panics\n\n**Scenario:** Event handler code panics (e.g., null pointer dereference).\n\n**Behavior:**\n- `shared_informer.go` has `defer handleCrash()` around handler invocation\n- Panic is logged and recovered (does not crash informer)\n- Event is skipped; next event is processed\n- **Controller misses the update** \u2192 object state diverges\n\n**Prevention:**\n- Robust event handler code (validate inputs, handle nil)\n- Wrap event handler in try-catch wrapper (if using a framework)\n- Thorough testing of event handler edge cases\n\n---\n\n#### Work Queue Starvation\n\n**Scenario:** Controller reconciliation is slower than event arrival rate.\n\n**Behavior:**\n- Work queue grows unbounded (all in-memory)\n- Memory usage increases linearly\n- CPU usage on controller increases (processing items)\n- No per-item timeout; items never expire\n\n**Prevention:**\n- Monitor work queue depth and controller processing latency\n- Scale controller replicas if using leader election\n- Optimize reconciliation logic (avoid nested loops, expensive API calls)\n- Use exponential backoff to reduce retry rate for hot items\n\n**Limits:**\n- Work queue is unbounded in memory (no max size)\n- Typical deployments: 1000-10000 items before memory becomes an issue\n- Large clusters (10k+ objects): Consider sharded controllers or leader-based routing\n\n---\n\n#### High Cardinality Labels\n\n**Scenario:** Custom index function returns many keys per object.\n\n**Example:** Indexing pods by all container image tags (a pod may have 5+ containers, each with unique tags).\n\n**Impact:**\n- Indexer memory usage multiplies\n- Index query latency degrades (linear scan of matching keys)\n- Cache consistency checker may produce false positives\n\n**Prevention:**\n- Index only low-cardinality fields\n- Avoid indices with cross-product keys\n\n---\n\n### 4.2 Design Limits\n\n| Limit | Value | Notes |\n|-------|-------|-------|\n| Watch stream timeout | 5min - 10min (random) | Randomized to spread reconnections |\n| Resync period | Configurable (default 10min) | Smaller = more LIST requests, larger = slower recovery |\n| Max in-flight requests (QPS \u00d7 Burst) | QPS \u00d7 Burst (default 5 \u00d7 10) | Configured per `rest.Config` |\n| Work queue max size | Unbounded (memory limit) | Typical safe limit: 10k items |\n| Informer handler timeout | None (unbounded) | Slow handlers block the informer |\n| Cache consistency window | Resync period | After resync: cache is consistent |\n| Lister lookups per object | O(1) by key, O(n) by index | Index efficiency depends on cardinality |\n\n---\n\n### 4.3 Design Tradeoffs\n\n#### Tradeoff 1: Consistency vs. Availability\n\n**Choice:** Favor availability and eventual consistency over strong consistency.\n\n**Rationale:**\n- Strong consistency requires synchronous client-server communication on every operation\n- Kubernetes at scale (10k+ nodes) cannot afford this cost\n- Eventual consistency with resync cycles and re-listing provides sufficient consistency for controllers\n\n**Implication:**\n- Controllers must be idempotent (can tolerate reprocessing the same object)\n- Controllers must tolerate missing intermediate states (e.g., a pod created and deleted before the controller sees it)\n\n---\n\n#### Tradeoff 2: Polling vs. Watching\n\n**Choice:** Favor WATCH over polling, with fallback to re-list on errors.\n\n**Rationale:**\n- WATCH is more efficient (server pushes changes, not client pulls)\n- Resync period ensures consistency without continuous polling\n- Hybrid approach recovers from watch failures gracefully\n\n**Implication:**\n- Controllers see changes with latency of watch network RTT (typically <100ms)\n- Controller re-processing not guaranteed to be in order across multiple objects\n- Controllers must tolerate watching the same state multiple times (resync)\n\n---\n\n#### Tradeoff 3: Code Generation vs. Dynamic\n\n**Choice:** Generate type-safe clients for built-in APIs; provide dynamic client for extensibility.\n\n**Rationale:**\n- Type safety catches errors at compile time (build-time safety)\n- Dynamic client enables CRDs without code generation (runtime flexibility)\n- Hybrid approach suits both operator developers and CRD authors\n\n**Implication:**\n- Custom resources require either code generation or dynamic client usage\n- Informers are type-safe for built-in types but less ergonomic for CRDs\n- Operators targeting multiple versions must either code-gen for each or use dynamic informers\n\n---\n\n#### Tradeoff 4: Caching Overhead\n\n**Choice:** Maintain a full local cache (memory overhead) to enable efficient reads.\n\n**Rationale:**\n- Controllers read from local cache frequently (no API latency)\n- API server is not bottleneck (no per-read request)\n- Memory cost is acceptable for typical cluster state sizes (10k-100k objects)\n\n**Implication:**\n- Every controller replica maintains a full copy of relevant objects in memory\n- Scaling to 10k controller replicas: each has a copy of cached state (memory \u00d7 10k = significant cost)\n- Large objects (e.g., 1MB ConfigMaps) amplify the cost\n\n**Optimization:**\n- Metadata-only informers (using `PartialObjectMetadata`) reduce memory\n- Shared informer factory reduces per-informer overhead\n- Custom indices enable efficient querying without full list iteration\n\n---\n\n#### Tradeoff 5: Event Ordering\n\n**Choice:** Guarantee ordering per object, not across objects.\n\n**Rationale:**\n- Per-object ordering is sufficient for idempotent reconciliation\n- Cross-object ordering would require distributed coordination (expensive)\n- Allows parallelism: multiple controllers can process different objects simultaneously\n\n**Implication:**\n- Controller cannot rely on seeing Pod-A update before Pod-B update (even if A updated first)\n- Controllers must use `resourceVersion` or timestamps to detect ordering issues if needed\n- Useful for detecting lost events (e.g., if Pod A was updated but cache shows old version, a watch event was lost)\n\n---\n\n## Part 5: Extension Points and Customization Hooks\n\n### 5.1 REST Transport Customization\n\n**Extension Point:** `rest.WrapperFunc` and `http.RoundTripper` chain.\n\n**Use Cases:**\n- Inject custom auth headers (e.g., API key passed through a secret service)\n- Add request/response logging\n- Metrics collection (latency, error rates)\n- Request tracing (OpenTelemetry integration)\n\n**Example:**\n\n```go\n// Custom transport middleware\nfunc MyAuthMiddleware(rt http.RoundTripper) http.RoundTripper {\n    return &myCustomRT{rt: rt}\n}\n\ntype myCustomRT struct{ rt http.RoundTripper }\nfunc (m *myCustomRT) RoundTrip(req *http.Request) (*http.Response, error) {\n    // Fetch auth token from external service\n    token, _ := fetchTokenFromService()\n    req.Header.Set(\"Authorization\", \"Bearer \" + token)\n    return m.rt.RoundTrip(req)\n}\n\n// Apply to rest.Config\nconfig, _ := clientcmd.BuildConfigFromFlags(\"\", kubeconfig)\nconfig.WrapTransport = MyAuthMiddleware\n```\n\n**Risks:**\n- Broken middleware can silently drop requests (ensure error handling)\n- Middleware overhead applies to every request registration.\n\n**Use Cases:**\n- Support custom wire formats (e.g., MessagePack instead of JSON)\n- Compression (gzip encoding in Accept header)\n- Custom error codes\n\n**Mechanism:**\n```go\nconfig.NegotiatedSerializer = myCustomSerializer\n```\n\n**Risks:**\n- Must implement full codec interface (both encode and decode)\n- Incompatibility with API server serialization breaks requests\n\n---\n\n### 5.3 Reflector Customization\n\n**Extension Point:** `ListerWatcher`, `TransformFunc`, `WatchErrorHandler`.\n\n**Use Cases:**\n- Filter watch events (transform function to strip sensitive data)\n- Custom error handling for watch failures\n- Alternative list/watch sources (e.g., cache server, proxy)\n\n**Example: Transform Function**\n\n```go\n// Strip sensitive fields before caching\nfunc stripSecrets(obj interface{}) (interface{}, error) {\n    secret := obj.(*v1.Secret)\n    // Remove Data field to avoid memory bloat\n    secret.Data = nil\n    return secret, nil\n}\n\nreflector := cache.NewReflector(\n    listerWatcher,\n    &v1.Secret{},\n    deltaFIFO,\n    reflector.NewReflectorOptions().WithTransformer(stripSecrets),\n)\n```\n\n**Risks:**\n- Transform function overhead applies to every object in initial LIST + every watch event\n- Transformed objects are what's cached; controllers see truncated state (may cause bugs if not carefully designed)\n\n---\n\n### 5.4 Cache Indexing Customization\n\n**Extension Point:** `SharedIndexInformer.AddIndexers()`, custom `IndexFunc`.\n\n**Use Cases:**\n- Enable efficient lookups by custom criteria (e.g., \"list pods by node\", \"list deployments by label key\")\n- Avoid O(n) scans for filtered queries\n\n**Example: Index by Node Affinity**\n\n```go\nindexers := cache.Indexers{\n    \"affinity\": func(obj interface{}) ([]string, error) {\n        pod := obj.(*v1.Pod)\n        var nodes []string\n        if pod.Spec.Affinity != nil && pod.Spec.Affinity.NodeAffinity != nil {\n            for _, term := range pod.Spec.Affinity.NodeAffinity.RequiredDuringSchedulingIgnoredDuringExecution.NodeSelectorTerms {\n                for _, expr := range term.MatchExpressions {\n                    for _, val := range expr.Values {\n                        nodes = append(nodes, val)\n                    }\n                }\n            }\n        }\n        return nodes, nil\n    },\n}\npodInformer.Indexer().AddIndexers(indexers)\n\n// Later, efficient lookup\npods, _ := podInformer.Indexer().ByIndex(\"affinity\", \"node-name\")\n```\n\n**Risks:**\n- Index cardinality explosion (if index function returns too many keys per object, memory usage explodes)\n- Index coherency bugs (if index function is non-deterministic or changes over time)\n\n---\n\n### 5.5 Event Handler Customization\n\n**Extension Point:** `ResourceEventHandler` and `ResourceEventHandlerFuncs`.\n\n**Use Cases:**\n- Custom event processing (filtering, aggregation)\n- Per-handler rate limiting (different resync periods per handler)\n- Context propagation (passing request ID through event handler)\n\n**Example: Custom Handler with Filtering**\n\n```go\ntype MyEventHandler struct {\n    filter func(obj interface{}) bool\n    queue  workqueue.RateLimitingInterface\n}\n\nfunc (h *MyEventHandler) OnAdd(obj interface{}, _ bool) {\n    if !h.filter(obj) {\n        return // Skip objects that don't match filter\n    }\n    key, _ := cache.MetaNamespaceKeyFunc(obj)\n    h.queue.Add(key)\n}\n\npodInformer.AddEventHandler(MyEventHandler{\n    filter: func(obj interface{}) bool {\n        pod := obj.(*v1.Pod)\n        return pod.Labels[\"app\"] == \"critical\"\n    },\n    queue: workqueue.NewRateLimitingQueue(...),\n})\n```\n\n**Risks:**\n- Blocking event handler blocks the informer (no parallelism in handler invocation)\n- Handler panics crash the informer (use defer handleCrash())\n\n---\n\n### 5.6 Work Queue Customization\n\n**Extension Point:** `RateLimiter`
