# Architecture Document: Envoy HTTP Connection Manager

**Repository:** envoyproxy/envoy
**Output:** Write your document to `/workspace/documentation.md`

## Task

Produce an architecture document for Envoy's **HTTP Connection Manager** (HCM) subsystem. The document must explain how the major components interact to process an HTTP request from arrival on a downstream connection through to upstream dispatch. Do not simply list APIs — explain the **design**, **data flow**, and **extension points**.

## Scope

Your document must cover these four components and how they work together:

### 1. ConnectionManagerImpl
The central network filter that owns the connection lifecycle. Explain:
- How it implements `Network::ReadFilter` to receive raw bytes
- How it creates and manages `ActiveStream` objects for each HTTP request
- Its role in codec creation (lazy codec instantiation for H1/H2 vs. H3)
- Connection-level concerns: drain decisions, overload management, watermark-based flow control

### 2. FilterManager and the HTTP Filter Chain
The per-stream filter chain execution engine. Explain:
- The decoder filter chain (request path) and encoder filter chain (response path)
- How `FilterChainFactory` creates filters for each stream
- Filter iteration: how `decodeHeaders`/`decodeData`/`decodeTrailers` propagate through decoder filters
- How filters can stop iteration, modify headers, or send local replies
- The distinction between `StreamDecoderFilter`, `StreamEncoderFilter`, and `StreamFilter` (dual)

### 3. Router Filter
The terminal decoder filter that forwards requests upstream. Explain:
- Route selection: how the router uses `RouteConfiguration` to pick a cluster
- How it obtains an HTTP connection pool from `ClusterManager`
- Upstream request lifecycle: `UpstreamRequest` creation, retry logic, timeout handling
- Shadow routing and hedged request support (if applicable)

### 4. Cluster Manager and Upstream Connectivity
The upstream connection pool and cluster management layer. Explain:
- How `ClusterManagerImpl` provides connection pools per cluster
- Load balancing: how the router obtains a host from the cluster's load balancer
- Connection pool mechanics: the relationship between logical connection pools and physical connections
- How health checking and outlier detection feed back into the load balancer

## Document Requirements

1. **Component Responsibilities** — what each component owns
2. **Data Flow** — the path of an HTTP request from downstream bytes to upstream dispatch, and the response path back
3. **Extension Points** — where users extend Envoy (HTTP filters, cluster extensions, load balancers, access loggers)
4. **Error Handling** — how errors at each stage (codec errors, filter errors, upstream failures) are handled
5. **relevant source files** — reference the actual source files in the envoyproxy/envoy repository

## Anti-Requirements

- Do NOT generate a simple API listing or header-file dump
- Do NOT fabricate class names or file paths that don't exist in the repository
- Do NOT cover Envoy components outside the HCM request path (e.g., xDS config delivery, listener management)

Stream response trailers downstream via encoder_->encodeTrailers()\n```\n\n### Retry Logic\n\n**Location**: `source/common/router/retry_state_impl.h` and `.cc`\n\nRetries are controlled by:\n1. **Route retry policy** \u2014 defines which conditions trigger retries\n2. **Request headers** \u2014 can override route policy via `x-envoy-retry-on`, `x-envoy-max-retries`\n3. **Response status code** \u2014 certain 5xx codes and connection errors trigger retries\n\n```cpp\nRetryPolicy retry_conditions = {\n  RETRY_ON_5XX,                    // 500-599 status codes\n  RETRY_ON_GATEWAY_ERROR,          // 502, 503, 504\n  RETRY_ON_CONNECT_FAILURE,        // Connection reset, refused\n  RETRY_ON_RETRIABLE_4XX,          // 409, 429\n  RETRY_ON_RESET,                  // Stream reset\n  RETRY_ON_REFUSED_STREAM,         // HTTP/2 refused stream\n  // ... and others\n};\n```\n\nWhen a retryable condition is detected:\n- Current upstream request is destroyed\n- Upstream connection pool releases the request\n- New upstream request attempts the next host (via load balancing) or same host\n- Per-try timeout is reset for the new attempt\n\n**Maximum Retries**: Limited by route policy and `x-envoy-max-retries` header\n\n### Timeout Handling\n\nThe router implements **three levels of timeouts**:\n\n#### 1. Global Timeout\nThe entire request lifecycle from decoding headers to receiving final response:\n```cpp\nTimeoutData timeout = FilterUtility::finalTimeout(route, headers, ...);\nstd::chrono::milliseconds global_timeout = timeout.global_timeout_;\n\n// Set timer\nglobal_timeout_timer_ = dispatcher.createTimer([this]() {\n  onUpstreamTimeoutAbort(...);  // Send 504 downstream\n});\n```\n\n#### 2. Per-Try Timeout\nIndividual upstream attempt timeout, independent of retries:\n```cpp\nstd::chrono::milliseconds per_try_timeout = timeout.per_try_timeout_;\n\nupstream_request->per_try_timeout_timer_ = dispatcher.createTimer([this]() {\n  onPerTryTimeout(*upstream_request);  // Retry if eligible\n});\n```\n\n#### 3. Per-Try Idle Timeout\nTimeout if no data flows for a period during a single attempt:\n```cpp\nupstream_request->per_try_idle_timer_ = dispatcher.createTimer([this]() {\n  onPerTryIdleTimeout(*upstream_request);  // Retry or fail\n});\n```\n\nThe timer in **Configuration**:\n- Specified in route config (`timeout`, `per_try_timeout`)\n- Can be overridden by request header `x-envoy-upstream-rq-timeout-ms`\n\n### Hedged Requests\n\n**Location**: `source/common/router/router.cc` (hedging logic in `Filter::startUpstreamRequest()`)\n\nHedged requests allow **concurrent upstream attempts** to reduce tail latency:\n\n```\nInitial request attempt:\n  Send to upstream host A with per-try-timeout of 100ms\n\nIf no response after 80ms AND hedge is enabled:\n  Launch second concurrent attempt to upstream host B\n\nWhichever upstream responds first wins:\n  - Response passed downstream\n  - Other attempt canceled (reset)\n  - Upstream connection returned to pool\n```\n\n**Configuration**:\n- Route-level hedge policy: `hedge_policy.hedge_on_per_try_timeout`\n- Header override: `x-envoy-hedge-on-per-try-timeout`\n\n### Shadow Routing\n\n**Location**: `source/common/router/router.cc` (shadow routing in `Filter::decodeHeaders()`)\n\nShadow routing sends **read-only copies** of requests to alternate clusters for traffic analysis:\n\n```cpp\n// After selecting primary route and cluster\nfor (const auto& shadow_policy : route->shadowPolicies()) {\n  // Determine if we should shadow this request\n  if (FilterUtility::shouldShadow(shadow_policy, random)) {\n    // Create separate upstream request to shadow cluster\n    // Send request data\n    // Discard response (don't affect client)\n  }\n}\n```\n\n**Key Properties**:\n- Shadow requests are **not retried**\n- Shadow responses are **not sent to client**\n- Used for canary testing, traffic shadowing, A/B testing\n- Controlled by `x-envoy-shadowed-header` to avoid infinite loops\n\n### Response Processing\n\nWhen upstream response headers arrive:\n\n```cpp\nUpstreamRequest::decodeHeaders(ResponseHeaderMap& headers, bool end_stream)\n  \u2193\nRouter::Filter::onUpstreamResponse(...)\n  \u2193\n1. Check if response is retryable (e.g., 503)\n2. If retry eligible and retries remaining: trigger retry (goto new attempt)\n3. Otherwise:\n   - Apply response header transformations\n   - Update stream info with upstream details\n   - Call parent_->encoder_->encodeHeaders() to send to client\n   - If end_stream=true, response complete\n   - Otherwise, prepare to receive body data\n```\n\n---\n\n## Component 4: Cluster Manager and Upstream Connectivity\n\n### Location\n- **Interface**: `envoy/upstream/cluster_manager.h`\n- **Implementation**: `source/common/upstream/cluster_manager_impl.h` and `.cc`\n- **Connection Pools**: `source/common/http/conn_pool_base.h`, `http1/conn_pool.h`, `http2/conn_pool.h`\n\n### ClusterManager: The Upstream Abstraction\n\n`ClusterManager` is Envoy's **central registry of upstream clusters**. **Least request** \u2014 choose host with fewest active requests\n- **Ring hash** \u2014 consistent hash of headers/cookies for session stickiness\n- **Random** \u2014 random host selection\n- Single pool handles many concurrent requests\n- Implements queue-based request scheduling if all connections busy\n\n**Physical Connection** (to specific host):\n- Underlying TCP or TLS connection to upstream\n- HTTP/1.1: One physical connection = one request at a time\n- HTTP/2: One physical connection = many concurrent requests (multiplexed)\n- HTTP/3: One QUIC connection = many concurrent streams\n\n```\nHttp::ConnectionPool::Instance\n\u251c\u2500\u2500 ActiveConnection (HTTP/1.1)\n\u2502   \u251c\u2500\u2500 CodecClient \u2192 HttpCodecImpl\n\u2502   \u2514\u2500\u2500 Physical socket\n\u251c\u2500\u2500 ActiveConnection (HTTP/2)\n\u2502   \u251c\u2500\u2500 CodecClient \u2192 HTTP2CodecImpl\n\u2502   \u251c\u2500\u2500 Multiplexed Stream A\n\u2502   \u251c\u2500\u2500 Multiplexed Stream B\n\u2502   \u2514\u2500\u2500 Physical socket\n\u2514\u2500\u2500 PendingStream queue (if all connections busy)\n```\n\n### Connection Pool Request Flow\n\nWhen router submits request to connection pool:\n\n```cpp\nHttp::ConnectionPool::Cancellable* cancellable = pool->newStream(\n    upstream_request->response_decoder_,\n    upstream_request->pool_callbacks_,\n Has idle connection \u2192 Immediately calls callbacks->onPoolReady()\n// 2. Has available multiplexing capacity \u2192 Create new stream, onPoolReady()\n// 3. No capacity, queue pending \u2192 Waits for available capacity\n// 4. Cannot connect \u2192 Calls callbacks->onPoolFailure()\n```\n\n### Connection Pool Callbacks\n\nWhen pool decides on an action:\n\n```cpp\nstruct Http::ConnectionPool::Callbacks {\n  // Pool successfully allocated a stream\n  void onPoolReady(RawAsyncStream& async_stream,\n                   UpstreamHost& upstream_host);\n\n  // Pool could not allocate (all connections exhausted, host down, etc.)\n  void onPoolFailure(PoolFailureReason reason,\n                     absl::string_view failure_reason,\n                     UpstreamHostDescription& upstream_host);\n};\n```\n\n**Router's implementation**:\n- `onPoolReady()` \u2192 Encodes request headers via stream encoder\n- `onPoolFailure()` \u2192 Checks retry eligibility, retries or sends error downstream\n\n### Circuit Breaking and Overload\n\n**Location**: `envoy/upstream/resource_manager.h`\n\nClusters enforce **circuit breaker limits**:\n```cpp\nCircuitBreakerStats limits {\n  max_connections: 100,      // Max concurrent connections to cluster\n  max_pending_requests: 100,  // Max queued requests\n  max_requests: 100,          // Max concurrent active requests\n  max_retries: 100,           // Max concurrent retry attempts\n};\n```\n\nIf limits exceeded:\n- Connection pool returns `onPoolFailure()`\n- Router checks if retry is allowed (doesn't consume retry budget)\n- If not retriable, returns 503 to client\n\n### Health Checking and Outlier Detection\n\n**Location**: `envoy/upstream/health_check_host_monitor.h`\n\nClusters maintain **health status** updated by:\n\n1. **Active Health Checking** \u2014 Periodic probes to upstream hosts\n   - If probe fails \u2192 Mark unhealthy\n   - Load balancer skips unhealthy hosts\n\n2. **Outlier Detection** \u2014 Passive monitoring of request failures\n   - Tracks error rates, latencies, consecutive errors\n   - Ejects underperforming hosts (temporarily)\n   - Periodically re-admits ejected hosts\n\nWhen a host becomes unhealthy:\n- Load balancer routes to healthy alternatives\n- Failing requests may be retried to healthy hosts\n\n### Connection Pool Mechanics: HTTP/1.1 vs HTTP/2\n\n#### HTTP/1.1 Connection Pool\n```\nOne request per physical connection:\n\nRequest 1 \u2192 Connection A \u2192 Upstream\nRequest 2 \u2192 Connection B \u2192 Upstream\nRequest 3 \u2192 Queue (waiting for Connection A or B to free up)\n\nWhen Connection A response completes:\n  Dequeue Request 3\n  Reuse Connection A for Request 3\n```\n\n#### HTTP/2 Connection Pool\n```\nMultiple requests per physical connection (multiplexed):\n\nRequest 1 \u2510\nRequest 2 \u251c\u2192 Connection A (single physical socket, multiple HTTP/2 streams)\nRequest 3 \u2518\nRequest 4 \u2500\u2192 Connection B (second physical socket if pool limit reached)\n\nEach request gets a unique HTTP/2 stream ID within the connection\n```\n\n---\n\n## Data Flow: Request from Bytes to Upstream and Back\n\nThis section traces a complete request lifecycle through all four components.\n\n### **Phase 1: Bytes Arrive at Downstream Connection**\n\n```\n1. TCP socket receives HTTP bytes from client\n                \u2193\n2. Network layer invokes ConnectionManagerImpl::onData(Buffer&, bool end_stream)\n                \u2193\n3. Is codec created?\n   NO \u2192 CreateCodec (lazily create HTTP/1.1 or HTTP/2)\n                \u2193\n4. Pass buffer to codec for parsing\n   codec.onData(buffer)\n                \u2193\n5. Codec parses HTTP frames and invokes:\n   ServerConnectionCallbacks::newStream(ResponseEncoder&)\n```\n\n### **Phase 2: Stream Creation and Filter Chain Initialization**\n\n```\n6. ConnectionManagerImpl::newStream() [ServerConnectionCallbacks]\n                \u2193\n7. Create ActiveStream(stream_id, filter_manager, ...)\n                \u2193\n8. FilterManager instantiates all configured HTTP filters\n                \u2193\n9. Return RequestDecoder from ActiveStream to codec\n                \u2193\n10. Codec invokes RequestDecoder::decodeHeaders(headers, end_stream)\n```\n\n### **Phase 3: Request Flows Through Decoder Chain**\n\n```\n11. ActiveStream::decodeHeaders()\n                \u2193\n12. FilterManager::decodeHeaders()\n                \u2193\n13. Iterate decoder filters:\n    For each filter in config order:\n      - filter.decodeHeaders(headers, end_stream)\n      - If StopIteration: halt, wait for continueDecoding()\n      - If Continue: proceed to next filter\n                \u2193\n14. Last filter = Router Filter\n                \u2193\n15. Router::decodeHeaders():\n      a) callbacks_->route() to get RouteEntry\n      b) Extract cluster name from route\n      c) Get connection pool from cluster_manager_\n      d) Submit request to connection pool\n      e) Return StopIteration (wait for upstream response)\n```\n\n### **Phase 4: Upstream Request Execution**\n\n```\n16. Connection pool either:\n    - Returns idle connection \u2192 onPoolReady() immediately\n    - Creates new connection and calls onPoolReady()\n    - Queues pending stream if no capacity available\n                \u2193\n17. Router::onPoolReady(request_encoder)\n                \u2193\n18. Encode request headers:\n    request_encoder->encodeHeaders(upstream_headers, end_stream=false)\n                \u2193\n19. If client request has body:\n    codec.onData() \u2192 ActiveStream::decodeData()\n                \u2193\n20. FilterManager::decodeData()\n                \u2193\n21. Router::decodeData() (typically passes through)\n                \u2193\n22. Router forwards to upstream:\n    request_encoder->encodeData(buffer, end_stream=false)\n                \u2193\n23. If client request has trailers:\n    codec.onTrailers() \u2192 ActiveStream::decodeTrailers()\n                \u2193\n24. Router::decodeTrailers()\n                \u2193\n25. request_encoder->encodeTrailers(trailers)\n```\n\n### **Phase 5: Upstream Response Returns**\n\n```\n26. Upstream connection receives response bytes\n                \u2193\n27. Upstream codec parses response and invokes:\n    ResponseDecoder::decodeHeaders(response_headers, bool end_stream)\n                \u2193\n28. Router::UpstreamRequest::decodeHeaders() (as ResponseDecoder impl)\n                \u2193\n29. Router::onUpstreamResponse(response_headers)\n      a) Check if response is retryable\n      b) If retryable AND retries remaining:\n         - Cancel per-try timers\n         - Destroy current upstream request\n         - Go back to connection pool (Phase 4)\n      c) Otherwise:\n         - Update response headers (strip hop-by-hop, add routing headers)\n         - Call encoder_->encodeHeaders(response_headers, end_stream)\n                \u2193\n30. FilterManager::encodeHeaders()\n                \u2193\n31. Iterate encoder filters in REVERSE order:\n    For each filter (in reverse config order):\n      - filter.encodeHeaders(headers, end_stream)\n      - If StopIteration: halt, wait for continueEncoding()\n      - If Continue: proceed to previous filter\n                \u2193\n32. Headers encoded to downstream connection via response_encoder\n```\n\n### **Phase 6: Response Body Flows Downstream**\n\n```\n33. Upstream continues sending body chunks:\n    Upstream codec \u2192 UpstreamRequest::decodeData()\n                \u2193\n34. Router::UpstreamRequest::decodeData()\n                \u2193\n35. Router forwards via:\n    encoder_->encodeData(buffer, end_stream=false)\n                \u2193\n36. FilterManager::encodeData()\n                \u2193\n37. Encoder filters process data (in reverse)\n                \u2193\n38. Data written to downstream response_encoder\n                \u2193\n39. Downstream codec serializes to TCP socket\n```\n\n### **Phase 7: Response Completion and Cleanup**\n\n```\n40. Upstream sends trailers or final chunk:\n    UpstreamRequest::decodeTrailers() or final decodeData(end_stream=true)\n                \u2193\n41. Router forwards trailers:\n    encoder_->encodeTrailers(trailers)\n                \u2193\n42. FilterManager::encodeTrailers()\n                \u2193\n43. Downstream response_encoder finalizes response\n                \u2193\n44. ConnectionManagerImpl detects response complete\n                \u2193\n45. ActiveStream::completeRequest():\n    - Log request (access logs)\n    - Collect timing stats\n    - Cleanup resources\n    - Dequeue next request if pipelined (HTTP/1.1) or multiplex (HTTP/2)\n                \u2193\n46. Upstream connection pool returns request to pool\n                \u2193\n47. If pool has pending requests: allocate connection to next pending request\n                \u2193\n48. If no pending requests: mark connection idle, reset activity timers\n```\n\n---\n\n## Error Handling Strategies\n\n### **Codec Errors**\n\nWhen the codec encounters malformed HTTP data:\n```\nHTTP/1.1 codec invalid frame\n              \u2193\nCodec invokes error callback\n              \u2193\nConnectionManagerImpl::onCodecError()\n              \u2193\nGenerate 400 Bad Request to client\n              \u2193\nClose downstream connection\n```\n\n**Stats**: `http.downstream_rq_invalid` counter incremented\n\n### **Filter Errors**\n\nIf a filter throws exception or returns error status:\n```\nFilter::decodeHeaders() throws\n              \u2193\nFilterManager catches exception\n              \u2193\nIf filter returned error/exception before headers sent:\n  - Generate 500 Internal Server Error (local reply)\n  - Track in stats\n\nIf filter error after headers sent:\n  - Reset stream with error code\n  - Connection may close if unrecoverable\n```\n\n**Stats**: `http.{filter_name}.error` counter\n\n### **Upstream Failures**\n\n#### Connection Pool Failure\n```\npool->newStream() \u2192 onPoolFailure()\n              \u2193\nReason: No healthy hosts, circuit breaker limit, connection refused\n              \u2193\nRouter checks route policy for retryability\n              \u2193\nIf retryable AND retries available:\n  - Increment retry counter\n  - Attempt alternate host\n  - Go back to pool acquisition\n              \u2193\nIf not retryable or retries exhausted:\n  - Generate 503 Service Unavailable\n  - Update stream info with CoreResponseFlag\n  - Lazy Initialization**\n- Codec created only when needed (on first data)\n- Connection pool created on first upstream request to cluster\n- Reduces startup latency and memory usage\n\n### **2. Thread-Local Everything**\n- Thread-local cluster instances\n- Thread-local connection pools per thread\n- No locks needed on hot paths\n\n### **3. Stop-and-Resume Pattern**\n- Filters return `StopIteration` to pause chain\n- Explicitly call `continueDecoding()/continueEncoding()` to resume\n- Enables arbitrary async operations within filter chain\n\n### **4. Bidirectional Filter Chain**\n- Same filters process both request and response\n- Encoder filters iterate in reverse of decoder order\n- Symmetric processing: filter A on request, filter A on response\n\n### **5. Callback-Based Async**\n- Connection pool returns via callbacks (not blocking)\n- Upstream response via `ResponseDecoder` callbacks\n- All I/O non-blocking, event-driven\n\n### **6. Circuit Breaking and Backpressure**\n- Limits prevent resource exhaustion (connections, requests)\n- Watermarks throttle slow consumers\n- Overload detection sheds load\n\n---\n\n## Summary\n\nThe HTTP Connection Manager in Envoy is a sophisticated, multi-layered architecture:\n\n1. **ConnectionManagerImpl** \u2014 Acts as the bridge between raw TCP bytes and logical HTTP streams, managing codec selection, stream lifecycle, and connection-level concerns.\n\n2. **FilterManager** \u2014 Executes a bidirectional filter chain where each filter can intercept and modify requests/responses, with clear control flow semantics for stopping and resuming iteration.\n\n3. **Router Filter** \u2014 The terminal decoder filter that selects upstream destinations, obtains connection pools, manages request lifetime, retry logic, and timeouts.\n\n4. **ClusterManager & Upstream Connectivity** \u2014 Abstracts upstream clusters and connections, providing load balancing, health checking, and connection pooling across multiple protocols and strategies.\n\nTogether, these components provide a composable, extensible platform for building HTTP proxies and API gateways with sophisticated traffic control, observability, and reliability features.\n\n---\n\n## Relevant Source Files Reference\n\n**Core HTTP Connection Manager:**\n- `source/common/http/conn_manager_impl.h` \u2014 Main ## Overview

The Envoy HTTP Connection Manager (HCM) is a sophisticated network filter that handles the complete lifecycle of HTTP requests flowing through a proxy. It sits at the intersection of downstream connections (client to Envoy) and upstream connections (Envoy to backend services), orchestrating the flow of data through multiple processing stages including codec translation, filter chains, routing, and request forwarding.

This document describes how the four major components—**ConnectionManagerImpl**, **FilterManager**, **Router Filter**, and **Cluster Manager**—work together to transform incoming HTTP requests into upstream communications and responses back to clients.
