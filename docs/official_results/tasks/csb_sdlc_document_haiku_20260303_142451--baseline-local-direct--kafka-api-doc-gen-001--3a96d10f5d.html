timeout\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(1000));\n```\n\n**What happens during `poll(timeout)`:**\n\n1. **Rebalancing (if needed):**\n   - If subscription changed or group needs rebalancing, triggers rebalance\n   - Calls `onPartitionsRevoked()` callback\n   - Calls `onPartitionsAssigned()` callback\n   - These callbacks can take significant time\n   - **poll() may block longer than timeout during rebalance**\n\n2. **Heartbeat (if group-managed):**\n   - Sends heartbeat if needed (managed internally)\n   - Receives group assignment updates\n\n3. **Fetching:**\n   - Sends fetch request to broker for assigned partitions\n   - Waits up to `timeout` for broker response\n   - Returns immediately if records available (up to `max.poll.records`)\n   - Returns empty if timeout expires with no records\n\n4. **Auto-commit (if enabled):**\n   - On some poll() calls (every `auto.commit.interval.ms`), commits offsets\n   - Happens asynchronously in background\n\n**Blocking Behavior:**\n\n```java\n// Scenario 1: Rebalance happening\nconsumer.poll(Duration.ofMillis(100));\n// May block for seconds if rebalance callbacks are slow!\n// Actual block time = rebalance callback duration + fetch timeout\n\n// Scenario 2: No records available\nconsumer.poll(Duration.ofMillis(100));\n// Blocks for full 100ms if no data\n\n// Scenario 3: Records immediately available\nconsumer.poll(Duration.ofMillis(100));\n// Returns immediately (may take <1ms)\n\n// Scenario 4: Rebalance + callback crashes\nconsumer.poll(Duration.ofMillis(100));\n// Throws CommitFailedException or other exception from callback\n```\n\n### Record Ordering Guarantees\n\n```java\n// Records are ordered per partition\nList<ConsumerRecord<String, String>> records = consumer.poll(...).records(partition);\n// records.get(0).offset() <= records.get(1).offset() <= ...\n\n// Records from different partitions have no ordering guarantee\nConsumerRecords<String, String> all = consumer.poll(...);\n// Partitions may be in any order: partition 2, then 0, then 1, etc.\n```\n\n### Poll Timeout Best Practices\n\n```java\n// Timeout should be set based on:\n// 1. Processing latency: poll should happen before max.poll.interval.ms expires\n// 2. Latency requirement: how quickly you want to process records\n// 3. Resource usage: longer timeout = less CPU spinning\n\n// Typical values:\nconsumer.poll(Duration.ofMillis(100));    // Low latency, frequent polling\nconsumer.poll(Duration.ofMillis(500));    // Balanced\nconsumer.poll(Duration.ofSeconds(1));     // High throughput, fewer CPU cycles\n\n// WRONG: Polling with very long timeout\nconsumer.poll(Duration.ofMinutes(1));  // May cause max.poll.interval.ms violation\n```\n\n### Empty Poll Handling\n\n```java\nwhile (running) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n    if (records.isEmpty()) {\n        System.out.println(\"No records, probably waiting for data\");\n        continue;\n    }\n\n    for (ConsumerRecord<String, String> record : records) {\n        process(record);\n    }\n}\n```\n\n---\n\n## Offset Management\n\n### Offset Concepts\n\n**Critical distinction between offset concepts:**\n\n| Concept | Definition | Usage |\n|---------|-----------|-------|\n| **Position** | Current offset the consumer will fetch next | Internal state, incremented on poll() |\n| **Committed Offset** | Last offset consumer agreed to process | Recovered on rebalance/restart |\n| **Beginning Offset** | Earliest offset available in partition | Seek target, retention boundary |\n| **End Offset** | Latest offset + 1 (high water mark) | Seek target, end-of-partition marker |\n| **Log Start Offset (LSO)** | Earliest offset after log compaction/retention | May differ from beginning if deleted |\n| **LSO (read_committed)** | Last stable offset for transactional reads | Messages after LSO are in-flight |\n\n**The Off-By-One Relationship:**\n\n```\nAssuming we read offset 41 and want to resume from there later:\n\nrecord.offset() = 41\n// To resume, we set:\nconsumer.seek(partition, 42)  // NOT 41! ## Overview

The `KafkaConsumer` is the primary API for consuming records from Apache Kafka topics. It provides:

- **Flexible subscription models**: automatic group-managed partition assignment or manual partition assignment
- **Offset management**: automatic or manual offset commit strategies with exactly-once and at-least-once semantics
- **Rebalancing support**: coordinated partition reassignment via `ConsumerRebalanceListener` callbacks
- **Flow control**: pause/resume per consumer instance
- **Position tracking**: separate concepts of current position, committed offset, and log start/end offsets

### Key Characteristics

- **Single-threaded event loop**: all network I/O and coordination happens in the poll() call
- **Lazy resource acquisition**: network connections created on first poll()
- **Group-based coordination**: consumer group membership maintained via heartbeat and generation tracking
- **Stateful position management**: maintains position per assigned partition independently of offset commits
