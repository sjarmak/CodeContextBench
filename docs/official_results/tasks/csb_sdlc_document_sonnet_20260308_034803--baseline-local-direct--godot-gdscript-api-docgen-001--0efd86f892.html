`docs/gdscript_vm_reference.md` has been created. Here's a summary of what it covers:

**VM Architecture**
- Register-based with a flat `Variant` array (not a push/pop stack machine)
- Computed-goto dispatch on GCC/Clang; `switch` on MSVC
- Execution entry point and call-depth limit (2048)

**Instruction Word Layout**
- Each instruction: one or more 32-bit `int` words from `GDScriptFunction::code`
- Address encoding: upper 8 bits = address space type (stack/constant/member), lower 24 bits = index
- Three reserved fixed stack slots: `self` (index 0), `class` (index 1), `nil` (index 2)
- Variable-arity instructions use a `LOAD_INSTRUCTION_ARGS` prefix block

**Complete Opcode Table** organized by category:
- Arithmetic & logic (`OPCODE_OPERATOR`, `OPCODE_OPERATOR_VALIDATED` with signature caching)
- Type tests (`is` operator — builtin/array/native/script variants)
- Property/key access — generic and validated fast-path variants
- Member variable access — instance props via `ClassDB`, static vars, GDScript members
- Assignment, type casts, construction (literals: array, typed array, dictionary)
- Function calls — dynamic, pre-bound (`MethodBind`), static, utility, self-base (super), builtin-type, async
- Lambda creation with variable capture
- Control flow — jumps, conditional jumps, default-argument dispatch
- Return with type checking variants
- Iteration — specialized per-type begin/step opcodes for all 20 iterable types
- Type-adjust, global storage, assert/breakpoint/line debug opcodes

**Function Call Convention** — argument placement at stack[3+], default argument handling, return by value, depth limiting

**Variable Access Patterns** — locals on stack, GDScript `var` via `ADDR_TYPE_MEMBER`, engine properties via `ClassDB`, constants, static variables

## VM Architecture

The GDScript VM is **register-based with a flat Variant array ("stack")**. It is not a push/pop stack machine; every instruction operand is an explicit address referencing a slot in one of three address spaces. The instruction pointer (`ip`) advances by a fixed number of words per instruction (the exact increment is hard-coded per opcode in the handler body).

### Dispatch mechanism

On GCC/Clang the VM uses **computed goto** (`goto *switch_table_ops[opcode]`) for the inner dispatch loop, which avoids the branch-predictor miss of a `switch`. On MSVC a `switch` statement is used instead. Both paths are guarded by the macros `OPCODE_SWITCH`, `DISPATCH_OPCODE`, etc. defined in `gdscript_vm.cpp:371-404`.
