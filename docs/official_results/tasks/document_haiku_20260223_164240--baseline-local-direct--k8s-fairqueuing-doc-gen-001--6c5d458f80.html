Virtual Time System\n\nThe fair queuing algorithm tracks two parallel timelines:\n\n**Real time:** Actual clock advancement (`lastRealTime`, `clock.Now()`)\n\n**Virtual time (R progress meter):** Advances based on active work:\n$$\\frac{dR}{dt} = \\min(\\text{seatsRequested}, \\text{ConcurrencyLimit}) / \\text{activeQueues}$$\n\n**Invariants:**\n- R strictly increases (or remains constant)\n- When R reaches `highR` threshold, epoch advance subtracts `rDecrement` from all R values to prevent overflow of uint64\n- Each queue has independent `nextDispatchR` start time for fairness calculation\n\n---\n\n## End-to-End Request Flow\n\n### Phase 1: Request Arrival and Enqueuing\n\n#### Entry Point: `StartRequest()` (lines 308\u2013356)\n\n```\nRequest arrives with:\n- workEstimate: estimated initial & final seats\n- hashValue: entropy for queue selection\n- flowDistinguisher: flow identifier for metrics\n- fsName: FlowSchema name\n```\n\n**Step 0: Zero-Queue Mode Check**\n- If `qCfg.DesiredNumQueues < 1`, skip queuing entirely\n- Check if request can fit within concurrency limit\n- If yes, call `dispatchSansQueueLocked()` and return immediately\n- If no, reject and return\n\n**Step 1: Shuffle Sharding and Enqueue**\n- Call `shuffleShardAndRejectOrEnqueueLocked()`:\n  1. Queue Saturation\n\n**Condition:** `totSeatsInUse >= ConcurrencyLimit` AND all queues at `QueueLengthLimit`\n\n**Behavior:** New requests are rejected with reason=\"queue-full\"\n\n**Impact:** Clients see rejection instead of queuing\n\n**Mitigation:**\n- Size `QueueLengthLimit` and `ConcurrencyLimit` appropriately\n- Use work estimators to avoid overestimating required seats\n- Monitor rejection metrics: `apiserver_flowcontrol_rejected_requests_total{reason=\"queue-full\"}`\n\n### 2. Virtual Time Overflow\n\n**Condition:** `currentR` approaches `MaxSeatSeconds` (uint64 max)\n\n**Detection:** `currentR >= highR` (= 3/4 of max)\n\n**Recovery:** `advanceEpoch()` subtracts `rDecrement` from all R values\n\n**Edge Case:** Sustained high load preventing epoch advance could theoretically overflow. Mitigated by:\n- Conservative threshold (`highR` = 75% of max)\n- Epoch advance during any `lockAndSyncTime()` call\n- Comprehensive logging of underflow/overflow conditions\n\n### 3. Request Timeout / Context Cancellation\n\n**Condition:** Client context cancels before request is dispatched\n\n**Decision Mechanism:** The Promise's `Get()` waits on both:\n- Dispatch signal channel\n- Context done channel\n\nIf context fires first: `Set(decisionCancel)`\n\n**Behavior:** Request returns without executing, counted as timeout-rejection\n\n**Tradeoff:** Allows clients to abandon waiting without garbage collection overhead\n\n### 4. Anti-Windup Oscillation\n\n**Problem:** Queues that cannot fully utilize their fair share accumulate negative virtual time drift, creating unfair advantage later.\n\n**Example:**\n- Queue A normally requests 2 seats but only gets 1 due to concurrency constraints\n- Virtual time for Queue A lags behind\n- When capacity opens, Queue A's requests have very early virtual finish times\n- Queue A floods with unfair dispatches\n\n**Solution:** `boundNextDispatchLocked()` prevents `nextDispatchR` from going before the oldest waiting request's `arrivalR`\n\n**Cost:** Occasional conservative dispatching order; benefit: prevents pathological fairness violations\n\n### 5. Seat Borrowing Constraints\n\n**Limit:** A single request cannot occupy more seats than `ConcurrencyLimit`\n\n**Handling:**\n```go\nif request.MaxSeats() > ConcurrencyLimit {\n    if totRequestsExecuting == 0 {\n        // Allow it to start (will hog the entire server)\n        // TODO: future borrowing framework\n    } else {\n        // Reject or wait for all to finish\n        return false\n    }\n}\n```\n\n**Design Gap:** No true borrowing yet; this is TODO in future KEPs\n\n### 6. Configuration Change During Operation\n\n**Scenario:** Desired queue count decreases while requests are executing\n\n**Behavior:**\n- New queues NOT created above desired count\n- Existing queues drain via attrition (removed when empty)\n- Ensures graceful migration without request loss\n\n---\n\n## Design Tradeoffs\n\n### 1. Virtual Time vs. Real Measurement\n\n**Decision:** Use virtual time (`R` progress meter) instead of real clock for fairness decisions\n\n**Rationale:**\n- **Decouples fairness from actual performance:** Slow or fast machines behave the same\n- **Enables predictable work accounting:** Requests have reproducible virtual costs\n- **Handles dynamic capacity:** Fairness adjusts as concurrency limit changes\n\n**Tradeoff:** Assumes work is proportional to estimated seats \u00d7 duration (breaks for compute-bound or I/O-bound skew)\n\n### 2. Shuffle Sharding vs. Global Queue\n\n**Decision:** Multiple queues with shuffle sharding instead of single global queue\n\n**Comparison:**\n\n| Aspect | Single Queue | Shuffle Sharding |\n|--------|------|----------|\n| **Lock contention** | High | Low (per queue) |\n| **Fair guarantee** | Perfect | Approximate (hand-size dependent) |\n| **Memory overhead** | Minimal | O(deckSize) |\n| **Pathological case** | Lock thrashing at high throughput | Occasional hand collisions |\n\n**Shuffle sharding benefit:** Reduces coordination cost while maintaining fairness within statistical bounds\n\n### 3. Promise-Based Dispatch\n\n**Decision:** Use non-blocking WriteOnce promise for dispatch decisions\n\n**vs. Alternative (channel-based):**\n- Promises allow context-cancellation to set decision without coordination\n- No explicit goroutine per request needed\n- Lock-free on the waiting side (only reads)\n\n**Tradeoff:** Slightly more memory per request; enables clean context semantics\n\n### 4. Two-Phase Execution (Initial + Final)\n\n**Decision:** Split request lifecycle into initial and final seat phases\n\n**Use Case:** Tracking when a request stops consuming resources vs. when it's fully released\n\n**Example:** Watch request that sends `initial` response but `additional latency` for event cleanup\n\n**Tradeoff:** Complexity in work estimation; enables more accurate resource accounting\n\n### 5. Anti-Windup Over Predictive\n\n**Decision:** Bound `nextDispatchR` retroactively instead of predictive weighting\n\n**Why not predictive?** Would require estimating which queues will underutilize (hard)\n\n**Why retroactive?** Simple, local check: if oldest request hasn't arrived yet in virtual time, reset to arrival\n\n**Tradeoff:** Occasionally serializes unfairly; prevents pathological cases\n\n---\n\n## Extension Points and Customization\n\n### 1. Performance Characteristics\n\n#### Lock Contention\n\n**Hotspot:** `queueSet.lock` serializes all decisions on a single priority level\n\n**Mitigation:**\n- Multiple queues reduce per-queue lock hold time\n- Shuffle sharding distributes request arrival burden\n\n**Bottleneck risk:** Single CPU core bottleneck at extreme throughput (>50k req/s with few queues)\n\n#### Virtual Time Advancement\n\n**Cost:** O(1) per `lockAndSyncTime()` call\n\n**Optimization:** Only updates progress meter, not queue state\n\n**Scaling:** Virtual time ratio calculation is O(queues), acceptable since usually \u2264 128 queues\n\n#### Dispatch Selection\n\n**Cost:** O(queues \u00d7 handSize) per dispatch in worst case\n\n**Typical:** O(queues) with early termination when found best queue\n\n**Impact:** Dispatch is the most expensive synchronized operation\n\n#### Epoch Advance\n\n**Frequency:** Every ~10^10 seat-seconds of work (very rare)\n\n**Cost:** O(queues \u00d7 waiting requests) to update all R values\n\n**Mitigation:** Walk-through happens with lock held; log if slow\n\n### 2. # Kubernetes APF QueueSet: Algorithmic Deep-Dive

## Overview

The Kubernetes Advanced Priority and Fairness (APF) QueueSet subsystem implements a distributed fair queuing algorithm for managing API server request concurrency. It enforces prioritized, fair allocation of execution capacity across competing flows of requests while minimizing tail latencies through sophisticated scheduling based on virtual time.

**Key Purpose:** QueueSet transforms a single server concurrency limit into multiple independent priority levels, each managing their requests through fair queuing that separates requests into queues via shuffle sharding and dispatches them according to their virtual finish time in a weighted fair queuing system.

---

## Subsystem Boundaries and Dependencies

### Scope

The QueueSet subsystem is **not** responsible for:
- Classifying requests into priority levels (handled by the APF controller via FlowSchema matching)
- Work estimation for request weight (handled by dedicated work estimators)
- Metrics collection (delegated to metrics subsystem)
- Configuration management (handled by APF controller)

The QueueSet **is** responsible for:
- Assigning requests to specific queues via shuffle sharding
- Queuing requests when concurrency limit is reached
- Dispatching requests fairly based on virtual time
- Tracking execution phases and managing seat allocation
- Adapting to dynamic configuration changes
