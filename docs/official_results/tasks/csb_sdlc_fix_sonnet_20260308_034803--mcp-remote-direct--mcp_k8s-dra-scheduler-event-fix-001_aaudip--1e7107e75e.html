# big-code-k8s-bug-001: Missing ResourceSlice Event Handler in Kubernetes Scheduler

## Task

Investigate a bug in the Kubernetes scheduler where pods requesting Dynamic Resource Allocation (DRA) devices become permanently stuck in an Unschedulable state when the DRA driver starts after the pod is created. Trace the execution path through the scheduler's event-driven re-queuing architecture to identify why ResourceSlice creation/update events are silently dropped.

## Context

- **Repository**: github.com/sg-evals/kubernetes--v1.30.0 (mirror of kubernetes/kubernetes) (Go, ~3.5M LOC)
- **Category**: Bug Investigation
- **Difficulty**: hard
- **Entry Point**: `pkg/scheduler/schedule_one.go` — `ScheduleOne()` method

## Symptom

When a pod requests a DRA device and the DRA driver starts *after* the pod is created, the pod becomes permanently stuck in `Pending` with reason `Unschedulable`. The scheduler logs `"cannot allocate all claims"` because no ResourceSlices exist at scheduling time. After the driver starts and publishes ResourceSlice objects, the pod should be rescheduled — but it never is.

The bug is a race condition: if the driver starts first, scheduling succeeds immediately. But if the pod arrives before the driver, the scheduler never retries because ResourceSlice events are not wired into the scheduling queue's re-evaluation mechanism.

## Bug Summary

The root cause is that `ResourceSlice` — the Kubernetes object that DRA drivers publish to advertise available device resources — is **completely absent from the scheduler's event-driven re-queuing architecture**. There are three missing pieces:

### 1. Missing GVK constant (`framework/types.go:96`)
`ResourceSlice` is not defined alongside the other DRA GVK constants (`ResourceClaim`, `ResourceClass`, `ResourceClaimParameters`, `ResourceClassParameters`, `PodSchedulingContext`).

### 2. Missing `EventsToRegister()` entry (`dynamicresources.go:386-412`)
The DRA plugin's `EventsToRegister()` never declares interest in `ResourceSlice` Add/Update events. So the scheduling queue has no record that pods rejected by this plugin should be retried when ResourceSlices appear.

### 3. Missing informer handler (`eventhandlers.go:395-543`)
`addAllEventHandlers()` has no `case framework.ResourceSlice:` branch. Even if the GVK were registered, the informer event would be silently dropped (the `default:` branch requires dot-separated GVK strings; plain `"ResourceSlice"` would fail the `strings.Count < 2` check).

### The race condition
- **Driver starts first** → ResourceSlices exist when `PreFilter` calls `newResourceModel()` → scheduling succeeds immediately ✓
- **Pod arrives first** → `newResourceModel()` returns empty resources → pod marked Unschedulable → pod enters unschedulable queue → driver starts and creates ResourceSlices → **no event wired to trigger retry** → pod stuck forever ✗
