Its primary responsibilities are:\n\n- **Request Routing**: Dispatch incoming HTTP requests to appropriate handlers based on URL patterns and API resource metadata\n- **Request Processing Pipeline**: Chain filters that handle cross-cutting concerns (authentication, authorization, audit, admission control)\n- **API Resource Management**: Register and expose API groups, versions, and resources through a discovery mechanism\n- **Object Storage Abstraction**: Provide a unified interface to persistent storage backends (etcd) with support for serialization, versioning, and transformations\n- **Request Context Enrichment**: Attach authentication/authorization information, metadata, and trace context to requests flowing through the system\n- **Response Handling**: Serialize objects according to negotiated content types and handle errors consistently\n\n### Subsystem Boundaries\n\n**What the apiserver library handles:**\n- HTTP request/response plumbing\n- Authentication and authorization enforcement\n- Admission control (validation and mutation)\n- Audit logging\n- OpenAPI/Swagger documentation generation\n- Storage interface for CRUD operations\n- Common error handling and response formatting\n\n**What it delegates:**\n- **Authentication mechanism implementation**: Delegated to pluggable authenticators (clients must provide `authenticator.Authenticator`)\n- **Authorization decision logic**: Delegated to pluggable authorizers (clients must provide `authorizer.Authorizer`)\n- **Persistent storage backend**: Delegated to storage implementations (etcd3, testing backends)\n- **Admission plugin implementation**: Delegated to admission controllers (clients implement `admission.MutationInterface` or `admission.ValidationInterface`)\n- **Resource business logic**: Delegated to REST storage handlers\n- **Binary startup and configuration**: Delegated to consuming binaries (kube-apiserver, custom API servers)\n\n### Upstream Dependencies\n\nThe apiserver library depends on:\n- **k8s.io/apimachinery**: Core runtime, serialization, object marshaling/unmarshaling\n- **k8s.io/client-go**: For loopback client configuration and informers\n- **k8s.io/component-base**: Shared components like logging, metrics, feature gates, tracing\n- **go-restful**: HTTP routing and web service registration\n- **etcd client libraries**: For storage backend interaction\n- **kube-openapi**: OpenAPI schema and documentation generation\n\n### Downstream Dependents\n\nThe library is consumed by:\n- **kube-apiserver**: Main Kubernetes API server\n- **aggregation-apiserver**: API server for aggregated APIs\n- **Custom API servers**: User-built servers using the library (sample-apiserver, etc.)\n- **apiextensions-apiserver**: Handles CustomResourceDefinitions\n\n---\n\n## 2. Key Components and Responsibility Division\n\n### 2.1 Server Core (`pkg/server/`)\n\n**GenericAPIServer** (`genericapiserver.go`): The central orchestrator that manages the complete server lifecycle.\n\n**Responsibilities:**\n- Holds server configuration (authentication, authorization, admission, storage)\n- Manages the HTTP handler chain and handler director\n- Coordinates API group installation\n- Manages lifecycle hooks (PreStartHooks, PostStartHooks)\n- Coordinates graceful shutdown\n- Tracks health status (healthz, livez, readyz checks)\n\n**Config** (`config.go`): The comprehensive configuration struct that accumulates all settings needed to construct a working API server.\n\n**Responsibilities:**\n- Captures authentication/authorization configuration\n- Stores admission control, audit backend, and storage transformers\n- Contains request/response tuning parameters (timeouts, body size limits, in-flight limits)\n- Defines the handler chain builder function\n- Stores feature gates, serializers, and OpenAPI configuration\n\n**APIServerHandler** (`handler.go`): Routes incoming requests to appropriate handlers.\n\n**Responsibilities:**\n- Maintains GoRestful container for REST API registration\n- Maintains NonGoRestfulMux for non-standard endpoints\n- Implements director pattern to intelligently route requests\n- Falls through to appropriate handler based on registered web services\n\n### 2.2 Request Filtering Pipeline (`pkg/server/filters/`, `pkg/endpoints/filters/`)\n\nThe handler chain applies a series of filters that process requests before they reach API handlers. Filters are applied **in order**, and each wraps the next:\n\n**Critical Filters (in application order):**\n1. **AuditInit** (`filters/audit_init.go`): Initialize audit logging context\n2. **PanicRecovery** (`filters/panic_recovery.go`): Recover from panics and prevent crashes\n3. **RequestInfo** (`filters/request_info.go`): Parse URL and attach RequestInfo to context (resource, verb, namespace, etc.)\n4. **MuxAndDiscoveryComplete** (`filters/mux_discovery_complete.go`): Signal when mux is ready\n5. **RequestReceivedTimestamp** (`filters/request_received_time.go`): Mark when request was received\n6. **HTTPLogging** (`filters/http_logging.go`): Log HTTP requests at debug level\n7. **LatencyTrackers** (`filters/latency_tracker.go`): Track filter execution latencies\n8. **ProbabilisticGoaway** (HTTP/2): Send periodic GOAWAY to rebalance load\n9. **HSTS** (`filters/hsts.go`): Add HSTS security headers\n10. **RetryAfter** (`filters/retry_after.go`): Signal shutdown via Retry-After header\n11. **CacheControl** (`filters/cachecontrol.go`): Add cache control headers\n12. **TimeoutForNonLongRunningRequests** (`filters/timeout.go`): Apply request timeout\n13. **RequestDeadline** (`filters/request_deadline.go`): Track and enforce request deadline\n14. **WaitGroup** (`filters/wait_group.go`): Track request completion for shutdown\n15. **WatchTerminationDuringShutdown** (`filters/watch_termination.go`): Handle watch during shutdown\n16. **Authentication** (`filters/authentication.go`): Validate user credentials; on failure, invoke unauthorized handler\n17. **Audit** (`filters/audit.go`): Log request/response for audit trail\n18. **Impersonation** (`filters/impersonation/`): Handle user/group impersonation\n19. **PriorityAndFairness** (`server/filters/`): Apply flow control and priority/fairness queuing\n20. **Authorization** (`filters/authorization.go`): Verify user has permission for action\n\nEach filter can examine/modify the request and context, and can short-circuit by directly writing a response (e.g., auth failure).\n\n### 2.3 Endpoint Installation and Handler Registration (`pkg/endpoints/`)\n\n**APIInstaller** (`installer.go`): Registers API resource handlers into the GoRestful container.\n\n**Responsibilities:**\n- Introspects REST storage objects to discover verbs (GET, POST, PUT, PATCH, DELETE, WATCH)\n- Creates gorestful routes for each verb\n- Constructs RequestScope with serialization, validation, and storage configuration\n- Associates handlers (get, create, update, delete, watch, patch, etc.) with routes\n- Generates OpenAPI documentation for discovered resources\n\n**APIGroupVersion** (`groupversion.go`): Represents a single API group version (e.g., \"apps/v1\").\n\n**Responsibilities:**\n- Holds VersionedResourcesStorageMap mapping resource names to REST storage implementations\n- Provides REST storage interfaces for each resource\n- Collaborates with APIInstaller to expose resources as HTTP endpoints\n\n### 2.4 Request Handlers (`pkg/endpoints/handlers/`)\n\nThese handlers implement CRUD operations on resources. Each handler operates within a **RequestScope** which provides:\n- Serializer: For encoding/decoding objects\n- Storage: The backend REST storage\n- Authorizer: For authorization checks\n- Admission: For validation/mutation\n- FieldManager: For server-side apply and managed fields tracking\n\n**Key Handlers:**\n- **create.go** (`CreateResource`): Handle POST requests; calls admission, then storage.Create\n- **get.go** (`GetResource`): Handle GET requests; calls storage.Get, applies authorization checks\n- **list.go** (via rest.go): Handle LIST requests; applies filtering, pagination\n- **update.go** (`UpdateResource`): Handle PUT requests; validates, mutates, then storage.Update\n- **patch.go** (`PatchResource`): Handle PATCH requests; applies JSONPatch or strategic merge patch, then storage.Update\n- **delete.go** (`DeleteResource`): Handle DELETE requests; calls storage.Delete with cascading delete strategy\n- **watch.go** (`WatchResource`): Handle WATCH requests; streams server-sent updates from storage\n\n**Shared Responsibility Pattern:**\nEach handler follows this pattern:\n1. Decode the incoming request body\n2. Apply defaults (if applicable)\n3. Run **mutating admission** (allows modification)\n4. Validate the object\n5. Run **validating admission** (no modification allowed)\n6. Call **storage** interface\n7. Encode and write response\n\n### 2.5 Admission Control (`pkg/admission/`)\n\nAdmission control is a two-phase system that allows pluggable policies to validate and mutate requests:\n\n**Interfaces:**\n- **MutationInterface** (`interfaces.go`): Can examine and modify objects\n- **ValidationInterface** (`interfaces.go`): Can only validate; no mutations allowed\n\n**Phases:**\n1. **Mutating Admission** (before validation): Plugins can modify the object (e.g., DefaultServiceAccount, LimitRanger)\n2. **Validating Admission** (after mutation): Plugins validate the final object (e.g., ResourceQuota, PodSecurityPolicy)\n\n**Attributes Interface** (`attributes.go`): Provides request context to admission plugins:\n- Operation (CREATE, UPDATE, DELETE, CONNECT)\n- Resource identity (namespace, name, resource, kind)\n- User information\n- Object and old object\n- Dry-run flag\n- Annotations capability (allows plugins to leave notes on the audit trail)\n\n**Chain** (`chain.go`): Chains multiple admission plugins into a single interface; short-circuits on first error.\n\n### 2.6 Authentication and Authorization\n\n**Authentication** (`pkg/authentication/`):\n- **Authenticator interface**: Examines request and produces `user.Info` containing username, uid, groups, and extra attributes\n- **Multiple authenticator chains** can be composed (union authenticator)\n- Pluggable implementations: X.509 certificates, bearer tokens, basic auth, service account tokens, OIDC, etc.\n\n**Authorization** (`pkg/authorization/`):\n- **Authorizer interface**: Examines request + user and returns Allow/Deny/NoOpinion\n- **Pluggable implementations**: RBAC (role-based), ABAC (attribute-based), webhook, node authorization, etc.\n- **RuleResolver** interface: Lists which rules/roles apply to a user in a namespace (used for discovery)\n\n### 2.7 Storage (`pkg/storage/`)\n\n**Storage Interfaces** (`storage/interfaces.go`):\n- **Interface** (base): New(), Destroy() methods\n- **Getter**: Get(ctx, key) \u2192 object\n- **Lister**: List(ctx, opts) \u2192 objects\n- **Creator**: Create(ctx, key, obj, ttl) \u2192 object\n- **Updater**: Update(ctx, key, obj, fn) \u2192 object\n- **Deleter**: Delete(ctx, key, obj) \u2192 deleted object\n- **Watcher**: Watch(ctx, opts) \u2192 events channel\n\n**Storage Backend** (`etcd3/`): Implements the interfaces using etcd3 as persistent storage.\n\n**Cacher** (`cacher/`): Wraps storage with in-memory list/watch caching for performance.\n\n**Responsibilities:**\n- CRUD operations on versioned objects\n- Watch streaming (sends updates as objects change)\n- Atomic conditional updates (CAS)\n- TTL/deletion grace periods\n- Encryption transformers (for secrets)\n- Resource version tracking\n\n### 2.8 Request Context (`pkg/endpoints/request/`)\n\nThe request context carries information through the pipeline:\n\n**RequestInfo** (`requestinfo.go`): Parsed from URL; contains:\n- Verb (get, list, create, update, delete, watch, patch, proxy, connect)\n- Resource, subresource, name, namespace\n- IsResourceRequest (distinguishes API resource requests from system endpoints)\n\n**Request Context Keys** (`context.go`):\n- User information (set by authentication filter)\n- Namespace (set by RequestInfo filter)\n\n**Decorators** that can be attached to context:\n- ReceivedTime: When request arrived at server\n- RequestDeadline: Deadline for request execution\n- WebhookDuration: Time spent in webhooks\n\n### 2.9 Discovery and OpenAPI (`pkg/endpoints/discovery/`)\n\n**Discovery Handlers** (`discovery/`):\n- **Root discovery** (`discovery/root.go`): Lists API groups at `/apis`\n- **Group version discovery** (`discovery/group.go`): Lists resources in a version at `/apis/group/v1`\n- **Legacy discovery** (`discovery/legacy.go`): Lists core API v1 resources\n- **Aggregated discovery** (`discovery/aggregated/`): Efficient discovery via gzip and ETags\n\n**OpenAPI** (`endpoints/openapi.go`):\n- Generates OpenAPI v2 spec from registered resources\n- Generates OpenAPI v3 spec\n- Exposes at `/openapi/v2`, `/openapi/v3`\n\n---\n\n## 3. Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### Authentication Failure\n- **Symptom**: 401 Unauthorized response\n- **Root causes**: Invalid token, expired certificate, unknown authenticator\n- **Impact**: User cannot access API\n- **Recovery**: User obtains new credentials\n- **Design choice**: Fail-closed for security; no request proceeds without authentication\n\n#### Authorization Failure\n- **Symptom**: 403 Forbidden response\n- **Root causes**: RBAC rule doesn't permit action, webhook denies request\n- **Impact**: User cannot perform action\n- **Recovery**: Admin updates RBAC rules\n- **Design choice**: Fails fast before admission plugins run; prevents wasted work\n\n#### Admission Webhook Timeout\n- **Symptom**: Request eventually fails (default 30s timeout); webhook never responds\n- **Root causes**: Webhook server crash, network partition, slow webhook\n- **Impact**: Request is rejected and user receives 500 error\n- **Recovery**: Fix webhook, retry request\n- **Design choice**: Webhooks have timeouts to prevent unbounded request latency\n- **Evidence**: `handlers/create.go:95` - admission is called within request timeout\n\n#### Storage Failure (etcd unavailable)\n- **Symptom**: 500 Internal Server Error response; readiness check fails\n- **Root causes**: etcd cluster down, network partition, disk full\n- **Impact**: Cannot persist data; all mutations fail\n- **Recovery**: Restore etcd cluster\n- **Mitigations**: Caching layer provides read availability; watch buffering masks brief outages\n- **Evidence**: `storage/cacher/cacher.go` - cacher wraps storage for high availability\n\n#### Request Timeout\n- **Symptom**: Request context exceeds deadline; connection reset\n- **Root causes**: Long-running operation, slow storage, overloaded system\n- **Impact**: Client receives error or connection closes\n- **Recovery**: Automatic retry (if client implements it)\n- **Design choice**: Non-long-running requests get strict timeout; long-running (watch) get graceful timeout\n- **Evidence**: `config.go:228` - RequestTimeout parameter; `filters/timeout.go` - timeout enforcement\n\n#### Flow Control Rejection\n- **Symptom**: 429 Too Many Requests response\n- **Root causes**: Request queue full, priority level saturated\n- **Impact**: Client must retry with backoff\n- **Recovery**: Automatic retry by client\n- **Design choice**: Protects system from overload; fair queuing prevents thundering herd\n- **Evidence**: `config.go:254-259` - flow control configuration; `server/filters/priorityandfairness.go`\n\n#### Panic in Handler\n- **Symptom**: 500 Internal Server Error response\n- **Root causes**: Bug in handler code, admission plugin, storage, or business logic\n- **Impact**: Request fails but server continues running\n- **Recovery**: Fix bug, redeploy\n- **Design choice**: Panic recovery prevents cascading failures\n- **Evidence**: `filters/panic_recovery.go` - wraps handler chain\n\n### 4.2 Design Limits\n\n#### MaxRequestBodyBytes\n- **Parameter**: `config.go:253`\n- **Default**: Typically 1.5MB for kube-apiserver\n- **Purpose**: Prevent unbounded memory usage from large request bodies\n- **Impact**: Large objects (e.g., big ConfigMaps) can hit this limit\n- **Mitigation**: Clients can split large objects; etcd has its own 1.5GB limit\n\n#### MaxRequestsInFlight / MaxMutatingRequestsInFlight\n- **Parameters**: `config.go:256-259`\n- **Default**: 200 read, 100 write for kube-apiserver\n- **Purpose**: Bound concurrency to prevent resource exhaustion\n- **Impact**: Beyond limit, requests queue and receive 429 if queue full\n- **Modern alternative**: Priority and Fairness (`config.go:140`) is more sophisticated\n\n#### RequestTimeout / MinRequestTimeout\n- **Parameters**: `config.go:228-231`\n- **Default**: 1 minute read, 1 hour watch\n- **Purpose**: Prevent unbounded connections\n- **Tradeoff**: Too short and legitimate operations fail; too long and broken clients hold connections\n- **Evidence**: `filters/timeout.go` - timeout is applied per RequestScope\n\n#### Storage Watch Buffer\n- **Location**: `storage/cacher/cacher.go`\n- **Default**: Small bounded buffer for watch events\n- **Purpose**: Decouple event production from consumption\n- **Tradeoff**: Clients watching many objects can fall behind; Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### Authentication Failure\n- **Symptom**: 401 Unauthorized response\n- **Root causes**: Invalid token, expired certificate, unknown authenticator\n- **Impact**: User cannot access API\n- **Recovery**: User obtains new credentials\n- **Design choice**: Fail-closed for security; no request proceeds without authentication\n\n#### Authorization Failure\n- **Symptom**: 403 Forbidden response\n- **Root causes**: RBAC rule doesn't permit action, webhook denies request\n- **Impact**: User cannot perform action\n- **Recovery**: Admin updates RBAC rules\n- **Design choice**: Fails fast before admission plugins run; prevents wasted work\n\n#### Admission Webhook Timeout\n- **Symptom**: Request eventually fails (default 30s timeout); webhook never responds\n- **Root causes**: Webhook server crash, network partition, slow webhook\n- **Impact**: Request is rejected and user receives 500 error\n- **Recovery**: Fix webhook, retry request\n- **Design choice**: Webhooks have timeouts to prevent unbounded request latency\n- **Evidence**: `handlers/create.go:95` - admission is called within request timeout\n\n#### Storage Failure (etcd unavailable)\n- **Symptom**: 500 Internal Server Error response; readiness check fails\n- **Root causes**: etcd cluster down, network partition, disk full\n- **Impact**: Cannot persist data; all mutations fail\n- **Recovery**: Restore etcd cluster\n- **Mitigations**: Caching layer provides read availability; watch buffering masks brief outages\n- **Evidence**: `storage/cacher/cacher.go` - cacher wraps storage for high availability\n\n#### Request Timeout\n- **Symptom**: Request context exceeds deadline; connection reset\n- **Root causes**: Long-running operation, slow storage, overloaded system\n- **Impact**: Client receives error or connection closes\n- **Recovery**: Automatic retry (if client implements it)\n- **Design choice**: Non-long-running requests get strict timeout; long-running (watch) get graceful timeout\n- **Evidence**: `config.go:228` - RequestTimeout parameter; `filters/timeout.go` - timeout enforcement\n\n#### Flow Control Rejection\n- **Symptom**: 429 Too Many Requests response\n- **Root causes**: Request queue full, priority level saturated\n- **Impact**: Client must retry with backoff\n- **Recovery**: Automatic retry by client\n- **Design choice**: Protects system from overload; fair queuing prevents thundering herd\n- **Evidence**: `config.go:254-259` - flow control configuration; `server/filters/priorityandfairness.go`\n\n#### Panic in Handler\n- **Symptom**: 500 Internal Server Error response\n- **Root causes**: Bug in handler code, admission plugin, storage, or business logic\n- **Impact**: Request fails but server continues running\n- **Recovery**: Fix bug, redeploy\n- **Design choice**: Panic recovery prevents cascading failures\n- **Evidence**: `filters/panic_recovery.go` - wraps handler chain\n\n### 4.2 Design Limits\n\n#### MaxRequestBodyBytes\n- **Parameter**: `config.go:253`\n- **Default**: Typically 1.5MB for kube-apiserver\n- **Purpose**: Prevent unbounded memory usage from large request bodies\n- **Impact**: Large objects (e.g., big ConfigMaps) can hit this limit\n- **Mitigation**: Clients can split large objects; etcd has its own 1.5GB limit\n\n#### MaxRequestsInFlight / MaxMutatingRequestsInFlight\n- **Parameters**: `config.go:256-259`\n- **Default**: 200 read, 100 write for kube-apiserver\n- **Purpose**: Bound concurrency to prevent resource exhaustion\n- **Impact**: Beyond limit, requests queue and receive 429 if queue full\n- **Modern alternative**: Priority and Fairness (`config.go:140`) is more sophisticated\n\n#### RequestTimeout / MinRequestTimeout\n- **Parameters**: `config.go:228-231`\n- **Default**: 1 minute read, 1 hour watch\n- **Purpose**: Prevent unbounded connections\n- **Tradeoff**: Too short and legitimate operations fail; too long and broken clients hold connections\n- **Evidence**: `filters/timeout.go` - timeout is applied per RequestScope\n\n#### Storage Watch Buffer\n- **Location**: `storage/cacher/cacher.go`\n- **Default**: Small bounded buffer for watch events\n- **Purpose**: Decouple event production from consumption\n- **Tradeoff**: Clients watching many objects can fall behind; buffer overflow causes watch restart\n\n### 4.3 Design Tradeoffs\n\n#### GoRestful vs. Custom Routing\n- **Choice**: Uses go-restful library for REST endpoint registration\n- **Advantages**: Structured OpenAPI documentation, parameter marshaling, content negotiation\n- **Disadvantages**: Added latency, less flexible than custom routing\n- **Alternative considered**: Custom HTTP mux\n- **Evidence**: `handler.go:73-100` - NewAPIServerHandler uses restful.Container\n\n#### Request Filtering Chain vs. Middleware\n- **Choice**: Sequential filter chain (each filter wraps next)\n- **Advantages**: Order is explicit and debuggable;
