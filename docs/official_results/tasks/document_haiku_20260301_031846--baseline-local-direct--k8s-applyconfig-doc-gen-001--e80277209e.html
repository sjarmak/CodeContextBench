Conflict Handling and Resolution Strategies\n\n### Conflict Anatomy\n\nA conflict occurs when:\n1. The current apply request (manager A) tries to modify field F\n2. Field F is currently owned by a different manager (manager B)\n3. Manager A does not have `force=true`\n\nExample:\n```yaml\n# Current object (after kubectl and operator both applied)\nmetadata:\n  managedFields:\n  - manager: kubectl\n    fields:\n      f:spec:\n        f:replicas: {}\n  - manager: operator\n    fields:\n      f:spec:\n        f:image: {}\n\n# User runs: kubectl apply --server-side\n# New patch sets: spec.replicas=5 (kubectl manager)\n# But operator manager owns spec.replicas \u2192 CONFLICT\n```\n\n### Conflict Detection\n\n**File: `/workspace/staging/src/k8s.io/apimachinery/pkg/util/managedfields/internal/conflict.go`**\n\n```go\nfunc NewConflictError(conflicts merge.Conflicts) *errors.StatusError {\n    causes := []metav1.StatusCause{}\n    for _, conflict := range conflicts {\n        causes = append(causes, metav1.StatusCause{\n            Type:    metav1.CauseTypeFieldManagerConflict,\n            Message: fmt.Sprintf(\"conflict with %v\", printManager(conflict.Manager)),\n            Field:   conflict.Path.String(),\n        })\n    }\n    return errors.NewApplyConflict(causes, getConflictMessage(conflicts))\n}\n```\n\nReturns HTTP 409 with details of conflicting fields and which manager owns each.\n\n### Resolution Strategies\n\n#### Strategy 1: Fail (Default)\n\n```\nRequest: apply with fieldManager=\"kubectl\" (no force)\nResponse: HTTP 409 Conflict\nError: \"Apply failed with 1 conflict: conflict with manager 'operator': spec.replicas\"\n```\n\nUser must either:\n- Manually remove the conflicting manager from object (rare)\n- Use `--force-conflicts` / `force=true` on the new apply request\n\n#### Strategy 2: Force Ownership Transfer\n\n```bash\nkubectl apply --server-side --force-conflicts\n```\n\nSets `force=true` in PatchOptions:\n\n```go\n// Overrides conflict: new manager takes ownership\nMerge(..., force=true) \u2192\n    Removes old manager from managedFields for conflicting fields\n    Adds new manager to managedFields for those fields\n    Result: HTTP 200, object updated\n```\n\n**Risks of force:**\n- Silently removes another tool's field tracking\n- Can cause thrashing if two tools continuously force-apply conflicting changes\n- Should be used only when you're certain you're the intended owner\n\n#### Strategy 3: Three-Way Merge Prevention\n\nBest practice: **Design ownership boundaries.**\n\n- **kubectl** manages user-specified fields (image, replicas, labels)\n- **operator** manages system fields (status, internal configuration)\n\nUse different field sets to avoid conflicts:\n```yaml\n# kubectl manages these:\nmetadata:\n  labels:\n    app: myapp\nspec:\n  replicas: 3\n  template:\n    spec:\n      containers:\n      - name: app\n        image: myapp:1.0\n\n# operator manages these (via separate apply):\nstatus:\n  conditions: [...]\nspec:\n  internalConfig: {...}\n```\n\nBoth can apply without conflict because they never both claim the same field.\n\n#### Strategy 4: Re-Apply After Conflict\n\nSome workflows retry with `force=true`:\n\n```go\nfor attempt := 0; attempt < maxRetries; attempt++ {\n    resp, err := applyWithForce(false) // force=false\n    if err != nil && isConflict(err) && attempt < maxRetries-1 {\n        applyWithForce(true) // Try with force\n    }\n}\n```\n\n### Conflict Prevention in Practice\n\n#### Test Case: `TestUpdateApplyConflict`\n\n**File: Failure Modes, Limits, and Design Tradeoffs\n\n### Failure Mode 1: Corrupted managedFields\n\n**Cause:** Buggy mutating admission webhook modifies managedFields incorrectly\n\n**Detection:**\n```go\nif err := managedfields.ValidateManagedFields(encodedManagedFields); err != nil {\n    // Admission corrupted managedFields\n}\n```\n\n**Recovery:**\n```go\n// NewManagedFieldsValidatingAdmissionController\nif err := managedfields.ValidateManagedFields(managedFieldsAfterAdmission); err != nil {\n    objectMeta.SetManagedFields(managedFieldsBeforeAdmission)  // Reset\n    warning.AddWarning(ctx, \"\", \"managedFields was in an invalid state after admission\")\n}\n```\n\n**User Impact:** Fields marked corrupted are reset; user sees warning but apply succeeds.\n\n### Failure Mode 2: Uncontrolled managedFields Growth\n\n**Cause:** Too many different field managers (e.g., flapping between tools)\n\n**Example:** 100 different CI systems apply the same resource with different manager names\n\n**Limit:** `DefaultMaxUpdateManagers = 10`\n\n**Recovery:**\n```go\n// CapManagersManager merges oldest entries\nif len(managers) > 10 {\n    // Merge manager #1 into manager #2\n    // Consolidates ownership records\n}\n```\n\n**Consequence:** Very old manager records are lost; field ownership history truncated\n\n**Design tradeoff:**\n- Could make this configurable, but default 10 is conservative\n- Prevents unbounded etcd growth but loses granular historical tracking\n- Acceptable: most clusters have ~3-5 managers per resource\n\n### Failure Mode 3: Patch Too Large (Objects)\n\n**Cause:** Apply configuration + live object + managedFields exceeds storage limit\n\n**Limit:** Typically ~1 MB per object (etcd limit)\n\n**Recovery:**\n```go\nif isTooLargeError(err) && p.patchType != types.ApplyYAMLPatchType && ... {\n    // Clear managedFields and retry\n    // (Not done for SSA to preserve field tracking)\n    p.updatedObjectInfo = rest.DefaultUpdatedObjectInfo(nil,\n        p.applyPatch,\n        p.applyAdmission,\n        func(...) (runtime.Object, error) {\n            accessor, _ := meta.Accessor(obj)\n            accessor.SetManagedFields(nil)  // Clear managedFields, try again\n            return obj, nil\n        })\n    result, err = requestFunc()\n}\n```\n\n**User Impact:** Auto-retry after clearing managedFields (non-SSA only). For SSA, the request fails.\n\n**Design tradeoff:**\n- SSA explicitly does NOT clear managedFields because field tracking is critical to correctness\n- Client must reduce object complexity or split into multiple objects\n\n### Failure Mode 4: Resource Version Conflicts (Optimistic Locking)\n\n**Cause:** Two concurrent requests modify the same resource\n\n**Example:**\n1. Request A reads object (resourceVersion=100)\n2. Request B reads object (resourceVersion=100)\n3. Request B writes (resourceVersion=101)\n4. Request A tries to write with resourceVersion=100 \u2192 CONFLICT\n\n**Recovery:**\n```go\n// In storage layer: GuaranteedUpdate loop\nfor {\n    obj := storage.Get(key)  // Get latest\n    transformedObj := transform(obj)\n    if !storage.Update(key, transformedObj, obj.resourceVersion) {\n        continue  // Retry with latest\n    }\n    break\n}\n```\n\n**User Impact:** Transparent retry. Apply succeeds after a few milliseconds (unless load is extreme).\n\n### Failure Mode 5: Type Converter Cache Stale\n\n**Cause:** OpenAPI schema changes after FieldManager is initialized\n\n**Cause:** Upgrade cluster, update CRD schema\n\n**Impact:** Three-way merge uses outdated field merge strategies\n\n**Prevention:**\n- FieldManager is initialized per API server process\n- Schema changes require API server restart to pick up\n- CRD changes are watched, but integration is complex\n\n### Design Tradeoff 1: Ownership Model\n\n**Choice:** Field-level ownership via managedFields\n\n**Alternatives:**\n- Resource-level: Track which manager last touched the object (simpler, less precise)\n- Operation-level: Track every operation's conflicts (more history, unbounded storage)\n\n**Why fields?**\n- Allows multiple managers to control different fields of same object\n- Kubectl (labels, spec) + operator (status) on same Pod without conflict\n- Matches mental model: \"I own this part of the spec\"\n\n**Cost:**\n- Larger managedFields entries\n- More complex merge algorithm\n- Harder to explain to users (\"why do I have a conflict?\")\n\n### Design Tradeoff 2: Three-Way Merge vs. Two-Way\n\n**Choice:** Three-way: compare desired + current + previously-owned\n\n**Alternatives:**\n- Two-way: just compare desired + current (traditional JSON merge patch behavior)\n- Client-driven: client computes diff and sends operations (JSON Patch)\n\n**Why three-way?**\n- Prevents data loss: can detect \"did someone else modify this field?\"\n- Enables force: can precisely know what to take over\n- Supports multiple managers: can track ownership lineage\n\n**Cost:**\n- Harder to debug (\"why did my change not apply?\")\n- Requires accurate managedFields (garbage in = garbage out)\n\n### Design Tradeoff 3: Apply Semantics for Lists\n\n**Choice:** Merge lists by default (can be overridden per-field in OpenAPI)\n\n**Example:**\n```yaml\nspec:\n  containers:\n  - name: app\n    image: myapp:1.0\n```\n\nIf you apply again with:\n```yaml\nspec:\n  containers:\n  - name: app\n    image: myapp:2.0\n  - name: sidecar\n    image: sidecar:1.0\n```\n\nResult: Both containers (merged), not just the sidecar.\n\n**Alternatives:**\n- Replace: Applying this would remove the original container\n- Add: Applying would append always (no updates)\n\n**Why merge?**\n- Intuitive: \"I want containers A and B\" means \"ensure I have A and B\"\n- Matches declarative paradigm: state what you want, server reconciles\n\n**Cost:**\n- List order can matter for some workloads\n- Users sometimes expect replace semantics\n- Requires OpenAPI hint to override (e.g., `x-kubernetes-patch-strategy: replace`)\n\n### Operational Limits\n\n| Limit | Value | Impact |\n|-------|-------|--------|\n| Max field managers per object | 10 | After 10, oldest entries merged |\n| Max managedFields entry size | ~10 KB | Very large field sets need chunking |\n| Max patch size | Usually 1-10 MB | Limited by API server's max request body |\n| Patch request timeout | Usually 30-60s | Long-running merges on huge objects |\n| Max retries (conflicts) | Unlimited in FieldManager | HTTP handler may timeout first |\n\n---\n\n## 6. Conflict Handling and Resolution Strategies\n\n### Conflict Anatomy\n\nA conflict occurs when:\n1. The current apply request (manager A) tries to modify field F\n2. Field F is currently owned by a different manager (manager B)\n3. Manager A does not have `force=true`\n\nExample:\n```yaml\n# Current object (after kubectl and operator both applied)\nmetadata:\n  managedFields:\n  - manager: kubectl\n    fields:\n      f:spec:\n        f:replicas: {}\n  - manager: operator\n    fields:\n      f:spec:\n        f:image: {}\n\n# User runs: kubectl apply --server-side\n# New patch sets: spec.replicas=5 (kubectl manager)\n# But operator manager owns spec.replicas \u2192 CONFLICT\n```\n\n### Conflict Detection\n\n**File: `/workspace/staging/src/k8s.io/apimachinery/pkg/util/managedfields/internal/conflict.go`**\n\n```go\nfunc NewConflictError(conflicts merge.Conflicts) *errors.StatusError {\n    causes := []metav1.StatusCause{}\n    for _, conflict := range conflicts {\n        causes = append(causes, metav1.StatusCause{\n            Type:    metav1.CauseTypeFieldManagerConflict,\n            Message: fmt.Sprintf(\"conflict with %v\", printManager(conflict.Manager)),\n            Field:   conflict.Path.String(),\n        })\n    }\n    return errors.NewApplyConflict(causes, getConflictMessage(conflicts))\n}\n```\n\nReturns HTTP 409 with details of conflicting fields and which manager owns each.\n\n### Resolution Strategies\n\n#### Strategy 1: Fail (Default)\n\n```\nRequest: apply with fieldManager=\"kubectl\" (no force)\nResponse: HTTP 409 Conflict\nError: \"Apply failed with 1 conflict: conflict with manager 'operator': spec.replicas\"\n```\n\nUser must either:\n- Manually remove the conflicting manager from object (rare)\n- Use `--force-conflicts` / `force=true` on the new apply request\n\n#### Strategy 2: Force Ownership Transfer\n\n```bash\nkubectl apply --server-side --force-conflicts\n```\n\nSets `force=true` in PatchOptions:\n\n```go\n// Overrides conflict: new manager takes ownership\nMerge(..., force=true) \u2192\n    Removes old manager from managedFields for conflicting fields\n    Adds new manager to managedFields for those fields\n    Result: HTTP 200, object updated\n```\n\n**Risks of force:**\n- Silently removes another tool's field tracking\n- Can cause thrashing if two tools continuously force-apply conflicting changes\n- Should be used only when you're certain you're the intended owner\n\n#### Strategy 3: Three-Way Merge Prevention\n\nBest practice: **Design ownership boundaries.**\n\n- **kubectl** manages user-specified fields (image, replicas, labels)\n- **operator** manages system fields (status, internal configuration)\n\nUse different field sets to avoid conflicts:\n```yaml\n# kubectl manages these:\nmetadata:\n  labels:\n    app: myapp\nspec:\n  replicas: 3\n  template:\n    spec:\n      containers:\n      - name: app\n        image: myapp:1.0\n\n# operator manages these (via separate apply):\nstatus:\n  conditions: [...]\nspec:\n  internalConfig: {...}\n```\n\nBoth can apply without conflict because they never both claim the same field.\n\n#### Strategy 4: Re-Apply After Conflict\n\nSome workflows retry with `force=true`:\n\n```go\nfor attempt := 0; attempt < maxRetries; attempt++ {\n    resp, err := applyWithForce(false) // force=false\n    if err != nil && isConflict(err) && attempt < maxRetries-1 {\n        applyWithForce(true) // Try with force\n    }\n}\n```\n\n### Conflict Prevention in Practice\n\n#### Test Case: `TestUpdateApplyConflict`\n\n**File: Failure Modes, Limits, and Design Tradeoffs\n\n### Failure Mode 1: Corrupted managedFields\n\n**Cause:** Buggy mutating admission webhook modifies managedFields incorrectly\n\n**Detection:**\n```go\nif err := managedfields.ValidateManagedFields(encodedManagedFields); err != nil {\n    // Admission corrupted managedFields\n}\n```\n\n**Recovery:**\n```go\n// NewManagedFieldsValidatingAdmissionController\nif err := managedfields.ValidateManagedFields(managedFieldsAfterAdmission); # Kubernetes Server-Side Apply and Apply Configurations: A Deep Technical Guide

## Executive Summary

Kubernetes Server-Side Apply (SSA) is a control plane subsystem that implements declarative resource management through field-level ownership tracking and intelligent merge semantics. Unlike client-side apply (which computes diffs locally), SSA shifts conflict resolution and ownership management to the API server, enabling robust multi-manager scenarios and reducing client-side complexity.

This guide explains how the subsystem is architected, how requests flow through the system, where conflicts are detected and resolved, and the operational tradeoffs involved.

---

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose

Server-Side Apply solves three core problems:

1. **Ownership Tracking**: Who last modified which field? Without this, concurrent edits by different tools (kubectl, Helm, operators) cause data loss or unwanted overwrites.

2. **Conflict Detection and Resolution**: When one manager tries to modify a field controlled by another, SSA detects the conflict and can either (a) fail with a clear error, (b) allow forced takeover, or (c) use structured merge semantics to prevent unintended overwrites.

3. **Declarative State Management**: Clients declare *what they want* (via apply configurations) rather than *how to change it* (via imperative patches). The server reconciles the desired state with the live state and field ownership information.
