# Architecture Document: Istio Pilot Discovery Service

**Repository:** istio/istio
**Output:** Write your document to `/workspace/documentation.md`

## Task

Produce an architecture document for **Istio Pilot's Discovery Service** subsystem. The document must explain how Pilot serves xDS (Envoy Discovery Service) configuration to proxies, translates Istio's high-level configuration into Envoy-specific resources, integrates with service registries, and supports multi-cluster deployments. Do not simply list APIs — explain the **design**, **data flow**, and **extension points**.

## Scope

Your document must cover these four components and how they work together:

### 1. DiscoveryServer — The xDS gRPC Server
The central component implementing Pilot's gRPC interface for Envoy's xDS APIs. Explain:
- How it implements Envoy's xDS v3 protocol (ADS, incremental/state-of-the-world modes)
- The role of `adsClients` in tracking active proxy connections
- How configuration pushes are coordinated: debouncing, rate limiting, concurrent push limits
- Cache management for xDS resources (CDS, LDS, RDS, EDS)
- The lifecycle: initialization, handling new connections, shutdown

### 2. ADS and the Connection Model
The Aggregated Discovery Service stream handler. Explain:
- How `Stream()` establishes a bidirectional gRPC stream with an Envoy proxy
- The `Connection` type: wrapping the stream with proxy metadata and state
- Request processing: how `processRequest()` determines what configuration to send
- Push coordination: how `pushConnection()` transmits updates to a single proxy
- The `StartPush()` broadcast mechanism for fleet-wide updates

### 3. PushContext — Configuration Snapshot and Translation
The immutable configuration snapshot used during a push. Explain:
- What `PushContext` holds: services, destination rules, virtual services, sidecars, gateways
- How `InitContext()` rebuilds the push context when Istio config changes (CRD watches)
- Config translation: how Pilot converts Istio's `VirtualService` and `DestinationRule` into Envoy's `RouteConfiguration` and `Cluster` resources
- The relationship between push triggers (config changes, service registry updates) and push context regeneration
- Metrics and error tracking during push generation

### 4. Service Registry and Multi-Cluster Support
The service discovery layer feeding the DiscoveryServer. Explain:
- How `ServiceRegistry` (Kubernetes controller) watches K8s Services, Endpoints, and Pods
- The aggregate controller pattern for multi-registry support (K8s services + `ServiceEntry` resources)
- Multi-cluster support: how Pilot aggregates services from multiple clusters
- How endpoint updates (pod scale-up, health changes) trigger EDS pushes
- The relationship between the service registry and xDS: when does a service change trigger CDS vs. EDS?

Rate limit incoming requests\n    if err := s.WaitForRequestLimit(stream.Context()); err != nil {\n        return status.Errorf(codes.ResourceExhausted, \"request rate limit exceeded\")\n    }\n\n    // 3. Authenticate client (extract SPIFFE identity)\n    ids, err := s.authenticate(ctx)\n\n    // 4. Initialize the push context\n    s.globalPushContext().InitContext(s.Env, nil, nil)\n\n    // 5. Create connection and delegate to xds.Stream\n    con := newConnection(peerAddr, stream)\n    con.ids = ids\n    con.s = s\n    return xds.Stream(con)\n}\n```\n\n#### Connection Initialization: `initConnection()`\n\nOnce a stream is established, the first DiscoveryRequest arrives with the proxy's node metadata:\n\n```go\nfunc (s *DiscoveryServer) initConnection(node *core.Node, con *Connection, identities []string) error {\n    // 1. Extract proxy metadata from node\n    proxy, err := s.initProxyMetadata(node)\n\n    // 2. Apply cluster aliases if configured\n    if alias, exists := s.ClusterAliases[proxy.Metadata.ClusterID]; exists {\n        proxy.Metadata.ClusterID = alias\n    }\n\n    // 3. Set baseline push context to prevent losing updates\n    proxy.LastPushContext = s.globalPushContext()\n\n    // 4. Assign connection ID and store in adsClients\n    con.SetID(connectionID(proxy.ID))\n    con.proxy = proxy\n    s.addCon(con.ID(), con)\n\n    // 5. Authorize the connection (RBAC check)\n    if err := s.authorize(con, identities); err != nil {\n        return err\n    }\n\n    // 6. Complete proxy initialization (compute initial state)\n    if err := s.initializeProxy(con); err != nil {\n        s.closeConnection(con)\n        return err\n    }\n\n    return nil\n}\n```\n\nThis initialization ensures:\n- Proxy metadata is extracted and validated\n- RBAC policies are checked (via `s.authorize()`)\n- The proxy state is computed with initial configuration\n- The connection is registered before any concurrent pushes can arrive\n\n#### Connection Teardown: `closeConnection()`\n\nWhen a proxy disconnects (stream ends or client closes):\n```go\nfunc (s *DiscoveryServer) closeConnection(con *Connection) {\n    if con.ID() == \"\" {\n        return  // Not yet initialized\n    }\n    s.removeCon(con.ID())  // Remove from adsClients map\n}\n```\n\n### Request Processing\n\n#### Request Entry Point: `processRequest()`\n\nWhen the proxy sends a DiscoveryRequest:\n\n```go\nfunc (s *DiscoveryServer) processRequest(req *discovery.DiscoveryRequest, con *Connection) error {\n    stype := v3.GetShortType(req.TypeUrl)  // \"cds\", \"lds\", \"rds\", etc.\n\n    // Handle special cases (debug, health check, etc.)\n    if req.TypeUrl == v3.HealthInfoType {\n        return s.handleWorkloadHealthcheck(con.proxy, req)\n    }\n    if strings.HasPrefix(req.TypeUrl, v3.DebugType) {\n        return s.pushXds(con, req.TypeUrl, ...)\n    }\n\n    // Normal processing: delegate to xds.Stream handler\n    // The stream's request processor determines if this is:\n    // - An ACK (nonce matches): no response needed\n    // - A NACK (nonce mismatch): resend the last response\n    // - A new watch: trigger generation and send response\n}\n```\n\nThe request contains:\n- **TypeUrl**: resource type (e.g., \"type.googleapis.com/envoy.config.cluster.v3.Cluster\")\n- **ResourceNames**: specific resources the proxy wants (e.g., [\"service1\", \"service2\"])\n- **VersionInfo**: version of resources the proxy currently has\n- **ResponseNonce**: token to match responses\n\n#### Watch Management\n\nThe base `xds.Stream` (from go-control-plane) handles:\n- Tracking which resource types each proxy is watching\n- Matching ACKs to responses\n- Detecting retries (NACK on version mismatch)\n- Resource name tracking per connection\n\n### Push Delivery: `pushConnection()`\n\nWhen a configuration push is triggered, each connected proxy receives updates via `pushConnection()`:\n\n```go\nfunc (s *DiscoveryServer) pushConnection(con *Connection, pushEv *Event) error {\n    pushRequest := pushEv.pushRequest\n\n    // 1. For full pushes, compute current proxy state\n    if pushRequest.Full {\n        s.computeProxyState(con.proxy, pushRequest)\n    }\n\n    // 2. Check if this proxy needs any updates\n    pushRequest, needsPush := s.ProxyNeedsPush(con.proxy, pushRequest)\n    if !needsPush {\n        return nil  // Skip push if proxy has latest config\n    }\n\n    // 3. Send updates for each resource type in order\n    // Push order is: CDS \u2192 EDS \u2192 LDS \u2192 RDS \u2192 SDS \u2192 (others)\n    // This ensures Envoy sees dependencies in correct order\n    wrl := con.watchedResourcesByOrder()\n    for _, w := range wrl {\n        if err := s.pushXds(con, w, pushRequest); err != nil {\n            return err\n        }\n    }\n\n    return nil\n}\n```\n\n**Push Order** is defined to respect Envoy's dependency model:\n1. **CDS** (Cluster Discovery) first \u2014 proxies learn about upstream clusters\n2. **EDS** (Endpoint Discovery) \u2014 learn which endpoints are in each cluster\n3. **LDS** (Listener Discovery) \u2014 configure listeners for traffic\n4. **RDS** (Route Discovery) \u2014 configure routes within listeners\n5. **SDS** (Secret Discovery) \u2014 TLS certificates\n6. Others (NDS, ECDS, etc.)\n\nThe order prevents Envoy from having incomplete configurations (e.g., routes without listeners).\n\n### Push Queue Management\n\nPushes to individual connections are coordinated via `PushQueue`:\n\n```go\ntype PushQueue struct {\n    pending map[*Connection]*model.PushRequest  // Waiting to be processed\n    queue []*Connection                         // FIFO ordering\n    processing map[*Connection]*model.PushRequest  // Currently being pushed\n}\n```\n\n**Enqueue**: When a push request arrives for a connection:\n- If already pending, the requests are merged (full=true if either is full)\n- If being processed, mark for re-queue after current push completes\n- Add to queue if new\n\n**Dequeue**: Workers process connections FIFO, respecting the semaphore limit:\n- Each `Dequeue()` returns one connection and its merged push request\n- Processing is marked until `MarkDone()` is called\n- If the connection was re-enqueued, it goes back in the queue\n\n---\n\n## 3. Send updates for each resource type in order\n    // Push order is: CDS \u2192 EDS \u2192 LDS \u2192 RDS \u2192 SDS \u2192 (others)\n    // This ensures Envoy sees dependencies in correct order\n    wrl := con.watchedResourcesByOrder()\n    for _, w := range wrl {\n        if err := s.pushXds(con, w, pushRequest); err != nil {\n            return err\n        }\n    }\n\n    return nil\n}\n```\n\n**Push Order** is defined to respect Envoy's dependency model:\n1. **CDS** (Cluster Discovery) first \u2014 proxies learn about upstream clusters\n2. **EDS** (Endpoint Discovery) \u2014 learn which endpoints are in each cluster\n3. **LDS** (Listener Discovery) \u2014 configure listeners for traffic\n4. **RDS** (Route Discovery) \u2014 configure routes within listeners\n5. **SDS** (Secret Discovery) \u2014 TLS certificates\n6. Others (NDS, ECDS, etc.)\n\nThe order prevents Envoy from having incomplete configurations (e.g., routes without listeners).\n\n### Push Queue Management\n\nPushes to individual connections are coordinated via `PushQueue`:\n\n```go\ntype PushQueue struct {\n    pending map[*Connection]*model.PushRequest  // Waiting to be processed\n    queue []*Connection                         // FIFO ordering\n    processing map[*Connection]*model.PushRequest  // Currently being pushed\n}\n```\n\n**Enqueue**: When a push request arrives for a connection:\n- If already pending, the requests are merged (full=true if either is full)\n- If being processed, mark for re-queue after current push completes\n- Add to queue if new\n\n**Dequeue**: Workers process connections FIFO, respecting the semaphore limit:\n- Each `Dequeue()` returns one connection and its merged push request\n- Processing is marked until `MarkDone()` is called\n- If the connection was re-enqueued, it goes back in the queue\n\n---\n\n## 3. Instead, **generators** read from PushContext and produce Envoy protobuf configs:\n\n#### Service \u2192 Cluster (CDS)\n\nWhen generating Clusters, the `cds` generator reads:\n- `PushContext.ServiceIndex` to find the service\n- `PushContext.destinationRuleIndex` to find load balancing policy\n- `PushContext.AuthzPolicies` for authorization context\n\nExample lookup flow:\n```\nRequest: \"Generate CDS for proxy in namespace 'default'\"\n1. Query: ps.ServiceIndex.HostnameAndNamespace[\"api.default.svc.cluster.local\"][\"default\"]\n2. Get Service object with endpoints, ports\n3. Query: ps.destinationRuleIndex for \"api\" in \"default\"\n4. Merge load balancing policy (round-robin, consistent-hash, etc.)\n5. Generate Envoy Cluster protobuf with Endpoints (EDS)\n```\n\n#### VirtualService \u2192 Route (RDS)\n\nWhen generating Routes, the `rds` generator reads:\n- `PushContext.virtualServiceIndex` to find VirtualService\n- Route destinations reference services in PushContext\n- `PushContext.EnvoyFilters` may patch the generated route\n\nExample:\n```\nRequest: \"Generate RDS for proxy in gateway 'istio-system/my-gateway'\"\n1. Query: ps.virtualServiceIndex.publicByGateway[\"istio-system/my-gateway\"]\n2. For each VirtualService:\n   - Extract http/tcp routes\n   - Resolve destination.host to cluster names\n   - Build Envoy RouteConfiguration\n3. Apply EnvoyFilters for patches\n```\n\n#### DestinationRule \u2192 Cluster Subset (CDS)\n\nDestinationRules define subsets of endpoints (e.g., \"v1\", \"v2\" versions):\n\n```\nDestinationRule:\n  spec:\n    subsets:\n    - name: v1\n      labels:\n        version: v1\n    - name: v2\n      labels:\n        version: v2\n```\n\nCDS generator creates separate Cluster for each subset:\n- `api-v1` cluster: endpoints with label `version=v1`\n- `api-v2` cluster: endpoints with label `version=v2`\n\nQueries PushContext:\n- `ps.destinationRuleIndex` for the DestinationRule\n- `ps.ServiceIndex.instancesByPort` for labeled endpoints\n\n### Push Metrics and Error Tracking\n\nPushContext tracks push statistics:\n\n```go\ntype ProxyPushStatus struct {\n    PushStartTime      int64\n    PushDuration       int64  // milliseconds\n    Err                string\n    // ... per-type status\n}\n```\n\nCaptured during push:\n- Successful resource generation\n- Errors (missing services, invalid config, etc.)\n- Per-proxy metrics for debugging\n\n### Relationship to Service Registry\n\nPushContext reads from the service registry via `env.ServiceRegistry()` and `env.GetServices()`:\n- **Services** come from: Kubernetes Services + ServiceEntry resources\n- **Endpoints** come from: Kubernetes Endpoints/EndpointSlice + ServiceEntry\n- **Service Accounts** come from: Kubernetes Pods + manual service account mapping\n\nPush trigger flow:\n```\nService Registry Event (e.g., pod added)\n\u2193\nConfigUpdate() \u2192 pushChannel\n\u2193\ndebounce (100ms - 10s)\n\u2193\nPush() \u2192 initPushContext() \u2192 new PushContext\n\u2193\nPushContext.initServiceRegistry() queries registry\n\u2193\nGenerators use PushContext.ServiceIndex\n\u2193\nxDS resources generated and sent to proxies\n```\n\n---\n\n## 4. Kubernetes Event: Pod \"api-v2-xyz\" created\n   - Ready status: Running\n   - IP: 10.0.1.5\n   - Labels: app=api, version=v2\n   - Service selector match: app=api\n\n2. Kube Controller Detects Change\n   - Watch fires for EndpointSlice\n   - Queries Pod cache for pod metadata\n   - Creates IstioEndpoint for 10.0.1.5:8080\n\n3. Push Trigger\n   - ServiceHandler callback from kube controller\n   - Calls ConfigUpdate() with reason: model.ServiceUpdate\n   - Push request goes to pushChannel\n\n4. Debounce\n   - Default: wait 100ms after last change\n   - Max: 10 seconds absolute\n   - If no new changes in 100ms, proceed to push\n\n5. PushContext Initialization\n   - initPushContext() called by Push()\n   - PushContext.initServiceRegistry() queries registry\n   - ServiceIndex rebuilt with new endpoint\n   - Lookup: ps.ServiceIndex[\"api.default\"][8080] \u2192 includes 10.0.1.5\n\n6. xDS Generation\n   - For each connected proxy:\n     - Call pushConnection() with PushContext\n     - CDS generator: checks if proxy needs cluster \"api\" (yes)\n     - EDS generator: loads endpoints from PushContext.ServiceIndex\n     - Generates EDS update with [10.0.1.2, 10.0.1.3, 10.0.1.5]\n     - Send: DiscoveryResponse with new EDS version\n\n7. Proxy Update\n   - Envoy proxy receives EDS response\n   - Adds 10.0.1.5 to connection pool\n   - Starts health checking\n   - Traffic gradually shifts to new endpoint\n```\n\n### Example: Applying VirtualService\n\n```\n1. Istio Event: VirtualService \"api\" created\n   kubectl apply -f virtualservice.yaml\n\n2. Config Watcher Detects Change\n   - ConfigStore watches VirtualService CRD\n   - Fires ConfigUpdate() with reason: model.ConfigChange\n   - ConfigsUpdated: {Kind: VirtualService, Name: \"api\", Namespace: \"default\"}\n\n3. Proxy Update\n   - Envoy receives RDS with new routes\n   - Traffic to \"api\" now respects defined routes\n   - Headers, timeouts, retries applied\n```\n\n---\n\n## 6. Extension Points\n\n### 1. # Istio Pilot Discovery Service Architecture

## Executive Summary

Istio Pilot's Discovery Service is a distributed control plane component that translates Kubernetes-native and Istio-specific network configuration into Envoy sidecar proxy configuration. The service operates as a gRPC server implementing Envoy's xDS (Envoy Discovery Service) protocol, enabling dynamic configuration delivery to a fleet of proxies without restarts.

The system is organized around four primary components:
1. **DiscoveryServer** — the gRPC xDS server managing proxy connections
2. **ADS (Aggregated Discovery Service)** — bidirectional streaming protocol between Pilot and proxies
3. **PushContext** — immutable configuration snapshot used during configuration pushes
4. **ServiceRegistry** — abstraction over service discovery backends (Kubernetes, ServiceEntry, multi-cluster)

## 1. DiscoveryServer — The xDS gRPC Server

**Source Files:** `pilot/pkg/xds/discovery.go`, `pilot/pkg/xds/ads.go`

### Purpose

The `DiscoveryServer` is Pilot's core gRPC implementation of Envoy's xDS APIs (CDS, LDS, RDS, EDS, SDS). It serves as the central hub that:
- Maintains active connections to all connected Envoy proxies via `adsClients`
- Coordinates configuration pushes to proxies when Istio resources change
- Implements request rate limiting and push throttling
- Manages cache for xDS resources to avoid recomputation

### Key Responsibilities

#### xDS Protocol Implementation
The DiscoveryServer implements the Envoy xDS v3 protocol with two operational modes:
- **Aggregated Discovery Service (ADS)**: A single bidirectional gRPC stream where all resource types (CDS, LDS, RDS, EDS) flow over one connection
- **Delta XDS**: An incremental update protocol for efficient bandwidth usage
